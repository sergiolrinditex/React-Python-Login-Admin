## Decisions (D1–D10)

| ID | Decision | Source |
|----|----------|--------|
| D1 | No `CREATE EXTENSION vector` in this migration | Vector not referenced by any T001 table; belongs to P02-S01-T001 | TECHNICAL_GUIDE §10.3 / task pack §9.2 |
| D2 | No `DROP EXTENSION pgcrypto` in downgrade | Standard Alembic practice — extension may be shared; idempotent `IF NOT EXISTS` is harmless | task pack §4.1 |
| D3 | ORM models split by bounded context: identity in `user.py` (User/EmployeeProfile/Role/Permission/UserRole), session/audit in `auth.py` (RefreshToken/MfaTotpSecret/PasswordResetToken/AuditLog) | Clean Architecture: bounded context boundaries mirror §3.1 | task pack §6 |
| D4 | `Base = DeclarativeBase` lives in `app/db/base.py` (write_set extension) | Avoids user↔auth import cycle; precedent `app/core/__init__.py` (P00-S01-T003) | task pack §6.3 |
| D5 | `alembic/env.py` single-line change `target_metadata = None` → `Base.metadata` (write_set extension) | Predicted by its own docstring; without this Alembic has no schema to diff | task pack §6.5 |
| D6 | `refresh_tokens.user_id` and `password_reset_tokens.user_id` declared NOT NULL | Tighter than §10.3 raw DDL; a token without an owner has no meaning; validator approved | handoff §K, validator observation 2 |
| D7 | 7 indexes declared: `user_roles_role_id_idx`, `refresh_tokens_token_hash_idx`, `refresh_tokens_user_id_revoked_idx`, `password_reset_tokens_token_hash_idx`, `audit_logs_actor_created_idx`, `audit_logs_entity_idx`, `audit_logs_created_at_idx` | Critical query patterns for P01-S02 and P04-S03 audit page | TECHNICAL_GUIDE §10.3 / task pack §9.2 D7 |

| D9 | No roles/permissions seed in migration | Belongs to verification_data loader (P00-S02-T003) | task pack §4.4 |
| D10 | No `updated_at` DB trigger | App-controlled (P01-S02-T007 service layer); keeps DB simple | task pack §9.2 D10 |

Additional implementation note: Python attribute `extra_metadata` maps to DB column `metadata` via `mapped_column("metadata", JSONB, ...)` in both EmployeeProfile and AuditLog, because SQLAlchemy 2.x reserves `metadata` as a class-level DeclarativeBase attribute. ORM consumers use `extra_metadata`; raw SQL/loader.py uses `metadata`. Decision documented in handoff and model docstrings.

## Open Items / Follow-ups

| ID | Kind | Severity | Title | Status |
|----|------|----------|-------|--------|
| FU-20260511145446 | bug | medium | fix verification_data loader `:meta::jsonb` SQL cast | proposed (not blocking T001 — out of write_set scope; severity medium) |

`FU-20260511145446` origin is P00-S02-T003. The `:meta::jsonb` cast syntax in `loader.py` line ~280 mixes SQLAlchemy named params with Postgres cast syntax, which psycopg3 cannot parse. The bug was dormant while tables didn't exist; P01-S01-T001 tables' creation activates it. Fix: change cast to `json.dumps(ep.metadata or {})` + `cast(..., Jsonb)` or pass Python dict (psycopg3 handles dict → JSONB automatically). Main-orchestrator to decide promotion/waiver.

Non-blocking observation from validator (info-only): `audit_logs` uses `actor_user_id, action, entity_type, entity_id, metadata jsonb` structure per TECHNICAL_GUIDE §10.3, while `01-non-negotiables.md §Security/Audit log` lists `ip, user_agent, request_id` as columns. These will live inside the `metadata` JSONB. If structured columns are required, that is a source-of-truth amendment for hardening phase (P04). Not a T001 defect.
