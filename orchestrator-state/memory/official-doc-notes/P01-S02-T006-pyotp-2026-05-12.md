## Q2 — `pyotp.TOTP(secret).verify(otp, for_time, valid_window)` exact API

Verified from GitHub source `src/pyotp/totp.py`:

```python
def verify(self, otp: str, for_time: Optional[datetime.datetime] = None, valid_window: int = 0) -> bool:
    """
    Verifies the OTP passed in against the current time OTP.

    :param otp: the OTP to check against
    :param for_time: Time to check OTP at (defaults to now)
    :param valid_window: extends the validity to this many counter ticks before and after the current one
    :returns: True if verification succeeded, False otherwise
    """
    if for_time is None:
        for_time = datetime.datetime.now()

    if valid_window:
        for i in range(-valid_window, valid_window + 1):
            if utils.strings_equal(str(otp), str(self.at(for_time, i))):
                return True
        return False

    return utils.strings_equal(str(otp), str(self.at(for_time)))
```

**Key facts**:
- Return type: `bool` — `True` if valid, `False` otherwise. **Never raises on valid string input**.
- Second parameter name: `for_time` (NOT `timestamp` — Context7 uses `timestamp` as an alias in its doc summary, but the source code and docstring use `for_time`). Task pack §G uses `valid_window=AUTH_MFA_VERIFY_WINDOW` without the second positional arg — this is correct.
- `valid_window=1` accepts codes from intervals `[-1, 0, +1]`, i.e. the previous 30s step, current step, and next 30s step. **Effective acceptance window ≈ 90s** (not 30s, not 60s).
- Uses `utils.strings_equal` (constant-time comparison) internally — timing-safe by design.
- **On malformed `otp` (non-digit, wrong length, empty)**: `str(otp)` cast means pyotp does NOT raise; it simply compares a stringified version of whatever was passed to the expected code string. A non-6-digit string will never match a valid TOTP code (which is always exactly 6 chars like `"028307"`), so `verify()` returns `False`. It does **not** raise `ValueError` or any exception for malformed input.

**IMPORTANT NOTE FOR DEVELOPER**: Because pyotp never raises on malformed code, the Pydantic validator in `MfaVerifyRequest` is the correct and only place to reject non-6-digit input. If the code somehow bypasses Pydantic (e.g. a bug), `verify()` will silently return `False`, which is the correct safe behavior. No try/except is needed around the `verify()` call for malformed input.

**RESOLVED**: Task pack §G flow step 4 is correct. `pyotp.TOTP(seed).verify(code, valid_window=AUTH_MFA_VERIFY_WINDOW)` returns `bool` directly; no exception to catch for verification failure.

## Q3 — `code` argument type: str vs bytes; pitfalls

- The `otp` parameter is **typed `str`** in the method signature.
- Internally, pyotp does `str(otp)` before comparison, so passing `bytes` would result in `str(b"123456") == "b'123456'"` which would never match — effectively a guaranteed `False` without raising.
- **FastAPI + Pydantic delivers `code` as a Python `str`** (Pydantic v2 parses JSON strings as str). No encoding issues.
- **Leading zeros**: pyotp always zero-pads generated codes to 6 digits (e.g. `"028307"`). `str(otp)` preserves leading zeros as long as the input is already a str. If the frontend sends `"028307"` as a JSON string, Pydantic delivers `"028307"` — correct.
- **Whitespace**: the Pydantic `code_must_be_six_digits` validator calls `v.isdigit()` which returns False for `" 12345"` (leading space) and `"123 56"` (internal space). So whitespace is caught at Pydantic level before pyotp is ever called.
- **Unicode digits**: `"１２３４５６"` (fullwidth digits, U+FF11-U+FF16) pass `str.isdigit()` in Python (returns True!). The Pydantic validator `v.isdigit()` would accept these, but pyotp's `str(otp)` would compare them against an ASCII-only code string and return `False`. **No security issue** (attacker cannot bypass this way), but the validator should use `v.isdecimal()` or `v.encode('ascii').isdigit()` to be strict. Using `re.match(r'^\d{6}$', v)` is the most robust option (matches only ASCII digits 0-9).

**RECOMMENDATION for developer**: Change the Pydantic validator from `v.isdigit()` to `re.fullmatch(r'\d{6}', v) is not None` to guarantee ASCII-only digits and exact length in one step. This is a minor hardening — not a blocking discrepancy, as `v.isdigit()` + `len(v) != 6` will reject all realistic bad inputs from a browser.

- **Recommended sanitization before `verify()`**: None needed if Pydantic validates. If called from a non-Pydantic path: `code.strip()` + `re.fullmatch(r'\d{6}', code)` check before calling verify.

**RESOLVED**: Pass `code` (str, already Pydantic-validated) directly to `pyotp.TOTP(seed).verify(code, valid_window=1)`. No `.encode()`, no cast, no strip needed in the service layer.

## Q4 — TOTP secret encoding for `pyotp.TOTP(secret)`

From GitHub source `src/pyotp/otp.py`:

```python
# Secret stored as string:
self.secret = s

# Decoded via:
def byte_secret(self) -> bytes:
    secret = self.secret
    missing_padding = len(secret) % 8
    if missing_padding != 0:
        secret += "=" * (8 - missing_padding)
    return base64.b32decode(secret, casefold=True)
```

**Key facts**:
- `pyotp.TOTP(secret)` expects the `secret` argument as a **plain base32 string** (e.g. `"JBSWY3DPEHPK3PXP"`). NOT bytes.
- `casefold=True` means lowercase base32 is also accepted (e.g. `"jbswy3dpehpk3pxp"`). The project verification data uses uppercase — correct.
- **Padding**: pyotp **automatically adds `=` padding** when the base32 string length is not a multiple of 8. The verification secret `"JBSWY3DPEHPK3PXP"` is exactly 16 chars (a multiple of 8), so no padding is added. Unpadded base32 strings of any length are accepted correctly.
- **Invalid base32**: `base64.b32decode(secret, casefold=True)` raises `binascii.Error` (subclass of `ValueError`) if the string contains characters outside the base32 alphabet after padding. **pyotp does NOT catch this** — it propagates up. The developer should either: (a) ensure the secret is validated at enrollment time (T002 path already validates TOTP secret format before storing), or (b) wrap the `pyotp.TOTP(secret)` construction in a try/except for this error in `mfa_crypto.py`.
- **Recommended secret length (RFC 4226 §4)**: minimum 128 bits = 26 base32 chars. Recommended 160 bits = 32 base32 chars. `"JBSWY3DPEHPK3PXP"` is 16 chars = 80 bits — this is a **test secret only** (explicitly labeled in task pack). Production secrets should be ≥26 base32 chars (160 bits). Note: this is a concern for the enrollment endpoint (future slice), not for T006 which only reads existing secrets.

**RESOLVED**: `pyotp.TOTP(decrypt_totp_secret(encrypted))` is the correct call form. The secret is a plain base32 str, no `.encode()` needed. pyotp handles padding automatically. Invalid base32 raises `binascii.Error` — recommend wrapping in try/except in `mfa_crypto.py` or the service, mapping to `MfaSecretMissingError` with audit reason `no_secret`.

## Q5 — Replay-attack guidance + CVE check

### RFC 6238 §5.2 mandate on replay prevention

RFC 6238 §5.2 states:
> "Note that a prover may send the same OTP inside a given time-step window multiple times to a verifier. The verifier MUST NOT accept the second attempt of the OTP after the successful validation has been issued for the first OTP, which ensures one-time only use of an OTP."

**pyotp does NOT enforce one-shot consumption**. There is no built-in deny-list, database, or in-memory store in pyotp. Calling `verify()` twice with the same code within the same time step returns `True` both times. **The application MUST track consumed codes/jtis to prevent replay.** This confirms Decision F.1 (in-memory `jti` consume store).

- **GitHub security advisories for pyauth/pyotp**: "There aren't any published security advisories" (verified from https://github.com/pyauth/pyotp/security/advisories on 2026-05-12).

## Discrepancy check — `valid_window=1` vs task pack §F.2

Task pack §F.2 states `valid_window=1`. This is confirmed as a reasonable production choice:
- `valid_window=0` = only current 30s step (too strict for browser UX).
- `valid_window=1` = current ± 1 step = ~90s window (standard for clock skew).
- `valid_window=2` = ~150s window (RFC 6238 recommends considering, but widens the replay surface).

Official pyotp default is `valid_window=0`. **No official doc recommendation for a specific production value** — the choice is application-dependent. `valid_window=1` is the widely-used production standard for TOTP (used by Google Authenticator spec, RFC 6238 guidance, and most 2FA implementations).

**NO DISCREPANCY** on `valid_window=1`. The task pack decision stands.
