## Q1 — Vite `server.proxy` — VERIFIED

**Sources**: Context7 /vitejs/vite v8.0.10 (https://github.com/vitejs/vite/blob/v8.0.10/docs/config/server-options.md); http-proxy-3 README (https://github.com/sagemathinc/http-proxy-3/blob/master/README.md)

### Set-Cookie and changeOrigin

- `changeOrigin: true` only rewrites the **Host header** sent to the upstream server. It does NOT rewrite the `Domain` attribute in `Set-Cookie` response headers from upstream.
- Cookie `Domain` rewriting requires the separate `cookieDomainRewrite` option.
- `cookieDomainRewrite` default: `false` (disabled — cookies pass through unchanged).
- `cookiePathRewrite` default: `false` (disabled — path passes through unchanged).
- **Implication for Strategy A**: With `changeOrigin: false` (planner pack config), the Host header is NOT rewritten, and cookies pass through entirely unchanged. Set-Cookie from `:8000` arrives at the browser with original attributes intact.
- **Implication for Strategy A with `changeOrigin: true`** (which Vite docs use in their example): Still does NOT rewrite Set-Cookie Domain. Cookies still pass through unchanged unless `cookieDomainRewrite` is explicitly set.

### Path attribute survival under proxy

- With `changeOrigin: false`, the proxy forwards requests to `http://localhost:8000` with the original path. `Set-Cookie: Path=/api/v1/auth` from the upstream arrives unchanged. Under Strategy A, the browser sees the cookie as belonging to `:5173` (same-origin). The Path attribute `/api/v1/auth` is preserved verbatim.

### Recommended minimal config (Strategy A)

From Vite v8 official docs: no `cookieDomainRewrite` or `cookiePathRewrite` needed when proxy target is `localhost` (same host). `secure: false` is only needed if proxying from `http://` to `https://` upstream. For `http://localhost:8000` target, `secure: false` is irrelevant (target is already http).

```ts
proxy: {
  "/api": {
    target: "http://localhost:8000",
    changeOrigin: false,   // host header unchanged → backend already accepts localhost
    // secure: false not needed (target is http)
  }
}
```

### Custom headers (X-Request-ID)

- http-proxy-3 passes all request headers through to the upstream by default. Custom headers like `X-Request-ID` are forwarded unchanged.
- Response headers from upstream (including `X-Request-ID` echo and `Set-Cookie`) are passed back to the browser by default.
- No explicit configuration needed.

### WebSocket gotcha under same `/api` prefix

- Vite proxy uses `ws: true` specifically for WebSocket upgrade handling.
- HTTP requests (including SSE chunked transfer) do NOT need `ws: true`.
- WebSocket is a separate protocol upgrade; if `/api` is used for both HTTP and WebSocket, a `ws: true` sub-entry is needed only for the WebSocket sub-path, not the HTTP sub-path.
- Vite docs note: use `rewriteWsOrigin` with caution (CSRF risk for WebSocket).
- **For this project**: No WebSocket endpoints exist. SSE (P02-S04) uses HTTP chunked transfer, not WebSocket — `ws: true` is not needed and should NOT be added to the `/api` proxy entry.

## Q2 — FastAPI CORSMiddleware — VERIFIED

**Sources**: Context7 /websites/fastapi_tiangolo (https://fastapi.tiangolo.com/reference/middleware, https://fastapi.tiangolo.com/tutorial/cors); Starlette docs (https://www.starlette.io/middleware/)

### Why OPTIONS returns 405 without CORSMiddleware

FastAPI routes are registered only for the declared HTTP methods. A route decorated with `@router.post("/refresh")` registers ONLY POST. FastAPI/Starlette's `ExceptionMiddleware` returns `405 Method Not Allowed` for any method not registered on a matched route. There is NO implicit OPTIONS handler generated — only `CORSMiddleware` (or an explicit `@app.options()` decorator) adds OPTIONS handling. This is expected, documented behavior.

**CITATION**: FastAPI tutorial/cors — "CORS preflight requests are OPTIONS requests... The middleware intercepts these requests, responds with appropriate CORS headers..."

### Correct registration

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:5173"],   # explicit list required with allow_credentials=True
    allow_credentials=True,
    allow_methods=["GET", "POST", "PATCH", "PUT", "DELETE", "OPTIONS"],
    allow_headers=["Authorization", "Content-Type", "X-Request-ID"],
    expose_headers=["X-Request-ID"],
    max_age=600,
)
```

### Middleware order

From Starlette docs: "Middleware is evaluated from top-to-bottom." `app.add_middleware(X)` wraps the app such that X becomes the outer layer. If both `CORSMiddleware` and a custom `X-Request-ID` middleware are registered via `add_middleware`, the LAST one registered via `add_middleware` becomes the outermost layer.

**Critical**: CORSMiddleware should be added LAST (outermost) so CORS headers are applied even to error responses. Starlette docs state: "wrapping your entire application with CORSMiddleware as the outermost layer... ensures CORS headers are applied even to error responses generated by unhandled exceptions."

```python
# Correct order — CORSMiddleware is outermost (registered last):
app.add_middleware(RequestIDMiddleware)  # inner
app.add_middleware(CORSMiddleware, ...)  # outermost — registered last
```

### allow_credentials=True + allow_origins=["*"]

**Explicitly forbidden per CORS spec and Starlette docs**: "If `allow_credentials` is True, `allow_origins`, `allow_methods`, and `allow_headers` cannot be set to `['*']` and must be explicitly specified." Using `["*"]` with `allow_credentials=True` is a spec violation — browsers reject it.

## Q3 — SameSite=Lax cookie behavior — DISCREPANCY #1

**Sources**: MDN Glossary/Site (https://developer.mozilla.org/en-US/docs/Glossary/Site); web.dev same-site-same-origin (https://web.dev/articles/same-site-same-origin); MDN Set-Cookie/SameSite (https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie#samesite); MDN Cookies Guide (https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/Cookies)

### DISCREPANCY with task-pack §B.3

**Task pack §B.3 states**: *"SameSite=Lax does NOT block cross-origin XHR by itself — it blocks cross-site third-party — but combined with `credentials:'include'` from the browser, the cross-origin preflight (OPTIONS) must pass."*

**What the task pack implies**: That SameSite=Lax is not the obstacle; the obstacle is the CORS preflight (OPTIONS 405). Fix the preflight → cookie flows.

**What official docs say**: The task pack's framing mixes up two separate mechanisms:

1. **SameSite** controls whether the browser *includes* the cookie in the outgoing request.
2. **CORS** controls whether the browser allows the JS code to read the response.

These are independent. Even with valid CORS headers, if SameSite=Lax blocks the cookie from being sent, the backend never receives it and the refresh fails.

**The port question resolves this**: Per MDN Glossary/Site and web.dev same-site-same-origin:
- Same-site is determined by **scheme + eTLD+1**, with port **ignored**.
- `localhost:5173` and `localhost:8000` share scheme (`http`) and eTLD+1 (`localhost`).
- **They ARE same-site.**

Per MDN Set-Cookie and web.dev:
- `SameSite=Lax`: sends cookie on same-site requests. For **subresource requests** (fetch/XHR), Lax sends the cookie only with **top-level navigations** (safe method, address bar changes). For same-site XHR/fetch POST from `:5173` to `:8000`, Lax behavior **per spec is navigation-only** — but since they are SAME-SITE (not cross-site), the restriction actually does not apply the same way as cross-site.

**The authoritative resolution** (web.dev, Google Chrome team):
- `SameSite=Lax` restricts cookies for **cross-site** subresource requests.
- Since `:5173` and `:8000` are **same-site**, `SameSite=Lax` does NOT block the cookie on XHR/fetch POST between them.
- Under Strategy B (direct cross-origin :5173→:8000): The request is cross-origin (different ports → different origin) but SAME-SITE (port ignored for same-site). Therefore, `SameSite=Lax` DOES allow the cookie on these requests because they are same-site — the lax restriction is about cross-site, not cross-origin.

**Task pack §B.3 conclusion is DIRECTIONALLY CORRECT** (cookie is not the blocker; CORS preflight is). But the reasoning in the task pack conflates same-site and same-origin imprecisely. The correct statement is: *localhost:5173 and localhost:8000 are same-site (port ignored for SameSite determination). SameSite=Lax allows the cookie on same-site requests including XHR/fetch POST. The blocker is CORS preflight (a separate mechanism), not SameSite.*

**Impact on Strategy A**: Under Strategy A (vite proxy, browser sees same-origin :5173), SameSite=Lax trivially works — same-origin is a subset of same-site.

**Impact on Strategy B**: Under Strategy B (direct :5173→:8000, cross-origin but same-site), SameSite=Lax also works for cookie sending — the cookie travels on the cross-origin same-site request. BUT the CORS preflight must pass for the browser to allow the XHR at all (separate mechanism). Strategy B conclusion remains valid.

**Severity**: Low — the conclusion (Strategy A recommended) is unaffected. The imprecise §B.3 reasoning does not change the implementation decision.

**Suggested resolution for developer**: In ADR-002 and §11.4, use the precise language: *"localhost:5173 and localhost:8000 are same-site (port not included in same-site determination), so SameSite=Lax does not block cookie sending between them. The cross-origin barrier is a CORS concern (separate from SameSite)."*

### Strategy A preserves cookie contract

Under Strategy A, browser sees `:5173` (same-origin). `Set-Cookie: refresh_token; HttpOnly; Secure; SameSite=Lax; Path=/api/v1/auth` is set on `:5173`. All attributes preserved verbatim by http-proxy-3 (cookieDomainRewrite=false by default). Cookie is sent on all same-origin requests to `:5173/api/v1/auth/**` — contractually identical to prod nginx topology.

## Q4 — X-Request-ID header preservation — VERIFIED

**Source**: http-proxy-3 README; Vite server-options docs (v8.0.10)

- **Request path (browser → vite → backend)**: http-proxy-3 forwards all request headers including custom ones (`X-Request-ID`) to the upstream by default. No configuration needed.
- **Response path (backend → vite → browser)**: http-proxy-3 passes all response headers including `X-Request-ID` echo back to the browser by default. No configuration needed.
- **Strategy B `expose_headers`**: If Strategy B is chosen, `expose_headers=["X-Request-ID"]` must be set in `CORSMiddleware` so browser JS (e.g., `response.headers.get("X-Request-ID")`) can read it. Without `expose_headers`, the browser receives the header but the CORS spec blocks JS access to non-safelisted response headers. The task pack's Strategy B config already includes this correctly.

## Q5 — Dev/prod alignment — DISCREPANCY #2 (nginx Set-Cookie forwarding)

**Sources**: nginx official docs (https://nginx.org/en/docs/http/ngx_http_proxy_module.html); first WebFetch pass returned a misleading result that was then corrected by a second authoritative fetch.

### nginx proxy_pass and Set-Cookie (CORRECTED)

- **Set-Cookie IS forwarded by default** through `proxy_pass`. No `proxy_pass_header Set-Cookie` directive is needed.
- The default hidden headers list is: `Date`, `Server`, `X-Pad`, `X-Accel-*`. Set-Cookie is NOT in this list.
- `proxy_ignore_headers Set-Cookie` only affects cache processing; it is NOT enabled by default.
- Custom response headers (e.g., `X-Request-ID`) are also forwarded by default through `proxy_pass`.

**This CONFIRMS Strategy A's prod claim** (§11.4 draft in task pack): nginx `proxy_pass http://backend/` forwards `Set-Cookie` from the backend to the browser unchanged, including all cookie attributes (`HttpOnly; Secure; SameSite=Lax; Path=/api/v1/auth`). No extra nginx directive needed.

**The initial WebFetch result was misleading** (it cited `proxy_ignore_headers` incorrectly as implying Set-Cookie is hidden by default). The corrected authoritative source confirms Set-Cookie passes through.

### Vite proxy = recommended dev mirror of prod nginx

Vite docs explicitly position `server.proxy` as the dev-time mechanism: "Set up proxy rules for the dev server." The pattern (vite proxies `/api` → `localhost:8000` in dev; nginx proxies `/api` → backend in prod) is the canonical dev/prod parity approach documented in the Vite guides. No official Vite, FastAPI, or OWASP doc recommends CORSMiddleware as the preferred pattern when prod uses a reverse proxy.

OWASP (from prior researcher note P01-S03-T001): `SameSite=Lax` + HttpOnly + in-memory access token + BFF-style same-origin is the recommended pattern. Strategy A aligns with this: the vite proxy creates the BFF-style same-origin in dev.
