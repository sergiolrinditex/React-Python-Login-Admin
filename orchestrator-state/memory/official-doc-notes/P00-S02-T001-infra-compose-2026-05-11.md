## 5. CRITICAL DISCREPANCY: nerdctl compose healthcheck support

- INTERNAL (task pack §F, §G): The docker-compose.yml uses `healthcheck:` on services AND `depends_on: { service: { condition: service_healthy } }` for service ordering.
- OFFICIAL: **nerdctl compose v2.2.2 (latest stable) does NOT support `services.<SERVICE>.healthcheck`**. It is explicitly listed as an unimplemented field. As a consequence, `condition: service_healthy` in `depends_on` is also non-functional.
- GitHub issue #2386 (opened 2023-07-21) "depend_on and healthcheck is ignored when running nerdctl compose up" — **still open**, labeled `enhancement`.
- SOURCE: https://github.com/containerd/nerdctl/blob/v2.2.2/docs/compose.md, https://github.com/containerd/nerdctl/issues?q=healthcheck+compose+service_healthy
- IMPACT: **WARNING** — The compose file will be syntactically valid (YAML parses fine), but when Rancher Desktop is configured to use `containerd` runtime (nerdctl), the `healthcheck` stanzas are silently ignored and `condition: service_healthy` dependencies are NOT enforced. Services may start out-of-order on nerdctl/containerd backend. On `dockerd (moby)` backend everything works correctly.
- SUGGESTED FIX (options for developer to choose):
  1. **No code change needed for moby users** — if the human uses Rancher Desktop in `moby` mode, the file is correct as-is.
  2. **Add warning comment** in docker-compose.yml above each `condition: service_healthy` block: `# NOTE: healthcheck conditions are ignored by nerdctl/containerd — use dockerd (moby) backend in Rancher Desktop`
  3. **Rancher Desktop workaround**: user switches to `dockerd (moby)` in Rancher Desktop > Preferences > Container Engine. Task pack §G already notes both modes are supported; this discrepancy clarifies that the fully-functional path requires moby.
  4. **Alternative**: provide a `docker-compose.override.yml` without healthchecks for nerdctl users (complex; not recommended for scaffold).
- RESOLUTION NEEDED: Developer should add a comment block to docker-compose.yml noting the nerdctl limitation AND the verify-slice instructions should specify that `/verify-slice` uses `dockerd (moby)` backend.

## 6. Docker Compose v2 spec — VERIFIED OK

- OFFICIAL: Top-level `version:` key is **obsolete** (informational only, triggers a warning, ignored by Compose). Task pack §F already says "No `version:` al inicio (deprecado en spec v2)" — CORRECT.
- `condition: service_healthy` in `depends_on`: **VALID** in the Compose v2 spec (docker compose / moby). Three valid values: `service_started`, `service_healthy`, `service_completed_successfully`.
- `restart: on-failure`: valid syntax in v2 spec. Note: `restart_policy` under `deploy:` is the Swarm/deploy equivalent; `restart: on-failure` at service level is still valid for standalone compose.
- SOURCE: https://docs.docker.com/compose/compose-file/05-services/#depends_on, https://docs.docker.com/compose/compose-file/04-version-and-name/
