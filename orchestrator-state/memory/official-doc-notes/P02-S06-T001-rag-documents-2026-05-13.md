## DISCREPANCY — Q1: python-multipart missing from requirements.txt

**Question:** Is `python-multipart` a transitive dep of `fastapi==0.135.2`, or must we add it?

**Official source:** https://github.com/fastapi/fastapi/blob/master/pyproject.toml (fetched 2026-05-13)
**Official source:** https://pypi.org/project/python-multipart/0.0.28/ (fetched 2026-05-13)

**Finding:** `python-multipart` is NOT in FastAPI's base `dependencies` array. It appears only in `[project.optional-dependencies]` under `standard` and `all` extras with constraint `>=0.0.18`. A plain `pip install fastapi==0.135.2` does NOT install it.

**Conflict with internal doc:** `backend/requirements.txt` does NOT include `python-multipart`. Any route using `UploadFile`, `File()`, or `Form()` will raise `ImportError` at application startup (FastAPI 0.60.2+ checks eagerly).

**Latest stable version:** `0.0.28` (released 2026-05-10; Production/Stable; Python >=3.10).

**Proposed reconciliation:** Add `python-multipart==0.0.28` to `backend/requirements.txt` in the `§ T003 §2.0 — PDF / Office backends` or a new `§ P02-S06 — File upload` section before any implementation work on the upload endpoint.

**Pydantic v2 compatibility:** Confirmed. `UploadFile` + `Form()` + Pydantic v2 `FormData(BaseModel)` pattern is fully documented for FastAPI 0.113.0+.

**TestClient multipart:** Use `httpx`/`requests` multipart form encoding:
```python
response = client.post(
    "/api/v1/admin/rag/documents",
    files={"file": ("doc.pdf", pdf_bytes, "application/pdf")},
    data={"collection_id": str(collection_id), "title": "Test Doc", "language": "es"},
)
```
No additional library beyond `python-multipart` (installed server-side) is needed for `TestClient`.

RESOLVED: _(developer fills after adding python-multipart==0.0.28 to requirements.txt)_

## VERIFIED — Q2: MIME sniffing approach

**Finding:** Hand-rolled magic bytes + zipfile structural check + parse-attempt via `python-docx`/`pypdf` is production-grade per OWASP defense-in-depth. OWASP explicitly states signature-only validation is insufficient standalone — combining with extension allow-list and library parse-attempt closes the gap. No CVE found for `python-magic` 2024–2026 in NVD. `python-magic` not recommended due to native `libmagic` OS dependency. Known DOCX pitfall: encrypted DOCX passes outer ZIP magic but lacks `word/document.xml`.

**Source:** https://cheatsheetseries.owasp.org/cheatsheets/File_Upload_Cheat_Sheet.html (2026-05-13)

RESOLVED: No action needed. Use hand-rolled approach as planned.

## VERIFIED — Q3: FastAPI UploadFile chunked read + body-size middleware

**Finding:** `await file.read(chunk_size)` is correct and fully documented. Starlette 0.49+ has NO built-in `MaxRequestBodySizeMiddleware`. Custom ASGI middleware wrapping the `receive` callable is the recommended pattern. `request.form(max_part_size=N)` is per-part only, not total body.

**Source:** https://github.com/kludex/starlette docs (2026-05-13); https://fastapi.tiangolo.com/reference/uploadfile (2026-05-13)

RESOLVED: No discrepancy. Implement custom ASGI body-size middleware.

## VERIFIED — Q4: boto3 put_object against MinIO

**Finding:** `put_object` is single-part regardless of file size. `ContentType` is optional but must be passed explicitly (otherwise defaults to `application/octet-stream`). `endpoint_url` pattern is the standard boto3 pattern for S3-compatible stores. No KMS needed. No quirks for <25 MiB files. Keep boto3, do not switch to `minio` SDK.

**Source:** https://docs.aws.amazon.com/boto3/latest/reference/services/s3/client/put_object.html (2026-05-13)
**Source:** https://github.com/boto/boto3/blob/develop/docs/source/guide/s3.rst (2026-05-13)

RESOLVED: No discrepancy.

## VERIFIED — Q5: Celery apply_async from async def route

**Finding:** `apply_async` is a synchronous blocking call (no `async`/`await`). Celery 5.6.3 has no asyncio-native dispatch variant. Calling directly from `async def` route blocks the event loop. Must wrap: `await asyncio.to_thread(task.apply_async, args=[...])`.

**Source:** https://docs.celeryq.dev/en/stable/userguide/calling.html (2026-05-13)
**Source:** Celery 5.3–5.6 changelog (no asyncio dispatch entries, 2026-05-13)

RESOLVED: No discrepancy with pack assumption. Use `asyncio.to_thread` pattern.
