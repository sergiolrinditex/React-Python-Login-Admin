## Q1 — Resend Python SDK 2026

**Source**: PyPI live JSON (2026-05-12) + Context7 /resend/resend-python + resend.com/docs/send-with-python
**Verified version**: `resend==2.30.0` (released 2026-05-04, latest stable)
**Cache valid until**: 2026-05-19

### Import and initialization

```python
import os
import resend

resend.api_key = os.environ["RESEND_API_KEY"]
```

Global module-level `resend.api_key` assignment is the canonical pattern (confirmed in README + official docs page). No `Resend(api_key=...)` client-class pattern — the Python SDK uses a module-level singleton, unlike the Node SDK.

### Send method signature

```python
params: resend.Emails.SendParams = {
    "from": "People <noreply@inditex-sandbox.com>",  # verified domain required in prod
    "to": ["user@example.com"],                        # list[str], max 50
    "subject": "Reset your password",
    "html": "<p>…</p>",
    "text": "…",              # optional plain-text fallback
    "reply_to": "support@example.com",  # optional, snake_case in Python SDK
    "headers": {"X-Custom": "value"},   # optional dict
    "tags": [{"name": "type", "value": "password_reset"}],  # optional
}

# Sync
email: resend.Emails.SendResponse = resend.Emails.send(params)
email_id = email["id"]  # dict with "id" key (str UUID)

# With idempotency key (supported officially, Python param = "idempotency_key")
options: resend.Emails.SendOptions = {"idempotency_key": "reset-<request_id>"}
email = resend.Emails.send(params, options)
```

**Return type**: `resend.Emails.SendResponse` — a TypedDict with at minimum `{"id": str}`. The `id` is a UUID string for the sent email.

**IMPORTANT NOTE (from official llms-full.txt)**: The Node SDK uses camelCase (`replyTo`, `scheduledAt`). The Python SDK uses snake_case (`reply_to`). The official docs page for Python shows `reply_to` in the SendParams TypedDict.

### Async API — CONFIRMED NATIVE

```python
# Async — native, does NOT require run_in_threadpool
email: resend.Emails.SendResponse = await resend.Emails.send_async(params)
# Also available: get_async, list_async, batch send_async
```

`resend.Emails.send_async(params)` is a first-class coroutine in the SDK — **not** a thread wrapper. This is the preferred pattern for FastAPI. No `run_in_threadpool` needed.

### Exceptions (`resend.exceptions.*`)

```python
import resend.exceptions

# Hierarchy (base → specific):
# resend.exceptions.ResendError          ← catch-all base class
#   .ValidationError                     ← 4xx validation
#   .MissingApiKeyError                  ← missing Authorization header

#   .RateLimitError                      ← 429 (has .headers with rate-limit info)
#   .MissingRequiredFieldsError          ← missing from/to/subject

```

Attributes on `ResendError`: `.code`, `.error_type`, `.message`, `.suggested_action`.

### Idempotency-Key

Officially supported on `POST /emails`. Python SDK param: `idempotency_key` (string, max 256 chars). Format: any unique string per request (e.g. `"reset-<request_id>"`). Source: resend.com/docs/dashboard/emails/idempotency-keys

## Q3 — SMTP fallback in Python 2026

**Source**: Context7 /cole/aiosmtplib; PyPI live JSON (2026-05-12)
**aiosmtplib version**: `5.1.0` (released 2026-01-25, Development Status: Production/Stable)
**Python requirement**: `>=3.10`
**No external dependencies** — pure asyncio

### Recommendation: `aiosmtplib` over `smtplib` for FastAPI

`smtplib` is sync and blocks the event loop in FastAPI/uvicorn without `run_in_threadpool`. `aiosmtplib` is natively async and integrates cleanly.

### STARTTLS (port 587) — canonical pattern

```python
from email.message import EmailMessage
import aiosmtplib

async def send_smtp_fallback(
    host: str,
    port: int,
    username: str,
    password: str,
    from_addr: str,
    to_addr: str,
    subject: str,
    html: str,
    text: str,
) -> None:
    message = EmailMessage()
    message["From"] = from_addr
    message["To"] = to_addr
    message["Subject"] = subject
    message.set_content(text)
    message.add_alternative(html, subtype="html")

    # STARTTLS is automatic by default on port 587 when server supports it
    await aiosmtplib.send(
        message,
        hostname=host,
        port=port,         # 587 → STARTTLS (recommended)
        username=username,
        password=password,
        # start_tls=None (default) — upgrade if server supports it
    )
```

### STARTTLS 587 vs SMTPS 465

- **Port 587 + STARTTLS**: Recommended by RFC 6409 (Mail Submission) and preferred in aiosmtplib docs. `start_tls=None` (default) upgrades automatically if server supports STARTTLS.
- **Port 465 + SMTPS**: Use `use_tls=True`. Older standard, still widely supported (Gmail, Outlook). Valid but not the default recommended port in RFC 6409.
- **2026 recommendation**: Port 587 + STARTTLS is the broadly recommended standard for SMTP submission (RFC 6409 §3.3). aiosmtplib defaults to this correctly.

## Q4 — OWASP Forgot Password + Anti-enumeration 2026

**Source**: cheatsheetseries.owasp.org/cheatsheets/Forgot_Password_Cheat_Sheet.html (fetched 2026-05-12)
         cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html (fetched 2026-05-12)

### Anti-enumeration (confirmed)

From **Forgot Password Cheat Sheet** — Section "Forgot Password Request":
> "Return a consistent message for both existent and non-existent accounts."
> "Ensure that responses return in a consistent amount of time to prevent an attacker enumerating which accounts exist."

From **Authentication Cheat Sheet** — Section "Authentication and Error Messages → Password recovery":
> "If that email address is in our database, we will send you an email to reset your password."

**Status**: CONFIRMED. Body byte-equal + timing-equal is OWASP 2026 official guidance. Task pack §H-item-4 and §M.2 are aligned.

### Token generation

OWASP states tokens must be:
> "Randomly generated using a cryptographically safe algorithm."

OWASP does NOT specify `secrets.token_urlsafe(32)` by name, but this is the canonical Python stdlib pattern for cryptographically secure 256-bit tokens (43 chars base64url). This is consistent with the "cryptographically safe algorithm" requirement.

**`secrets.token_urlsafe(32)` = 32 bytes = 256 bits entropy. CONFIRMED SUFFICIENT.** No HMAC, no PBKDF2 required for the token itself.

### Token storage

OWASP states:
> "Stored securely."

References Password Storage Cheat Sheet for specifics, but does not prescribe SHA-256 vs HMAC-SHA256 for token storage specifically.

**Analysis**: `token_urlsafe(32)` has 256 bits of entropy. SHA-256 without salt is sufficient for defense-in-depth — a DB-dump attacker would need to brute-force 2^256 values, which is computationally infeasible regardless of parallelism. HMAC-SHA256 with a server secret adds one more layer (dictionary precomputation impossible even with DB+code leaked simultaneously), but is not required when the token already has 256 bits of entropy. Task pack §H-item-6 (`token_hash = sha256(raw)`) is CONFIRMED SUFFICIENT per OWASP intent.

### Token TTL

OWASP states:
> "Single use and expire after an appropriate period."

No specific TTL number is given by OWASP. Common industry practice: 1 hour (3600s) for email-based reset flows. Task pack default `AUTH_RESET_TOKEN_TTL_SECONDS=3600` (1h) is aligned with industry consensus and the "appropriate period" guidance.

## Q5 — Resend SDK + FastAPI async

**Confirmed**: `resend.Emails.send_async(params)` is a **native coroutine** (confirmed via Context7 /resend/resend-python README and llms.txt). It can be called directly with `await` inside an `async def` FastAPI route or service. No `run_in_threadpool` wrapper needed.

```python
# Inside FastAPI service (async context):
email = await resend.Emails.send_async(params)
```

No separate `AsyncResend` class. The async methods are `send_async`, `get_async`, `list_async`, `send_async` on Batch.
