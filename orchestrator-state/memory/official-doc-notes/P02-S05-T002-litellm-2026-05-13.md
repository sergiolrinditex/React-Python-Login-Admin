## Q1 — Non-streaming completion shape

### ANSWER

#### Function signature (non-streaming)
```python
response = await litellm.acompletion(
    model="openai/gpt-4o-mini",            # or "gpt-4o-mini" — provider-prefixed preferred
    messages=[{"role": "user", "content": prompt}],
    stream=False,                           # default is False; explicit is clearer
    api_key="sk-...",                       # decrypted plain-text key, NEVER logged
    api_base="http://localhost:4000",       # only when using self-hosted proxy / provider.base_url
    timeout=30,                             # optional; seconds
)
```
Source: https://docs.litellm.ai/docs/completion/input — parameters section.

#### Response shape (official JSON schema)
```json
{
  "id": "chatcmpl-...",
  "choices": [
    {
      "finish_reason": "stop",
      "index": 0,
      "message": {
        "content": "Hi!",
        "role": "assistant",
        "tool_calls": null,
        "function_call": null
      }
    }
  ],
  "created": 1721955063,
  "model": "gpt-3.5-turbo",
  "object": "chat.completion",
  "system_fingerprint": null,
  "usage": {
    "prompt_tokens": 10,
    "completion_tokens": 20,
    "total_tokens": 30
  }
}
```
Source: https://docs.litellm.ai/docs/providers/custom_llm_server (canonical example) +
        https://docs.litellm.ai/docs/completion/output (LiteLLM completion response example).

#### Reading assistant text
```python
output_text = response.choices[0].message.content
```
Confirmed. `choices[0].message.content` is the string field. Non-streaming response is a single
complete `ModelResponse` object (not an iterator), so `.content` is populated immediately.

#### Token usage on NON-streaming response
```python
tokens_in  = response.usage.prompt_tokens
tokens_out = response.usage.completion_tokens
total      = response.usage.total_tokens
```
Usage is always present on non-streaming completion responses — no `stream_options` needed.
The `usage` field is populated in the top-level response object (not in chunks).
Source: https://docs.litellm.ai/docs/completion/output — usage dict present in all response examples.

#### completion_cost() with non-streaming response
```python
from litellm import completion_cost
cost = completion_cost(completion_response=response)  # pass full ModelResponse
# OR
cost = completion_cost(
    model="openai/gpt-4o-mini",
    prompt=prompt_text,
    completion=output_text,
)
```
Both forms confirmed to work. `completion_response=response` is the cleanest form for non-streaming
because the full token counts are already in `response.usage`.
Source: https://docs.litellm.ai/docs/completion/token_usage — "Calculate Completion Cost" section.

`response._hidden_params["response_cost"]` is also available post-completion call (non-streaming):
```python
cost = response._hidden_params["response_cost"]  # float USD, set by LiteLLM internally
```
Source: https://docs.litellm.ai/docs/completion/token_usage — "Get Response Cost in LiteLLM Completion".

Recommended: prefer `completion_cost(completion_response=response)` for explicit control; use
`_hidden_params["response_cost"]` as a secondary check when available and non-None.

#### Does stream_options apply to non-streaming?
NO. Official docs state: "Only set this when you set `stream: true`" for the `stream_options`
parameter. Do NOT pass `stream_options` when `stream=False`.
Source: https://docs.litellm.ai/docs/completion/input — stream_options parameter description.

### RESOLVED
The developer must build `backend/app/llm_gateway/complete_chat.py` with:
```python
response = await litellm.acompletion(
    model=model_str,
    messages=[{"role": "user", "content": prompt}],
    stream=False,                   # explicit; do NOT pass stream_options
    api_key=api_key,                # decrypted, never logged
    api_base=api_base or None,      # provider.base_url if set
    timeout=30,
)
output  = response.choices[0].message.content
t_in    = response.usage.prompt_tokens
t_out   = response.usage.completion_tokens
try:
    cost = float(completion_cost(completion_response=response) or 0.0)
except Exception:
    cost = _estimate_cost_from_pricing(model, t_in, t_out)
```
No monkeypatching needed for this shape — it is standard OpenAI-format output confirmed by
official docs across multiple providers.

For test mocks: use the `mock_response` parameter (see Q3) or create a plain object that
matches the shape: `.choices[0].message.content`, `.usage.prompt_tokens`, `.usage.completion_tokens`.

## Q2 — Error taxonomy

### ANSWER

All exceptions importable from `litellm` directly (no sub-module needed in v1.83.14):
```python
from litellm import (
    AuthenticationError,       # 401 — invalid api_key
    RateLimitError,            # 429 — provider rate-limit
    Timeout,                   # 408 — connect or inference timeout (inherits openai.APITimeoutError)
    APIConnectionError,        # 500 — network/connection errors, unmapped upstream errors
    NotFoundError,             # 404 — invalid model name / model not found at provider
    BadRequestError,           # 400 — bad request (e.g. malformed messages)
    ServiceUnavailableError,   # 503 — provider down
    InternalServerError,       # 500+ — provider server error
    APIError,                  # generic fallback for unmapped errors
)
```
"All exceptions can be imported from `litellm`."
"All our exceptions inherit from OpenAI's exception types."
Each exception carries `.status_code`, `.message`, `.llm_provider` extra attributes.
Source: https://docs.litellm.ai/docs/exception_mapping — full exception mapping table.

#### Per question mapping

| Scenario | Official LiteLLM exception | HTTP status |
|---|---|---|
| (a) network/connection error | `litellm.APIConnectionError` | 500 |
| (b) timeout | `litellm.Timeout` (= `openai.APITimeoutError`) | 408 |
| (c) rate-limit by provider | `litellm.RateLimitError` | 429 |
| (d) invalid api_key | `litellm.AuthenticationError` | 401 |
| (e) model not found at provider | `litellm.NotFoundError` | 404 |

Also catch `litellm.ServiceUnavailableError` (503) and `litellm.InternalServerError` (500+) as
upstream failures that should also produce our 502 `AI_PROVIDER_TEST_FAILED`.

### RESOLVED

Wrapping into our domain errors for `complete_chat.py`:

```python
import litellm

try:
    response = await litellm.acompletion(...)
except litellm.Timeout as exc:
    raise LiteLLMTimeoutError(
        message="Model test timed out.",
        cause=exc,
    )
except (
    litellm.AuthenticationError,
    litellm.NotFoundError,
    litellm.RateLimitError,
    litellm.APIConnectionError,
    litellm.ServiceUnavailableError,
    litellm.InternalServerError,
    litellm.APIError,
) as exc:
    raise ModelTestFailedError(
        message=f"Provider returned {type(exc).__name__}.",
        cause=exc,
    )
```

Mapping summary:
- `litellm.Timeout` → `LiteLLMTimeoutError` → `ai_model_tests.status='timeout'` → HTTP 502
- All other provider/network errors → `ModelTestFailedError` → `status='failure'` → HTTP 502
- `ModelTestFailedError` must be added to `errors.py` (new class, subclass of `LlmGatewayError`)

Note: `litellm.AuthenticationError` and `litellm.NotFoundError` are ALSO upstream failures in the
model-test context (wrong api_key or model ID at the provider) — they should produce 502
`AI_PROVIDER_TEST_FAILED`, NOT 401/404 at our API level. Our 404 is only for `ai_models` row not
found in OUR database before we even call LiteLLM.

## Q3 — Sandbox/dry-run

### ANSWER

NO native "dry-run" flag exists in the `litellm.acompletion` public API for v1.83.14.
There is no `dry_run=True` parameter.

HOWEVER, LiteLLM does expose a `mock_response` parameter (confirmed via the token_usage docs page
and existing usage in `litellm_client.py` comments). It is NOT a dry-run that avoids billing by
talking to the provider — rather it is a local mock: the library returns a fake `ModelResponse`
WITHOUT making an HTTP call at all. Shape is identical to a real response.

```python
response = await litellm.acompletion(
    model="openai/gpt-4o-mini",
    messages=[{"role": "user", "content": "hello"}],
    stream=False,
    mock_response="Hello world",    # str — returned as choices[0].message.content
)
# response.choices[0].message.content == "Hello world"
# response.usage is populated with synthetic token counts
# No HTTP call is made; no real billing occurs
```
Source: https://docs.litellm.ai/docs/completion/token_usage — example uses `mock_response="Hello world"`.

This is the correct tool for TEST FIXTURES (T01 happy path in `test_model_test.py`) instead of
a full monkeypatch. However, for integration tests we SHOULD still use `monkeypatch.setattr` or
`mock_response` — both are valid. `mock_response` is simpler and avoids attribute patching.

The task pack §E.2 says: "monkeypatch.setattr('litellm.acompletion', _fake_acompletion)". With
`mock_response` the developer can pass it through the real `acompletion` call in the service
layer by having the test set it via a fixture that patches the call kwargs — OR they can keep
the monkeypatch approach. Either is valid.

### RESOLVED

No dry-run for actual billing avoidance exists. The "sandbox" qualifier in the acceptance
criterion (`D.3`) refers to the verification-data LiteLLM proxy container
(`base_url=http://localhost:4000`), NOT a flag in the Python API.

For tests:
- Use `mock_response="<fake output>"` parameter OR `monkeypatch.setattr("litellm.acompletion", ...)`.
- `mock_response` is the cleaner approach as it exercises the real `acompletion` code path.
- For T08/T09 (error path tests): monkeypatch to raise the specific LiteLLM exception.

The "Model-test sandbox dry-run flag" in §F.4 is correctly listed as a FU candidate.

## Q4 — completion_cost None cases

### ANSWER

The official docs do not enumerate an explicit list of providers where `completion_cost()` returns
`None`. However, from the token_usage page and observed behavior in our existing code (lines 186-193
of `litellm_client.py`), `completion_cost()` returns `None` (or raises) in these cases:

1. **Custom/self-hosted models** — when `model` is not in LiteLLM's built-in pricing registry
   (e.g. a model deployed at `http://localhost:4000` with a non-standard model ID).
2. **Provider-prefixed models without pricing data** — some providers (Ollama, Groq custom,
   Together AI custom fine-tunes, self-hosted Bedrock, Azure deployments with custom names)
   may not have per-token pricing in LiteLLM's internal `model_cost.json`.
3. **Models with id that doesn't match LiteLLM's registry key** — e.g. using `"my-company/gpt-4"`.
4. **When `usage` is None or missing** — `completion_cost()` may return 0.0 or raise if token
   counts are unavailable.
5. **Anthropic/Claude models via proxy** — when routed through the LiteLLM proxy using a custom
   alias, the proxy-level model name may not match the pricing registry entry.

The existing implementation in `litellm_client.py` already handles this correctly:
```python
estimated_cost = float(
    litellm.completion_cost(completion_response=chunk) or 0.0
)
# OR on exception:
estimated_cost = _estimate_cost_from_pricing(model, tokens_in, tokens_out)
```
The `or 0.0` guard handles `None` return; the `except Exception` block handles any pricing
lookup failure.

Source: Inferred from https://docs.litellm.ai/docs/completion/token_usage (partial — no explicit
None-condition list documented) + existing `litellm_client.py` D-COST1 comment (lines 40-41, 186-193).

### RESOLVED

The `complete_chat.py` non-streaming wrapper must use the SAME double-guard pattern as streaming:
```python
try:
    cost = float(litellm.completion_cost(completion_response=response) or 0.0)
except Exception:
    cost = _estimate_cost_from_pricing(model, tokens_in, tokens_out)
```
Where `_estimate_cost_from_pricing` is the existing helper in `litellm_client.py` (lines 333-347)
that reads `ai_models.pricing` JSONB `{input_cost_per_token, output_cost_per_token}`.

The developer should import and reuse `_estimate_cost_from_pricing` from `litellm_client.py`
or make it a module-level export in `backend/app/llm_gateway/__init__.py`.

## Q5 — Proxy vs direct OpenAI

### ANSWER

#### Calling the LiteLLM proxy via `litellm.acompletion`

When `provider.base_url = "http://localhost:4000"` (our LiteLLM proxy container):
```python
response = await litellm.acompletion(
    model="litellm_proxy/your-model-name",  # OR "openai/gpt-4o-mini"
    messages=messages,
    api_base="http://localhost:4000",        # proxy URL
    api_key="sk-1234",                       # the LITELLM PROXY key (master key or virtual key)
    stream=False,
)
```
The `api_key` is the **LiteLLM proxy's own key** (set via `LITELLM_MASTER_KEY` on the proxy
container, or a virtual key created via the proxy `/key/generate` endpoint).
The proxy handles the actual provider credential internally. The Python client does NOT need to
know the underlying OpenAI/Anthropic API key when routing through the proxy.

Source: https://docs.litellm.ai/docs/providers/litellm_proxy — "api_key: your-litellm-proxy-api-key".

#### Calling direct OpenAI (no proxy)

When `provider.base_url` is None or empty:
```python
response = await litellm.acompletion(
    model="openai/gpt-4o-mini",   # or just "gpt-4o-mini"
    messages=messages,
    api_key="sk-...",             # the REAL OpenAI API key (decrypted by caller)
    # no api_base
    stream=False,
)
```
No `api_base` needed. LiteLLM resolves to `api.openai.com` automatically.
Source: https://docs.litellm.ai/docs/providers/openai — standard direct call.

#### api_key semantics: are they identical?

PARAMETER NAME is identical: `api_key=...` in both cases.
SEMANTICS are different:
- Direct provider: `api_key` = the real provider credential (e.g. `sk-abc123` OpenAI key,
  `sk-ant-...` Anthropic key). Our `service.py` decrypts it from `AiProviderCredential` and
  passes it here.
- Via LiteLLM proxy: `api_key` = the LiteLLM proxy master key or virtual key
  (`sk-1234`). The proxy itself holds the real provider key in its config.yaml / DB.

#### Authorization header
No special custom header is needed when calling via `litellm.acompletion`. The library sends
`Authorization: Bearer <api_key>` automatically for both cases. No manual header injection.

When using the proxy, `Authorization: Bearer <LITELLM_MASTER_KEY>` is what the proxy expects.

#### Model string format
- Via proxy with explicit prefix: `model="litellm_proxy/your-model-name"` + `api_base="http://localhost:4000"`
- Via proxy with provider prefix: `model="openai/gpt-4o-mini"` + `api_base="http://localhost:4000"`
- Direct OpenAI: `model="openai/gpt-4o-mini"` (no `api_base`)

The proxy URL override (`api_base`) is what tells LiteLLM where to send the request.
The model string prefix is secondary — both `litellm_proxy/` and `openai/` work with proxy.

Our stack's `litellm_client.py` pattern (lines 134-142) already implements the correct approach:
```python
kwargs = {
    "model": model_str,
    "messages": messages,
    "api_key": api_key,
}
if provider.base_url:
    kwargs["api_base"] = provider.base_url
response = await litellm.acompletion(**kwargs)
```
This works for BOTH proxy and direct cases — `api_base` is only added when set.

### RESOLVED

The `complete_chat.py` wrapper must follow the SAME `api_base` conditional pattern as
`litellm_client.py`. The `api_key` passed to it must be:
- When provider has a proxy URL: the LiteLLM proxy key (already stored/encrypted in
  `AiProviderCredential.encrypted_secret` for that provider, decrypted by `service.py`).
- When provider has no proxy URL: the real provider API key.

No extra header injection. No environment variable manipulation. The `api_key` kwarg is
sufficient for both paths.
