## DISCREPANCY 2 — Exception class for 502 mapping is SDK-specific, not raw httpx

**Official docs say:**
`streamable_http_client` (from `mcp.client.streamable_http`) raises:
- `StreamableHTTPError` — base transport exception (SDK-specific)
- `ResumptionError` — subclass of `StreamableHTTPError`
- `httpx.StreamError` — HTTP streaming errors
- After `MAX_RECONNECTION_ATTEMPTS = 2` retries, connection failures surface.

The `sse_client` (from `mcp.client.sse`) raises:
- Network/timeout failures via `httpx` standard timeout mechanisms (no SDK-specific wrapper)
- `SSEError` for SSE stream processing errors

For both transports, `httpx.TimeoutException` and `httpx.ConnectError` are the underlying
network exceptions before the SDK wraps them.

**Internal doc says:**
Task pack line 170: "Si transport error / timeout / non-2xx → 502 `MCP_SERVER_UNREACHABLE`"
— does not specify which exception class to catch; left to developer to infer.

TECHNICAL_GUIDE §6.2 line 281: `POST /api/v1/admin/ai/mcp/servers/{id}/sync ... errores: 401,403,404,502`
— does not document the exception catch pattern.

**Implication:**
`mcp.client.discover_tools()` wrapper must catch (in order of precedence):
1. `mcp.client.streamable_http.StreamableHTTPError` (for streamable HTTP transport)
2. `httpx.TimeoutException` (covers `ConnectTimeout`, `ReadTimeout`, `WriteTimeout`, `PoolTimeout`)
3. `httpx.ConnectError` (TCP-level connection refused / DNS failure)
4. `httpx.HTTPStatusError` (non-2xx from the MCP server)
5. `Exception` as last-resort catch-all → map to 502

All of the above map to HTTP 502 + `MCP_SERVER_UNREACHABLE` in the service layer.

**Recommendation reconcile:**
In `mcp/client.py`, wrap the SDK call in:
```python
from mcp.client.streamable_http import StreamableHTTPError
import httpx

try:
    # SDK call
except (StreamableHTTPError, httpx.TimeoutException, httpx.ConnectError, httpx.HTTPStatusError) as exc:
    raise McpServerUnreachableError(str(exc)) from exc
```
No source-of-truth amendment needed — this is implementation detail, not a contract change.

## DISCREPANCY 3 — Transport naming: `http` vs `streamable_http` (SDK) and `sse` (legacy)

**Official docs say:**
The MCP protocol spec (2025-06-18) defines two transports: `stdio` and `Streamable HTTP`.
The older `HTTP+SSE` (protocol 2024-11-05) is deprecated in the spec but maintained for backwards compat.
The Python SDK exposes:
- `mcp.client.streamable_http.streamable_http_client()` — Streamable HTTP (current standard)
- `mcp.client.sse.sse_client()` — legacy HTTP+SSE (still functional, 159 lines active code in `main`)
- `mcp.client.stdio.stdio_client()` — stdio

**Internal doc says:**
Task pack / instrucciones.md: `transport ∈ {http, sse}` — these are **DB domain values**
for `mcp_servers.transport_type`, not SDK class names.
`stdio` is rejected at POST /servers (D-TRANSPORT).

**Implication:**
NO contract discrepancy — the DB values `{http, sse}` are business domain identifiers.
The developer maps them at client-construction time:
- `transport_type='http'` → `streamable_http_client(url, http_client=...)`
- `transport_type='sse'` → `sse_client(url, headers=..., timeout=..., auth=...)`
This is the correct pattern. No source-of-truth amendment needed.

## DISCREPANCY 4 — Auth injection: SDK does NOT expose auth on streamable_http_client constructor

**Official docs say:**
`streamable_http_client(url, *, http_client: httpx.AsyncClient | None = None)`:
- Constructor accepts ONLY `url` + optional pre-built `httpx.AsyncClient`.
- Auth MUST be injected via the `httpx.AsyncClient` before passing it in.
- No `headers=` or `auth=` parameters on `streamable_http_client` directly.

`sse_client(url, *, headers=None, timeout=5.0, sse_read_timeout=300.0, auth=None)`:
- SSE transport accepts `headers` and `auth` directly on the factory function.
- `auth` accepts any `httpx.Auth` subclass.

OAuth2 uses `OAuthClientProvider` + `TokenStorage` protocol. `TokenStorage.set_tokens()` is
the hook for persisting rotated refresh tokens to DB. `OAuthClientProvider` calls
`set_tokens()` automatically on token refresh.

**Internal doc says:**
`mcp_credentials.auth_type` values (TECHNICAL_GUIDE §10.3 line 482):
`auth_type TEXT NOT NULL` — no enum constraint in DDL; candidate values from task pack:
`{none, api_key, bearer, oauth2}`.
`encrypted_refresh_token TEXT` + `expires_at TIMESTAMPTZ` — schema supports oauth2 rotation.

**Implication:**
Auth injection pattern per `auth_type` and transport:

| auth_type | transport=http (streamable_http_client) | transport=sse (sse_client) |
|---|---|---|
| none | No http_client override needed (default httpx.AsyncClient) | No auth/headers param |
| api_key | Build `httpx.AsyncClient(headers={"X-Api-Key": secret})` | `headers={"X-Api-Key": secret}` |
| bearer | Build `httpx.AsyncClient(headers={"Authorization": f"Bearer {secret}"})` | `headers={"Authorization": f"Bearer {secret}"}` |
| oauth2 | Build `httpx.AsyncClient(auth=OAuthClientProvider(...))` with custom `TokenStorage` | `auth=httpx.BearerAuth(token)` or full OAuthClientProvider |

For `oauth2` with refresh: implement `TokenStorage.set_tokens()` to write updated
`encrypted_refresh_token` + `expires_at` back to DB via `mcp_credentials` (using
`encrypt_secret()` on the new refresh token). The developer should implement a
`DbTokenStorage(server_id, db_session)` class in `mcp/client.py`.

Timeout for streamable HTTP: inject via `httpx.AsyncClient(timeout=10.0)` — there is NO
timeout parameter on `streamable_http_client()` itself.
Timeout for SSE: use `sse_client(..., timeout=10.0)` directly.

**Recommendation reconcile:**
No source-of-truth amendment needed. This is the correct implementation pattern for
`mcp/client.py`. Developer should document the `DbTokenStorage` pattern in `client.py`
docstring.

## RESOLVED (debugger cycle 1, 2026-05-13)

**Decision**: For this slice (P02-S07-T001) the MCP discovery client is implemented as a synchronous JSON-RPC 2.0 client over `httpx` (artisanal) instead of adopting `mcp==1.27.1` (the official Anthropic SDK pinned by the researcher).

**Why**:
1. The official SDK exposes async-only API (`async with streamable_http_client(...)`). The FastAPI sync endpoint chain for admin MCP registry calls (POST /servers, POST /sync) is sync end-to-end; adopting the SDK would require turning all four endpoints + service + repository into async, which is scope expansion beyond this slice (KISS / YAGNI per .claude/rules/01-non-negotiables.md).
2. The MCP wire protocol (JSON-RPC 2.0 + `initialize` handshake + `tools/list`/`resources/list`/`prompts/list` requests) is documented at modelcontextprotocol.io and stable in its discovery surface. Implementing just this minimal surface is ~80 LoC of testable code (split across `client.py` and `client_handshake.py` to honour the 300-LoC cap); the SDK adds value only when invoking tools (`tool/call`), which is out of scope for this slice — handled in P02-S08.
3. The `initialize` handshake (added by debugger cycle 1, CRITICAL-2 fix) brings the artisanal client to spec-compliance for discovery (MCP 2025-06-18 §6.1). T26/T27 in `test_mcp_registry.py` exercise the real `discover()` path with a mocked `httpx` transport and assert the wire-order `initialize → notifications/initialized → tools/list → resources/list → prompts/list`, plus fast-fail on missing `result.protocolVersion`.

**Discrepancy-by-discrepancy resolution**:
- **Discrepancy 1 (SDK pinning)**: NOT adopted in this slice. Rationale above. Trade-off: hand-rolled client risks divergence; mitigated by (a) restricted ownership to two module files within `backend/app/mcp/`, (b) follow-up FU to replace TECHNICAL_GUIDE.md line 60 `<SDK MCP Python candidato>` placeholder with `mcp==1.27.1` reference (TECHNICAL_GUIDE edits are blocked by `hook_write_scope_guard.py` during an active DAG slice; reconciliation must happen in a dedicated maintenance pass).
- **Discrepancy 2 (exception classes)**: N/A — SDK not adopted; `client.py` and `client_handshake.py` wrap `httpx.{TimeoutException, ConnectError, RequestError}` + non-2xx + JSON-decode errors + JSON-RPC error envelopes in `McpServerUnreachableError` directly.
- **Discrepancy 3 (transport naming `http` vs `streamable_http`)**: N/A — DB domain values `{http, sse}` are stable business identifiers, not SDK class names.
- **Discrepancy 4 (auth injection)**: N/A — `_build_headers()` applies `Authorization: Bearer` / `X-API-Key` via httpx headers in-process; OAuth2 refresh rotation (`DbTokenStorage`) is deferred to a future slice when `tool/call` lands.

**Recommendation for future slice** (P02-S08 — Agents endpoints, or first slice doing real `tool/call`): re-evaluate adopting `mcp==1.27.1` together with an async/sync bridge. Today's slice is a safe defensible choice that keeps the slice scope bounded.
