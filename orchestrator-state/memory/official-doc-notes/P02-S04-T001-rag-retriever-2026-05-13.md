## Q1 (HIGH) — `cosine_distance()` ORM column method: recommended API, `<=>` pushdown, HNSW activation

### ANSWER: YES — confirmed correct API; HNSW index is activated by this pattern.

### EVIDENCE

Context7 `/pgvector/pgvector-python` (canonical snippet from llms.txt):

```python
# Find by cosine distance
results = session.scalars(
    select(Document)
    .order_by(Document.embedding.cosine_distance(query_embedding))
    .limit(5)
).all()
```

The ORM column method `.cosine_distance(x)` is the **official recommended API** for producing the SQL `<=>` operator in SQLAlchemy 2.x (sync). The retriever code uses:

```python
distance = DocumentEmbedding.embedding.cosine_distance(query_embedding)
stmt = select(...).order_by(distance.asc()).limit(filters.k)
```

This is the exact pattern shown in the official pgvector-python documentation. The `.asc()` call is compatible — `order_by(distance)` (ascending by default) and `order_by(distance.asc())` are equivalent.

For HNSW index pushdown: pgvector `vector_cosine_ops` triggers the `<=>` operator path. When the query is `ORDER BY embedding <=> $1 LIMIT k` without a `WHERE` on the vector column itself, PostgreSQL's planner will use the HNSW index (`document_embeddings_vector_idx`) if the table has rows and the index is present. With an empty table (0 rows), the planner chooses a seq scan (which is still correct — no rows, no cost). Context7 pgvector README also notes that NULL vectors and zero vectors are not indexed — both are excluded by `WHERE de.embedding IS NOT NULL` in `_build_select`.

### VERDICT: matches task pack assumption — CONFIRMED, no change needed.

## Q3 (MEDIUM) — HNSW empty-table behavior: does query return `[]` cleanly?

### ANSWER: YES — returns empty result set cleanly; no error or warning.

### EVIDENCE

From Context7 `/pgvector/pgvector` README (Troubleshooting section):

> "Fewer results after adding an HNSW index can be due to the `hnsw.ef_search` parameter limiting
> the dynamic candidate list. Dead tuples or filtering conditions can also reduce results.
> `NULL` vectors and zero vectors (for cosine distance) are not indexed."

The HNSW index "has no training step" (confirmed in prior note P02-S01-T001-pgvector-2026-05-13.md) — meaning it can be created on an empty table and queried immediately. An empty table has no rows to return, so `SELECT ... ORDER BY embedding <=> $1 LIMIT k` returns 0 rows (empty result set).

PostgreSQL behavior on empty table + `ORDER BY ... LIMIT k`: standard SQL — no error, returns 0 rows. The pgvector `<=>` operator does not raise on empty input; it simply has no rows to process.

The retriever `_rows_to_dtos([])` returns `[]` (empty list). The `retrieve()` function returns `[]` cleanly.

Additional note: pgvector troubleshooting confirms "zero vectors (for cosine distance) are not indexed" but this does NOT cause an error — they are simply excluded from index scan (still found by seq scan fallback if needed).

### VERDICT: expected behavior confirmed — A8 test case (`[]` on empty table) is correct.

## Q4 (LOW) — cosine_distance(x, x) exact value and float tolerance band

### ANSWER: The official return value for identical-direction vectors is `0` (mathematically). In practice, float64 arithmetic may produce small epsilons; `[0.0, 1.001]` tolerance band is safe.

### EVIDENCE

Context7 `/pgvector/pgvector` canonical function reference:

```sql
SELECT cosine_distance('[1,2]'::vector, '[2,4]'::vector);  -- 0 (same direction)
```

The scalar function returns `float8` (double precision / 64-bit IEEE 754). For mathematically identical or same-direction vectors:
- `cosine_distance('[1,2]', '[2,4]')` = 0 (confirmed in official snippet — these are parallel vectors)
- `cosine_distance('[1,0]', '[0,2]')` = 1 (orthogonal)
- `cosine_distance('[1,1]', '[-1,-1]')` = 2 (opposite)

The range is `[0, 2]` per the pgvector docs: "ranging from 0 for identical directions to 2 for opposite directions."

**For unit-normalised identical vectors** (x == y, ||x|| = 1): mathematically `cosine_distance = 0.0`. However, since the vectors are stored as `float4` (PostgreSQL `vector` type uses 32-bit floats internally), accumulation errors in the dot-product and norm computations can produce a small epsilon. The official docs do NOT specify a tolerance band.

**Practical implication for the retriever code:**
- `score = 1.0 - cosine_distance` where `cosine_distance` is a Python float (64-bit) received from psycopg3.
- If `cosine_distance` is `0.0` exactly → `score = 1.0`.
- If `cosine_distance` is a small epsilon (e.g., `-1e-7` due to float rounding in 32-bit internal pgvector ops) → `score = 1.0000001`.
- The code's tolerance band `[0.0, 1.001]` (`score >= 0.0 and score <= 1.001`) is appropriate: it covers both directions of float imprecision.

**PostgreSQL `vector` internal precision:** the `vector` type stores components as `float4` (single-precision). Cosine distance computation is done in float4 arithmetic, then returned as `float8`. This is the root cause of potential epsilon > 0.

Official docs do not contradict the `[0.0, 1.001]` band. The band is a safe engineering decision for float4 → float8 conversion artifacts.

### VERDICT: matches task pack assumption (R6 score range `[0.0, 1.001]`) — CONFIRMED. No change needed.

## Q5 (LOW) — Official LangChain/pgvector pattern for language-aware retrieval with ES fallback

### ANSWER: No official LangChain/pgvector "language-aware retrieval with fallback" pattern exists. Hilo's custom two-pass approach is the correct design. Decision D-RR2 (fallback delegated to caller) is appropriate.

### EVIDENCE

Context7 pgvector-python docs show no language-aware or i18n-specific retrieval pattern. The library's responsibility is vector similarity search; language metadata filtering is application logic.

LangChain pgvector integration (langchain_community.vectorstores.PGVector) provides:
- `similarity_search(query, k, filter={})` — filter dict maps to metadata WHERE clauses.
- This could encode `{"language": "fr"}` as a filter, but there is no built-in "fallback to ES if 0 hits" logic.

The pgvector README has no language-awareness section. Language-aware retrieval is a business logic concern, not a vector store concern.

**Conclusion for Decision D-RR2:** "Fallback delegated to caller" is the correct design. The retriever returns `[]` for zero hits; the caller (chat endpoint / future agent) implements the ES fallback loop. This is consistent with the official separation of concerns — the vector store handles similarity; the application handles multi-language fallback policy.

### VERDICT: no official pattern to conflict with — D-RR2 is correct, no discrepancy.
