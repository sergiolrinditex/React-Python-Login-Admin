## redis-py ping() return value and exceptions — VERIFIED OK

- OFFICIAL:
  - `redis.Redis.ping()` (sync) — returns `bool` (True on PONG response). Annotated as `bool | Awaitable[bool]`.
  - On connection failure: raises `redis.exceptions.ConnectionError` (direct subclass of `redis.exceptions.RedisError`).
  - On timeout: raises `redis.exceptions.TimeoutError` (also a direct subclass of `RedisError`, separate from `ConnectionError`).
  - `from_url(url, **kwargs)` — class method. Accepts `socket_timeout`, `socket_connect_timeout`, `decode_responses` as kwargs passed to the connection pool.
  - Exception hierarchy relevant to health ping:
    ```
    redis.exceptions.RedisError
      ├── ConnectionError  (connection refused / network unreachable)
      │     ├── AuthenticationError
      │     ├── BusyLoadingError
      │     └── MaxConnectionsError
      └── TimeoutError     (socket_timeout / socket_connect_timeout exceeded)
    ```
- INTERNAL (task pack §Contracts item 4, §R4):
  - Recommends `redis.Redis.from_url(url, decode_responses=True, socket_timeout=2, socket_connect_timeout=2)`.
  - Says capture `redis.ConnectionError` on failure.
  - Task pack does NOT mention `TimeoutError` separately.
- DISCREPANCY: **Minor — incomplete exception capture.** Task pack only mentions `redis.ConnectionError`. For a robust health probe, the handler should also catch `redis.exceptions.TimeoutError` (timeout exceeded) separately from `ConnectionError` (refused). Both result in "error" state but have distinct meanings for ops teams. If the developer catches only `ConnectionError`, a timeout will propagate as an unhandled exception (500 instead of 503).
