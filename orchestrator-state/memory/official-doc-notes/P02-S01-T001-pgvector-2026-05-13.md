## Q1 — Index type recommendation for vector(1536), empty table, 10k–100k rows

### What the internal TECHNICAL_GUIDE §10.3 says

```sql
CREATE INDEX document_embeddings_vector_idx
  ON document_embeddings USING ivfflat (embedding vector_cosine_ops);
```
No `lists` parameter specified; index created on empty table.

### What official pgvector 0.8.2 docs say

**HNSW is the recommended index type for production:**
> "It has better query performance than IVFFlat (in terms of speed-recall tradeoff)"

**Context7 llms.txt summary** (authoritative excerpt):
> "The HNSW index is recommended for most production workloads due to its speed-recall tradeoff,
> while IVFFlat is suitable for faster index builds or lower memory usage."

**IVFFlat on empty table — EXPLICITLY WARNED AGAINST:**
> "Create the index *after* the table has some data"
> (IVFFlat has a training/clustering step; on empty table the lists are degenerate)

**HNSW on empty table — EXPLICITLY SAFE:**
> "an index can be created without any data in the table since there isn't a training step"

**IVFFlat lists parameter formula:**
- `rows / 1000` for up to 1M rows → for 10k rows: 10 lists; for 100k rows: 100 lists
- No default lists value is provided by the extension; omitting it defaults to 100

**HNSW default parameters (official):**
- `m = 16` (max connections per layer)
- `ef_construction = 64` (dynamic candidate list during build)
- Both are production-safe defaults

### DISCREPANCY #1 — CRITICAL

| Field | Internal guide | Official docs |
|---|---|---|
| Index type | `ivfflat` | **HNSW recommended for production** |
| Empty table safety | Creates index on empty table | IVFFlat on empty table is **explicitly discouraged**; HNSW is safe on empty table |
| `lists` parameter | Not specified (defaults to 100) | Must match expected row count: `rows/1000` formula |

**Recommended reconciliation for developer:**

Option A (recommended): Switch to HNSW — safe on empty table, better production performance:
```sql
CREATE INDEX document_embeddings_vector_idx
  ON document_embeddings USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);
```

Option B (if ivfflat is kept for build-time reasons): Defer index creation to a separate migration
run AFTER initial data load, NOT in the schema migration. Use:
```sql
CREATE INDEX document_embeddings_vector_idx
  ON document_embeddings USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 10);  -- adjust upward as row count grows; re-run REINDEX after data load
```

Option A is the official recommendation. Option B adds operational complexity.

**ALPINE VARIANT: DOES NOT EXIST** for pg17 (confirmed — no alpine tag in Docker Hub listing).

**My cache from 2026-05-11** (Hook 6, P00-S02-T003 note) already confirmed:
> "the correct tag is `pgvector/pgvector:pg17-trixie` (NOT `pg17-alpine` — alpine variant is not an
> official pgvector tag; must compile from source on alpine)"

**Recommended tag for production-like pinning:**
```yaml
image: pgvector/pgvector:0.8.2-pg17
```
Or for latest Debian 13 base:
```yaml
image: pgvector/pgvector:0.8.2-pg17-trixie
```

**Compatibility with standard postgres image:**
- The `pgvector/pgvector` image is built FROM the official `postgres:17` (or `postgres:17-trixie`) base.
- It is drop-in compatible: same `pg_isready` healthcheck, same volume mounts (`/var/lib/postgresql/data`), same env vars (`POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`).
- Changing from `postgres:17-alpine` to `pgvector/pgvector:0.8.2-pg17` changes the base from Alpine to Debian. Any Alpine-specific shell scripts (`/bin/sh` without bash) should still work since `sh` is available on Debian. If `apk` is used anywhere in init scripts, switch to `apt-get`.

**Extension availability:**
- pgvector extension IS pre-installed in the image binaries, but `CREATE EXTENSION IF NOT EXISTS vector;` in the migration is STILL REQUIRED to activate it in the database.
- The migration's `CREATE EXTENSION` line is correct and necessary.

### DISCREPANCY #2 — MEDIUM (image tag change required)

The docker-compose.yml must change from `postgres:17-alpine` to `pgvector/pgvector:0.8.2-pg17` (or `pg17-trixie` variant). The developer confirmed this was pre-approved by the user. This is not a surprise — it is the expected change. Documented here for traceability.

No alpine tag exists; using Debian-based image is the ONLY official option without building from source.

## Q3 — SQLAlchemy ORM mapping for vector(1536)

### What the internal guide implies

The migration uses raw SQL via `op.execute(...)`. The ORM model would need a column type.

### What official pgvector-python 0.4.2 docs say

**Import path — DISCREPANCY #3 (low severity):**

The current project pins `pgvector==0.4.2`. The official README for the current codebase shows:

```python
from pgvector.sqlalchemy import VECTOR
# NOT: from pgvector.sqlalchemy import Vector
```

The type class is `VECTOR` (all caps), not `Vector`. This matters for ORM model files that will be created
in subsequent slices.

**Full canonical ORM pattern (SQLAlchemy 2.x, verified):**
```python
from pgvector.sqlalchemy import VECTOR
from sqlalchemy.orm import mapped_column, DeclarativeBase

class DocumentEmbedding(Base):
    __tablename__ = "document_embeddings"
    embedding = mapped_column(VECTOR(1536))
```

**SQLAlchemy 2.0.49 compatibility:** Confirmed. The official README shows `mapped_column()` and
`select()` patterns throughout — full SQLAlchemy 2.x API style. No breaking changes noted.

**Alembic index creation — no official helper:**

Official pgvector-python docs show NO Alembic-specific helper. Two valid patterns:

Pattern A — SQLAlchemy `Index` object (preferred for ORM-integrated code):
```python
# In models.py or migration
from sqlalchemy import Index

index = Index(
    'document_embeddings_vector_idx',
    DocumentEmbedding.embedding,
    postgresql_using='hnsw',
    postgresql_with={'m': 16, 'ef_construction': 64},
    postgresql_ops={'embedding': 'vector_cosine_ops'}
)
# In Alembic migration: index.create(op.get_bind())
```

Pattern B — raw SQL in Alembic (valid for pure-migration approach):
```python
# In Alembic migration upgrade():
op.execute("""
    CREATE INDEX document_embeddings_vector_idx
    ON document_embeddings USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64)
""")
```

Both patterns are valid. Pattern B is what the internal guide uses (raw SQL); it remains valid
if the developer switches from ivfflat to hnsw. No helper package needed.

**register_vector() call:**
In older pgvector-python (<0.3.0), `register_vector(conn)` was required for psycopg2.
For SQLAlchemy with psycopg3 (psycopg==3.3.4 as in pyproject.toml), this is NOT needed.
The `VECTOR` type registers automatically when imported.
