## Q1 — cosine_distance API for pgvector.sqlalchemy 0.4.2

**ANSWER: CONFIRMED. `cosine_distance` IS the correct method name in 0.4.2.**

From the v0.4.2 README (SQLAlchemy section):
> "Also supports `max_inner_product`, `cosine_distance`, `l1_distance`, `hamming_distance`, and `jaccard_distance`"

The canonical pattern:
```python
from pgvector.sqlalchemy import Vector  # NOTE: 0.4.2 uses Vector (title-case), not VECTOR

session.scalars(
    select(Item)
    .order_by(Item.embedding.cosine_distance([3, 1, 2]))
    .limit(5)
)
```

**Does `cosine_distance` emit `<=>`?**
Yes. pgvector defines three distance operators:
- `<->` = L2 (Euclidean) distance → `l2_distance`
- `<=>` = cosine distance → `cosine_distance`
- `<#>` = negative inner product → `max_inner_product`

The HNSW index with `vector_cosine_ops` uses the `<=>` operator. `cosine_distance()` emits `<=>`.
The HNSW index WILL be used (index scan, not seqscan) when ordering by `cosine_distance`.

**IMPORTANT — Version drift on import name:**
- v0.4.2 README shows: `from pgvector.sqlalchemy import Vector` (title-case `Vector`)
- Context7 current docs (latest) show: `from pgvector.sqlalchemy import VECTOR` (all-caps)
- The P02-S01-T001 note resolved this as VECTOR (all-caps) — that reflects the CURRENT/latest version.
- For 0.4.2 specifically, the README uses `Vector` (title-case). Both may work as aliases, but
  the canonical export at 0.4.2 tag is `Vector`. Confirm by checking `pgvector/sqlalchemy/__init__.py`
  at v0.4.2 if needed. Safe approach: `from pgvector.sqlalchemy import Vector` for 0.4.2.

RESOLVED: `cosine_distance()` is the correct method; emits `<=>` ; HNSW `vector_cosine_ops` index is used.

## Q2 — vector binding pattern: psycopg3 + SQLAlchemy 2.x async + AsyncSession

**ANSWER: CONFIRMED with important registration requirement.**

### Python type for query_vec

In the v0.4.2 SQLAlchemy section, list literals are used directly:
```python
.order_by(Item.embedding.cosine_distance([3, 1, 2]))
```
Plain `list[float]` works. `numpy.ndarray` also works (the raw psycopg3 examples use `np.array`).
For the ORM `cosine_distance(query_vec)` call specifically: **plain `list[float]` is fine**.

### register_vector IS required for SQLAlchemy + psycopg3

Even when using the SQLAlchemy ORM (not raw psycopg3), registration is required.
The v0.4.2 README shows the exact pattern:

**Sync engine + psycopg3:**
```python
from pgvector.psycopg import register_vector
from sqlalchemy import event

@event.listens_for(engine, "connect")
def connect(dbapi_connection, connection_record):
    register_vector(dbapi_connection)
```

**Async engine + psycopg3 (AsyncSession):**
```python
from pgvector.psycopg import register_vector_async
from sqlalchemy import event

@event.listens_for(engine.sync_engine, "connect")
def connect(dbapi_connection, connection_record):
    dbapi_connection.run_async(register_vector_async)
```

**KEY POINTS:**
1. `engine.sync_engine` — for async engines, listen on the underlying sync engine.
2. `dbapi_connection.run_async(register_vector_async)` — NOT `await register_vector_async(conn)`.
   The event listener is sync; `run_async` bridges to the async registration call.
3. This registration is per-new-connection (via the event), NOT per-call. Once wired at engine
   creation time, every new connection in the pool auto-registers. No per-query overhead.
4. This IS required even with the ORM — the VECTOR type codec must be registered in the
   psycopg3 connection layer regardless of SQLAlchemy layer presence.

RESOLVED: Use `list[float]` for `query_vec`. Wire `register_vector_async` via
`event.listens_for(engine.sync_engine, "connect")` once at engine creation.
No per-call registration needed after that.

## Q3 — HNSW ef_search tuning for small test fixtures (~30 chunks)

**ANSWER: Default ef_search=40 is sufficient for ~30 rows. `SET LOCAL` is optional but recommended for determinism.**

### Default ef_search value

From pgvector server README (confirmed via Context7):
> "hnsw.ef_search (integer) — Size of the dynamic candidate list for search (default 40)."
> "A higher value improves recall but reduces speed."

For a table with ~30 chunks, ef_search=40 already exceeds the table size — the index will
effectively scan ALL nodes. No tuning needed for correctness on small fixtures.

### Canonical usage:
```sql
-- Global (persists for session):
SET hnsw.ef_search = 100;

-- Per-query within transaction:
BEGIN;
SET LOCAL hnsw.ef_search = 100;
SELECT * FROM items ORDER BY embedding <=> '[...]' LIMIT 5;
COMMIT;
```

### Determinism of HNSW on synthetic/one-hot embeddings

HNSW is an **approximate** nearest neighbor algorithm. Even on small tables, the
graph traversal order is influenced by insertion order and index structure.

For smoke tests:
- **Do NOT assert exact ordering of equidistant neighbors** — HNSW graph structure
  can return ties in different orders across runs.
- **DO assert that the top-k results contain the expected document IDs** (set membership),
  not that they are in a specific rank order when distances are equal.
- For one-hot vectors at equal magnitude, L2/cosine distances between distinct vectors
  are deterministic (all equal). Only the TIE-BREAKING order is non-deterministic.
- Practical fix: use embeddings where the correct document has STRICTLY LOWER distance
  than all others (e.g., query = [1.0, 0, 0, ...] and correct chunk has [0.99, 0, 0, ...],
  others have zero in that dimension). This avoids tie-breaking ambiguity entirely.

Official docs do NOT explicitly state determinism guarantees for HNSW — non-determinism
is expected for ANN indexes. Test assertions must be distance-margin-based, not rank-based.

RESOLVED: Default ef_search=40 is sufficient for ~30 rows (ef_search > table_size = full scan).
Write smoke tests asserting set membership with strict distance gap, not rank order.
