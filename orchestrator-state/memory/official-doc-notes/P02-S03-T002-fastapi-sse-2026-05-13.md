## M.4 — FastAPI StreamingResponse + cancellation + headers

### Question
CancelledError propagation into async generator on client disconnect, required SSE headers,
and flushing behavior.

### Answer

#### EventSourceResponse vs StreamingResponse for SSE
FastAPI 0.135.0 introduced native SSE support via `EventSourceResponse` and `ServerSentEvent`
from `fastapi.sse`. This is the RECOMMENDED approach for SSE in modern FastAPI.

```python
from fastapi.sse import EventSourceResponse, ServerSentEvent

@app.post("/chat/stream", response_class=EventSourceResponse)
async def stream_chat(body: ChatRequest) -> AsyncIterable[ServerSentEvent]:
    async for chunk in litellm_stream:
        yield ServerSentEvent(data={"delta": chunk}, event="chunk")
    yield ServerSentEvent(data={"message_id": "..."}, event="done")
```

Source: https://fastapi.tiangolo.com/tutorial/server-sent-events/ — "SSE was added in FastAPI 0.135.0"

#### Headers set automatically by EventSourceResponse
FastAPI's `EventSourceResponse` sets these automatically — no manual header configuration needed:
- `Content-Type: text/event-stream`
- `Cache-Control: no-cache`
- `X-Accel-Buffering: no` (prevents nginx buffering)
- Keep-alive `:` ping comment every 15 seconds (prevents proxy timeouts)

Official docs verbatim: "FastAPI implements some SSE best practices out of the box... You don't
have to do anything about it, it works out of the box."

Source: https://fastapi.tiangolo.com/tutorial/server-sent-events/ — "Technical Details" section

#### CancelledError propagation and generator cleanup
From official FastAPI docs (advanced/custom-response):
- "An `async` task can only be cancelled when it reaches an `await`."
- "If there is no `await`, the generator can not be cancelled properly and may keep running."
- Official pattern for safe cancellation: add `await anyio.sleep(0)` to give the event loop a
  chance to handle cancellation at each iteration step.
- Starlette's `StreamingResponse` iterates the generator; when the client disconnects, anyio
  cancels the coroutine at the next `await` point, which raises `asyncio.CancelledError` inside
  the generator.

Pattern for disconnect-safe generator:
```python
async def sse_generator():
    try:
        async for chunk in litellm_response:
            yield ServerSentEvent(data={"delta": chunk.choices[0].delta.content or ""}, event="chunk")
            await anyio.sleep(0)   # allow cancellation + flush
    except asyncio.CancelledError:
        await litellm_response.aclose()   # clean up LiteLLM stream
        # persist partial content here
        raise   # re-raise so Starlette/anyio knows we're done
```

NOTE: `EventSourceResponse` (added 0.135.0) inherits from Starlette `StreamingResponse`; the same
cancellation mechanics apply.

#### Flushing
- **Automatic per `yield`** — Starlette/ASGI flushes after each yield from the generator.
- `await anyio.sleep(0)` is needed only for cancellation checkpointing, not for flushing per se.
- No manual flush call is needed.

Source: FastAPI docs — each `yield` produces the SSE event immediately; `await anyio.sleep(0)` is
for "allow the event loop to handle cancellation," not flushing.

### Status
DISCREPANCY (minor — approach, not blocking)

The task pack (§C response contract) assumes plain `StreamingResponse` with manual SSE framing
(`f"event: chunk\ndata: {json}\n\n"` style bytes). FastAPI 0.135.0 adds native `EventSourceResponse`
+ `ServerSentEvent` that handles formatting, headers, and keep-alive automatically.

The pack does NOT specify which class to use. Using `StreamingResponse` with manual SSE bytes
is valid W3C-compliant SSE and works with `fetch+ReadableStream`. Using `EventSourceResponse` is
cleaner and handles headers automatically.

PROPOSED_RESOLUTION: Either approach is correct wire-format-wise. Developer may choose:
  - Option A (preferred, less boilerplate): Use `EventSourceResponse` from `fastapi.sse`; yield
    `ServerSentEvent(data=payload_dict, event="chunk")` objects. Headers are auto-set.
  - Option B (pack-compatible, more explicit): Use `StreamingResponse(generator, media_type="text/event-stream")`
    with manual `f"event: chunk\ndata: {json.dumps(payload)}\n\n"` bytes; manually add headers
    `{"Cache-Control": "no-cache", "X-Accel-Buffering": "no"}`.
  Both produce identical wire output. The developer should confirm which matches the existing
  codebase pattern (other SSE endpoints if any). No source-of-truth amendment required — pack
  is silent on which class.

RESOLVED: Developer chose Option B (StreamingResponse with manual SSE byte framing via
`app/chat/streaming/sse.py`). This matches the rest of the codebase which has no prior SSE
endpoints and avoids `fastapi.sse` import (only available ≥0.135.0 — not pinned in pyproject.toml
at time of implementation). Wire format is identical; headers set manually. No source-of-truth
amendment required. Decision documented as D-SSE1/D-SSE2 in `sse.py` module docstring.

## M.5 — SSE framing for fetch+ReadableStream

### Question
Canonical SSE wire format, multi-event writes, id/retry optionality, fetch vs EventSource.

### Answer

#### Canonical SSE wire format (W3C / MDN confirmed)
```
event: chunk\n
data: {"delta":"Tienes "}\n
\n
event: done\n
data: {"message_id":"abc","request_id":"rid"}\n
\n
```

Rules:
- Each event block is terminated by a **double newline (`\n\n`)**.
- `event:` line is optional. If omitted, client receives `message` event type.
- `data:` line is required.
- `id:` line is optional. Sets `EventSource.lastEventId` for resume.
- `retry:` line is optional. Integer milliseconds for reconnection delay.
- Lines starting with `:` are comments (e.g., keep-alive pings like `: ping\n\n`).

Source: https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events/Using_server-sent_events

#### Multiple events in one write
YES — valid. Multiple event blocks separated by `\n\n` in a single TCP write are parsed correctly
by both `EventSource` and `fetch+ReadableStream` manual parsers. The SSE spec is line-based and
does not require one event per TCP packet.

#### id: and retry: — required?
Both OPTIONAL. The task pack marks them optional — confirmed correct.
- `id:` is useful for reconnect resume (EventSource sends `Last-Event-ID` header).
- `retry:` sets reconnect delay (only relevant if client uses EventSource, not fetch).
- For a fetch+ReadableStream client (as declared in pack §3.5.3 D-CHATSTREAM), neither `id:` nor
  `retry:` is functionally required. They can be included without harm.

#### fetch+ReadableStream vs EventSource
The task pack explicitly chose `fetch+ReadableStream` (D-CHATSTREAM decision in TECHNICAL_GUIDE §3.5.3)
over native `EventSource`. This is valid:
- The wire format is IDENTICAL for both consumers.
- fetch+ReadableStream requires client-side manual SSE line parsing (split on `\n\n`, then parse
  `event:` and `data:` lines).
- EventSource auto-parses and handles reconnect; fetch+ReadableStream gives more control (supports
  POST, custom headers, auth tokens — which EventSource does not).
- MDN confirms fetch+ReadableStream is the correct modern approach when POST or custom auth headers
  are needed.

#### Pack event names vs MDN framing
Pack declares events: `meta`, `chunk`, `citation`, `usage`, `error`, `done`.
MDN framing for these:
```
event: meta\ndata: {"message_id":"..."}\n\n
event: chunk\ndata: {"delta":"text"}\n\n
event: citation\ndata: {"document_id":"..."}\n\n
event: usage\ndata: {"tokens_in":342,"tokens_out":58,...}\n\n
event: error\ndata: {"code":"...","message":"..."}\n\n
event: done\ndata: {"message_id":"...","request_id":"..."}\n\n
```
All valid SSE. `data:` must always be a single line (no embedded raw newlines in JSON values).
Use `json.dumps(payload, ensure_ascii=False)` to produce a single-line JSON string.

and fetch+ReadableStream choice are all confirmed correct by MDN and FastAPI SSE official docs.
