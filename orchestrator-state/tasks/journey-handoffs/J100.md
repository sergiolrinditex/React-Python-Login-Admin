## Why this waiver is being granted

The end-to-end browser-MCP verification of journey J100 surfaced a real, documented frontend defect in the auth store's concurrent refresh handling. On every `/chat` page reload (F5), the frontend fires two simultaneous `POST /auth/refresh` requests with the same cookie; the first succeeds and rotates the cookie, the second carries the now-revoked cookie and gets `401 AUTH_SESSION_EXPIRED`, logging the user out. Documented exhaustively by `slice-verifier` in `orchestrator-state/tasks/handoffs/P05-S01-T001.md` (## verify-slice section) and `orchestrator-state/tasks/evidence/P05-S01-T001/verify-07-reload-result.png` + `verify-network-summary.json`.

The defect is out of scope for the originating slice P05-S01-T001 (write_set was strictly `orchestrator-state/tasks/evidence/journeys/J100/**`). It was correctly registered as `FU-20260519121908-auth-store-deduplicate-concurrent-refresh-token-` (severity=high, kind=bug, scope_classification=out_of_scope, write_set=frontend/src/**, journey_refs=[J100]) and promoted on 2026-05-19 to canonical DAG task **P05-S01-T007** (PR #67 merged to main).

P05-S01-T007's `journey_refs=[J100]` creates a chicken-and-egg block: `pending_journey_verifications=[J100]` would defer the claim of P05-S01-T007 itself, even though P05-S01-T007 is precisely the fix needed to verify J100 cleanly.

This waiver releases the journey gate temporarily so the fix can land. The full end-to-end journey re-verification will occur AFTER P05-S01-T007 closer.
