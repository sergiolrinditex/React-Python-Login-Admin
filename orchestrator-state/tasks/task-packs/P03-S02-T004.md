### 3.3 Contratos API afectados

#### `GET /api/v1/users/me` (Auth: Bearer)

- **Already implemented**: P01-S02-T007. Backend file `backend/app/users/routers/me.py`.
- **Response 200**: `{data: UserProfile, meta, errors:[]}` where `UserProfile` exactly matches `frontend/src/features/auth/domain/types.ts:UserProfile` (id, email, full_name, status, preferred_language, roles[], employee_profile|null, created_at, updated_at).
- **Errors**: 401 → aggregate `AUTH_SESSION_EXPIRED` (already handled by authFetch refresh interceptor). No 403 path documented for this endpoint.
- **Reuse decision**: developer MUST reuse the same `UserProfile` type from `features/auth/domain/types.ts` and re-export from `features/user/domain/types.ts`. **Never re-declare**. See §D-T004-USER-DOMAIN-REUSE.

#### `PATCH /api/v1/users/me/language` (Auth: Bearer)

- **Already implemented**: P01-S02-T007. Backend file `backend/app/users/routers/me.py:118+` + `backend/app/users/schemas.py:146-180` (request body strict whitelist `'es'|'en'|'fr'`).
- **Request body**: `{"language": "es" | "en" | "fr"}` (strict — non-string or other literal → 400 `AUTH_INVALID_PAYLOAD` with `field='language'`).
- **Response 200**: **full `UserProfile`** in envelope `{data: UserProfile}`. **NOT 204**. Pinned by `backend/app/users/schemas.py:127-140` and PROGRESS.md (T007 DISCREPANCY-1 resolved → 200+body).
- **Errors**:
  - `400 AUTH_INVALID_PAYLOAD` → UserValidationError (defensive — UI should never send invalid since selector is constrained, but treat as `error_validation` state if it ever happens).
  - `401 AUTH_SESSION_EXPIRED` → handled by authFetch refresh; final 401 surfaces as `UserAuthExpiredError`.
  - `422` documented in TECHNICAL_GUIDE §6.2 but the strict Pydantic mode in T007 collapses both 400 and 422 onto 400. Treat 422 as 400-equivalent (defensive).
- **Side effects**: `audit_log` row with `action="users.language.update"` (no PII), `users.preferred_language` updated, `users.updated_at` set via `func.now()`.

#### `POST /api/v1/auth/logout` (Auth: Bearer)

- **Already implemented**: P01-S02-T004 + cookie-Path fix P01-S02-T011.
- **Already consumed**: `useAuth().logout()` in `features/auth/presentation/AuthProvider.tsx:193-213` fully wires logout HTTP + token clear + queries clear + state reset. **AccountPage will call `useAuth().logout()` directly — no new repository method, no new hook**. See §D-T004-LOGOUT-REUSE.
- **Post-logout side effects (defensive, always run, even on 401)**: clear in-memory token, set status='unauthenticated', clear React Query cache, redirect to `/auth/sign-in`.

## 13. Riesgos y decisiones previas

### 13.1 Riesgos para esta slice

- **R1 — i18n race condition under React 18 StrictMode**: `i18n.changeLanguage()` is asynchronous; React 18 double-invokes effects in dev mode. If `useUpdateLanguage` calls `changeLanguage` inside `onSuccess`, two mutations in StrictMode could resolve in wrong order. **Mitigation**: track `lastIntendedLanguage` in a ref inside the hook; ignore stale `onSuccess` callbacks if `lastIntendedLanguage` no longer matches. Test "race condition" in §11.1.
- **R2 — Optimistic revert UX**: if the optimistic FR is applied and PATCH 5xx fires, reverting to ES is jarring. Acceptable for V1 (rare error path), but log the event clearly and surface a brief inline error.
- **R3 — `i18n.changeLanguage` and react-i18next re-render fan-out**: every component using `useTranslation` re-renders. AccountPage has ~10–15 t() call sites; ChatHomePage has more. Performance is acceptable but cap re-renders with `useTranslation([...])` namespace lists (already standard in this codebase). No additional mitigation needed.
- **R4 — `useMe` vs `useAuth().user` consistency**: AuthProvider's `user` state and `useMe` query may diverge after a language change. **Mitigation**: `useUpdateLanguage.onSuccess` updates BOTH the query cache AND fires a custom event (or directly calls `setQueryData(['user','me'], newProfile)`). The AuthProvider's `user` field can stay stale for V1 since AccountPage reads from `useMe` not `useAuth().user` — but document this clearly in the handoff so future slices know which source-of-truth to use.
- **R5 — Verification data reset between attempts**: if the developer or tester runs `/verify-slice` multiple times without resetting, `preferred_language` will be stuck on the last value. **Mitigation**: explicit reset SQL or `--reset` documented in §12.
- **R6 — Empty state interpretation**: UX_CONTRACT explicitly omits `empty` from `/account` required states. Validator must not flag this as missing. Pinned in §D-T004 + VISUAL_CONTRACT_CHECK.
