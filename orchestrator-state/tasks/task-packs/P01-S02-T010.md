### Why a manual fix (commit 570b702) was needed in spite of the merger

Reading `git show 570b702 -- orchestrator-state/tasks/registry.json` reveals the registry state just before the manual fix was:

```
"status": "ready_for_close",
"last_updated_by": "tester",
"last_outcome": "pass",
"last_stop_at": "2026-05-11T18:57:00+00:00",
```

These are **tester's trailer values**, not closer's. The closer's commit `0d22bcc` had landed on `main` 2h earlier, but the registry never advanced past `ready_for_close`. Two plausible root causes (not mutually exclusive):

1. **Closer's SubagentStop hook did not produce the expected `status=done, last_outcome=committed` write before bootstrap --refresh ran**. The refresh then preserved `ready_for_close/tester/pass` as "live runtime" because that was what was on disk.
2. **The current `_apply_preserved_runtime` only preserves fields present in `old`. If `old["last_outcome"]=="pass"` and `old["status"]=="ready_for_close"`, the merger faithfully copies those. There is no asymmetric "if old is closer-final, refuse to refresh status" guard.**

For this task, the in-scope fix is **defensive hardening + a regression test that exhaustively pins behavior**. We do not chase the upstream closer-hook race here (that would be a separate FU); we make bootstrap idempotent against any closer-final task already on disk, and we add a test that demonstrates this.

## Closer-final state contract

Verified against `.claude/orchestrator-contract.json -> trailer_schema.roles`:

- `closer.outcome_values: [committed, blocked]`; `closer.next_status_values: [done, blocked]`.
- `deployer.outcome_values: [deployed, planned, blocked, failed]`; `deployer.next_status_values: [done, blocked]`.
- `tester.next_status_values: [ready_for_close, needs_debug, blocked]` — `ready_for_close` is NOT closer-final.
- `debugger.next_status_values: [validator_tester_pending, blocked]` — not closer-final.
- `developer.next_status_values: [validator_tester_pending, blocked]` — not closer-final.

**Canonical closer-final set used by this slice:**

```python
CLOSER_FINAL_STATUSES = {"done", "blocked", "skipped"}
CLOSER_FINAL_OUTCOMES = {"committed", "deployed"}
```

A task is "closer-final" iff `task.status in CLOSER_FINAL_STATUSES` AND `task.last_outcome in CLOSER_FINAL_OUTCOMES`. Note `blocked` and `skipped` may occur without `last_outcome=committed/deployed`; in those cases the task is "lifecycle-terminal" but not strictly closer-final. The FU acceptance is conservative — preserve everything when EITHER condition holds, since the existing `_RUNTIME_TASK_FIELDS_TO_PRESERVE` already does that for all listed fields.

## Implementation guidance (for developer)

The minimal fix is two pieces:

1. **The new test file** (the most important deliverable — pins the behavior).
2. **Optional: defensive hardening in `_apply_preserved_runtime`**. If, after writing the test, the existing merger already passes all 7 cases (likely — preservation logic LOOKS correct on read), no code change is needed in `bootstrap_three_docs.py`. The test alone closes the FU because it makes any future regression visible.

If the test reveals an actual gap (most plausible: `last_outcome` not in preserved set, or some downstream step overwrites it), then the developer adds the minimal patch:

- Extend `_RUNTIME_TASK_FIELDS_TO_PRESERVE` if a field is missing.
- Add an explicit guard at the end of `_apply_preserved_runtime`: after the copy loop, for every task whose `old["status"]` is in `{done, blocked, skipped}`, assert (or re-assign) the preserved lifecycle tuple to be exactly equal to `old`'s values. This makes the invariant impossible to break by future refactors.

**Do not** rewrite `bootstrap_three_docs.py` into a new `build_registry.py` file. The FU Write set names that file but the actual implementation is in-place in `bootstrap_three_docs.py`.
