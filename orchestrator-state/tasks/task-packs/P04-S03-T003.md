### Unknowns / decisions for the developer to take and document

1. **Max window**: usage.py uses 90 days max; this endpoint should likely do the
   same. The developer decides between (a) no window cap, (b) 90-day cap with
   422 `AUDIT_WINDOW_TOO_WIDE`, or (c) require both `from` and `to`. Recommend
   following usage.py's precedent: both required, 90-day cap.
2. **Default sort**: `created_at DESC` (most recent first) is the obvious
   default for an audit log; document explicitly.
3. **Cursor format**: opaque base64-encoded `(created_at_iso, id_uuid)`. The
   developer may simplify to offset pagination if the AuditLogPage MVP wants
   simpler — but cursor is more correct for an append-only table and matches
   the §API contract rule "Cursor or offset pagination on list endpoints; no
   unbounded lists". Recommend cursor.
4. **Page size**: 50 default, 200 max (consistent with admin GETs in this repo
   if any precedent exists; otherwise document the choice).
5. **`metadata` JSONB serialization**: ORM attr is `extra_metadata`, DB column
   is `metadata`. The Pydantic response field should be named `metadata` for
   API consistency with the rest of `{data, meta, errors}` (the ORM rename was
   to avoid SQLAlchemy clash, NOT to expose `extra_metadata` in JSON).

### Risks

- **Security**: audit data exposes `actor_user_id`, `action`, `metadata`.
  Metadata may already contain `ip`, `user_agent`, `request_id`, `email_domain`,
  `email_local_len` (per existing audit writers) — but NEVER raw PII or
  credentials (existing writers are validated to redact). The endpoint must NOT
  add any extra surface to the response shape.
- **Performance**: `audit_logs` is append-only; index `(created_at)` and
  `(actor_user_id, created_at)` already exist (per migration 0001 §10.3). The
  filter SQL must be index-friendly: `WHERE created_at BETWEEN :from AND :to
  [AND actor_user_id = :actor] [AND action = :action] ORDER BY created_at DESC, id DESC
  LIMIT :limit`. Reject unbounded windows.
- **Test isolation**: integration tests must commit fixture users (employee +
  admin + inline auditor) so the ASGI request handler can see them (same
  pattern as `test_admin_ai.py`).
- **Auditor verification user gap**: described above (real auditor JWT vs
  `super_admin` superset). `/verify-slice` should still pass with `super_admin`
  per D-PERM1; if reviewers want a real `people_auditor` JWT, that follow-up is
  out of scope.
