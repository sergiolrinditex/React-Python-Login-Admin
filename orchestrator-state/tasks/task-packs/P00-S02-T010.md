### 6.3 Decision — fix loader to match real schema (NOT vice-versa)

The schema IS the source-of-truth (§10.3 + migration 0003 + ORM models). The seed schema (`schemas/admin_ai.py`) is the *seed input contract*; the loader's job is to **map** seed input → real DB rows. The loader is the only thing wrong.

**`AiProviderSeed` schema does NOT need editing** for T010. The seed JSON shape (`name, provider_type, api_key_env, api_key_backup_env, base_url, is_active, description`) stays — it expresses intent, not the row shape. The loader translates:

| `provider_type` | `ai_providers.provider_type` |

### 6.4 Idempotency contract (changes required)

Today's loader uses `ON CONFLICT (name) DO UPDATE` for `ai_providers` and `ON CONFLICT (name) DO UPDATE` for `ai_models`. **Neither table has a UNIQUE constraint on `name`** (verified in migration 0003: only `id` PK and `uq_ai_models_provider_id_model_id` UNIQUE on `(provider_id, model_id)`).

So the existing `ON CONFLICT (name)` would itself raise `InvalidColumnReference` *even if* the columns existed. The fix is two-pronged:

1. **`ai_providers`**: idempotency by `(name)` requires either (a) a partial unique index on `name` (schema change — out of scope) or (b) the loader does **SELECT-then-INSERT** pattern. → **Use SELECT-then-INSERT.**
2. **`ai_provider_credentials`**: no natural unique key beyond `id`. Idempotency = "one active credential per provider": the loader does `DELETE FROM ai_provider_credentials WHERE provider_id = :pid` then `INSERT` (rotation pattern), inside the same `engine.begin()` transaction so a concurrent reader never sees zero credentials. Acceptable for seed CLI (no concurrent admin writes).
3. **`ai_models`**: real UNIQUE index `uq_ai_models_provider_id_model_id` exists. → `ON CONFLICT (provider_id, model_id) DO UPDATE SET model_type=EXCLUDED.model_type, enabled=EXCLUDED.enabled, auto_discovered=EXCLUDED.auto_discovered`.

Note the planner-memory §121 exit-code matrix: missing fixture = 1, dir missing = 2, table missing = 0 (skip). T010 must NOT introduce a new exit code — only fix the SQL inside the existing `if ai_prov_exist:` / `if ai_model_exist:` branches.

### 6.5 Logging policy (changes required)

Current `_logger.debug("seed.admin_ai.upsert_provider.before", api_key_masked="[resolved_from_env]")` is fine. But the loader **must not** log the resolved key value, the encrypted token, or the Fernet master key source — only the env var name and a status. `01-non-negotiables.md §Logging` + `HILO_PEOPLE_TECHNICAL_GUIDE.md §10.5` (BEFORE/AFTER/ERROR with redaction of `api_key`, `encrypted_secret`).

Add structured before/after logs around the credential insertion:

```python
_logger.debug("seed.admin_ai.credential.before", provider_name=provider.name, auth_type="api_key", env_var=provider.api_key_env)
_logger.debug("seed.admin_ai.credential.after", provider_name=provider.name)
```

NEVER log `cred_token`, `plaintext_key`, `provider.api_key`. The token (Fernet output) is technically not the cleartext, but `01-non-negotiables.md` redacts `encrypted_secret` too — same posture.

## 9. Riesgos y decisiones previas

### R1 — Fernet master key in dev (HIGH for the tester, NOT for the developer)

Dev `.env` has `PROVIDER_ENCRYPTION_KEY=dev-encryption-key-placeholder`. This is **not** a valid Fernet key (Fernet requires URL-safe base64 of 32 random bytes). When `app.core.security.encrypt_secret` is called with this value, `Fernet(key_bytes)` raises `ValueError: Fernet key must be 32 url-safe base64-encoded bytes`. T011 (`data` follow-up) owns the dev-env fix.

### Step 3 — Insert the credential when `api_key_env` resolves

For each provider with `provider.api_key_env` set:

1. The current `resolve_env_var(provider.api_key_env, required=True)` already runs and returns the cleartext.
2. Call `app.core.security.encrypt_secret(cleartext)` — this is the **only** place T010 invokes Fernet. Same module the discover-models endpoint uses. Returns a Fernet token string.
3. Open a NEW `engine.begin()` block:
   ```sql
   DELETE FROM ai_provider_credentials WHERE provider_id = :pid;
   INSERT INTO ai_provider_credentials (provider_id, auth_type, encrypted_secret, expires_at)
   VALUES (:pid, :auth_type, :token, NULL);
   ```
4. Increment `report.rows_inserted`.

If `provider.api_key_env` is NOT set AND `provider.api_key` is set (synthetic bundle), the loader currently uses the plaintext directly. **For T010 in productive bundles this branch should not fire** (productive seed JSON has only `api_key_env` references). For the synthetic codepath, still encrypt with `encrypt_secret(provider.api_key)` so `ai_provider_credentials.encrypted_secret` is uniformly Fernet across both bundle types — symmetry with the endpoint's `decrypt_secret` consumer.

If neither `api_key_env` nor `api_key` is present (public provider — e.g., `docs-langchain` if it ever lands), skip the credential insert with `_logger.debug("seed.admin_ai.credential.skip", provider_name=..., reason="no auth")`.

`auth_type` mapping:
- LiteLLM master key (`provider_type='litellm'`, `api_key_env='VERIFICATION_LITELLM_MASTER_KEY'`) → `auth_type='master_key'`.
- All others (`gemini`, `openai`) → `auth_type='api_key'`.
