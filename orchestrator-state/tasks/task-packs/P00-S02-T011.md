# Task Pack: P00-S02-T011 — Dev `.env` ENCRYPTION_KEY hygiene (replace legacy `PROVIDER_ENCRYPTION_KEY=dev-encryption-key-placeholder` with a valid Fernet key under canonical name)

## 0. Resumen ejecutivo (qué cambia y por qué)

Slice `data` (no UI, no backend code, no DB migration). Cierra el último runtime follow-up de la cadena P00-S02 (T006).

**Problema observado en /verify-slice P00-S02-T006**:
- `.env` (línea 12) lleva `PROVIDER_ENCRYPTION_KEY=dev-encryption-key-placeholder`. Eso NO es base64 url-safe de 32 bytes → Fernet lanza `ValueError` al construirse.
- P01-S01-T002 ya renombró el campo a `ENCRYPTION_KEY` en `.env.example` y `backend/app/core/config.py`, pero el `.env` real del dev nunca se sincronizó.
- El backend `core/security.py` introdujo un fallback transitorio (ENCRYPTION_KEY → PROVIDER_ENCRYPTION_KEY → settings.encryption_key) **pero `backend/app/seeds/loader/auth.py:_get_fernet()` lee solo `ENCRYPTION_KEY` y no usa el fallback**. Resultado live:
  - (a) `python -m app.seeds.bootstrap_verification_data --only auth` falla con `ENCRYPTION_KEY env var is required for MFA TOTP encryption` y deja auth + admin_ai sin seed.
  - (b) `POST /api/v1/admin/ai/providers/{id}/discover-models` devuelve 502 `CryptoError` en cada intento porque Fernet rechaza la clave inválida.
- Mitigación temporal en /verify-slice: `export ENCRYPTION_KEY=$(python -c 'from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())')` por shell.

**Solución persistente (KISS)**: actualizar `.env` con un Fernet key válido bajo el nombre canónico `ENCRYPTION_KEY`, y hacer `setup-from-scratch.sh` idempotente para que un clon nuevo deje siempre un key válido sin intervención manual. Documentarlo en TECHNICAL_GUIDE §11.1 (es el home del env contract).

**Por qué es importante AHORA**: J100 y J103 dependen de credenciales cifradas funcionales. Sin esto, el verify-journey J100/J103 de hitos posteriores fallaría con el mismo error en cada terminal nuevo. Cerrar este slice deja el dev workflow auto-sufficient sin requerir tribal knowledge.

### 5.2 Riesgos de rotación de claves sobre datos cifrados existentes

**Real**: si la DB local YA tiene filas en `ai_provider_credentials.encrypted_secret` o `mfa_totp_secrets.secret_encrypted` cifradas con la clave anterior (el placeholder inválido era inutilizable, así que probablemente NO hay datos cifrados — el endpoint era 502 antes), las nuevas filas estarían cifradas con la nueva clave y no podrían descifrar las viejas.

No hay datos productivos en local, así que no hay riesgo real. El developer debe documentar este punto en el handoff: **"hard reset es idempotente porque la clave anterior nunca llegó a cifrar nada en disco — Fernet fallaba antes de escribir"**.

**Nota para producción** (NO scope de este slice, pero relevante para evitar repetirlo): rotación real de ENCRYPTION_KEY exige plan documentado (multi-key reader pattern o re-encrypt-in-place). Eso aterriza en P02-S02-T001 cuando se introduzcan datos productivos cifrados. Si la planner detecta que P02-S02-T001 no tiene ADR sobre rotación, abrir follow-up `low` después del cierre — NO ahora.

```bash
ensure_encryption_key() {
  local env_file="$PROJECT_ROOT/.env"
  local example_file="$PROJECT_ROOT/.env.example"

  if [ ! -f "$env_file" ] && [ -f "$example_file" ]; then
    log "Creating .env from .env.example"
    cp "$example_file" "$env_file"
  fi

  if [ ! -f "$env_file" ]; then
    return 0
  fi

  # 1) Drop legacy PROVIDER_ENCRYPTION_KEY line silently if present.
  if grep -q '^PROVIDER_ENCRYPTION_KEY=' "$env_file"; then
    log "Removing legacy PROVIDER_ENCRYPTION_KEY from .env (renamed to ENCRYPTION_KEY in P01-S01-T002)"
    grep -v '^PROVIDER_ENCRYPTION_KEY=' "$env_file" > "$env_file.tmp" && mv "$env_file.tmp" "$env_file"
  fi

  # 2) Detect missing or placeholder ENCRYPTION_KEY.
  local current_key
  current_key="$(grep -E '^ENCRYPTION_KEY=' "$env_file" | head -1 | cut -d= -f2- || true)"

  local needs_new=0
  if [ -z "$current_key" ] || [ "$current_key" = "<change-me>" ] || [ "$current_key" = "dev-encryption-key-placeholder" ]; then
    needs_new=1
  fi

  if [ "$needs_new" -eq 1 ]; then
    if ! command -v python3 >/dev/null 2>&1; then
      warn "python3 not found; cannot generate Fernet key. Set ENCRYPTION_KEY manually in .env."
      return 0
    fi
    local generated_key
    generated_key="$(python3 -c 'from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())' 2>/dev/null || true)"
    if [ -z "$generated_key" ]; then
      warn "Fernet not available (cryptography pkg not installed yet?). Run backend setup first, then rerun this script."
      return 0
    fi
    if grep -q '^ENCRYPTION_KEY=' "$env_file"; then
      # in-place replace without echoing the key
      python3 - "$env_file" "$generated_key" <<'PY'
import pathlib, re, sys
path = pathlib.Path(sys.argv[1]); key = sys.argv[2]
text = path.read_text()
text = re.sub(r'^ENCRYPTION_KEY=.*$', f'ENCRYPTION_KEY={key}', text, count=1, flags=re.M)
path.write_text(text)
PY
    else
      printf '\nENCRYPTION_KEY=%s\n' "$generated_key" >> "$env_file"
    fi
    log "Generated dev ENCRYPTION_KEY (Fernet) — value not echoed for security"
  fi
}

ensure_encryption_key
```

Reglas duras del bloque:
- **Nunca echo del valor de la clave a stdout/stderr**. Solo "Generated dev ENCRYPTION_KEY — value not echoed for security".
- **Idempotente**: si ya hay `ENCRYPTION_KEY=<valid>`, no hace nada.
- **No depende de servicios externos**: `cryptography` ya viene en backend deps (instalado por el step previo del setup).
- Si el backend aún no instaló deps al primer run → `warn` y sigue (el segundo run completará).
- Borra **silenciosamente** el legacy `PROVIDER_ENCRYPTION_KEY=` si existe (deuda histórica).

## 11. Riesgos y decisiones previas

### 11.1 Riesgos visibles

- **R1 (low)**: Si `cryptography` no está instalado al primer run de `setup-from-scratch.sh` (clon completamente fresco), el bloque `ensure_encryption_key` no puede generar la clave. Mitigación: el script ya tiene un `warn` y sigue; el segundo run (tras `pip install -e ".[dev]"` que el script declara en otro paso) completará. Documentar en TECHNICAL_GUIDE §11.1.
