### Logging architecture (current state, before this slice)

`backend/app/core/logging.py` (157 lines) wires structlog with a `processors`
chain ending in `structlog.stdlib.ProcessorFormatter.wrap_for_formatter`
plus a stdlib `StreamHandler` whose formatter is
`structlog.stdlib.ProcessorFormatter(processor=renderer, foreign_pre_chain=...)`.

```python
shared_processors = [
    structlog.contextvars.merge_contextvars,
    structlog.stdlib.add_log_level,
    structlog.stdlib.add_logger_name,
    structlog.processors.TimeStamper(fmt="iso"),
    _redaction_processor,                 # only event_dict KEYS, not frame locals
    structlog.processors.StackInfoRenderer(),
    structlog.dev.set_exc_info,           # populates event_dict["exc_info"]
]

if verbose:
    renderer = structlog.dev.ConsoleRenderer()  # ← uses RichTracebackFormatter
                                                #    with show_locals=True (default)
else:
    renderer = structlog.processors.JSONRenderer()
```

**The leak path** (CWE-532, confirmed by T002 verify-slice evidence):

1. Caller does `_logger.error("...", exc_info=True)`.
2. `structlog.dev.set_exc_info` extracts `sys.exc_info()` and binds it onto
   `event_dict["exc_info"]`.
3. `_redaction_processor` runs **before** the renderer but only walks
   `event_dict.keys()`. The exc_info tuple is opaque to it.
4. `ConsoleRenderer.__call__` receives the dict, sees `exc_info`, and
   delegates to `RichTracebackFormatter` (default `show_locals=True`) which
   walks every frame in the traceback and prints
   `local_var = repr(value)` to stdout.
5. asyncpg/SQLAlchemy connect path keeps `cparams = {host, user, password,
   port, database}` and `ConnectionParameters(...)` in stack frames at the
   moment of `OperationalError`. Those leak verbatim. T002 evidence captured
   `password=hilopeople_dev_pwd` printed **14+ times** in a single 503 log.

**The JSONRenderer path** (verbose=false / production) renders `exc_info`
as a stringified Python traceback that **does NOT include locals by default**
(stdlib `traceback.format_exception` does not show locals). So the leak is
specific to dev-mode verbose=true. Verify both paths anyway — defense in
depth, plus we want the same scrubbing if a future change swaps renderers.

### Why `_redaction_processor` is insufficient (architectural insight)

`structlog`'s redaction-style processors operate on `event_dict` (a flat
dict). Frame locals captured by `RichTracebackFormatter` live inside the
exception tuple, not inside the dict. There is **no built-in structlog
processor** that walks frame locals — the project must either:

- Disable frame-local rendering at the source: `RichTracebackFormatter(
  show_locals=False)`, OR
- Wrap the renderer in a custom processor that replaces `event_dict[
  "exc_info"]` with a sanitized formatted string before the renderer sees it
  (advanced; risk of breaking line numbers and class names).

§10.5 already mandates redaction of `password`, `token`, `secret`, `api_key`,
`encrypted_secret` — but it predates the discovery that frame locals bypass
the dict-level redactor. **This slice closes that gap.**

## Strategy hints for developer (evaluate options; do not blindly pick)

The acceptance accepts EITHER `show_locals=False` OR a frame-locals
redaction processor. Three viable options, ordered by simplicity:

### Option A — Pure `show_locals=False` on the Rich traceback formatter

```python
import structlog
import structlog.dev

# Build a non-locals-leaking traceback formatter.  Pass it to ConsoleRenderer.
_safe_traceback = structlog.dev.RichTracebackFormatter(show_locals=False)
renderer = structlog.dev.ConsoleRenderer(
    exception_formatter=_safe_traceback,
)
```

**Pros**: minimal code, no custom processor, easy to reason about. Defends
against all callers automatically. **Cons**: loses the dev-side debugging
value of seeing frame locals during local development. **Acceptable
trade-off** — production never used `show_locals=True` anyway (JSON path).

### Option B — Keep `show_locals=True` + custom frame-locals redaction processor

Insert a processor BEFORE the renderer that walks `event_dict["exc_info"]`,
re-builds the traceback with `traceback.TracebackException` and explicitly
filters out locals matching the key/value patterns from the acceptance.
**Higher complexity, higher risk of escaping a key pattern. Not
recommended unless dev-side locals are truly required.**

### Option C — Hybrid (defense in depth) — RECOMMENDED

`show_locals=False` by default + extend `_REDACTED_KEYS` to also include
`pwd`, `dsn`, `database_url`, `connection_string` (acceptance #1 lists
them explicitly). Even if a future contributor reverts Option A or
introduces a non-Rich exception_formatter, the dict-level redactor still
catches the obvious cases.

```python
_REDACTED_KEYS = frozenset({
    # ... existing keys ...
    "pwd",
    "dsn",
    "database_url",
    "connection_string",
})

renderer = structlog.dev.ConsoleRenderer(
    exception_formatter=structlog.dev.RichTracebackFormatter(show_locals=False),
)
```

**Recommended choice**: Option C. Smallest code change with maximum
defense.

### Researcher must validate (cache-shallow ≤5s)

- structlog 25.5.0 canonical name and constructor signature for
  `RichTracebackFormatter` (confirm `show_locals` kwarg exists at this
  version; the kwarg name has been stable since structlog 21.x, but
  Rich's own `Traceback` had a rename in rich==13).
- Whether `ConsoleRenderer` accepts `exception_formatter=` in 25.5.0
  (it has — kwarg added in structlog 22.x — but verify on the docs).
- Any newer redaction-friendly API in structlog 25.x that we should
  prefer (e.g. a built-in DSN scrubber). If yes, prefer that.
