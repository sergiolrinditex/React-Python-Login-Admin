## Bug analysis

### Primary bug — SQLAlchemy `text()` cast syntax (line ~280)

Current code (`backend/app/verification_data/loader.py` ~277-291, inside `load_users()`):

```python
session.execute(
    text(
        "INSERT INTO employee_profiles"
        " (user_id, employee_id, brand, society, center, country, department, metadata)"
        " VALUES (:uid, :eid, :brand, :soc, :ctr, :cntry, :dept, :meta::jsonb)"
    ),
    {
        "uid": user_id, "eid": ep.employee_id, "brand": ep.brand,
        "soc": ep.society, "ctr": ep.center, "cntry": ep.country,
        "dept": ep.department,
        "meta": str(ep.metadata or {}).replace("'", '"'),
    },
)
```

Two defects in the same statement:

1. **Cast syntax**: `:meta::jsonb` — SQLAlchemy's `text()` named-parameter parser treats `::` after a bind parameter as ambiguous and either raises a parser error or fails to substitute the value correctly against psycopg/SQLAlchemy 2.x. The canonical valid forms accepted by `text()` are:
   - `CAST(:meta AS JSONB)` — SQL-standard, recommended.
   - `(:meta)::jsonb` — parenthesized bind, also accepted.
2. **JSON serialization**: `str(ep.metadata or {}).replace("'", '"')` — fragile Python-repr-as-JSON trick. Breaks for string values containing single quotes, booleans (`True`/`False` vs `true`/`false`), `None` vs `null`, datetimes, nested dicts in some edge cases. The canonical fix is `json.dumps(ep.metadata or {})` — exactly the same pattern the same file already uses at lines ~402, ~415, ~594, ~607.

### Recommended fix (developer's choice between two valid forms)

```python
import json  # add at module top with the other stdlib imports

# inside load_users(), the INSERT path:
session.execute(
    text(
        "INSERT INTO employee_profiles"
        " (user_id, employee_id, brand, society, center, country, department, metadata)"
        " VALUES (:uid, :eid, :brand, :soc, :ctr, :cntry, :dept, CAST(:meta AS JSONB))"
    ),
    {
        "uid": user_id, "eid": ep.employee_id, "brand": ep.brand,
        "soc": ep.society, "ctr": ep.center, "cntry": ep.country,
        "dept": ep.department,
        "meta": json.dumps(ep.metadata or {}),
    },
)
```

`CAST(:meta AS JSONB)` is preferred over `(:meta)::jsonb` for clarity and SQL-standard compliance. Either is acceptable. Both must be paired with `json.dumps(...)`, not `str(...)`.

## Impact analysis — code paths exercised by the acceptance tests

### `backend/tests/integration/test_dev_restart_reset.py`

Two tests, both use real Postgres + real alembic CLI binary + real `app.verification_data.bootstrap` CLI:

1. `test_db_reset_cycle_exits_zero` — runs `alembic downgrade base` → `alembic upgrade head` → `python -m app.verification_data.bootstrap --source data/verification` and asserts exit 0.
2. `test_reset_exits_zero_and_is_idempotent` — runs the above cycle twice and asserts both exit 0.

After `alembic upgrade head`, migration 0001 creates `users` and `employee_profiles`. The bootstrap then enters `load_users()`, finds tables exist, and on the first run hits the INSERT branch at line ~280 — that is the SQL `text()` execution that currently fails.

On the second run (idempotency), `load_users()` hits the UPDATE-or-skip branches (`_upsert_user` lines ~140-199 which use plain `text()` with simple bindings — no jsonb cast) AND the `existing_ep is not None` branch at line ~294 which only increments a counter (no SQL). So the second run does NOT hit the buggy line.

This means a **single insertion attempt** must succeed once, and the rest of the test is essentially a no-op verifying idempotency at counter-level only. The fix is small.

### `backend/tests/integration/test_verification_data_bootstrap.py`

Eight tests using `pg_engine` fixture (real Postgres but NO migrations applied by the fixture):

- AC1 idempotency, AC2 missing dir, AC3 missing field, AC4 --only filter, AC5 invalid --only, AC6 --dry-run, AC8 deferred when tables missing, AC9 redaction, AC11 verbose off.

**Critical observation**: this test module's `run_bootstrap()` helper does NOT run `alembic upgrade head`. It relies on the `pg_engine` fixture's pre-existing state. Currently the tester pipeline in T003 was green because tables didn't exist → `_table_exists()` returned False → `deferred` for all groups → exit 0.

**Now that T001 is merged**, the same Postgres database has tables (because previous test runs left tables persisted, OR the fixture creates them, OR the developer runs `alembic upgrade head` outside the test). The acceptance command does `alembic upgrade head` BEFORE pytest, so by the time these 8 tests run, tables DO exist and `load_users()` will exercise the buggy line.

This means:

- AC1 (idempotency, real load): hits line ~280 INSERT on first run, then UPDATE/skip on second. The fix unblocks AC1.
- AC2, AC3, AC5: error-path tests that don't reach line ~280. Not affected.
- AC4 `--only auth`: hits line ~280. Affected.
- AC6 `--dry-run`: status='dry_run' branch — does NOT execute SQL. Not affected.
- AC8 deferred-when-tables-missing: **may now actually fail** because tables are no longer missing! Test logic accepts `status in ("deferred", "ok", "dry_run")`, so `"ok"` (which is what we get post-migration) is accepted. Should still pass after fix.
- AC9 redaction, AC11 verbose-off: hit line ~280 indirectly through bootstrap.main(). Affected.

The developer must verify all 8 tests AC pass after the fix. If AC8's `pg_engine` fixture isolation logic (transactional rollback per test) leaves tables but rolls back rows, the test is still meaningful — it just transitions from "deferred" to "ok" with zero net rows.
