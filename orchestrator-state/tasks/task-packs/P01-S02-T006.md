## A. Why this slice exists

`POST /api/v1/auth/2fa/verify` closes the second leg of the J100 journey
`SignInPage /auth/sign-in → TwoFactorPage /auth/2fa → ChatHomePage /chat`.

The contract is binary:

- Given a valid `mfa_challenge_token` (issued by T002 sign-in when the user has TOTP
  `enabled=True`) and a valid 6-digit TOTP `code`, the endpoint returns
  `{data:{access_token, user}}` and sets the HttpOnly refresh cookie — exactly
  the same shape the no-MFA sign-in branch returns, so the frontend `authStore`
  can land the employee on `/chat`.
- Any failure returns a byte-identical 401 (signature, missing user, no MFA secret,
  wrong code, replay) so the API cannot be used as an enumeration oracle.
  Expired challenge is the only documented 410 case.

This is the first endpoint in the project that **consumes** the
`mfa_challenge_token` minted by `tokens.encode_mfa_challenge_token` (T002) and
the **first** endpoint that touches `mfa_totp_secrets.secret_encrypted` (encrypted
at rest with Fernet via `MFA_ENCRYPTION_KEY`). All other auth endpoints
(sign-up/sign-in/refresh/logout/forgot/reset) are already done.

## F. Decisions (LOCKED — developer must follow)

### F.1 — Replay-prevention strategy (CHOSEN: in-process token-bucket store keyed by `jti`, with TTL = exp)

- **Decision (D-MFA-REPLAY):** Track one-shot consumption of the `mfa_challenge_token` by its `jti` claim. Store: in-memory dict `_consumed_jtis: dict[str, float]` (jti → epoch expiry), guarded by `threading.Lock`. On every successful verify, insert `jti → exp` AFTER the rest of the path succeeds. On entry, reject if `jti in _consumed_jtis and now < _consumed_jtis[jti]`.
- **Cleanup**: opportunistic prune of expired entries on every insert (`for k, exp in list(...): if now > exp: del ...`).
- **Why in-memory and NOT a new DB table:** mirrors the existing rate-limit pattern (`rate_limit.py` _store, also in-memory) and the project-wide V1 plan ("Redis upgrade in P02-S02-T001"). Adding a `mfa_challenge_consumed(jti, expires_at)` table would require a migration outside this slice. KISS/YAGNI.
- **Why not Redis NOW:** Redis is up in compose, but no auth module talks to Redis yet (sign-in lockout also uses audit-log scan, not Redis). Introducing the redis client here would expand scope to a new dependency contract and would not match T002/T003 precedent. The TODO marker `# TODO(P02-S02-T001): replace with Redis SETNX` is canonical.
- **Failure mode**: a single-worker uvicorn (current dev setup) is safe. Multi-worker production would race; documented as "single-worker invariant until P02-S02-T001" in the handoff §K. The worst case is one extra valid verify (the loser still gets a 401 because the jti is consumed AFTER the success branch completes — see test T10).

### F.2 — TOTP verify window (CHOSEN: `valid_window=1`)

- **Decision (D-MFA-WINDOW):** Call `pyotp.TOTP(secret).verify(code, valid_window=1)`. This accepts the current 30s step and ±1 step (so the acceptance window is ~90s, clock-skew tolerant on both client and server).
- **Why 1 and not 2:** RFC 6238 §5.2 says "the prover and verifier MUST share or have knowledge of … a clock that is roughly synchronized". `valid_window=1` covers any clock skew up to 30s without opening a 2.5-minute window. The challenge token TTL is 300s (5 min, AUTH_MFA_CHALLENGE_TTL_SECONDS), so the user has time to type even with `valid_window=1`.
- **Why not 0:** Pure step matching is too strict in a real browser flow (user types code into a Bodoni-Moda 6-digit input while the step boundary passes).
- **Configurable via env**: `AUTH_MFA_VERIFY_WINDOW` (default `1`); developer reads it once at module import.

### F.3 — Anti-enumeration / timing equalization

- **Decision (D-MFA-ANTI-ENUM):** Every 401 path returns the same envelope:
  ```json
  {"data": null,
   "meta": {"request_id": "<uuid>"},
   "errors": [{"code": "AUTH_MFA_CODE_INVALID", "message": "Invalid 2FA code or challenge", "field": null, "details": null}]}
  ```
- **Internally** the use case may raise different typed errors (`MfaChallengeInvalidError`, `MfaCodeInvalidError`, `MfaSecretMissingError`, `MfaReplayError`) but the router maps ALL of them to the same code `AUTH_MFA_CODE_INVALID` + HTTP 401. The actual reason lives in `audit_logs.metadata.reason`.
- **Dummy-verify branch (timing equalisation, mirrors T002):** When the user_id from the challenge has NO row in `mfa_totp_secrets` OR has `enabled=false`, the use case STILL runs a dummy `pyotp.TOTP(_DUMMY_SECRET).verify(code, valid_window=1)` so timing doesn't reveal whether the user has MFA enabled. `_DUMMY_SECRET` is a constant base32 string computed once at module import. Argon2 dummy parallel: see `password.DUMMY_VERIFY_HASH` precedent.
- **Why the dummy verify**: T002 already issues the mfa_challenge token only when `mfa_totp_secrets.enabled=true`, so in theory the "no secret" path is unreachable in production. But a stolen challenge token cookie/JWT replay against a deleted MFA row IS possible, and not running the dummy would let an attacker probe for "this user_id had MFA but it was removed" by timing. The dummy verify costs ~50µs and closes the oracle.

### F.4 — HTTP status mapping (CHOSEN per §6.2 row 261)

| Outcome | HTTP | Envelope code | Audit reason |
|---|---|---|---|
| Success | 200 | — (success body) | `success` |
| Empty payload / not 6 digits / non-numeric code | 400 | `AUTH_INVALID_PAYLOAD` | `invalid_payload` |
| Challenge signature invalid / purpose mismatch / sub not UUID | 401 | `AUTH_MFA_CODE_INVALID` | `challenge_invalid` |
| Challenge `exp` in the past | **410** | `AUTH_MFA_CHALLENGE_EXPIRED` | `challenge_expired` |
| User has no MFA secret / `enabled=false` (dummy-verify still runs) | 401 | `AUTH_MFA_CODE_INVALID` | `no_secret` |
| Code does not match (within window) | 401 | `AUTH_MFA_CODE_INVALID` | `wrong_code` |
| Replay (jti already consumed) | 401 | `AUTH_MFA_CODE_INVALID` | `replay` |
| User inactive (status != 'active') | 401 | `AUTH_MFA_CODE_INVALID` | `user_inactive` |
| Rate limit exceeded | 429 | `AUTH_MFA_VERIFY_RATE_LIMITED` | `rate_limited` |
| Unexpected exception | 500 | `AUTH_MFA_VERIFY_INTERNAL_ERROR` | (log only) |

**§6.2 row 261 declares `400,401,410`** so 410 (expired) is in-contract. The 429 + 500 are operational additions, both already precedented in T002/T003 and in §6.4. Document the 410-vs-401 split:

- **410 (signature still valid, exp claim in the past)**: the user knew the challenge but waited too long. UX implication: TwoFactorPage shows a "challenge expired, please sign in again" CTA.
- **401 (signature invalid, purpose wrong, sub malformed)**: indistinguishable from "wrong code". Hidden in 401.

### F.5 — Rate limit (CHOSEN)

- **Bucket name**: `MFA_VERIFY` (separate from `SIGNIN`, `REFRESH`, `SIGNUP`, etc.).
- **Defaults**: `AUTH_MFA_VERIFY_RATE_PER_MINUTE=20`, `AUTH_MFA_VERIFY_RATE_BURST=5`. Justification: a real user enters 6 digits in <60s; 20/min/IP covers retries comfortably. Burst 5 = "type-and-paste-twice" before slowing. **D-MFA-RL1.**
- **Reuse**: `app/auth/rate_limit.py::_check_bucket("MFA_VERIFY", ip, rate_per_minute, burst)` + new `check_rate_limit_mfa_verify(ip)` wrapper. Add `"MFA_VERIFY": ("20", None)` to `_load_limits` defaults dict.
- **429 returns `Retry-After`** header (set via `_error_response(..., headers={"Retry-After": str(exc.retry_after)})`, same pattern as T002 sign-in).

### F.6 — Refresh cookie + access token (CHOSEN)

- **Refresh cookie:** REUSE `_set_refresh_cookie(json_resp, opaque_refresh)` from `routers/_helpers.py` — byte-identical to sign-in: `HttpOnly; Secure; SameSite=lax; Path=/api/v1/auth; Max-Age=AUTH_REFRESH_TTL_SECONDS`. **D-MFA-COOKIE.**
- **Access token:** `tokens.encode_access_token(user, ttl=AUTH_ACCESS_TTL_SECONDS)`. Same TTL as sign-in (1800s default).
- **Refresh row INSERT:** `RefreshTokenRepository(session).insert_new(user_id=user.id, token_hash=sha256(opaque), expires_at=now+AUTH_REFRESH_TTL_SECONDS)` — flushed within the main transaction; the caller commits after success audit row.
- **Response body shape (consumed by P03-S01-T005 TwoFactorPage):**
  ```json
  {"data": {"access_token": "<jwt>",
            "token_type": "Bearer",
            "expires_in": 1800,
            "user": {"id": "<uuid>",
                     "email": "<corp-email>",
                     "preferred_language": "es",
                     "roles": ["employee"]}},
   "meta": {"request_id": "<uuid>"},
   "errors": []}
  ```
- **User dict shape:** Mirror the §6.3 `User` Pydantic model minus password_hash. The `roles` field is built from `user.user_roles` if loaded; defaults to `["employee"]` (same logic `encode_access_token` already implements).
- **Why include `user` in the body (and not just `access_token` like sign-in success):** §6.2 row 261 explicitly says response is `{data:{access_token, user}}`. This saves one round-trip to `GET /users/me` on the TwoFactorPage success branch. The sign-in success branch returns only `access_token` because the no-MFA caller already has identity from somewhere else; the 2FA path is the first time the frontend learns who the authenticated user is.

### F.7 — Audit (D-S2 pattern reused from T003/T004)

- **Success path**: `AuthRepository(session).write_audit(action="auth.mfa.verify", entity_type="user", entity_id=user.id, actor_user_id=user.id, metadata={request_id, ip, user_agent, outcome:"success"})` — flushed within the main transaction.
- **Failure path**: a SEPARATE short transaction via `audit_session_scope()`:
  ```python
  with audit_session_scope() as audit_session:
      AuthRepository(audit_session).write_audit(
          action="auth.mfa.verify",
          entity_type="user", entity_id=user_id_or_none, actor_user_id=user_id_or_none,
          metadata={request_id, ip, user_agent, outcome:"failure", reason:"<F.4 label>"},
      )
  ```
- **Action name**: `auth.mfa.verify` (lowercase, dotted, mirrors `auth.sign_in`, `auth.password_reset.completed`).
- **Outcome values**: `success | failure`. Reason taxonomy: see F.4 table.

### F.8 — Decryption of `mfa_totp_secrets.secret_encrypted` (CHOSEN: thin facade in auth)

- **Decision (D-MFA-CRYPTO):** Create `backend/app/auth/mfa_crypto.py` containing one function `decrypt_totp_secret(encrypted: str) -> str` that simply calls `app.verification_data.crypto.decrypt_secret(encrypted)`. Same MFA_ENCRYPTION_KEY, same Fernet.
- **Why not import directly from `verification_data.crypto`:** Clean Architecture — the auth module owns the production decryption surface. `verification_data/` is a bootstrap-time module (loads test fixtures); production runtime must not have a runtime import from a fixture-loader package. The facade is one line and isolates the indirection in case we later replace Fernet with KMS.
- **Encoding contract:** `mfa_totp_secrets.secret_encrypted` is a Fernet token (UTF-8 string). Plaintext is a **base32 TOTP seed string** (e.g. `JBSWY3DPEHPK3PXP`, 16 chars). `pyotp.TOTP(secret_b32_str)` accepts the plain str directly — no `.encode()` needed (researcher to confirm in question #3).
- **MFA_ENCRYPTION_KEY hygiene risk**: `.env.example` line 84 shows `MFA_ENCRYPTION_KEY=replace-with-fernet-key-from-generate_key` — the placeholder is NOT idempotently rotated by `scripts/gen-dev-secrets.sh` (that script only rotates JWT keys per FU-20260512044309-d-T009-2). If the dev `.env` still has the placeholder, the decrypt branch will fail with `RuntimeError: MFA_ENCRYPTION_KEY is not a valid Fernet key`. **MITIGATION** (documented, not fixed in this slice): `verify-slice` must check `MFA_ENCRYPTION_KEY` validity before running. Test fixtures use `Fernet.generate_key()` per call (same pattern as `test_auth_signin.py::T02`).

### F.9 — Verbose logging contract

- **BEFORE log** (INFO if verbose, DEBUG otherwise): `auth.mfa.verify.received request_id=%s ip=%s`.
- **AFTER success** (INFO): `auth.mfa.verify.success user_id=%s request_id=%s`.

- **NEVER log**: `code`, `mfa_challenge_token`, `secret_encrypted`, plaintext TOTP seed, raw `opaque_refresh`, JWT access token, full email (domain only).

```
     - InvalidTokenError → MfaChallengeInvalidError (401, audit="challenge_invalid")
     - ExpiredSignatureError → MfaChallengeExpiredError (410, audit="challenge_expired")
  2. Replay check: jti not in _consumed_jtis (else MfaReplayError, audit="replay")
  3. Load user (must exist + status="active")
     - else MfaCodeInvalidError, audit="user_inactive" (also runs dummy verify)
  4. Load mfa_totp_secrets row by user_id:
     - row missing or enabled=false → dummy_verify(code) + MfaSecretMissingError
       (401, audit="no_secret")
     - row present + enabled=true → decrypt → pyotp.TOTP(seed).verify(code,
       valid_window=AUTH_MFA_VERIFY_WINDOW)
       → False → MfaCodeInvalidError (401, audit="wrong_code")
       → True  → continue
  5. Atomic in main tx: insert refresh_tokens row + write success audit + commit
  6. Post-commit: mark jti as consumed (replay store)
  7. Return VerifyMfaResult(user, access_token, refresh_token, expires_in)
      │
      ▼
DB (PostgreSQL via SQLAlchemy 2.0)
  ├─ users (R)                : id, email, status, preferred_language, password_hash
  ├─ user_roles (R, optional) : for roles list in JWT claims
  ├─ mfa_totp_secrets (R)     : user_id, secret_encrypted, enabled
  ├─ refresh_tokens (W)       : INSERT id, user_id, token_hash, expires_at, revoked_at=NULL
  └─ audit_logs (W)            : INSERT action='auth.mfa.verify', actor_user_id,
                                  metadata={request_id, ip, user_agent, outcome, reason}
```

### Request schema (Pydantic v2)

```python
class MfaVerifyRequest(BaseModel):
    challenge_id: str = Field(..., min_length=30, max_length=2048,
                              description="JWT mfa_challenge_token from sign-in MFA branch.")
    code: str = Field(..., min_length=6, max_length=6,
                      description="6-digit TOTP code from the user's authenticator app.")

    @field_validator("code")
    @classmethod
    def code_must_be_six_digits(cls, v: str) -> str:
        if not v.isdigit() or len(v) != 6:
            raise ValueError("code must be 6 numeric digits")
        return v
```

- `challenge_id` name matches §6.2 row 261 exactly. The value IS the JWT (not a separate id). Min 30 is a payload guard; real JWTs are ~180 chars.
- `code` strict 6-digit numeric. Pydantic 422 here on payload garbage; the use case never sees malformed code.
