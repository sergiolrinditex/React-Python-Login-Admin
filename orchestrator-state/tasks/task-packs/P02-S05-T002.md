### D.3 — LiteLLM model-test semantics (acceptance interpretation)

The acceptance line says:

> Admin can test model via **LiteLLM sandbox** and usage summary updates.

Interpretation pinned by planner (subject to researcher confirmation — see §G):

- **Real call, not dry-run.** The endpoint MUST perform an actual `litellm.acompletion(model=..., messages=[{role:"user", content:prompt}], stream=False, api_key=<decrypted>, api_base=<provider.base_url>)` call to the configured provider. Reasoning:
  1. The acceptance phrase "usage summary updates" requires a real `llm_usage_logs` row with non-zero tokens — only a real call produces that.
  2. §10.4 LLM Gateway describes "Smoke: llamada a modelo sandbox oficial/reales/proporcionados con prompt corto y log de latencia." — explicitly a real (sandboxed) call.
  3. The PR for P04-S01-T004 ModelTestDrawer consumes this endpoint to display `output, latency_ms, cost` — placeholders would fail visual verification.
- **The "sandbox" qualifier** refers to the verification-data provider (`litellm_verification_sandbox` from `data/verification/admin_ai/providers/litellm_verification.json`, `base_url=http://localhost:4000`), which in dev points at the local LiteLLM proxy container. In tests we use a real model row pointing at the local proxy or — when the proxy is not reachable — a controlled error path (502 AI_PROVIDER_TEST_FAILED) is the verified happy-path of a failure test.
- **No dry-run flag exists** in the LiteLLM v1.83.14 API for a single completion. Confirm with researcher (Q1).

### Researcher questions
