## 4. Decisión arquitectónica clave — Drawer vs Page

`§D-T004-DRAWER-AS-PAGE` — El "drawer" del UX_CONTRACT/TECHNICAL_GUIDE se implementa como **página de ruta dedicada** (no como overlay con context provider), por estas razones:

1. **TECHNICAL_GUIDE §6.1 lista ruta propia** (`/admin/ai/models/:modelId/test`) — es page, no overlay.
2. **Deep-link debe funcionar** (instrucciones §6.4 navigation contract) — el admin puede llegar directo desde Slack/email; un overlay solo es accesible desde la lista.
3. **`useParams<{ modelId: string }>()`** es el patrón canónico React Router en este proyecto (no hay nada con `<Drawer>` overlay todavía).
4. **Visualmente** se renderiza con un layout de "panel lateral derecho" sobre fondo de la lista (logical wrapper, no Portal). Si en futuro hace falta superponer sobre la lista sin recargar, se introduce overlay; por ahora KISS/YAGNI.
5. **Patrón T003 precedente** — `ModelWizardPage` también es page y se llama "wizard" (no "page wizard"); el naming "Drawer" es estético, no técnico.

Conclusión: `ModelTestDrawer.tsx` es un componente de página completa registrado como `<Route path="/admin/ai/models/:modelId/test" element={<ModelTestDrawer />}>` dentro del bloque `RequireRole`.
