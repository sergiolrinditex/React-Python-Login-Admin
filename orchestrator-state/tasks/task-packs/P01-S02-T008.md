## Root cause inspeccionada (verbatim, ya confirmada por humano)

`scripts/dev-restart.profile.sh` líneas 347-360 (current file, after `# Loading verification_data` block):

```bash
log "Loading verification_data (real/provided fixtures)..."
cd "${HILO_BACKEND_DIR}"
# `python -m app.verification_data.bootstrap` is safe here: `app` is a real
# package at backend/app/, and there is no top-level module that shadows it.
if ! "${py}" -m app.verification_data.bootstrap \
      --source data/verification >>"${BACK_LOG}" 2>&1; then
  cd "${ROOT_DIR}"
  # WARN-only: P00-S02-T004 (FU-20260511145446) is the DAG task that fixes
  # the :meta::jsonb cast bug in backend/app/verification_data/loader.py.
  # The seed step legitimately fails until that slice is closed.
  warn "verification_data bootstrap failed — see ${BACK_LOG}."
  warn "This is expected until P00-S02-T004 closes (FU-20260511145446)."
  return 0
fi
cd "${ROOT_DIR}"
```

Two defects in this block:

1. **Path defect (the bug)**: `cd "${HILO_BACKEND_DIR}"` makes `cwd = <repo>/backend/`, then `--source data/verification` is resolved as `<repo>/backend/data/verification`, which does not exist. The actual fixtures dir is `<repo>/data/verification` (repo root). Bootstrap aborts with "verification data directory not found", `db_reset` swallows the failure with `return 0`, and the user must manually re-invoke with the correct path (the documented workaround). This breaks the acceptance contract "without falling back to the manual workaround".
2. **Stale warn-only branch (the silencer)**: the message "This is expected until P00-S02-T004 closes (FU-20260511145446)" is **stale**. P00-S02-T004 is already `done` (loader fix committed; PROGRESS.md confirms). Today, every legitimate seed failure is silenced by a misleading warning and `return 0` makes `scripts/dev-restart.sh --reset` exit 0 even when no users were inserted. Acceptance (a) "exits 0" must be earned by real success, not by silent fallback.

Both must be addressed together. Acceptance literally says "without falling back to the manual workaround", and ‘fallback’ here is the `return 0` after warn.

## Candidate implementations (developer decides — keep both, pick one)

The developer chooses; both are acceptable. The pack does NOT pre-commit to one — pick the most explicit per `.claude/rules/01-non-negotiables.md` (KISS + explicit > clever).

- **Option A — pass absolute `--source` (preferred for explicitness)**: keep the `cd "${HILO_BACKEND_DIR}"` (the python `-m app.verification_data.bootstrap` form still benefits from running with `backend/` ancestor for module resolution if `.venv` symlinks rely on it) but invoke `--source "${ROOT_DIR}/data/verification"`. Pro: cwd-independent, robust against future relocations. Con: one more inline reference to `${ROOT_DIR}`.
- **Option B — cd to `${ROOT_DIR}` before invoke**: `cd "${ROOT_DIR}"` (instead of `cd "${HILO_BACKEND_DIR}"`) before `"${py}" -m app.verification_data.bootstrap --source data/verification ...`. The python `-m` resolves `app` via `sys.path`; the package import already works from `${ROOT_DIR}` because the venv is configured (or because the earlier import check at line 320 already verified import works from `${ROOT_DIR}` without `cd`). Pro: minimal diff; matches the surrounding `cd "${ROOT_DIR}"` cleanup pattern. Con: relies on python import resolution from repo root (already proven by line 320 — both `cd ${HILO_BACKEND_DIR}` and root cwd are explicitly checked).

**Both options must also**: remove or rewrite the stale `warn "This is expected until P00-S02-T004 closes (FU-20260511145446)."` branch. Two valid replacements:

- Convert to hard failure: drop the `return 0`, replace with `fail "verification_data bootstrap failed; see ${BACK_LOG}"`. This matches acceptance "without falling back to the manual workaround" — a seed failure now correctly aborts `--reset` with non-zero exit, so CI / future humans get a real error instead of silent green.
- Or, if still wanting a soft escape hatch for environments missing `data/verification/` (e.g. clean clones before fixtures land), gate the warn on `[ ! -d "${ROOT_DIR}/data/verification" ]` and otherwise fail. The current behavior (warn-always, return 0) is the worst combination — it hides real bugs.

Either replacement is in scope; both fit `scripts/dev-restart.profile.sh` only.
