## §1 — Resumen ejecutivo

Bug-fix slice promovida desde la runtime follow-up `FU-20260514105909` (origin slice P03-S02-T001). El bootstrap de auth en `AuthProvider.tsx:124` llama `repo.refresh()` desde un `useEffect([])`. Bajo React 18 StrictMode (Vite dev por default) el effect se monta DOS VECES en desarrollo → DOS POST `/api/v1/auth/refresh` concurrentes. El backend rota refresh tokens atómicamente (`refresh_tokens.revoked_at` se setea on use, ver `backend/app/auth/routers/refresh.py:60`): el primero que llega revoca su token y emite uno nuevo; el segundo llega con un token ya revocado → **401**. El usuario ve "session expired" en el primer paint pese a haber loggeado correctamente; o peor, en multi-tab, abrir `/chat` en una segunda pestaña invalida la sesión de la primera.

La solución es un **single-flight** (inflight Promise dedupe) en la capa que llama `fetch(REFRESH_URL)` directamente desde el bootstrap: `AuthRepository.refresh()`. Patrón ya en uso en este repo en `httpClient.ts:50` (`_inflight`), pero httpClient cubre solo el camino `authFetch → 401 → retry` (tests T15–T17 de `auth.test.tsx`), NO el camino bootstrap. Esta slice añade single-flight a `AuthRepository.refresh()` para que cualquier número de callers concurrentes produzcan exactamente 1 fetch.

**Bug es real y reproducible** — no es un sentinel/no-repro (no es P-27).

## §3 — Acceptance criteria (verbatim del registry)

> Two consecutive AuthProvider mounts under StrictMode (or any two concurrent callers of `authRepository.refresh`) produce **exactly ONE** network POST `/api/v1/auth/refresh` (single-flight / inflight-dedupe). No 401 from `/auth/refresh` on a valid session. Multi-tab opening `/chat` in a second tab does not invalidate the first.

Tres invariantes derivados (cada uno debe pasar en `/verify-slice`):

1. **AC1 — StrictMode double-mount**: en `vite dev` (StrictMode activo por defecto), cargar `/chat` autenticado dispara exactamente **1** `POST /auth/refresh` por recarga.
2. **AC2 — Concurrent same-tab callers**: si por cualquier razón dos sitios del código llaman `repo.refresh()` antes de que termine la primera, ambos reciben la misma Promise y se emite **1** fetch.
3. **AC3 — Multi-tab no-invalidation**: abrir una 2ª pestaña en `/chat` mientras la 1ª está logueada no invalida la sesión de la 1ª. *Caveat real: este invariante depende de que ambas pestañas SE COMPORTEN bien individualmente (cada una emita exactamente 1 refresh). El dedupe cross-tab (BroadcastChannel/SharedWorker) es OUT-OF-SCOPE — ver §10.*

#### §D-T006-INFLIGHT-LOCATION — Where the dedupe lives

**Lives in `data/` (authRepository / refreshSingleFlight helper), NOT in `presentation/` (AuthProvider).**

Razones:
- AuthProvider es presentation — su useEffect debe seguir simple, idempotente y cancellable. - Espejo del pattern existente: `httpClient.ts._inflight` también vive en `data/`. Consistencia arquitectónica.

#### §D-T006-INFLIGHT-PROMISE — Shape of the singleton

```ts
// frontend/src/features/auth/data/refreshSingleFlight.ts
let _inflight: Promise<Result<string>> | null = null;

export async function refreshOnce(
  fetcher: () => Promise<Result<string>>,
): Promise<Result<string>> {
  if (_inflight !== null) {
    logVerbose("auth.repo.refresh.dedupe_hit");
    return _inflight;
  }
  logVerbose("auth.repo.refresh.start");
  _inflight = fetcher().finally(() => {
    _inflight = null; // reset on success AND error (try/finally semantics)
  });
  return _inflight;
}

/** Test utility only — mirrors httpClient._resetInflight. */
export function _resetRefreshInflight(): void {
  _inflight = null;
}
```

Invariantes:
- **El reset DEBE ser en `finally`**, no en `.then`/`.catch`. Si solo se resetea en éxito, una llamada fallida (401, network error) deja el dedupe "envenenado" hasta que se descargue el módulo. Esto fue el bug histórico §F1 en httpClient single-flight; no lo repitas.
- `Result<string>` (no `Promise<string>` que rejecta) porque `AuthRepository.refresh()` ya devuelve `Result<string>` (ver `authRepository.ts:421`). Cambiar la signature rompería el caller único `AuthProvider:124`.
- **Module-level singleton** = compartido en TODO el frontend bundle. Cualquier futuro caller dentro del mismo bundle automáticamente queda dedupedo. **Cross-tab dedupe = OUT-OF-SCOPE** (§10).

#### §D-T006-EXISTING-INFLIGHT-IS-DIFFERENT — Why httpClient._inflight does not cover this

`httpClient.ts:50` ya tiene un `_inflight: Promise<string>`. **No cubre este bug** porque:
- `_inflight` solo se dispara cuando `authFetch` recibe **401 después de ya haber hecho una request original**. Es el camino "401 → refresh → retry".
- `AuthProvider:124` llama `repo.refresh()` **directamente**, sin pasar por `authFetch`. Esto es el camino "bootstrap blind refresh" — no hay request original que 401.
- Por tanto el `_inflight` de httpClient nunca se activa en el bootstrap → no hay dedupe → bug.

La fix añade un singleton paralelo, no reemplaza el de httpClient. Tampoco los unifica (sería refactor especulativo no justificado por esta slice — YAGNI).

## §10 — Out-of-scope (explícito)

Esta slice **NO** hace:

1. **Cross-tab dedupe** (BroadcastChannel/SharedWorker). Una 2ª pestaña abierta a `/chat` mientras la 1ª está iniciando el bootstrap todavía dispara 2 fetches independientes (uno por bundle/tab). El AC3 está cumplido porque cada pestaña individualmente emite 1 fetch correcto; la rotación atómica del backend secuencia bien la pareja (la 2ª pestaña "gana" la rotación, emite un nuevo cookie, la 1ª pestaña sigue con el access_token que ya tiene). El riesgo de carrera real cross-tab solo aparece si ambas pestañas RECARGAN al MISMO milisegundo, y la mitigación requeriría infraestructura de sincronización (out-of-scope). Si esto re-aparece como bug productivo, FU separada.
2. **Refactor de `httpClient._inflight` para compartir el singleton con `refreshSingleFlight._inflight`**. ## §17 — Patterns reusables anotados (para planner MEMORY post-slice)

Cuando este slice cierre, planner debe anotar en MEMORY.md:

> **P-35 — Single-flight dedupe in repository layer (P03-S01-T006 pattern)**
>
> When a bug is "concurrent identical mutating calls produce different outcomes" (server rotates state on each call) and the caller is e.g. `useEffect([])` under StrictMode:
>
> 1. Dedupe lives in `data/` (repository), NOT in `presentation/` (provider/component). Reasons: presentation effects must stay simple/cancellable; data layer knows the endpoint; future callers within the same bundle auto-dedupe.
> 2. Extract to a tiny sibling module (`<feature>SingleFlight.ts`) instead of inlining into a fat repo file. Mirror the existing `httpClient._inflight` pattern.
> 3. Reset on `.finally(() => { _inflight = null; })` — NEVER only on success. Failure path that doesn't reset poisons the singleton.
> 4. Module-level singleton is fine in SPA (one bundle, one tab) — but document the cross-tab/cross-bundle limitation as out-of-scope explicitly.
> 5. Test surface: 8 unit cases (1-caller, 2-callers concurrent, 5-callers concurrent, sequential, error-reset, error-shared-result, manual-reset, log-dedupe-hit). PLUS one integration test wrapping `<StrictMode>` literally.
> 6. The existing httpClient `_inflight` does NOT cover bootstrap calls that bypass `authFetch`. Inspect actual call graph before assuming an existing single-flight covers the new caller.
