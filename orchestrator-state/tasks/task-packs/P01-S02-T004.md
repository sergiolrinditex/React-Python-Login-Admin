## Resolved decisions (pinned)

### D1 — Idempotency of repeated logout

**Decision**: 401 + `AUTH_SESSION_EXPIRED` on any path where the caller cannot
prove an active session (missing/invalid Bearer, missing/expired/unknown/revoked
cookie, or active cookie but Bearer invalid).

**Pinned by**:
- §6.2 row: `Auth: Sí` and `Errors: 401`. The endpoint requires authentication,
  so unauthenticated callers MUST get 401 per the contract.
- §3.1 "todo logout queda auditado" → 401 still produces an audit row with
  `outcome=failure, reason=<no_bearer|invalid_bearer|no_cookie|unknown_hash|
  expired|revoked>`. So the call is auditable even on failure.
- Aggregate anti-enumeration: byte-identical 401 body for ALL failure reasons,
  same as T003 refresh.

**Not idempotent at HTTP level by design.** Idempotency of the user-facing
"I clicked Logout twice" UX is a P03-S02-T004 (AccountPage) concern: the
frontend treats both 204 and 401 as "logged out" and redirects to /auth/sign-in.
Backend must remain truthful: no valid session → 401.

**Cookie clearing on 401**: yes — every 401 path also emits
`Set-Cookie: refresh_token=; Max-Age=0; ...` so a stale/invalid cookie cannot
linger in the browser and cause a loop.

### D2 — Auth requirement

**Decision**: Require **BOTH** Bearer access token AND the refresh cookie.

- Bearer access token: proves the caller is alive and identifies the user
  (claim `sub` → UUID → audit `actor_user_id`). Decoded via
  `app.auth.tokens.decode_token`. Bearer-missing or Bearer-invalid → 401.
- Refresh cookie: identifies WHICH specific session/device to revoke. SHA-256
  hash matched against `refresh_tokens.token_hash`. Cookie-missing or
  cookie-unknown/expired/revoked → 401.

**Both must match the same `user_id`** (cookie's row.user_id == JWT.sub). If
they don't match, that is a token-confusion/CSRF-style attempt → 401 +
audit `reason=user_mismatch`. This is the only failure reason that doesn't
also exist in T003 refresh.

**Pinned by**:
- §6.2 row: `Auth: Sí` (required) AND `refresh_tokens revoke` side-effect → the
  endpoint clearly needs both the user identity (Bearer) and the specific token
  (cookie). Cookie-only would not allow distinguishing logout-from-this-device
  from a stolen cookie used to forge logouts.
- §10.2: "Access token Bearer de vida corta. Refresh token en cookie HttpOnly"
  — both are part of the auth strategy; not either-or.
- Non-negotiables §Security: "Auth endpoints: login, register, refresh, logout
  minimum" — logout is auth-protected.

**Implication for the JWT.exp check**: If the Bearer is expired,
`jwt.ExpiredSignatureError` → 401 same envelope. The frontend should call
`/refresh` first then retry `/logout`. Backend is not lenient on Bearer expiry
because the only purpose of Bearer here is identity; an expired Bearer cannot
be trusted to identify the user.

### D3 — Revocation scope (single vs all)

**Decision**: **Single-session revocation only**. Revoke ONLY the
`refresh_tokens` row whose `token_hash` matches SHA-256(cookie). Other rows
for the same user (if any) are untouched.

**Pinned by**:
- §6.2 acceptance: "Logout revokes active refresh token" (singular).
- §3.1 only lists `logout` as one of the operations — no "logout all devices"
  feature mentioned.
- §3.6/§3.7 J102: no "log out everywhere" action.
- No "logout-all" endpoint declared in the §6.2 catalogue.

**"Logout all devices"** is explicitly **out-of-scope** for V1. Note that the
schema (`refresh_tokens(id, user_id, token_hash, expires_at, revoked_at)`)
ALREADY supports it (revoke all rows where `user_id=:uid AND revoked_at IS NULL`)
but a future slice with its own UI (Account → Security → "Sign out everywhere")
would own it.

### Audit row contract

Success (single audit row, in main transaction; main `session.commit()` writes
both the revoke and the audit atomically):

```python
write_audit(
    action="auth.logout",
    entity_type="user",
    entity_id=user_uuid,
    actor_user_id=user_uuid,
    metadata={
        "request_id": request_id,
        "ip": client_ip,
        "user_agent": user_agent[:255],   # cap; defensive
        "outcome": "success",
        "revoked_token_id": str(revoked_row.id),
    },
)
```

Failure (D-S2 pattern: own short session via `audit_session_scope()` so the
audit persists even if main session rolls back):

```python
with audit_session_scope() as short:
    AuthRepository(short).write_audit(
        action="auth.logout",
        entity_type="user",
        entity_id=actor_user_id_or_None,
        actor_user_id=actor_user_id_or_None,
        metadata={
            "request_id": request_id,
            "ip": client_ip,
            "user_agent": user_agent[:255],
            "outcome": "failure",
            "reason": "<no_bearer|invalid_bearer|expired_bearer|"
                      "no_cookie|unknown_hash|expired|revoked|user_mismatch>",
        },
    )
    short.commit()
```

### Rate limiting decision

**No new rate-limit bucket for logout in V1.** Rationale:

- Refresh has a dedicated bucket (`AUTH_REFRESH_RATE_PER_MINUTE`, default 30)
  because authStore retries refresh on every 401 — high natural traffic.
- Logout is intentional + one-shot per user action. Brute-forcing logout is
  not an enumeration vector (401 body is aggregate; no per-call signal).
- A future hardening slice (P02-S02-T001) brings the Redis rate limiter; at
  that point a generic per-IP cap will cover logout too.
- If validator disagrees and wants a bucket, the same 1-pattern reuse of
  `_check_bucket("LOGOUT", ip, …)` adds it in <20 LOC. Pre-declared as
  **not needed** here; if reverse decision required, do it as part of T004,
  not as a follow-up.
