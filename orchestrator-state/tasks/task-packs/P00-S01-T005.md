### 2.5 Formato de bundles — decisión técnica

**Decisión propuesta (KISS, ratificar con researcher si discrepa):**
- Bundles JSON estáticos servidos por Vite desde `frontend/public/locales/{lng}/{ns}.json`. Vite los expone como `/locales/{lng}/{ns}.json` en runtime.
- **Sin `i18next-http-backend`**: `frontend/package.json` ya tiene los namespaces preparados; los cargaremos **inline en build time** vía `import * as commonEs from "../../public/locales/es/common.json"` o equivalente — la forma exacta queda a discreción del developer, pero el contrato es: **sin HTTP backend, sin lazy load, todo en el bundle inicial**.
- Justificación: con 8 ns × 3 idiomas y ≤30 claves iniciales, el coste de bundle es despreciable (<5KB gzipped). Lazy-load añade complejidad innecesaria en P0. Si en una slice posterior el tamaño justifica lazy, se migra entonces (YAGNI).
- Interpolation: simple (`{{var}}`), **NO ICU** — instrucciones.md no exige plurales complejos en P0. Reutilizar `interpolation: { escapeValue: false }` ya configurado.

### 8.2 Riesgos

- **R1 (heredado de T002)**: `i18next-browser-languagedetector` deshabilitado en bootstrap actual por crash de jsdom. Mantener DISABLED en T005. Activación queda para AccountPage (P03-S02-T004), donde el cambio de idioma se persistirá vía `PATCH /api/v1/users/me/language` (P01-S02-T007) y el detector tendrá sentido en runtime real. - **R2**: Vite NO empaqueta automáticamente `frontend/public/**` como módulos importables — los expone como **estáticos** servidos desde `/`. Si el developer elige `import` JSON desde `src/`, el path debe ser RELATIVO desde `src/i18n/index.ts` (ej. `import esCommon from "../../public/locales/es/common.json"` — vite resuelve esto en build time). **Alternativa preferida (KISS, sin tocar tsconfig)**: copiar los JSON a `src/i18n/resources/{lng}/{ns}.json` durante el desarrollo. **Conflicto**: el write_set declarado es `frontend/public/locales/**` (por contrato), no `frontend/src/i18n/resources/**`. → **Decisión propuesta**: respetar el write_set canónico. Importar los JSON desde `public/` con paths relativos desde `src/i18n/index.ts`. Si Vite o tsconfig requieren `resolveJsonModule: true`, confirmar que ya está en `tsconfig.json` (lo está: `tsconfig.json` heredado de T004). **Requiere validación en developer step.**
- **R3**: Tipos i18next augmentation. Para que `useTranslation('auth').t('signIn.title')` sea type-safe, hay que augmentar el módulo `i18next` con `CustomTypeOptions`. Esto NO es bloqueante para acceptance (`t()` siempre devuelve string), pero es BEST PRACTICE recomendada para downstream. **Decisión KISS**: añadir un archivo `frontend/src/i18n/types.d.ts` opcional (parte del write_set `frontend/src/i18n/**`) con la augmentation mínima. Si genera fricción TS, deferir a follow-up.
- **R4**: ESLint no instalado (R1-T004 known). `tsc -b` es el lint gate. Los JSON no compilan TS pero `resolveJsonModule` los importa como tipos genéricos. No bloqueante.
- **R5**: Tamaño bundle. 8 ns × 3 idiomas × ~20 claves = ~480 entries. Estimado <8KB gzipped. Negligible. **YAGNI** lazy loading.
- **R6**: i18next v26 — confirmado en `frontend/package.json`. La API `i18n.use(initReactI18next).init({...})` es estable desde v23. El default export sigue siendo el singleton (planner MEMORY P-6). No hay breaking change esperado.
