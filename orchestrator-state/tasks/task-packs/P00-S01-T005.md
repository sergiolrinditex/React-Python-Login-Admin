## Decisions the developer must make (with planner recommendation)

### D1 — Productive vs scaffold copy → **PRODUCTIVE for keys in §6, MINIMAL placeholders for the rest**

`instrucciones.md §6` is authoritative. Ship the 15 keys from the table verbatim with the exact ES/EN/FR strings. **Do NOT invent additional productive copy** — every key not in §6 is invented scope, which violates DRY/KISS/YAGNI and "tests are REAL". For the gap (`mcp` namespace has no key in §6), seed exactly **one** key per locale that mirrors the patterns nearby:

```json
// frontend/public/locales/es/mcp.json
{ "servers": { "title": "Servidores MCP" } }
// en
{ "servers": { "title": "MCP servers" } }
// fr
{ "servers": { "title": "Serveurs MCP" } }
```

Reason: the bundle must not be `{}` (empty bundles pass `JSON.parse` but vacuously satisfy any drift test, violating "tests are REAL"). One seed key + a comment in the file documenting "P03/P04 will add productive keys for MCP screens" is enough.

Document the convention in `frontend/src/i18n/README.md`? **No** — adding a README is out-of-scope and would be undocumented scope expansion. Document instead in the docstring at the top of `frontend/src/i18n/index.ts` ("Adding keys: copy verbatim from `instrucciones.md §6` table; update all 3 locales in the same commit").

### D2 — Fallback chain → **`fallbackLng: 'es'`**

i18next accepts both `'es'` and `['es']`. The acceptance line says "fallback es" (singular). Use the canonical i18next form `fallbackLng: 'es'`. This matches what providers.tsx already declares.

### D3 — Loading strategy → **EAGER (synchronous resources)**

Two viable strategies in i18next:

- **Eager**: import the 24 JSON files at module load and pass them as `resources` to `i18n.init({...})`. Pros: zero network race in tests; `t()` resolves the first render synchronously; no `Suspense` boundary needed; smaller config surface. Cons: bundles are bundled into the JS payload (~few KB total — productive copy from §6 is tiny).
- **Lazy (`i18next-http-backend`)**: load JSON files at runtime from `/locales/{lng}/{ns}.json`. Pros: split per-locale bundle. Cons: requires extra dep, requires `Suspense` or `useTranslation({ useSuspense: false })`, complicates Vitest + jsdom (must mock fetch or serve via Vite).

**Recommendation: EAGER**. Justification: (a) bundles are tiny — 8 namespaces × 1 file each, plain text; (b) i18next-http-backend is NOT in `frontend/package.json` (T002 deliberately did not add it); adding it would expand T005 scope; (c) test reliability — Vitest+jsdom would need fetch mocks for the lazy path; (d) `TECHNICAL_GUIDE §4` declares both `frontend/src/i18n/` AND `frontend/public/locales/` paths but does not specify the loading strategy — it leaves the choice to implementation. Eager satisfies §4 by keeping the JSON canonically in `public/locales/` (so future lazy migration is trivial) while importing them via `import` (Vite handles JSON imports natively).

Concrete pattern for `frontend/src/i18n/index.ts`:

```ts
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';

import esCommon from '../../public/locales/es/common.json';
import esAuth from '../../public/locales/es/auth.json';
// ... 22 more imports

export const SUPPORTED_LANGUAGES = ['es', 'en', 'fr'] as const;
export const NAMESPACES = ['common', 'auth', 'chat', 'account', 'admin-ai', 'rag', 'mcp', 'errors'] as const;

if (!i18n.isInitialized) {
  i18n.use(initReactI18next).init({
    lng: 'es',
    fallbackLng: 'es',
    supportedLngs: SUPPORTED_LANGUAGES,
    ns: NAMESPACES,
    defaultNS: 'common',
    resources: {
      es: { common: esCommon, auth: esAuth, /* ... */ },
      en: { /* ... */ },
      fr: { /* ... */ },
    },
    interpolation: { escapeValue: false },
  });
}

export default i18n;
```

`languages.ts` exposes `SUPPORTED_LANGUAGES` and helper `isSupportedLanguage(x: string): x is SupportedLanguage`. Used downstream by P01-S02-T007 (PATCH /users/me/language) and P03-S02-T002..T004 (account language switcher).

### D5 — Drift detector test design

The verify command `npm --prefix frontend run test -- --run -t i18n` filters by test name pattern `i18n`. The single test file `frontend/src/i18n/__tests__/i18n.test.ts` should contain a `describe('i18n bundle structure', ...)` block (matches the filter) with the 6 assertions enumerated in §Impact analysis → "Files to create (tests)". Use real `import` of each JSON file (Vite/Vitest handle JSON natively), not file-system reads — file-system reads make the test sensitive to where Vitest resolves CWD.

For drift detection across locales, recursively flatten each bundle to dotted keys and compare sets:

```ts
function flatten(obj: unknown, prefix = ''): string[] {
  if (typeof obj !== 'object' || obj === null) return [prefix];
  return Object.entries(obj).flatMap(([k, v]) =>
    flatten(v, prefix ? `${prefix}.${k}` : k),
  );
}
expect(new Set(flatten(esCommon))).toEqual(new Set(flatten(enCommon)));
expect(new Set(flatten(esCommon))).toEqual(new Set(flatten(frCommon)));
// repeat for all 8 namespaces
```

This is the real value-add of the slice: any future PR that adds a key to ES but forgets EN/FR will break the gate.
