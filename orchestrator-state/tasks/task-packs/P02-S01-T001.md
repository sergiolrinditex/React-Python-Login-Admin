### C.1 — Chat conversacional con RAG (§3.1#chat-rag)

- **Entities**: `Conversation`, `Message`, `MessageCitation`, `LlmUsageLog` (compartido), más tablas RAG via FK.

- **Entities**: `AiProvider`, `AiProviderCredential`, `AiModel`, `AiModelTest`, `LlmUsageLog`.
- **Reglas**: credenciales cifradas (Fernet via `ENCRYPTION_KEY` env, §11.1); solo admins pueden activar modelos; debe existir 1 modelo chat default y 1 modelo embeddings default; ninguna API key se devuelve al frontend.
- **Tablas DB nuevas**: `ai_providers`, `ai_provider_credentials`, `ai_models`, `ai_model_tests`, `llm_usage_logs`.

### C.3 — RAG Admin y vectorización (§3.1#rag-admin)

- **Entities**: `RagCollection`, `Document`, `DocumentVersion`, `DocumentChunk`, `DocumentEmbedding`, `VectorizationJob`.
- **Reglas**: solo admins suben documentos; cada documento tiene idioma + vertical; no se indexan sin colección; embeddings se recalculan por versión; vectorización en Celery.
- **Tablas DB nuevas**: `rag_collections`, `documents`, `document_versions`, `document_chunks`, `document_embeddings`, `vectorization_jobs`.
- **Índice pgvector**: `document_embeddings_vector_idx` ivfflat sobre `embedding vector(1536)` con `vector_cosine_ops`.

### C.4 — MCP y Agents (§3.1#mcp-agents)

- **Entities**: `McpServer`, `McpCredential`, `McpTool`, `McpResource`, `McpPrompt`, `McpAgentBinding`, `Agent`, `AgentRun`, `McpToolInvocation`, `McpApproval`.

- **Tablas DB nuevas**: `mcp_servers`, `mcp_credentials`, `mcp_tools`, `mcp_resources`, `mcp_prompts`, `mcp_agent_bindings`, `agents`, `agent_runs`, `mcp_tool_invocations`, `mcp_approvals`.

### I.4 — Modelos SQLAlchemy: contratos por archivo

### I.5 — Idempotencia y rerun

- `CREATE EXTENSION IF NOT EXISTS vector` — idempotente.
- `op.create_table(...)` — no es idempotente; Alembic se basa en `alembic_version` para no reaplicar. Test idempotencia: upgrade → downgrade → upgrade debe terminar en mismo estado.
- En `downgrade()` ordenar `op.drop_index(..., if_exists=True)` o no — Alembic 1.18.4 acepta `if_exists`; para coherencia con 0001 (NO usa if_exists), no lo añadimos.

### J.2 — Riesgo medio: `pgvector==0.4.2` vs upstream actual

`pgvector` (lib Python) 0.4.2 está pinned (PyPI publicó 0.5.x con HNSW support). Upstream pgvector Postgres extension 0.5+ introdujo el tipo de índice `hnsw` que es estrictamente mejor que `ivfflat` para datasets pequeños y dinámicos (HNSW se construye incrementalmente; ivfflat requiere reindex tras crecer mucho la tabla). El acceptance dice "pgvector index", no especifica el tipo.

**Decisión propuesta**: usar `ivfflat` exactamente como §10.3 declara (literal). Si researcher confirma que la imagen `pgvector/pgvector:pg17` viene con extensión >=0.5 y soporta HNSW, considerar migrar a `hnsw` en una migración futura (no en 0002). Mantener 0002 alineado con §10.3 LITERAL preserva el contrato y simplifica validator.

**Para el researcher**: Q1 (§K) confirmará si pgvector 0.5+ es compatible con `pgvector==0.4.2` Python (lib client). Si HAY incompatibilidad → bloqueo; si NO la hay (forward-compatible) → seguir con 0.4.2 cliente + extension recent.

### J.3 — Riesgo medio: `ivfflat` con tabla vacía / pocos vectores

`ivfflat` requiere parámetro `lists` (defecto en pgvector extension: 100). Con tabla vacía y luego ~10k-100k embeddings, `lists=100` es razonable (recomendación pgvector: `sqrt(N)` o `N/1000` para tabla mediana). §10.3 no especifica `lists`, así que la migración usaría el default. Si el resultado del index es subóptimo en la fase de pruebas, esto será un follow-up de tuning, no de schema.

**Decisión propuesta**: NO especificar `lists` (default 100) en la migración. Confirmar con researcher Q1.
