## Diagnóstico real del bug residual (post FU-20260513201402)

**El fix histórico ya está en disco**: `_display_path` usa `workspace_relpath` (línea 78) — la string cosmética `details["handoff"]` no tira `ValueError`. **Lo que sí está roto**: la resolución del path donde se BUSCA el handoff.

Cadena de resolución actual (orden de precedencia, ver `common.py:135-157`):
```
handoff_path(task_id) -> per_slice_tasks_dir() / "handoffs" / f"{task_id}.md"
per_slice_tasks_dir() -> workspace_root() / "orchestrator-state/tasks"
workspace_root() ->
  1. $CLAUDE_WORKTREE_ROOT (si existe)
  2. $CLAUDE_WORKSPACE_ROOT
  3. $CLAUDE_PROJECT_DIR
  4. $PWD (cwd)
  -> _resolve_worktree_repo(base): sube buscando .git (dir o file)
```

**Modos de fallo reproducidos en vivo desde el worktree de T007**:

| Caller cwd | env workspace | Resultado actual | Esperado |
|---|---|---|---|
| `/tmp` (sin env) | ninguno | EXIT=2 "missing handoff" (mira `/tmp/orchestrator-state/tasks/handoffs/…`) | EXIT=0 (handoff existe en canonical) |
| canonical root (sin env) | ninguno | EXIT=0 (`workspace_root()==canonical`, handoff existe en canonical) | EXIT=0 ✓ |
| canonical root, env `CLAUDE_ORCHESTRATOR_ROOT=canonical` | ninguno | EXIT=0 (handoff existe en canonical) | EXIT=0 ✓ |
| desde worktree T007 cwd | sin env | EXIT=0 ya que `_resolve_worktree_repo` sube hasta el `.git` del worktree | EXIT=0 ✓ |
| desde worktree T007 cwd | env `CLAUDE_WORKTREE_ROOT=worktree`, busca handoff de OTRO task que NO está en este worktree | EXIT=2 "missing" aunque la canonical sí lo tenga | EXIT=0 (fallback canonical) |
| post-cleanup-worktrees | cwd ahora en worktrees padre o `/` | EXIT=2 false-negative | EXIT=0 (canonical autoritativo) |

**Origen mecánico** (per `.claude/rules/04-traceability.md` y CLAUDE.md "Canonical root vs task worktree"):
> *Slice artifacts to commit: `./orchestrator-state/tasks/handoffs/<TASK_ID>.md` … live in the active worktree for PR/Gitflow and are staged by `./scripts/git-add-slice.sh`*

Pero tras merge, el handoff acaba **también** en canonical. El closer pre-check puede correr en CUALQUIERA de los dos: (a) durante /verify-slice dentro del worktree → handoff en worktree; (b) tras squash-merge, en main + post-cleanup → handoff solo en canonical. El resolver actual elige uno de los dos por cwd/env y **no hace fallback** al otro.

## Fix propuesto (mínimo, dentro del write_set canónico)

Solo edita `.claude/bin/check_handoff_contract.py`. **No tocar `common.py`** (fuera de write_set; cualquier cambio ahí afectaría a TODOS los otros consumidores y rompería la política `Conflict group: infra:orchestrator-scripts`).

### Función nueva (privada al módulo): `_resolve_handoff_path(task_id) -> Path`

Política unificada con fallback ordenado, todo dentro de `check_handoff_contract.py`:

1. **Candidate A — workspace**: `common.handoff_path(task_id)` (el actual, vía `workspace_root()`). Si existe → usar.
2. **Candidate B — canonical**: construir `project_root() / STATE_DIR / "tasks" / "handoffs" / f"{task_id}.md"` usando `common.project_root()` + `common.STATE_DIR_NAME`. Si existe → usar.
3. **Fallback**: devolver Candidate A (preserva mensaje "missing handoff: <ruta workspace>" que ya espera el test 158 de `test_handoff_contract_rejects_missing_validator_tester_outcomes` y el contrato existente).

`_display_path()` sigue usando `workspace_relpath()`; si el path elegido vive en canonical y no está bajo workspace, `workspace_relpath()` ya hace fallback a `relpath(project_root())` automáticamente (verificado en `common.py:483-486`). No hace falta tocar `_display_path`.

### Defensa adicional: ValueError jamás propagado

Aunque el `_display_path` actual ya es robusto, envuelve el cuerpo de `main()` después de `parser.parse_args()` en `try/except Exception` (no `BaseException`; preserva `SystemExit`) que imprime el error en stderr y retorna `2`. Esto cumple la regla `01-non-negotiables.md` "never catch generic Exception" parcialmente: la excepción al error-handling se justifica porque es el `main()` de un script CLI cuyo contrato es exit-code estable; el catch convierte un crash interno en "Handoff contract FAILED — <task_id>\n- internal error: <type>: <msg>" + exit 2, NUNCA traceback crudo a stderr. Documentar la decisión con una docstring de 3 líneas sobre la función `main()`.

### Logging (per `.claude/rules/01-non-negotiables.md §Logging`)

Este script NO usa `logging`; es un CLI determinista invocado por hooks. La traza la cubre el ledger (`hook_update_ledger.py`). No introducir `logging` en la fila para no inflar la slice. Sí: añadir docstrings BEFORE/AFTER **conceptuales** en `_resolve_handoff_path` y `validate` (entradas declaradas, output esperado, exit-code semantics).
