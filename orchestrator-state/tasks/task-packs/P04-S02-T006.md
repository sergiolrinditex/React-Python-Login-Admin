## 11. Riesgos, edge cases y decisiones previas

### Riesgos

- **R1 (medium)**: El bug runtime no se reproduce porque la causa real depende de la
  versión de httpx instalada o de un side-effect (DB lock, encryption fallo). Mitigación:
  reproducir primero con curl real antes de tocar código (paso 1 §6).
- **R2 (low)**: Si el fix amplía el catch genéricamente, podría enmascarar bugs internos
  reales (DB down, repo error) que SÍ deben ser 500. Mitigación: el fix debe SIEMPRE
  identificar el tipo exacto de excepción y mapearla específicamente, no usar
  `except Exception: return 502`.
- **R3 (low)**: Tests existentes con mock siguen pasando aunque el runtime falle. Por
  eso el test de regresión nuevo DEBE no usar mock del client.discover y verificar el
  path real del transport error.
- **R4 (low)**: Edge cases adicionales a confirmar agrupados como `MCP_SERVER_UNREACHABLE`
  con `details=null` (no se expone tipo interno):
  - Connection refused (puerto cerrado) — caso primario.
  - DNS resolution failure (endpoint inválido).
  - TLS/SSL handshake error.
  - Timeout (≥`MCP_DISCOVERY_TIMEOUT_SECONDS`, default 10s).
  - HTTP non-2xx del servidor MCP (501/503/etc).
  - JSON inválido en la respuesta.
  - JSON-RPC error envelope (body.error presente).
  - Falta de `protocolVersion` en initialize.
  - Decrypt failure (raises `McpServerUnreachableError` por diseño, ver service_sync.py:98).

  **Decisión**: TODOS esos casos comparten `code=MCP_SERVER_UNREACHABLE` y
  `details=null`. La diferenciación queda en el log interno (`type(exc).__name__`),
  no en la respuesta pública.
