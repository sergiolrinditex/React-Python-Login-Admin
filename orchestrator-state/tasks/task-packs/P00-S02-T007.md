## 1. Pragmatic scope decision (USER-APPROVED, encoded here)

This task is a **MVP** of a feature-based frontend module that lets an admin invoke the
already-shipped `discover-models` endpoint and inspect the result. It is **not** the
canonical AdminAiModelsPage with full table — that is owned by **P04-S01-T002** per
TECHNICAL_GUIDE §6.1 line 229.

### 1.1 Approve/reject reality

The literal acceptance text in the registry says `user can review the diff and approve/reject`.
That cannot be implemented strictly because:

- FU-X1 (`POST /admin/ai/providers/{id}/discover-models`) **persists immediately** — there
  is no `dry_run` path in the endpoint shipped by P00-S02-T006. Persistence happens on the
  backend SELECT-then-INSERT diff; the wizard's view is **post-fact**.
- Per-model **reject (PATCH /admin/ai/models/{id})** lands in **P02-S05-T001**
  (TECHNICAL_GUIDE §6.2 line 265). It does not exist yet.

**MVP shape**:

1. User enters `provider_id` (UUID input — there is no GET providers list yet, that lands
   in P02-S05-T001).
2. User clicks **Discover** → the wizard performs `POST /admin/ai/providers/{id}/discover-models`.
3. Wizard renders the response: lists of `added` (new rows persisted with
   `auto_discovered=true`), `existing` (already present, untouched), `skipped` (with reason),
   plus summary counts.
4. **Approve** = informational confirmation button that simply navigates back to
   `/admin/ai/models` and shows a success toast. It does NOT toggle persistence (already done).
5. **Reject** for individual rows = explicit follow-up note in the handoff pointing at
   P02-S05-T001's PATCH endpoint. Surface this limitation in the UI with a `TrackedLabel`
   (e.g., "Reject pending P02-S05") so developer doesn't silently scope-creep into
   inventing a backend endpoint.

This is the contract the user already approved. Encode it; don't second-guess it.

### 1.2 Out-of-scope fence (explicit)

Developer must not ship any of these in T007:

- Real auth (Phase 1 — `P01-S01..S03`). T007 uses the P00 stub `Bearer dev-admin-<token>`.

## 3. Endpoint contract (mirror of P00-S02-T006 handoff)

### 3.1 Request

```http
POST /api/v1/admin/ai/providers/{provider_id}/discover-models HTTP/1.1
Host: 127.0.0.1:8000
Authorization: Bearer dev-admin-<any-suffix>
Accept: application/json
X-Request-ID: <uuid4>          # optional — backend generates if absent
```

- `provider_id`: path param, UUID v4. Backend returns 422 on malformed UUID
  (FastAPI auto-validates). 404 if the row is absent in `ai_providers`.
- No request body. The backend reads the encrypted credential from
  `ai_provider_credentials` (FK `provider_id`).

### 3.2 Response (200 OK)

```json
{
  "data": {
    "added":    [ { "id": "...", "provider_id": "...", "model_id": "...", "model_type": "chat|embedding|unknown", "capabilities": [], "enabled": true|false, "is_default": false, "auto_discovered": true } ],
    "existing": [ { ...same shape... } ],
    "skipped":  [ { "model_id": "...", "reason": "unsupported_model_type|parse_error|empty_model_id" } ],
    "total_seen": 50
  }
}
```

Schemas: `backend/app/features/admin_ai/schemas.py` — `AiModelOut`, `SkippedModel`,
`DiscoverModelsData`, `DiscoverModelsResponse`. Mirror the field names verbatim in TS.

### 3.3 Error codes

| HTTP | Source | UI mapping (translated copy via `errors` namespace + page-local fallback) |
|---|---|---|
| 401 | `require_admin` — header absent or not `Bearer ` | `permission_denied` state; copy: "Admin access required" |
| 403 | `require_admin` — token does not start with `dev-admin-` | `permission_denied` state |
| 404 | `ProviderNotFoundError` | `error_validation` state; copy: "Provider not found" |
| 422 | `UnsupportedProviderError` or malformed UUID | `error_validation` state; field-level error on the input |
| 502 | `MissingCredentialError` / `CryptoError` / `UpstreamProviderError` | `error_network` state; copy: "Upstream provider error — try again" |
| 5xx, network failure, timeout | client transport | `error_network` state |

The wizard's API client maps the FastAPI error envelope `{detail: {error: {code, message}}}`
to a typed `Result<T, AdminAiError>` object (no thrown exceptions across the layer
boundary, per `01-non-negotiables.md §Error handling`).

## 12. Testing strategy

### 12.1 Stack

- Test runner: **Vitest 4.1.5** (already installed).
- DOM: **jsdom 29.1.1** (configured in `vitest.config.ts` from T004).
- Component testing: **@testing-library/react 16.3.2** + **@testing-library/jest-dom 6.9.1**.
- Setup file imports `import '@testing-library/jest-dom/vitest'` per `official-doc-notes/P00-S01-T004`.
- HTTP mocking: **whatwg-fetch shim** if present, else mock `globalThis.fetch` per test
  (no MSW dependency added in T007). Justification: project does not yet have MSW; adding
  it is a separate dep slice. Direct fetch mock is sufficient for the API client unit tests.

### 12.2 Test layers

| Layer | What is tested | Mock strategy |
|---|---|---|
| API client (`discoverModels.ts`) | Happy path response shape + 6 error mappings (network, 401, 403, 404, 422, 502) | mock `fetch` per test |
| Domain (`diffSummary.ts`) | Pure-function counts | none |
| Component (`DiffReviewTable.tsx`) | Renders 3 lists + summary; empty state when `total_seen=0` | none — pure props |
| Component (`AdminAiModelsPage.tsx`) | Renders shell + CTA → correct href | `MemoryRouter` from `react-router-dom` |
| Component (`ModelWizardPage.tsx`) | Step 1 → 2 → 3 transitions; error state rendering on each error code | mock `fetch` per test |
| i18n drift | T005's existing test passes after admin-ai bundle extension | none |

### 12.3 Real vs mock

Per `01-non-negotiables.md §Tests are REAL`:

- **Component + API-client unit tests mock `fetch`** because that is the boundary between
  our code and an external HTTP endpoint we own. The endpoint itself has its own
  integration tests in T006 (3 PASS + 4 SKIP-able).
- **Verify-slice (human gate) hits the REAL backend** with a real seeded provider. That is
  the production-quality validation; the FE unit tests are guard-rails, not the
  proof-of-life. Document this split in the handoff.
- **No business-logic mocking** — all `discoverModels` mappings live in `discoverModels.ts`
  with real branching (no `jest.spyOn` on internal helpers).
