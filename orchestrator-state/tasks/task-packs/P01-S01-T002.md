### D. docker-compose.yml (lines 209, 269)

```yaml
LITELLM_PROXY_BASE_URL: http://litellm:4000   # backend service env override (line 209)
LITELLM_PROXY_BASE_URL: http://litellm:4000   # worker service env override (line 269)
```

**These are explicit `environment:` overrides on `backend` and `worker` services.** They override anything that comes through `env_file: .env`. After the rename:
- `.env.example` will have `LITELLM_BASE_URL=http://localhost:4000`.
- `config.py` will have `litellm_base_url: str` (env name `LITELLM_BASE_URL`).
- compose still injects `LITELLM_PROXY_BASE_URL=http://litellm:4000` → pydantic-settings will silently ignore it (because `extra="ignore"` and the new field reads `LITELLM_BASE_URL`).
- Effect: **inside the compose stack, `litellm_base_url` would fall back to its default `http://localhost:4000` instead of the compose hostname `http://litellm:4000`.** The backend container would try to reach `localhost:4000` (which is the container itself), fail, and break LiteLLM-dependent slices (P02-S05+).

This is a **real wiring regression** triggered by the rename. The acceptance criterion explicitly anticipated this ("docker-compose.yml env overrides updated to match"), but compose is NOT in `allowed_paths`.

The other "drifted" names visible in compose evidence (`JWT_SECRET`, `PROVIDER_ENCRYPTION_KEY`, `S3_BUCKET_RAG_DOCUMENTS` — visible at `orchestrator-state/tasks/evidence/P00-S02-T001/compose-config-resolved.yaml`) are **not explicit `environment:` overrides** in `docker-compose.yml` itself. They flow only through `env_file: .env`. So renaming them in `.env.example` propagates to the compose stack automatically — no compose edit is needed for those three. The only compose-level override that needs an edit is `LITELLM_PROXY_BASE_URL` at lines 209 + 269.

## Risks

1. **Logging redaction stale entries** — `backend/app/core/logging.py` `_REDACTED_KEYS` still lists `"jwt_secret"` and `"provider_encryption_key"`. After rename these never match. Mitigation: scoped follow-up to update `logging.py` (see Option 3 in Compose decision). New names contain `secret` / `key` / `password` substrings that are already redacted by broader entries (`secret`, `password`), so the immediate security regression is small, but the dead entries should be cleaned. **Track in handoff.**

2. **Compose service-to-service URL break** — if the slice ships without a compose edit, `LITELLM_PROXY_BASE_URL` will be ignored by the renamed `litellm_base_url` field, and `settings.litellm_base_url` falls back to `http://localhost:4000` inside containers. Breaks any LiteLLM consumer in P02-S05+. **Hard blocker** until resolved (see Compose decision).

3. **JWT shape change semantic, not just rename** — current `jwt_secret` is HMAC-style. §11.1 + §10.2 mandate asymmetric (`JWT_PRIVATE_KEY` + `JWT_PUBLIC_KEY`). The auth slice P01-S02-T001 must use RS256 (or another asymmetric algo). This rename slice prepares the field shape but does NOT implement signing — any startup validator the developer adds must be deferred to P01-S02-T001 (which is what the existing `Field(SecretStr(""), ...)` defaults already encode). **Do not add a startup validator that requires both keys in this slice** — it would block the dev stack until both keys are issued.

4. **`MAX_UPLOAD_MB` default change 50 → 25** — current is 50 MB; §11.1 dev/staging/prod is 25 MB. This is a **policy change**, not just a rename. There are no current consumers of this setting (RAG upload lands in P02-S06), so the change is safe today, but document the source-of-truth alignment in the handoff.

5. **`MCP_ALLOWLIST_DOMAINS` default change `""` → `"localhost"`** — current is empty (effectively "deny all"). §11.1 dev says `localhost`. Aligning to `localhost` widens the dev allowlist. No current consumers (MCP lands in P02-S07), so the change is safe today. Document in handoff.

6. **`S3_BUCKET_DOCUMENTS` default value mismatch** — §11.1 dev = `hilo-docs-dev`; current default = `hilopeople-rag-documents-dev`. Decision: keep current default value (this slice renames **the field**, not the bucket value, and the actual bucket name is an infra concern owned by the AWS/MinIO setup, not §11.1). Document in handoff.

7. **Pydantic v2 alias subtlety** — pydantic-settings v2.14.1 reads env vars by lowercasing field names against env keys (because `case_sensitive=False`). No `Field(alias=...)` is needed for the renames. `BaseSettings` matches `JWT_PRIVATE_KEY` env var to `jwt_private_key` field automatically. No `model_config` changes required.

8. **`.env` files in dev environments** — if any contributor has a local `.env` (not committed, gitignored) with the OLD names (`JWT_SECRET=`, `PROVIDER_ENCRYPTION_KEY=`, `LITELLM_PROXY_BASE_URL=`, `S3_BUCKET_RAG_DOCUMENTS=`), pydantic-settings will silently ignore them after rename and fall back to defaults. Mitigation: put a clear migration note in PROGRESS.md / handoff so contributors update their `.env` files manually.
