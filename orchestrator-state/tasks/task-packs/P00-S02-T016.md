## 9. Root cause hypotheses (ordered by likelihood)

### H1 (DOMINANTE, ya validada por T001 family) — Singleton `app.core.db._engine` reusado entre event loops

- `pytest-asyncio asyncio_mode=auto` + sin `loop_scope` declarado → **un loop nuevo por test function**.
- El handler `discover-models` invoca `get_session()` → `_get_session_factory()` → `_get_engine()` → la 1ª llamada cachea `_engine` en el loop del **primer test que ejecuta el handler con DB**.
- Tests 5/6 (401/403) salen antes de tocar DB → no tocan el singleton.
- Tests 1, 2, 4, 7 ejecutan el handler que consulta DB. El primero (orden alfabético `test_audit_log_written` < `test_discover_models_404_unknown_provider` < `test_discover_models_gemini_real` < `test_discover_models_idempotent` < `test_discover_models_litellm_empty`) inicializa el singleton; los siguientes lo reusan en otro loop → asyncpg revienta.
- Sin `.env.local`: GEMINI tests skipean → solo el 404 toca el handler con DB → 1 sola entrada → no hay reuso → **no hay bug**. Esto explica la condicionalidad del síntoma.
- Con `.env.local`: 4 tests tocan el singleton → reuso garantizado → 2 fallan determinísticamente (`idempotent` y `404`, los dos que se ejecutan después del primero que cachea).
- **Probabilidad**: muy alta. **Evidencia previa idéntica**: `test_auth_signup.py:99-127 reset_db_engine_singleton` y MEMORY.md gotcha doc en `test_auth_migration.py`/`test_audit_logs_compliance_cols.py`.

### H2 (descartada como causa primaria) — `loop_scope` incoherente entre fixture session-scoped y function-scoped

- `fernet_test_key` es `scope="session"` pero NO async (solo manipula `os.environ`). No abre conexiones, no participa en loops.
- `postgres_engine` (session) en `conftest.py` está abierto al loop de la primera fixture; pero **este fichero no lo importa**. Descartado.

### H3 (alternativa válida pero más invasiva) — Reemplazar ASGITransport por uvicorn live

- Cambiar `http_client` para usar `httpx.AsyncClient(base_url="http://127.0.0.1:8000")` contra el uvicorn dev real.
- Pros: elimina por completo el doble-loop (un solo loop, el del test).
- Contras: (a) los tests pasarían solo cuando uvicorn está up, no auto-skip; (b) `audit_log_written` y `idempotent` necesitan estado consistente que el uvicorn live no resetea (los providers de test quedarían colgando si crash); (c) cambio más grande de superficie. **Rechazada por coste/riesgo** vs H1.

### H4 (rescate) — Skip determinístico

- Marcar los 2 tests con `@pytest.mark.skipif(_uvicorn_live(), reason="known event-loop conflict — see FU-20260510104826")`.
- **Inaceptable como solución** (oculta el bug). Solo aceptable como fallback si H1+H3 fallaran.
