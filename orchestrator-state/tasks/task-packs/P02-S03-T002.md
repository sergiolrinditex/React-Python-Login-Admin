## A. Goal (1 line)

Implement `POST /api/v1/chat/conversations/{id}/stream` — an SSE endpoint that retrieves RAG
context, streams the LLM response, persists assistant message + citations + usage atomically.

---

## B. Acceptance criteria (verbatim from registry, decomposed)

> "SSE/fetch stream emits chunks, persists assistant message and citations, logs cost"

Decomposed verifiable sub-criteria:

- AC-1. Endpoint returns `Content-Type: text/event-stream` and streams **incremental** chunks
  (no buffer-then-flush). First byte arrives before the LLM completes.
- AC-2. Ownership: only the conversation owner (current user) can stream; other employees → 403,
  unknown conv → 404, no JWT → 401, body invalid → 400, LLM/provider upstream fail → 502.
- AC-3. After stream completion, exactly **one** `messages` row with `role='assistant'` is
  persisted with the full concatenated content and final `token_count`.
- AC-4. RAG citations (if any chunks retrieved) are persisted in `message_citations` linked
  to the new assistant message (NOT to the user message). Survives doc deletion (D-LATE).
- AC-5. One `llm_usage_logs` row is written with `user_id`, `model_id`, `conversation_id`,
  `tokens_in`, `tokens_out`, `estimated_cost`, `latency_ms`.
- AC-6. The user message from the request body is persisted **before** streaming starts
  (so it survives client disconnect mid-stream).
- AC-7. Mid-stream client disconnect: partial assistant content + citations + usage row are
  still persisted with a `failed`/partial marker (see §H.5 for the policy). The endpoint never
  silently drops a turn.
- AC-8. Verbose logging off → only warning+error lines visible; verbose on → BEFORE/AFTER for
  validate → retrieve → generate → persist. NEVER log: API key, raw prompt content, raw chunk
  text, assistant content. Only lengths, IDs, and counts.
- AC-9. Conversation `updated_at` is bumped to `now()` after persistence.
- AC-10. `request_id` (X-Request-ID) is propagated in all logs and emitted in the final
  `done` SSE event for end-to-end traceability.

## C. Endpoint contract (TECHNICAL_GUIDE §6.2 row 267, frozen)

### Request

```http
POST /api/v1/chat/conversations/{conversation_id}/stream HTTP/1.1
Authorization: Bearer <access_token>
X-Request-ID: <uuid v4>
Content-Type: application/json

{ "message": "¿Cuántos días de vacaciones me quedan?" }
```

- `conversation_id` path param: UUID. 404 if not found, 403 if not owner.
- `message` body field: 1–8000 chars (mirror `CreateConversationRequest.initial_message`
  bounds from `app/chat/schemas.py`). Empty/missing → 400 with code `CHAT_STREAM_BAD_REQUEST`.

### Response — happy path (200 OK, text/event-stream)

SSE wire format (W3C `text/event-stream`, fetch-stream compatible per §3.5.3 D-CHATSTREAM decision):

```
event: meta
data: {"message_id":"<assistant_msg_uuid>","model_id":"<uuid>","language":"es","request_id":"<rid>"}

event: chunk
data: {"delta":"Tienes "}

event: chunk
data: {"delta":"22 días"}

...

event: citation
data: {"document_id":"<uuid>","chunk_id":"<uuid>","label":"Política vacaciones, p.3","score":0.87}

event: usage
data: {"tokens_in":342,"tokens_out":58,"estimated_cost":0.000412,"latency_ms":1830}

event: done
data: {"message_id":"<assistant_msg_uuid>","request_id":"<rid>"}
```

Decisions pinned:
- `event:` types: `meta`, `chunk`, `citation`, `usage`, `error`, `done`. No other event names.
- `data:` is always a JSON object (one line per event). No raw text payload.
- Citation events are emitted **between** `meta` and `done` (after retrieval, before/during
  generation — see §H.4 for exact ordering).
- `done` is the **terminal** event. After it, the response stream is closed.
- Mid-stream errors → `event: error` with `{code, message}` then close; no `done`. Status was
  already 200 because headers were sent first — this is intentional SSE semantics.

### Error responses (before headers are sent; standard `{data:null,meta,errors:[...]}` envelope)

| HTTP | Code | Trigger |
|---|---|---|
| 400 | `CHAT_STREAM_BAD_REQUEST` | empty/missing `message`, length > 8000 |
| 401 | `AUTH_SESSION_EXPIRED` | no Bearer / invalid JWT (handled by `get_current_user`) |
| 403 | `CHAT_CONVERSATION_FORBIDDEN` | conv exists, owner mismatch |
| 404 | `CHAT_CONVERSATION_NOT_FOUND` | conv UUID absent |
| 502 | `CHAT_STREAM_UPSTREAM_FAILED` | LLM provider unreachable / fatal before first chunk |
| 502 | `AI_PROVIDER_NOT_CONFIGURED` | no `is_default=true` AND `enabled=true` chat model |

Reuse `ConversationForbiddenError` / `ConversationNotFoundError` from `app/chat/errors.py`.
Add two new errors: `ChatStreamBadRequestError`, `ChatStreamUpstreamError`,
`NoActiveChatModelError`.

|---|---|---|---|
| `messages` (user turn) | `id`, `conversation_id`, `role='user'`, `content=body.message`, `created_at=now()` | BEFORE streaming starts | survives disconnect (AC-6) |

### E.3 — LLM gateway (NEW module, in-scope WRITE_SET_DRIFT §K-LLM-GATEWAY)

§5.1 declares `llm_gateway` as a top-level backend module. §10.4 lists
`llm_gateway/litellm_client.py`. This slice **creates** the gateway with the minimum API
needed by chat streaming:

```python
# backend/app/llm_gateway/litellm_client.py
async def stream_chat(
    *,
    model: AiModel,
    provider: AiProvider,
    api_key: str,
    messages: list[dict[str, str]],   # [{"role":"system|user|assistant","content":"..."}]
    request_id: str,
) -> AsyncIterator[StreamEvent]:
    """Async generator yielding StreamEvent records.

    StreamEvent kinds:
      - 'delta'  → {"delta": "<text>"}
      - 'usage'  → {"tokens_in": int, "tokens_out": int, "estimated_cost": float, "latency_ms": int}
      - 'error'  → {"code": str, "message": str}  (only fatal mid-stream errors)
    """

async def embed_query(
    *,
    model: AiModel,           # embeddings model (model_type='embeddings', is_default=true)
    provider: AiProvider,
    api_key: str,
    text: str,
    request_id: str,
) -> list[float]:              # 1536-dim
    """Compute the embedding vector for a user query. Used by RAG retriever caller."""
```

LiteLLM is the canonical SDK (`litellm==1.83.14` already pinned). Use `litellm.acompletion(...,
stream=True)` for streaming and `litellm.aembedding(...)` for embeddings. Provider routing is
handled by LiteLLM via `model="<provider_type>/<model_id>"` syntax.

**Researcher questions live in §M.** This is the highest-risk integration in the slice.

## G. Pipeline & data flow (the 5 steps inside the handler)

```
[Step 1 VALIDATE]
  - get_current_user (Depends) → User | 401 JSONResponse
  - get conversation by id → 404/403/owner check (reuse get_conversation_detail_for_user logic
    or a thin variant that doesn't load all messages)
  - parse body → 400 if invalid

[Step 2 PERSIST USER MESSAGE]
  - INSERT messages(role='user', content=body.message) on the validated conversation
  - session.commit() — survives the rest of the pipeline (AC-6)

[Step 3 RETRIEVE RAG CONTEXT]
  - get active embeddings model + credentials (model_selector)
  - embed_query(text=body.message) → list[float] (1536-dim)
  - retrieve(session, query_embedding, filters=RetrieverFilters(language=conv.language, k=5))
    → list[RetrievedChunk]
  - if empty list: continue without citations (NOT an error — RAG is optional augmentation)

[Step 4 STREAM-GENERATE]
  - emit SSE "meta" event with assistant_message_id (pre-assigned UUID), model_id, language, rid
  - emit SSE "citation" event PER chunk (before generation, so UI can show "Sources..." early)
  - build system+user prompt:
      system = SYSTEM_PROMPT_PEOPLE_ASSISTANT (constant string — see §G.1)
              + "\n\nContext:\n" + "\n---\n".join(c.content for c in chunks)
      user   = body.message
  - call llm_gateway.stream_chat(model, provider, api_key, messages, request_id)
  - for each StreamEvent:
      kind=='delta' → emit SSE "chunk" + accumulate to assistant_content_buffer
      kind=='usage' → store usage payload (don't emit yet — emit just before "done")
      kind=='error' → emit SSE "error" event + break + go to Step 5 PARTIAL persist branch
  - if client disconnects mid-stream → see §H.5

[Step 5 PERSIST + CLOSE]
  - INSERT messages(id=assistant_message_id, role='assistant', content=buffer, token_count=usage.tokens_out)
  - INSERT message_citations bulk (N rows from chunks)
  - INSERT llm_usage_logs(user_id, model_id, conversation_id, tokens_in, tokens_out, est_cost, latency_ms)
  - UPDATE conversations SET updated_at=now() WHERE id=:cid
  - session.commit()  (single atomic tx for AC-3..AC-5, AC-9)
  - emit SSE "usage" event
  - emit SSE "done" event
  - close stream
```

### G.2 — Stream cancellation / client disconnect handling

FastAPI's `StreamingResponse` cancels the async generator via `asyncio.CancelledError` when
the client disconnects. The handler MUST:

1. Wrap the async generator in `try/except asyncio.CancelledError:` (and the broader `Exception`).
2. On cancel/error AFTER first chunk: still run **Step 5 PARTIAL persist** with the partial
   buffer + `token_count=None` + `llm_usage_logs.latency_ms=<time-so-far>` + a structured log
   line `chat.stream.client_disconnect partial_chars=N citations=M`.
3. Use a separate DB session for the persistence cleanup (the request-scoped session may
   already be in an aborted state). See `app.db.session.get_db_session` for the helper pattern;
   if no "background session" helper exists, create a minimal one in-module (`SessionLocal()`).
   Flag this as predeclared WRITE_SET_DRIFT §K-CHATSTREAM-BGSESSION.
4. NEVER swallow the error — log at WARNING level.

§E "no silent loss" invariant from §12 Constraints: *"Every streaming message is persisted
after completion or marked failed with audit trail."*

## H. Decisions (D-CHATSTREAM-x)

- **D-CHATSTREAM-WIRE**: include the streaming router in `chat/routers/__init__.py`
  aggregator. Do NOT touch `main.py` unless aggregator becomes architecturally awkward.
- **D-CHATSTREAM-SSE**: SSE wire format pinned to `event:` / `data: <JSON>`. Event names
  fixed (meta/chunk/citation/usage/error/done). Reason: §11.0 Library Discovery row
  "SSE/fetch streaming: CUSTOM — fetch reader propio con parser SSE pequeño". No EventSource
  polyfill needed; the frontend's fetch-stream parser will be authored in P03-S02-T002.
- **D-CHATSTREAM-EMBED**: query embedding is computed inline in the request via LiteLLM. NO
  caching in this slice. Cache layer is a future optimization slice. KISS.
- **D-CHATSTREAM-CITORDER**: citations emitted **before** the first chunk, so UI can render
  "Sources" pane immediately. Alternative ("citations after generation") rejected — worse UX
  and not aligned with `/chat/:conversationId` Estados UI "streaming" + "success".
- **D-CHATSTREAM-PARTIAL**: on client disconnect or LLM mid-stream error: persist partial
  assistant content with `token_count=NULL`; persist usage with `latency_ms` reflecting
  time elapsed; write a WARN log line. The conversation is NOT in an inconsistent state.
- **D-CHATSTREAM-AUTH**: reuse `app.users.deps.get_current_user` exactly. No new auth.
- **D-CHATSTREAM-RL**: rate-limit DEFERRED to P02-S02-T001 platform layer or future hardening.
  Same pattern as P02-S03-T001 D-RL1. Document explicitly so validator does not push back.
- **D-CHATSTREAM-AUD1**: chat streaming is NOT in §Security audit_log action list (§14 audit
  actions list only `chat_message_streamed` as a **metric**, not an audit action). NO
  `audit_logs` writes from this slice. Symmetric with §P02-S03-T001 D-AUD1.
- **D-CHATSTREAM-MODELTYPE**: `model_type='chat'` exact string (open enum from §10.3; seed
  fixture confirms). Cross-check `data/verification/admin_ai/models/*.json` at developer
  start; if seed uses a different string, raise `NoActiveChatModelError` and document in
  handoff §K — do NOT silently coerce.
- **D-CHATSTREAM-COST**: trust LiteLLM's `response_cost` field from the stream usage chunk
  for `estimated_cost`. Fallback to `ai_models.pricing.input_cost_per_token * tokens_in +
  output_cost_per_token * tokens_out` if LiteLLM does not emit cost (researcher Q3).
- **D-CHATSTREAM-LANG**: stream uses `conversation.language` (already enforced by D-LANG1
  in P02-S03-T001). The user CANNOT override language per-message in this slice.

## I. Security checklist (mini)

- [ ] **Ownership 403/404**: reuse the exact pattern from `get_conversation` in
  `app/chat/routers/conversations.py` — 404 first (not found), then 403 (wrong owner),
  never leak existence to non-owners.
- [ ] **API key never leaks**: `decrypt_secret(...)` result lives only in a local var inside
  the LiteLLM call. Never log, never JSON-dump, never return in any SSE event. Logs use only
  `model_id`, `provider_id` UUIDs.
- [ ] **PII redaction**: never log `body.message`, never log assistant `content`, never log
  chunk `content`. Only lengths and counts. Per §10.5: redact `password|token|secret|api_key|
  encrypted_secret|prompt|document.content`. Add `assistant_content` to the redaction list
  contractually (handoff §K).
- [ ] **request_id propagation**: every log line + final `done` event carries `request_id`.
  Mirror auth/chat pattern (`_get_request_id`).
- [ ] **No audit_log writes** (D-CHATSTREAM-AUD1).
- [ ] **Rate limit deferred** (D-CHATSTREAM-RL) — call out in handoff so the security
  reviewer does not flag missing limit. Inline DoS guard: `body.message` length capped to
  8000 (mirrors `CreateConversationRequest`).
- [ ] **No prompt injection mitigation in V1** beyond the system prompt instruction "NUNCA
  inventes…". Out of scope; defer to RAG hardening slice. Document in handoff §K.
