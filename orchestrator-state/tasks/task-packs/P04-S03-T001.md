## 5. Reglas de negocio y UX

- Roles que ven `/admin/audit`: **only `people_auditor` and `super_admin`**. `people_admin` is intentionally NOT allowed (the backend test asserts 403 for people_admin). This is a deliberate separation-of-duties: admins configure, auditors observe.
- `empleado` users: redirect to `/chat` (RequireRole behavior).
- PII contract on the wire:
  - `actor_user_id` is a UUID, not an email. The page does NOT have a name-resolution endpoint in v1; render the UUID truncated (first 8 chars + ellipsis) per the existing `_short_hash` precedent in backend logs. Add a tooltip showing the full UUID for accessibility (`<abbr title>` or `aria-label`).
  - `entity_type`/`entity_id` are opaque labels (e.g. `user`, conversation UUID) — render as-is.
  - `metadata` is a JSONB blob that may contain `ip`, `user_agent`, `request_id`, action-specific context. **DO NOT render the full metadata blob by default**. v1: render a compact summary (e.g. `request_id` if present; otherwise "—"). A full-detail drawer (the "abrir detalle evento" next action in §3.2) is OUT OF SCOPE for v1 — mark as a future-enhancement candidate, not a follow-up to promote now (no FU bloat).
- Filter validation:
  - `from`/`to` are ISO dates. Default range: **last 7 days** (mirrors UsagePage's "fixed range, no date picker" KISS approach — but for audit we may want from/to inputs since instrucciones §3.2 explicitly says "filtros de fecha válidos"). DECISION: implement read-only default `last 7 days`, BUT expose a single date-range control with two inputs (`<input type="date">` × 2) so the auditor can change the window. Server enforces max 90 days; client mirrors with a soft warning before fetching. See §10 decision D-T001-FILTER-UX.
  - `actor` filter: optional text input that accepts a UUID. If non-UUID is typed → inline `error_validation` (do not fetch).
  - `action` filter: optional dropdown of audit action strings observed in backend code:
    - `auth.sign_up`, `auth.sign_in`, `auth.mfa.verify`, `auth.logout`, `auth.refresh`, `auth.password_reset.requested`, `auth.password_reset.completed`, `users.language.update`, `admin.agent.tools.update`, `admin.agent.run.start`, `admin.agent.run.complete`.
    - More may exist (admin.provider.*, admin.mcp.*, admin.rag.*). v1 plan: render as a free-text input (server does exact-match) with a `<datalist>` containing the known actions. Avoid hardcoding a static select that would silently drop unknown actions. See D-T001-ACTION-FILTER.
- i18n: every visible string must come from `useTranslation("audit")`. Add the `audit` namespace inline in `frontend/src/i18n/index.ts` ES/EN/FR in lockstep + register in `I18N_NAMESPACES` (`languages.ts`). DO NOT add a separate JSON file under `public/locales/{lng}/audit.json` unless the sibling pattern adds it too (sibling T002 did NOT).
- Logging: `logVerbose("audit.page.render.start", { from, to, action_present, actor_present })`. NEVER log full audit rows, NEVER log actor UUIDs (use `_short_hash`-style truncation if needed), NEVER log IPs or metadata blobs.

Status codes the frontend must map:

- `200 OK` → render rows + handle empty.
- `401 AUTH_SESSION_EXPIRED` → `authFetch`'s single-flight refresh should hide this; if it surfaces, render the permission_denied view AND log out the user (the same path UsagePage uses).
- `403 AUTH_PERMISSION_DENIED` → `AuditForbiddenError` → `permission_denied` UX state. Note: with the RequireRole wrapper this branch is essentially unreachable from the page itself; it still must be handled defensively in `auditRepository` for the case where a session loses the role mid-flight.
- `422 AUDIT_WINDOW_INVALID` / `AUDIT_WINDOW_TOO_WIDE` → `AuditValidationError` → `error_validation` UX state with the specific message localized.
- `5xx` → `AuditServerError` → `error_network` UX state with retry CTA.
- Network failure → `AuditNetworkError` → `error_network` UX state with retry CTA.

Same-origin contract: use the relative URL `/api/v1/admin/audit` (vite proxy in dev, Nginx in prod). NEVER hardcode `http://localhost:8000` — sibling repositories already enforce this (ADR-002).

## 10. Key decisions (D-T001-*)

| ID | Decision | Rationale |
|---|---|---|
| D-T001-RBAC-SCOPE | Audit route guarded by `RequireRole(["people_auditor","super_admin"])`, separate from the existing admin RequireRole block. | Backend `require_auditor` accepts only those two roles. Reusing the admin block would either lock auditors out (no `people_auditor` in admin roles) or wrongly let `people_admin` reach /admin/audit (backend will 403 anyway, bad UX). Cleanest: own block, no regression on other admin pages. |
| D-T001-FILTER-UX | Two `<input type="date">` for from/to + default last 7 days + free-text actor UUID + free-text action with `<datalist>` of known actions. Client validates UUID format + range invariant before fetching. | Instrucciones §3.2 explicitly requires filtros de fecha válidos. UsagePage v1 used a fixed range (KISS) but audit has stronger filter need per instrucciones; still keep KISS by avoiding a heavy date-picker library. |
| D-T001-PAGINATION | `Load more` CTA below the table that consumes `next_cursor`; no infinite scroll. | Matches `HairlineTable` editorial aesthetic + accessibility (no scroll-triggered fetches). Server returns `has_more` + `next_cursor`. |
| D-T001-METADATA-RENDER | Render only `request_id` (when present) in the metadata column. Full metadata blob expansion is v2. | PII risk: metadata can contain IPs, user-agents, raw email. Future "abrir detalle" is the next action per §3.2 but not in v1 scope. |
| D-T001-ACTOR-RENDER | Render `actor_user_id` as `xxxxxxxx…` (first 8 hex) with full UUID in `aria-label` / `<abbr title>`. No name resolution endpoint in v1. | No backend endpoint resolves UUID → email/name; do not invent one. PII safe. |
| D-T001-ACTION-FILTER | Free-text input + `<datalist>` of known actions (see §5). Server enforces exact match. | Action set grows as new features add audit rows; a hardcoded `<select>` would silently drop unknown values. |
| D-T001-NEXT-ACTION | "Open event detail" CTA rendered DISABLED (with localized "coming soon" tooltip) below the table. | §3.2 declares this as the next action; v1 ships the surface but not the modal/drawer. Marks the contract for a future slice without bloating this PR. |
| D-T001-RANGE-INVARIANT | Client validates from < to AND `(to - from) <= 90 days` before fetching. Surfaces `ValidationErrorView` for either failure. | Mirrors backend service-layer guards. Avoids a guaranteed 422 round-trip. |
| D-T001-NO-COLOR | Use only monochrome tokens. No status dots needed (audit rows don't have status). | UX_CONTRACT §5: cero badges de color no declarados. |
| D-T001-A11Y | `<table>` + `<caption class="sr-only">` + `<th scope="col">` headers + `aria-busy` on loading + visible focus on filters. | UX_CONTRACT §7 + sibling UsagePage D-T002-TABLE-A11Y. |
