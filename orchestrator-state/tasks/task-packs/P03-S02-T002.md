## D. Required UI states (7) — per-state acceptance

| State | Trigger | What the UI MUST do | Anchor |
|---|---|---|---|
| `loading` | GET `/conversations/{id}` in-flight on initial mount or on resume from `/history`. | `MobileFrame` + skeleton header + `aria-busy="true"`. No streaming UI yet. | §D-T002-LOAD-STATE |
| `empty` | GET returns 200 but `messages == []` and no streaming queued. | Show "no messages yet — type a question" message in i18n; render the composer; no transcript. | §D-T002-EMPTY-STATE |
| `streaming` | A POST stream is in-flight (SSE open OR being read). | Show transcript with the last assistant message growing chunk-by-chunk, `aria-live="polite"`, `aria-busy="true"` on the streaming region, composer disabled, "stop" or "cancel" not required in V1 (Researcher Q5 to confirm StrictMode behavior). | §D-T002-STREAM-STATE |
| `error_network` | fetch failure (TypeError), 5xx, OR mid-stream connection drop. | Inline error banner with a retry CTA. Retry reopens the POST stream with the SAME message (see §D-T002-RETRY-SAME-PROMPT). Partial assistant text from the failed attempt is discarded. | §D-T002-NETWORK-STATE |
| `error_validation` | POST stream returns 400 `CHAT_STREAM_BAD_REQUEST` (empty / >8000 chars / malformed). | Inline error below the composer with the field-level message; do NOT clear the composer; do NOT retry automatically. | §D-T002-VALIDATION-STATE |
| `permission_denied` | GET or POST returns 403, OR GET returns 404 against an existing-id-but-foreign conversation. | Full-screen `ForbiddenView` ("you do not have access — return to chat"). The 404 case routes to `empty-not-found` (see §D-T002-403-VS-404). | §D-T002-FORBIDDEN-STATE |

The 7th state `success` is reached after `done` from streaming OR after the
initial GET when there are messages. Per §D-T002-DEEP-LINK the page must
distinguish two arrival modes: (1) navigation from ChatHomePage with a freshly

### 404 vs 403 (important)

- `404 CHAT_CONVERSATION_NOT_FOUND` → render an empty-state with text "conversation
  not found / start a new one" and a `SolidCTA` back to `/chat`. This is NOT
  `permission_denied`; users should not see "forbidden" when the row simply does

- `403 CHAT_CONVERSATION_FORBIDDEN` → `permission_denied` view.
- `401` after `authFetch`'s single-flight refresh is handled by RequireAuth /

        {
          "id": "uuid",

          "role": "user|assistant|system",
          "content": "string",
          "token_count": "int|null",
          "created_at": "ISO-8601"

      ],
      "citations": [
        {
          "id": "uuid",
          "message_id": "uuid",
          "document_id": "uuid|null",
          "chunk_id": "uuid|null",
          "label": "string",
          "score": "number 0.0-1.0"

  }

- Errors: 401 (handled upstream by `authFetch`), 403, 404. Error body envelope:
  `{ data: null, meta: { request_id }, errors: [{ code, message, field?, details? }] }`.

### E.2 POST `/api/v1/chat/conversations/{id}/stream` (write + stream)

- Request: `Content-Type: application/json`, body `{ "message": "<1..8000 chars>" }`
  (Pydantic schema `StreamRequest` in `app/chat/streaming/schemas.py:28`).
- Headers received by client: `Content-Type: text/event-stream; charset=utf-8`,
  `X-Request-ID: <uuid>`, `Cache-Control: no-cache`, `Connection: keep-alive`
  (verified in `router.py:191-198`).
- Errors **before stream opens** (JSON envelope, normal HTTP status):
  400 `CHAT_STREAM_BAD_REQUEST`, 401 `AUTH_SESSION_EXPIRED` (handled by
  `authFetch`), 403 `CHAT_CONVERSATION_FORBIDDEN`, 404
  `CHAT_CONVERSATION_NOT_FOUND`, 502 `AI_PROVIDER_NOT_CONFIGURED`.
- Errors **after stream opens** arrive as in-stream `event: error` frames; the
  parser must surface them as `CHAT_STREAM_ERROR` to the caller.

#### SSE wire format (W3C `text/event-stream`)

Each event is exactly two lines + blank line separator
(verified in `app/chat/streaming/sse.py:40-58`):

```
event: <name>\n
data: <json-payload>\n
\n
```

Backend emits **6 event names** (router docstring line 70, sse.py functions):

| Event | Payload shape | When | Frontend reaction |
|---|---|---|---|
| `meta` | `{ message_id, model_id, language, request_id }` | once, before any chunks | start a NEW assistant message with id=`message_id`; record language for the page. |
| `citation` | `{ document_id, chunk_id, label, score }` | 0..N times, interleaved before/around `chunk` events | append to the in-progress assistant message's citations list. **Citations are streamed as SEPARATE events, NOT inside `done` payload.** |
| `chunk` | `{ delta }` | many times | concat `delta` to the assistant message's `content`. |
| `usage` | `{ tokens_in, tokens_out, estimated_cost, latency_ms }` | once near the end | optional — UI does not currently display usage in V1 but parser MUST accept the event without crashing. |
| `error` | `{ code, message }` | 0..1 times, mid-stream fatal | abort the stream; transition to `error_network`; allow retry. |
| `done` | `{ message_id, request_id }` | once, terminal | close the reader cleanly; transition to `success`. **Citations already arrived as `citation` events — do NOT expect them in `done`.** |

> Backend code paths confirmed: `sse_meta`, `sse_citation`, `sse_chunk`, `sse_usage`,
> `sse_error`, `sse_done` in `app/chat/streaming/sse.py:61-171`. UTF-8 encoding,
> `ensure_ascii=False` so ES/FR characters are preserved as multibyte UTF-8
> (D-SSE2). Researcher Q1 must verify the runtime contract by reading the
> service caller `app/chat/streaming/service.py` to confirm the citation/chunk
> interleaving order in V1.

## G. Key decisions (binding)

### §D-T002-NO-EVENTSOURCE — use fetch + ReadableStream + TextDecoder, NOT EventSource

Canon from TECHNICAL_GUIDE §5.3 line 213: `EventSource` cannot send the
`Authorization` header. We use `authFetch` + `response.body!.getReader()` +
`TextDecoder({stream:true})`. This also lets us send `Content-Type: application/json`
+ a JSON request body, which `EventSource` cannot do.

### §D-T002-SSE-PARSE — incremental SSE parser

The parser consumes a `ReadableStreamDefaultReader<Uint8Array>` and yields typed
events. Algorithm:

1. Maintain a `buffer: string` carried across reads.
2. On each `read()` → decode bytes with `TextDecoder('utf-8', { fatal: false }).decode(value, { stream: true })`.
3. Append decoded chunk to `buffer`.
4. Split `buffer` on the SSE record separator `\n\n`. Keep the **last** fragment
   (incomplete record) in `buffer`; emit the rest.
5. For each complete record, find the `event: <name>\n` line and the
   `data: <json>\n` line. Parse JSON. Yield `{ kind, payload }` where `kind` is
   one of `meta|citation|chunk|usage|error|done` and `payload` is the typed object.
6. On `done` event, terminate normally. On `error` event, throw a
   `ChatStreamError`.
7. On reader done with leftover non-empty buffer that does not parse, log a
   warning but do not crash.
8. Always call `reader.releaseLock()` in a `finally` block, even on abort.

### §D-T002-RETRY-SAME-PROMPT — retry semantics on network failure

When the streaming POST fails (TypeError, 5xx pre-stream, or mid-stream
disconnection), the retry CTA **REOPENS the POST stream with the SAME body
the user sent**. It does NOT recreate the conversation, does NOT duplicate the
user message in the UI, and discards partial assistant text from the failed
attempt.

Backend caveat: `app/chat/streaming/service.py` (`run_stream`) persists the user
message **at the start of the request** and the assistant turn **at `done`**.
Therefore the same `POST .../stream` with the same `{message}` may result in
TWO persisted user-message rows if the first attempt crashed AFTER the user
message was committed but BEFORE `done`. **This is out of scope for the
frontend** — the UI shows one user message and the new assistant attempt;
deduplication on the server side is for a future hardening slice. Researcher
Q1 may confirm whether `service.py` already deduplicates or rolls back; if it
does not and the duplication is visible on the UI after refresh, we open a
follow-up (out-of-scope, no debugger fix here).

### §D-T002-ABORT-ON-UNMOUNT

`useChatStream` owns an `AbortController`. On unmount, or when the
`conversationId` route param changes, the hook calls `controller.abort()`,
which aborts the in-flight `fetch` (`authFetch` passes `signal` through). This
prevents the React "setState on unmounted component" warning and stops wasting
backend resources on closed tabs. Researcher Q5 will confirm StrictMode behavior
(double-invoke + abort in dev).

### §D-T002-CITATIONS-INLINE

Citations render inline as `<CitationInline label="[N]" onClick={openSource} />`
inside the assistant message paragraph. The transcript layout is **NOT
bubbles**: each turn is a vertical block of `TrackedLabel` + paragraph.

- User turn: `<TrackedLabel>You</TrackedLabel>` (sans, uppercase, tracked) +
  `<p style={{ fontFamily: 'var(--font-sans)' }}>{content}</p>`.
- Assistant turn: `<TrackedLabel>Assistant</TrackedLabel>` (sans, uppercase,
  tracked) + `<p style={{ fontFamily: 'var(--font-display)' }}>{content with
  inline citation chips}</p>`.

No avatars, no rounded containers, no colored backgrounds. The Hilo wordmark
already lives on the `MobileFrame` header (P00-S01-T004).

### §D-T002-DEEP-LINK

The page distinguishes 2 arrival modes:

1. **From ChatHomePage** (post-`POST /conversations` create): ChatHomePage calls
   `navigate('/chat/' + conversationId)`. There MUST be a pending first prompt.
   Today (P03-S02-T001) ChatHomePage does **not** pass anything in
   `location.state`. Therefore ConversationPage cannot rely on state for the
   first prompt; it must trigger streaming based on the conversation's last
   message. **Decision**: if GET returns a conversation whose last message is
   `role: 'user'` and there is no `assistant` reply yet, ConversationPage
   **automatically opens the stream** with that user's `content` as the
   `{message}` body, treating it as the first turn. This matches J101's narrative
   ("recibe streaming + citas" immediately after creating the conversation) and
   does not require a router change in T001's pack. If the last message is
   `assistant`, we are in "resume" mode and do not auto-stream.
2. **Deep link / resume from `/history`** (or browser refresh): GET returns the
   persisted transcript; no auto-stream; user types a new question, which calls
   POST stream with that new message.

### §D-T002-LANGUAGE-FROM-USER

The stream body `{message}` only — no `language` field (Pydantic schema in
`app/chat/streaming/schemas.py:28` only accepts `message`). The conversation's
language is fixed at creation time (T001 already passes it via
`POST /conversations`). Do NOT reintroduce a language picker here.

### §D-T002-403-VS-404

- HTTP 403 → `permission_denied` full-screen `ForbiddenView`.
- HTTP 404 → empty-state "conversation not found" with CTA back to `/chat`.
- HTTP 401 → handled by `authFetch` single-flight; on final 401, `onAuthFailure`
  triggers `logout()` and `RequireAuth` redirects to sign-in (already wired
  via `useAuth().logout` — reuse ChatHomePage's pattern).

### §D-T002-TANSTACK-CACHE

After `done` event, `useChatStream` writes the persisted assistant message +
citations into `queryClient.setQueryData(['conversation', id], …)` so that:

- A second mount (e.g. back-nav + re-enter) does NOT refetch.
- The user sees the same transcript as a fresh GET would yield.

The shape of cached data must match the `ConversationDetail` shape. Researcher
Q3 will confirm `setQueryData` vs `invalidateQueries` recommendation for
TanStack v5.

### §D-T002-NO-PII-IN-LOGS

Logger rules from `features/chat/data/logger.ts` apply: never log message
content, prompt text, token values, user email/full_name. Log lengths, ids, codes,
status, request_id only.

### §D-T002-ACCESSIBILITY

- Streaming region wrapper: `aria-live="polite"` and `aria-busy="true"` while
  streaming.
- Loading region: `aria-busy="true"`.
- Citation chips: `<CitationInline>` already exposes `aria-label`. Pass
  `"Source: <label>"`.
- Composer: re-used; already accessible.
- Tap targets ≥ 44 px (provided by `SolidCTA` and `EditorialInput`).
- Reduced motion: no enter/exit animations on chunks; only opacity is allowed.
