## 3. Reality check (verificado pre-spawn por main-orchestrator)

This is the cornerstone of Option B. The pack records the ground truth so

### 3.1 The original FU premise is FALSE in HEAD

The FU description says `_ping_db(engine).connect()` is called sync on an
**asyncpg AsyncEngine**, raising `MissingGreenlet`. That does not match the
code on `dev/P02-S03-T005`:

- `backend/app/api/router.py:121-138` — `_ping_db(engine: Engine)` is typed for
  `sqlalchemy.engine.Engine` (sync) and uses `with engine.connect() as conn` +
  `conn.execute(text("SELECT 1"))`.
- `backend/app/api/router.py:66-90` — `get_db_engine()` calls
  `create_engine(db_url, pool_pre_ping=True, pool_size=2)` with the **sync**
  psycopg dialect (`postgresql+psycopg://`). It is a separate sync engine
  singleton, **not** the async session engine.
- `backend/app/db/session.py:69-75` — the application session engine used by
  business endpoints is also **sync** (`create_engine` + `sessionmaker`),
  explicitly coercing `postgresql+asyncpg://` → `postgresql+psycopg://`
  (lines 61-64). No `AsyncEngine` exists anywhere in the codebase.
- Live probe (10/10 sample): `curl localhost:8000/ready` → 200 OK with
  `{"data":{"db":{"status":"ok"},"redis":{"status":"ok"},"litellm":{"status":"unknown"}}}`.
  No MissingGreenlet in back.log.

**Implication**: Acceptance (1) (`GET /ready returns 200 under happy path`) is
already satisfied by the existing code. Acceptance (2) and (3) are partially
satisfied — but the existing tests use `MagicMock` dependency overrides; they
do not hit real Postgres, so they do not catch the "future regression to
AsyncEngine" scenario the original FU was worried about.

### 3.2 The original write_set has two drifts vs. real layout

- `backend/app/db/engine.py` — **does not exist**. There is no
  `db/engine.py`. The sync engine for the app lives in `backend/app/db/session.py`;
  the sync engine for `/ready` lives in `backend/app/api/router.py::get_db_engine`.
  The FU author assumed an `engine.py` module that the project never created.
- `backend/tests/integration/test_health.py` — **does not exist**. Existing
  health tests live in `backend/tests/test_health.py` (top of `backend/tests/`,
  not under `integration/`). The existing tests use `MagicMock` dependency
  overrides — fast/isolated, but not a real integration test.

These are write_set drifts the developer must declare in the handoff (see §6).

### 3.3 What is ACTUALLY missing for the FU to be closed correctly

1. **Engine type sentinel**: no test asserts that `get_db_engine()` returns a
   sync `sqlalchemy.engine.Engine` (and **not** an `AsyncEngine`). If a future
   contributor swaps the engine factory to `create_async_engine`, no test will
   fail. The FU explicitly wants this regression signal.
2. **Real Postgres happy-path test**: existing tests inject `MagicMock` engines.
   The acceptance literally says "with real Postgres up" — that needs a test
   that does NOT override `get_db_engine`, so the real
   `create_engine(postgresql+psycopg://...)` runs against the live `:5432`.
3. **Real Postgres down test (optional but valuable)**: a test that points
   `get_db_engine()` at an unreachable DSN (e.g. port 1) and asserts
   `503 + db.status == "error"` against the real psycopg driver
   (`sa_exc.OperationalError`). This proves the 503 branch under real driver
   semantics, not just MagicMock.
4. **Verbose logging coverage under real engine**: the existing logging tests
   use `caplog.at_level`; they prove the logger emits start/ok markers but
   they do so against MagicMock. Add coverage that exercises the same log
   markers under the real engine path (or explicitly justify reuse of existing
   coverage when the real engine emits the same log calls).

## 4. Scope acordado con humano (Opción B — SENTINEL)

- **NO modificar product code** in `backend/app/api/router.py`, `backend/app/db/session.py`,
  or anywhere else in `backend/app/`. The /ready endpoint already satisfies
  acceptance (1).
- **CREATE** `backend/tests/integration/test_health_ready.py` with 4 test
  classes / test functions that close the gap above:
  - **TEST-1 — real-Postgres happy path**: with `DATABASE_URL` pointed at the
    live test Postgres (the integration test runner already requires it), call
    `TestClient(app).get("/ready")` WITHOUT overriding `get_db_engine` or
    `get_redis_client`. Assert `status_code == 200`, body envelope matches
    `{"data":{"db":{"status":"ok"},"redis":{"status":"ok"},"litellm":{"status":"unknown"}}}`,
    and the back log contains `health.ready.start` + `health.ready.db.ping.ok`
    (only when `ENABLE_VERBOSE_LOGGING=true`).
  - **TEST-2 — real-driver DB-down → 503**: override `get_db_engine` to a real
    `create_engine("postgresql+psycopg://invalid:invalid@127.0.0.1:1/none", pool_pre_ping=False)`
    so that `engine.connect()` raises a real `sa_exc.OperationalError` (not a
    MagicMock side_effect). Leave `get_redis_client` un-overridden. Assert
    `status_code == 503`, `body["data"]["db"]["status"] == "error"`,
    `body["data"]["redis"]["status"] == "ok"`. This proves the 503 branch
    under real psycopg semantics and exercises the `except sa_exc.OperationalError`
    catch in `_ping_db` against a real exception class.
  - **TEST-3 — SENTINEL: engine type contract**: call `get_db_engine()` and
    assert `isinstance(engine, sqlalchemy.engine.Engine)` **and**
    `not isinstance(engine, sqlalchemy.ext.asyncio.AsyncEngine)`. This is the
    explicit regression signal for the original FU premise. If anyone swaps
    `create_engine` for `create_async_engine` in the future, this single
    assertion fails the suite.
  - **TEST-4 — verbose logging under real engine**: parametrize the happy-path
    test with `ENABLE_VERBOSE_LOGGING` set via `monkeypatch.setenv` to
    `"true"` and `"false"`. Under `"true"`, assert log records include
    `health.ready.start`, `health.ready.db.ping.start`, `health.ready.db.ping.ok`,
    `health.ready.redis.ping.start`, `health.ready.redis.ping.ok`,
    `health.ready.ok`. Under `"false"`, assert no INFO records appear (only
    WARNING/ERROR — same contract as existing `test_logging_quiet_mode_*` but
    over the real engine path).
  - **Caveat for TEST-4**: the existing `router.py` reads
    `_VERBOSE = os.getenv("ENABLE_VERBOSE_LOGGING", "false").lower() == "true"`
    at **module import time** (lines 45-48). `monkeypatch.setenv` after import
    does not retroactively change `_LOG_LEVEL`. Two acceptable approaches:
    - (A) Test directly with `caplog.at_level(logging.INFO, logger="app.api.router")`
      to observe verbose behavior (mirrors existing tests #9/#10 in
      `backend/tests/test_health.py`), and document the "module-level _VERBOSE
      is read once" behavior as an observed contract.
    - (B) Reload the router module after `setenv` using `importlib.reload`
      (more thorough but heavier and may break the app `dependency_overrides`
      state). Developer chooses A unless A leaves the verbose contract
      unverified at the integration level; if so, B with proper teardown.
    Developer must pick one approach and document it in §K of the handoff
    (decision anchor). Recommendation: **A** for KISS — the existing unit
    tests already prove the module-level guard works; the new integration
    test just needs to observe the real log markers emitted at runtime.
- **DO NOT** create `backend/app/db/engine.py`. It is a phantom path from the
  FU triage — confirmed not to exist and not needed.
