## §8 — Architecture, logging, i18n runtime contract

### i18n runtime shape (CRITICAL — see `frontend/src/i18n/index.ts` lines 48-52, 59-376)

`i18n/index.ts` ships an inline `resources` object that is what i18next actually reads at runtime. The JSON files under `frontend/public/locales/{lng}/{ns}.json` are reference/co-documentation only — they are NOT fetched at runtime (no HTTP backend plugin, no lazy load — see lines 50-51 comment).

**Therefore the developer MUST add the new key in BOTH places** so they remain in lockstep (lines 56-58 contract: "Sync any key changes here WITH the corresponding JSON file"). Failing to update `i18n/index.ts` will compile fine but the runtime alert will fall back to the i18next key string (`chat:composer.errors.tooLong`) — visible immediately in /verify-slice and a critical UX bug.

Namespace structure today (es):
```json
"chat": {
  "empty": {...},
  "citation": {...},
  "composer": { "placeholder": "...", "send": "..." }
}
```

After this slice:
```json
"chat": {
  "empty": {...},
  "citation": {...},
  "composer": {
    "placeholder": "...",
    "send": "...",
    "errors": { "tooLong": "El mensaje no puede superar {{max}} caracteres." }
  }
}
```

i18next interpolation `{{var}}` is enabled with `escapeValue: false` (React already escapes — see `i18n/index.ts:410`). Use `t("chat:composer.errors.tooLong", { max: COMPOSER_MAX_LENGTH })` to pass the value. Do NOT inline the number in the translation — it's a parameter so the key stays correct if `COMPOSER_MAX_LENGTH` ever changes.

Existing `Composer.tsx` already has BEFORE/AFTER/WARN logs (`chat.Composer.submit.start`, `chat.Composer.submit.ok`, `chat.Composer.submit.validation.too_long`, `chat.Composer.submit.validation.empty`). ## §9 — Decisions (anchors for handoff / report)

### D-T005-I18N-KEY (CANONICAL KEY NAME)

**Decision**: Use `chat.composer.errors.tooLong` (nested under the existing `chat.composer` substructure).

**Alternatives rejected**:

1. `chat.errors.tooLong` (Coverage Registry example): rejected. The `errors` namespace at the **chat** sub-level (`chat.errors.*`) does not exist today — there is only the TOP-LEVEL `errors` namespace which holds backend error code constants (`AUTH_INVALID_CREDENTIALS`, `NETWORK`, `CHAT_STREAM_FAILED`, etc., all UPPERCASE — see `i18n/index.ts:149-163`). Adding `chat.errors.tooLong` would introduce a second `errors`-named subtree with different semantics (UI validation copy vs backend error mapping), making the namespace ambiguous.
2. `errors.COMPOSER_TOO_LONG` (top-level errors namespace): rejected. That namespace is reserved for backend error codes mapped to user-facing messages (per TECHNICAL_GUIDE §6.4 error format + the existing keys' UPPERCASE convention). Composer over-limit is a CLIENT-side validation that never reaches the backend — putting it in `errors:` would conflate client validation with server error mapping.
3. `chat.composer.tooLong` (flat, no `errors` nesting): rejected. Future composer validations (e.g., URL-only, empty after trim, blocked characters) will likely need their own keys; introducing the `errors` sub-namespace now under `composer` makes the structure extensible without renames.

**Consequence**: New nested key `chat.composer.errors.tooLong` with interpolation `{{max}}`. The Coverage Registry's `chat.errors.tooLong` example is non-binding (the row uses "e.g." prefix); the work-item acceptance only requires "a dedicated i18n key" with no specific name.

### D-T005-I18N-COPY (EXACT STRINGS)

Following the brand voice of existing chat copy (see `chat.empty.subtitle` ES: "Escribe tu pregunta para empezar" — direct, no exclamation, sentence case, ends with period only when complete sentence). The FU description suggested `'El mensaje no puede superar 4000 caracteres'`; we keep that ES wording but interpolate the max via `{{max}}`.

- **ES**: `El mensaje no puede superar {{max}} caracteres.`
- **EN**: `Messages cannot exceed {{max}} characters.`
- **FR**: `Le message ne peut pas dépasser {{max}} caractères.`

All three are short, present-tense, declarative, and follow the same a11y SR-friendly pattern. They do not include emoji, exclamation, "please/por favor/s'il vous plaît", or hardcoded numbers — the interpolated `{{max}}` keeps the copy synchronized with `COMPOSER_MAX_LENGTH`.

This is NOT a project-level ADR (does not meet TECHNICAL_GUIDE §15 threshold of "non-obvious decision with alternatives considered"); it lives only as a slice-level decision in this pack + handoff. If a project-wide i18n key-naming convention is needed later, that would be a separate FU.

### D-T005-PLACEHOLDER-UNCHANGED

Decision: `chat.composer.placeholder` ("Escribe tu mensaje..." / "Type your message..." / "Rédigez votre message...") REMAINS unchanged. The slice is only about decoupling the over-limit message from the placeholder; the placeholder itself is still correct UX for the empty composer state.

### D-T005-EMPTY-VALIDATION-UNCHANGED

Decision: The empty-input validation path (`Composer.tsx:155-159`, `setValidationError(t("common:states.empty"))`) is OUT OF SCOPE. It reuses `common:states.empty` ("Sin resultados") which is also a copy mismatch (the empty composer is not "no results found"), but the FU and work-item explicitly scope this slice to the over-limit path only. If the empty-validation copy is also bad UX, it is a separate FU candidate — do NOT fix it in this slice (would be scope creep, write_set drift, and the developer should NOT propose a FU for it inline; let validator/tester surface it post-slice if real).
