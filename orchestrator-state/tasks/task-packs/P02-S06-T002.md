## D. Reglas de negocio y UX (instrucciones.md + UX_CONTRACT.md)

- People Tech Admin gobierna qué colecciones RAG están activas (instrucciones.md §1.3 personas, §3.1#rag-admin).
- Only `people_admin` (or `super_admin` superset) can list and edit collections. `employee` → 403. `people_auditor` → 403 (auditor reads `audit_logs` view, not RAG admin write surface).
- Edits go to `audit_logs` (J104 evidence): action `admin.rag.collection.update`.
- Vertical / language are arbitrary strings at the schema level; TECHNICAL_GUIDE §6.2 only specifies that `{name?, vertical?, enabled?}` are editable. The schema also has a `language` column. **Decision (planner)**: include `language` in the PATCH body too. Rationale: (1) the column exists in the schema; (2) `language` will be needed in P04-S02-T002 (Collections page edit form already documents "edits vertical/language/enabled"); (3) the implementation checklist registry row for T002 says "Admin lists collections and toggles vertical/language settings" — toggling **language** is explicit acceptance. Risk: if reviewer reads §6.2 strictly without §6.1 context, this is a perceived deviation. **Mitigation**: declare `D-RAGCOLL-LANG-IN-PATCH` in handoff and reference both Coverage Registry acceptance row + UX_CONTRACT row.
- Cross-document consistency: UX_CONTRACT §3.2 RagCollectionsPage row also references "edits vertical/language/enabled" semantics. PATCH must allow updating any subset; require at least one editable field present (400 RAG_INVALID_PAYLOAD).
- `metadata` JSONB and `id` are NOT editable via PATCH (no contract authorizes it; admin uses `name` / `vertical` / `language` / `enabled`).
- Empty result is a valid 200 with `data: []`. There is no "create collection" endpoint in V1; admins seed collections via verification data bootstrap (data/verification/rag_chat/collections/*.json + `load_rag_collections()` loader). Document this explicitly in handoff.

### H.2 — PATCH /api/v1/admin/rag/collections/{id}

- Auth: `Depends(require_admin)`.
- Path param: `id: uuid.UUID` (FastAPI validates UUID format → 422 on malformed).
- Body (Pydantic v2 model `CollectionPatchIn`):
  ```python
  class CollectionPatchIn(BaseModel):
      name:     str  | None = Field(None, min_length=1, max_length=255)
      vertical: str  | None = Field(None, min_length=1, max_length=64)
      language: str  | None = Field(None, pattern=r"^(es|en|fr)$")
      enabled:  bool | None = None

      @model_validator(mode="after")
      def at_least_one_field(self):
          if self.name is None and self.vertical is None \
             and self.language is None and self.enabled is None:
              raise ValueError("at_least_one_field_required")
          return self
  ```
- Validation rules:
  - At least one of {name, vertical, language, enabled} must be present → else 400 `RAG_INVALID_PAYLOAD` field=`body`.
  - `name` trimmed; empty after trim → 400 `RAG_INVALID_PAYLOAD` field=`name`.
  - `vertical` similar.
  - `language` (if provided) must be one of `es|en|fr` (aligned with `documents.language` CHECK + employees' `preferred_language` D-IDX).
  - `enabled` is plain bool.
  - `metadata` and `id` are NOT editable.
- Response 200:
  ```json
  {
    "data": {
      "id": "...", "name": "...", "vertical": "...",
      "language": "...", "enabled": true, "metadata": {...}
    },
    "meta": {"request_id": "uuid"}
  }
  ```
- Errors:
  - 400 `RAG_INVALID_PAYLOAD` (empty body / empty trimmed name / invalid language).
  - 401 (no token).
  - 403 (employee / auditor).
  - 404 `RAG_COLLECTION_NOT_FOUND` (id not in `rag_collections`).
  - 422 (path UUID malformed — FastAPI default).
- Audit:
  - `write_admin_ai_audit(action="admin.rag.collection.update", entity_type="rag_collection", entity_id=<row.id>, metadata={request_id, ip, user_agent, changed_fields: ["enabled","language"], old: {...subset...}, new: {...subset...}})`.
  - audit runs AFTER successful DB commit, via D-S2 independent session (helper handles this internally).
  - On any 4xx (validation/not-found/forbidden) the slice writes **no** audit row (consistent with admin/providers and admin/model_catalog precedent — only successful state changes are audited; auth failures already logged elsewhere).
