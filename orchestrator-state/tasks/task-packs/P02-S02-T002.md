## C. Root cause confirmado (precis del orquestador)

**File**: `backend/app/auth/tokens.py`

```python
# Lines 57–60 — evaluated at IMPORT TIME, ONCE per Python process
_JWT_KEY: str = os.getenv("JWT_PRIVATE_KEY", "")
_JWT_ALGORITHM: str = os.getenv("JWT_ALGORITHM", "HS256")
_ACCESS_TTL: int = int(os.getenv("AUTH_ACCESS_TTL_SECONDS", "1800"))
_MFA_CHALLENGE_TTL: int = int(os.getenv("AUTH_MFA_CHALLENGE_TTL_SECONDS", "300"))

# Lines 62–73 — at-import warning that ALSO sees the bug
if not _JWT_KEY:
    logger.warning("tokens.jwt_key.missing JWT_PRIVATE_KEY env var is not set; ...")
elif len(_JWT_KEY.encode()) < 32:
    logger.warning("tokens.jwt_key.too_short JWT_PRIVATE_KEY is %d bytes; ...")
```

**Failure mechanism**:
1. pytest collects modules in alphabetical/order-of-discovery. If ANY test file imports a module that transitively imports `app.auth.tokens` BEFORE `JWT_PRIVATE_KEY` is in `os.environ` (e.g., the env-loading fixture has not run yet for THAT test session), `_JWT_KEY` is captured as `""` and **stays `""` for the entire pytest session**.
2. `decode_token` (line 196: `jwt.decode(token, _JWT_KEY, ...)`) and `encode_*` (lines 124, 163) then operate with an empty secret. `jwt.encode` with empty key produces a token signed with `""`; `jwt.decode` with empty key fails signature verification on any token signed with the real key — and vice-versa. The 6 auth test suites that share this module then produce mixed pass/fail depending on which collected first.
3. `test_users_me.py::TestGetMeSecurityShape` is the canonical reproducer because it mints its own token via `_mint_access_token` (line 196 of that file) using `_JWT_KEY` captured from env, then sends it to `GET /api/v1/users/me`, which decodes via `app.auth.tokens.decode_token`. If the two `_JWT_KEY`s drift (one from env at test-collect-time, one from env at app-import-time), decode fails with `InvalidSignatureError`.

**Important secondary detail**: `backend/tests/integration/test_users_me.py` **also** captures `_JWT_KEY` at import time (line 88), but because it imports `app.auth.tokens` (line 79) BEFORE evaluating its own line 88, the order is: tokens.py reads `os.getenv` (possibly empty) → test_users_me.py reads `os.getenv` (possibly different). Both must be lazy for the cross-process consistency to hold. **Both** files are in the write_set for exactly this reason.

### F.2 Decision: which constants to convert to lazy?

KISS + YAGNI. The only constant that breaks in full-suite ordering is `_JWT_KEY` because env-pinning happens via fixtures/profile, not via Python import. The others are environmental "knobs":

| Constant | Lazy? | Rationale |
|---|---|---|
| `_JWT_KEY` | **YES — lazy required** | Root cause. Fixtures may set env AFTER import on collection paths. |
| `_JWT_ALGORITHM` | **NO — keep at import** | No fixture in this repo monkeypatches `JWT_ALGORITHM`. Default `HS256` is hardwired by §10.2. Touching it expands surface and risks regressions. |
| `_ACCESS_TTL` | **NO — keep at import** | Same logic. No fixture rotates `AUTH_ACCESS_TTL_SECONDS` at runtime in any test. |
| `_MFA_CHALLENGE_TTL` | **NO — keep at import** | Same. |

If a future slice ever needs to rotate any of these at runtime, that slice will convert them. **Conservative wins.** Justified explicitly to forestall validator F-arch round-trip (P-19 pattern: "Co-existence over migration"; "KISS + Write_set + regression-safety triangle").

### F.3 Decision: `@lru_cache` semantics

The known anti-pattern is `@lru_cache` over a getter that returns `""` (the empty placeholder) — it would **pin** `""` for the rest of the process even after env is fixed. Two options:

- **Option A (recommended) — `@lru_cache(maxsize=1)` + raise on empty**
  - Mirrors `app/security/encryption.py:_get_fernet()` exactly.
  - The getter raises `RuntimeError` (or a typed `MissingJwtKeyError`) when env is empty.
  - Cache only caches the **successful** case (because raises never reach the cache).
  - **PyJWT semantics check**: `jwt.encode(payload, "", algorithm="HS256")` and `jwt.decode(token, "", algorithms=["HS256"])` both technically succeed at the API level (PyJWT does NOT validate key length); but the signed/verified bytes are not what production wants. **Raising early at the getter is the safer contract** and lets the existing warnings turn into a hard exception for prod.
  - Symmetric tradeoff: tests that fail to set `JWT_PRIVATE_KEY` will now raise on first call instead of silently producing a "" signature. Acceptable — `conftest`/profile must set the key (already does via `gen-dev-secrets.sh` + `dev-restart.profile.sh`).

- **Option B — no cache, getter inlined every call**
  - One `os.getenv` per encode/decode call. Cost is negligible (<1µs).
  - No risk of caching `""`; no need for a clear function.
  - **Disadvantage**: visually drifts from `_get_fernet()` pattern. Validator may push back.

**Recommendation: Option A.** Mirror `app/security/encryption.py`. Provide a small `_clear_jwt_key_cache()` test-only helper for monkeypatch scenarios (existing test `test_security.py:_sync_tokens_module_with_env` will be replaced by a direct call to `_clear_jwt_key_cache()` followed by env mutation; **but T001 test file is NOT in this slice's write_set — DO NOT touch it**). Validator may approve either A or B; document the trade-off in handoff §K so the choice is explicit.

### F.4 Decision: at-import warnings (lines 62–73 of current `tokens.py`)

The warnings serve TWO audiences:
1. **Prod startup**: detect mis-configured prod containers BEFORE first user request. Loud-at-startup is the right posture.
2. **Dev startup**: same.

If we make the key lazy, the warnings move to the **first call** of the getter. That changes the timing slightly (no longer on `import app.auth.tokens` but on first `encode_access_token` / `decode_token`).

**Recommendation**: Keep the at-import warnings as-is in `tokens.py`, reading `os.getenv("JWT_PRIVATE_KEY", "")` LOCALLY in the import-time branch. They serve only as a startup-time observability signal; they do NOT participate in the encode/decode flow. The encode/decode flow uses the lazy getter, which is independent. **No behavior regression**: prod still gets the startup warning; tests no longer suffer the import-order bug.

```python
# Sketch (developer to implement and refine)
@lru_cache(maxsize=1)
def _get_jwt_key() -> str:
    """Read JWT_PRIVATE_KEY lazily; raises if missing.

    Lazy so that pytest fixtures / dev tooling that set env AFTER first import
    still work. Cached for stable per-process value.
    """
    key = os.getenv("JWT_PRIVATE_KEY", "")
    if not key:
        raise RuntimeError("JWT_PRIVATE_KEY env var is not set")
    return key

def _clear_jwt_key_cache() -> None:
    """Test-only: clear the lazy JWT key cache. Used after env rotation in tests."""
    _get_jwt_key.cache_clear()

# At-import advisory (UNCHANGED — keeps prod ops observability)
_startup_key = os.getenv("JWT_PRIVATE_KEY", "")
if not _startup_key:
    logger.warning("tokens.jwt_key.missing JWT_PRIVATE_KEY env var is not set; ...")
elif len(_startup_key.encode()) < 32:
    logger.warning("tokens.jwt_key.too_short ...", len(_startup_key.encode()))
del _startup_key  # do not keep a closure to the read-once value
```

In `encode_access_token` / `encode_mfa_challenge_token` / `decode_token`, replace every `_JWT_KEY` use with `_get_jwt_key()`. Leave `_JWT_ALGORITHM`, `_ACCESS_TTL`, `_MFA_CHALLENGE_TTL` unchanged.

### F.5 Decision: `test_users_me.py` symmetric fix

`test_users_me.py` has the SAME anti-pattern at line 88 (`_JWT_KEY: str = os.getenv(...)`). The fix is symmetric:

```python
# Replace lines 88–89 with:
def _get_jwt_key() -> str:
    """Lazy JWT key getter, mirrors app.auth.tokens._get_jwt_key()."""
    return os.getenv("JWT_PRIVATE_KEY", "")
```

**Important**: Do NOT add `assert key, "JWT_PRIVATE_KEY must be non-empty"` here — `test_security.py:106` did, but here we want graceful skip-or-fail semantics. If env is empty in CI for some reason, let the production `_get_jwt_key()` raise; the test will fail with a clear message rather than an assertion in a helper.
