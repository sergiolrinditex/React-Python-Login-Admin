## Resolved decisions (R1–R6)

### R1 — RATE-LIMIT LIB

**Decision: custom Redis sliding-window using `redis==7.4.0` (already pinned).**

Rationale:

- `redis-py==7.4.0` is already a runtime dep and the valkey container is UP. No new pin, no library audit, no dep budget burn.
- Sliding-window over Redis using `INCR` + `EXPIRE` on a key keyed by `<prefix>:<ip>:<window_bucket>` is a well-documented primitive (Redis "Rate limiting pattern", `INCR` docs §"Pattern: Rate limiter"). No library wrapping is needed.
- The dependency is exposed as a callable `RateLimiter(prefix, per_minute, burst)` instance that FastAPI consumes as `Depends(rate_limiter)`. This matches `Depends` semantics and stays Clean (no middleware monkey-patching, no global state outside the Redis-backed store).
- Decision documented in pack ADR-note format below — closer/architect can lift it to TECHNICAL_GUIDE §ADR if desired.

**NEEDS_OFFICIAL_DOCS: no.** The Redis `INCR`+`EXPIRE` rate-limiter pattern is documented in the upstream Redis docs (`redis.io/glossary/rate-limiting/`); `redis-py` 7.x is the same SDK already in use by health endpoints (P00-S02-T002). No 2026 API drift risk above what we already accept for `redis-py`.

**Atomicity note for developer**: use a small Lua script (`EVAL`) or a `pipeline()` with `INCR` + conditional `EXPIRE` to keep the increment+expire atomic. Avoid two round-trips that can race under burst. KISS variant: `INCR key`, if return value == 1 → `EXPIRE key <window>`; this is the canonical Redis pattern and is race-safe for the "first hit creates the window" branch since the increment is atomic.

### R2 — ENCRYPTION KEY SPLIT

**Decision: `ENCRYPTION_KEY` and `MFA_ENCRYPTION_KEY` remain SEPARATE.** Confirmed by user.

Rationale:

- Blast radius isolation: MFA TOTP secrets and AI/MCP provider credentials are different threat classes (user enumeration vs. tenant key leakage). Compromise of one must not compromise the other.
- `MFA_ENCRYPTION_KEY` is consumed today by `app/verification_data/crypto.py` (loader path) and by `app/auth/mfa_crypto.py` (decrypt at sign-in time). That path stays untouched.
- `ENCRYPTION_KEY` is consumed by NEW `app/security/encryption.py` and (future) admin services for AI/MCP credentials.
- Current `.env` value is `ENCRYPTION_KEY=replace-with-dev-key` (placeholder, NOT a valid Fernet key — Fernet requires exactly 32-byte url-safe base64). `app/security/encryption.py` MUST raise `EncryptionKeyError` at first use with a clear message + repair command.

### R4 — Permission guards: reuse `users/deps.py::get_current_user`

**Decision: re-export, do NOT duplicate JWT decode logic.**

Current state in repo:

- `backend/app/users/deps.py::get_current_user(request, session) -> User | JSONResponse` — async FastAPI dep, decodes Bearer, loads `User` with eager `user_roles` + `employee_profile`, emits anti-enum 401 envelope on all failures.
- JWT payload (from `app/auth/tokens.py::encode_access_token`) contains `roles: list[str]` claim. Pulled from `user.user_roles[].role.name`, defaults to `["employee"]` if empty.

Design for `app/security/permissions.py`:

```python
# Pseudocode — informational, developer is the implementer.

from app.users.deps import get_current_user  # delegate

require_user = get_current_user  # alias — same async signature/return contract

def require_role(role_name: str):
    """Factory returning an async FastAPI dependency that requires `role_name`.

    Returns User on success; returns JSONResponse 403 PermissionDeniedError
    when authenticated but role is missing; returns JSONResponse 401 when
    require_user fails (delegated)."""
    async def _dep(request, user_or_response = Depends(get_current_user)):
        if isinstance(user_or_response, JSONResponse):
            return user_or_response          # propagate 401
        role_names = [ur.role.name for ur in user_or_response.user_roles if ur.role]
        if role_name not in role_names:
            # 403 envelope — anti-enum NOT applicable (auth succeeded; this is a real authz failure)
            return _emit_403(request_id, code="AUTH_PERMISSION_DENIED",
                             message="You do not have permission to perform this action.")
        return user_or_response
    return _dep

require_admin    = require_role("people_admin")     # §3.3 role name
require_auditor  = require_role("people_auditor")   # §3.3 role name
```

Notes:

- The §3.3 role names are `employee`, `people_admin`, `people_auditor`, `super_admin`. The §10.2 guard names are the canonical ones: `require_admin` aliases `require_role("people_admin")` (NOT `"admin"`). Developer MUST verify the seed `roles` rows (seeded by P00-S02-T003 from `data/verification/roles.json`) to confirm exact `roles.name` strings. If seed uses different name, fix the alias.
- `super_admin` is a superset: `require_admin` should accept either `people_admin` OR `super_admin` (DOM rule). Spec it in tests.
- 403 envelope: `{ data: null, meta: { request_id }, error: { code: "AUTH_PERMISSION_DENIED", message: "..." } }` — consistent with §10.2/§10.5 patterns and the existing 401 envelope shape from `app.auth.routers._helpers._error_response`.

#### rate_limit (target: 6 tests)

1. `test_rate_limit_allows_under_burst` — `RateLimiter("TEST_X", per_minute=10, burst=10)`: 10 consecutive requests from same IP → all 200.

4. `test_rate_limit_per_ip_isolation` — IP_A exhausts its bucket; IP_B is not affected (different key in Redis).
5. `test_rate_limit_per_prefix_isolation` — `RateLimiter("PREFIX_A", ...)` exhaustion does NOT affect `RateLimiter("PREFIX_B", ...)` for the same IP.
6. `test_rate_limit_logs_warning_on_exceeded_and_debug_under_verbose` — caplog assertion: exhaust bucket; `WARNING` record at `security.rate_limit.exceeded`; under `ENABLE_VERBOSE_LOGGING=true` env also `DEBUG` BEFORE/AFTER records present.
