### B.3 — TECHNICAL_GUIDE §10.2 (Auth strategy — cookie contract)

Refresh cookie attributes are: `HttpOnly; Secure; SameSite=Lax; Path=/api/v1/auth` (pinned by P01-S02-T011, ADR-001). The cross-origin strategy chosen here **must preserve all four attributes byte-identical**. In particular:

- `SameSite=Lax` allows the cookie on **same-site** top-level navigations and on **same-origin** XHR. In dev, with frontend at :5173 and backend at :8000, these are **cross-origin** (different ports). Lax does NOT block cross-origin XHR by itself — it blocks cross-site third-party — but combined with `credentials:'include'` from the browser, the cross-origin preflight (OPTIONS) must pass. Today it does not, because no CORSMiddleware is registered and FastAPI emits `405 Method Not Allowed` for OPTIONS.
- `Secure` in dev: most modern browsers (Chrome, Safari, Firefox) treat `http://localhost` as a secure context for cookie purposes, so `Secure` cookies are accepted over plain HTTP localhost. No change needed.
- A reverse-proxy strategy that makes the request **same-origin** (browser sees `:5173/api/v1/auth/...`, vite forwards to `:8000/api/v1/auth/...`) removes the cross-origin classification entirely: no preflight, no CORS headers, cookie travels naturally on the same origin.

### B.4 — TECHNICAL_GUIDE §11 (Deploy) — DOES NOT specify cross-origin

§11 declares `frontend`, `backend`, `postgres`, `redis`, `worker`, `litellm`, `minio` services in Docker Compose. Frontend is served by **Nginx as a static site**. There is **no current spec** for:

## C. Recommended strategy

### C.1 — Strategy A (recommended): vite `server.proxy` — same-origin reverse proxy in dev, mirrors prod

```ts
// frontend/vite.config.ts (additive, no behavior change for tests)
export default defineConfig({
  // ... existing ...
  server: {
    host: "0.0.0.0",
    port: 5173,
    strictPort: false,
    proxy: {
      "/api": {
        target: "http://localhost:8000",
        changeOrigin: false,
        secure: false,
        // Forward cookies and X-Request-ID untouched. Vite/http-proxy preserves Set-Cookie by default.
      },
    },
  },
});
```

```bash
# frontend/.env.example (NEW FILE — does not exist today)
# Hilo People — frontend dev env contract.
# Slice P01-S03-T002 — cross-origin strategy = vite proxy (ADR-002).
#
# Empty value → frontend uses RELATIVE same-origin paths. The vite dev server proxies
# /api → http://localhost:8000 (see vite.config.ts server.proxy). In production the same
# pattern is implemented by nginx in front of the static SPA (TECHNICAL_GUIDE §11).
#
# DO NOT set this to http://localhost:8000 in dev. That re-introduces cross-origin XHR
# and the browser will require CORS preflight again.
VITE_API_BASE_URL=

# Enables verbose [auth] console.info logs in the auth feature.
# Mirrors backend ENABLE_VERBOSE_LOGGING. Default true in dev for /verify-slice flows.
VITE_ENABLE_VERBOSE_LOGGING=true
```

```python
# on top would be defense-in-depth noise that masks Strategy A breaking later.
```

**Why this is the right call (rationale — also goes to ADR-002)**:

1. **Mirrors prod**. §11 declares "Frontend build estático servible por Nginx" — in real deploy, Nginx (or CloudFront / ALB / ingress) sits in front of both the static SPA bundle AND `/api/**` and the browser sees ONE origin. Dev should mirror that topology so the browser code is identical in dev and prod (no `if (DEV) … else …` branches around the API URL).

2. **No CORS attack surface**. With same-origin, the backend never has to declare an allowlist of trusted origins. There is no `Access-Control-Allow-Origin: *` (which is incompatible with `allow_credentials=true` per CORS spec). There is no preflight at all. Less code = fewer holes.

3. **Cookie story stays trivially correct**. `SameSite=Lax; Path=/api/v1/auth` works exactly as designed by §10.2 / ADR-001. No interaction with cross-origin XHR rules.

4. **Aligns with §10.2 BFF-style intent**. Although Hilo doesn't use a managed auth SDK with `localStorage` (the trigger for §01-non-negotiables.md mandatory BFF), the *spirit* of the rule — browser only talks to "its" origin — is preserved.

5. **FU author's stated recommendation**. The triage `why_not_debugger` field calls Strategy A out as "recommended; same pattern as nginx/cloudfront in prod, BFF-style same-origin."

**Caveats / risks**:

- Streaming endpoints (SSE — `POST /api/v1/chat/conversations/{id}/stream`) MUST work through the proxy. Vite proxy (http-proxy) supports streaming out of the box; `ws: true` is only needed for WebSocket upgrades (none here). The proxy must NOT buffer SSE responses. Vite's default behavior preserves chunked transfer; **the developer must explicitly verify this works** with a smoke for `POST /api/v1/chat/conversations/{id}/stream` (this is a J101 concern — out of scope today, but worth a 1-line `Notes for downstream` entry in the handoff).
- The two existing T001 files (`frontend/src/features/auth/data/httpClient.ts` line 34 and `frontend/src/features/auth/data/authRepository.ts` line 26) hardcode `?? "http://localhost:8000"` as the fallback. Strategy A relies on `VITE_API_BASE_URL=""` (empty string). `"" ?? fallback` does NOT trigger the nullish coalescing fallback (empty string is not nullish), so `API_BASE = ""` and `${API_BASE}/api/v1/...` → `/api/v1/...` (relative same-origin). **This works today without touching those files**, but it is a latent risk: if someone deletes `VITE_API_BASE_URL` from the env entirely, the env var becomes `undefined` and the fallback kicks back in. Documented in §H below as a Strategy A follow-up candidate, NOT in scope here (those files are not in this task's write_set).
- One existing X-Request-ID test (T18 in `auth.test.tsx`) checks that Bearer is injected on `/api/v1/users/me` and not on `/api/v1/auth/refresh` — this is path-based, agnostic to absolute vs relative URLs, so it still passes.

### C.2 — Strategy B (alternative): backend `CORSMiddleware`

```python
# backend/app/main.py (added near top, before include_router calls)
from fastapi.middleware.cors import CORSMiddleware

# Dev-only: tightened allowlist. Staging/prod override via env, NOT via wildcards.
_CORS_ALLOW_ORIGINS = [
    o.strip() for o in os.getenv("BACKEND_CORS_ALLOW_ORIGINS", "http://localhost:5173").split(",")
    if o.strip()
]

app.add_middleware(
    CORSMiddleware,
    allow_origins=_CORS_ALLOW_ORIGINS,
    allow_credentials=True,  # required for cookie-based refresh; forces explicit origin list (no "*")
    allow_methods=["GET", "POST", "PATCH", "PUT", "DELETE", "OPTIONS"],
    allow_headers=["Authorization", "Content-Type", "X-Request-ID"],
    expose_headers=["X-Request-ID"],  # so frontend can read request_id for traceability
    max_age=600,
)
```

```bash
# frontend/.env.example (under Strategy B)
VITE_API_BASE_URL=http://localhost:8000        # absolute, cross-origin
VITE_ENABLE_VERBOSE_LOGGING=true
```

```ts
// frontend/vite.config.ts — NO server.proxy block.
```

**Why this is the fallback, not the recommendation**:

1. It documents the cross-origin reality but does not mirror prod. If Nginx in prod proxies `/api/**` (which is the only sensible prod topology), prod and dev disagree on whether CORS is in the picture. That delta is a class of bugs.
2. Forces a permanent allowlist env var (`BACKEND_CORS_ALLOW_ORIGINS`) that grows over time and is a vector for misconfig (forgotten staging URL, accidental wildcard).
3. `allow_credentials=True` with `allow_origins=["http://localhost:5173"]` is correct, but the next person who needs to add another origin will be tempted to use `["*"]` — which CORS forbids combined with credentials, but the error mode is silent in some browsers.
4. The backend has to ship and maintain a piece of middleware that is purely a deployment concession. Prod with same-origin Nginx doesn't need it.

**When Strategy B is the right call (NOT today)**:

- Frontend served from a different origin than the API by **explicit product decision** (e.g. mobile app + web API, separate domains, multi-tenant SaaS with `<tenant>.example.com` UI talking to `api.example.com`). None of these apply to Hilo People as scoped.

### C.3 — Decision

**Strategy A** (vite `server.proxy` + matching prod nginx reverse-proxy spec in §11 + ADR-002). Documented rationale above. Strategy B is the documented runner-up so future bootstrap runs preserve the trade-off.

If the developer hits a blocker with Strategy A during implementation (e.g. vite proxy mangles a header that the backend requires byte-identical), they fall back to Strategy B and re-extract this section in the handoff — they do not invent a third option without re-opening planning.

## F. Logging contract

**Strategy A (recommended)**:
- vite logs proxy events to its own stdout at info level. Visible during `npm run dev` when `VITE_ENABLE_VERBOSE_LOGGING=true`. No new app-level logs.
- No new BEFORE/AFTER logs because no new code path is introduced — every existing endpoint already has its logging.

**Strategy B**:
- `CORSMiddleware` writes nothing to app logs by default. No BEFORE/AFTER needed.
- The handler for `OPTIONS` preflight is added by FastAPI/Starlette automatically when `CORSMiddleware` is present; no new log statement required.

In neither case may secrets leak. The X-Request-ID is already logged everywhere. PII/tokens redaction is already enforced in §10.5 logging contract.

## H. ADR-002 draft (to append at end of TECHNICAL_GUIDE §15)

> The developer must paste this verbatim or substantially equivalent. The format follows ADR-001.

```markdown
### ADR-002 — Cross-origin strategy: same-origin reverse proxy in dev (vite) and prod (Nginx)

**Fecha**: 2026-05-13

**Contexto**: El frontend dev sirve en :5173 (vite) y el backend dev en :8000 (uvicorn). Sin más, todo XHR del navegador desde :5173 a :8000 es cross-origin, dispara preflight OPTIONS y requiere CORSMiddleware. La cookie de refresh (`SameSite=Lax; Path=/api/v1/auth`) sí se envía cross-origin con `credentials:'include'`, pero solo si el preflight pasa. Hoy no existe ni `CORSMiddleware` ni `vite server.proxy`; el preflight devuelve 405 y todo flujo real de auth queda bloqueado (`/verify-slice P01-S03-T001` F3–F5).

**Decisión**: Adoptar reverse proxy same-origin en dev y prod:
- Dev: `vite.config.ts` declara `server.proxy["/api"] = { target: "http://localhost:8000", changeOrigin: false }`. El navegador habla a :5173 para TODO (HTML, assets, /api/**) y vite reenvía /api → :8000.
- Prod: Nginx delante del SPA estático también proxy-pasa `/api/**` al servicio backend. El navegador habla a UN solo origin (el de la app) en ambos entornos.

`VITE_API_BASE_URL=""` en `frontend/.env.example` pin de contrato: el cliente usa paths relativos.

**Alternativas descartadas**:
- `CORSMiddleware` con `allow_origins=["http://localhost:5173"]; allow_credentials=true` — funciona pero deja prod y dev en topologías distintas (en prod hay Nginx delante; en dev hay CORS), introduce middleware adicional permanente solo por dev, requiere allowlist de orígenes mantenida a mano. Rechazado por divergencia dev/prod.
- `CORSMiddleware` con `allow_origins=["*"]` — incompatible con `allow_credentials=true` por spec CORS; además inseguro. Rechazado.
- `VITE_API_BASE_URL=http://localhost:8000` (status quo implícito) — re-introduce el problema. Rechazado.

**Consecuencias**: 
- El frontend code path es idéntico en dev y prod; cero ramas `if (DEV)` alrededor de URLs.
- La cookie refresh no necesita razonar sobre cross-origin: `SameSite=Lax` simplemente funciona.
- El backend no necesita conocer la lista de orígenes UI; un servicio detrás de Nginx no necesita CORS.
- SSE / streaming (P02-S04 `POST /api/v1/chat/conversations/{id}/stream`) debe verificarse explícitamente a través del proxy cuando aterrice; vite preserva chunked transfer por defecto.
- Si en el futuro Hilo gana un consumer externo (app móvil nativa, integración B2B), se reabrirá esta decisión con un ADR que SUPERSEDE a ADR-002.
```
