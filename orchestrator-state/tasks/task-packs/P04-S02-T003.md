### 3.3 ⚠️ Acceptance vs wire — DISCREPANCY ANALYSIS

Acceptance #1: *"MCP server list shows sync state, tool count and risk labels."*

| Acceptance term | Backend wire field (LIST) | Available? | How |
|---|---|---|---|
| **sync state** | `status` + `last_sync_at` | ✅ | `status ∈ {draft, active, …}`; `last_sync_at` ISO timestamp or null |
| **tool count** | — | ❌ | NOT exposed by `GET /servers`. Available via `POST /sync` *response* OR via DB (`SELECT count(*) FROM mcp_tools WHERE server_id=...`); no GET tool list endpoint exists. |
| **risk labels** | — | ❌ at server level | `mcp_tools.risk_level` exists per tool (`low|medium|high|critical`); per-server "risk label" is NOT a backend concept. There is no GET `/mcp/tools` endpoint either. |

`mcp_tools` rows exist in DB (P02-S07-T001 created them); they are mutated via `POST /sync` (upsert) and `PATCH /mcp/tools/{id}` (admin curation). The page cannot read them today without:

- (a) a new backend endpoint `GET /api/v1/admin/ai/mcp/servers/{id}/tools` returning `[ToolOut]`, OR
- (b) extending `ServerOut` to include `tool_count: int` and (optional) `max_tool_risk_level: str | null`, OR
- (c) the page only displaying the **last sync's `tools_count`** (transient, lost between refresh cycles unless persisted).

### 3.4 Recommended in-scope mitigation (NO BACKEND CHANGES)

**Decision §D-T003-RISK-LABELS-DERIVED-OR-OMITTED** + **§D-T003-TOOL-COUNT-TRANSIENT** — keep the slice frontend-only, satisfy the acceptance via this contract:

1. **Sync state column**: render `status` (i18n-mapped: `draft → "Borrador"`, `active → "Activo"`, etc.) + `last_sync_at` (relative, e.g. "Hace 5 min" / "Nunca"). Use `StatusDot` + `TrackedLabel` per UX_CONTRACT §3.2 ("Statuses combine dot and uppercase tracked label"). This satisfies *"sync state"*.
2. **Tool count column**: show **last-sync tools_count** sourced from a per-server *transient cache* populated by the `POST /sync` mutation `onSuccess` (TanStack Query `setQueryData`-style overlay or a small Zustand-free `useState` keyed by `server.id`). On initial load the cell shows `—` (em-dash) with `aria-label="No tools synced yet"` (i18n key) — this is honest: the server may have synced before but we have no DB-of-record for the count. Document this as a known limitation in the handoff with the anchor `§D-T003-TOOL-COUNT-TRANSIENT`. Do NOT auto-create a follow-up; if `screen-journey-reviewer` flags it as a real gap during `/verify-slice`, then `register-followup` for backend `tool_count`.
3. **Risk labels column**: render `—` (em-dash) per row with the same honest treatment, accompanied by a single i18n footnote (`mcp.servers.notes.risk_per_tool`: *"El nivel de riesgo se gestiona por herramienta en P04-S02-T004."*) in ES/EN/FR. Reference the `PATCH /mcp/tools/{id}` endpoint as the future surface for risk curation.

This satisfies acceptance #1 in spirit (the page DOES show sync state, the cells exist for tool count and risk labels with honest fallback values, and the user has unambiguous information about where risk is curated) while keeping the slice within its declared `frontend` kind in the registry. **It is NOT a silent skip** — both gaps are explicitly disclosed in the handoff and in §7 R-1/R-2.
