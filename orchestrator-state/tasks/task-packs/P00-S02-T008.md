## Algoritmo de routing (especificación canónica)

**Input**: `message: str`, `subagents: list[AgentSpec]` (cada uno con `subagent_topics: list[str]`).

**Algoritmo**:
1. Tokenizar `message` con regex Unicode-aware: `re.findall(r"\w+", message.lower(), flags=re.UNICODE)` → `set[str]`.
2. Para cada subagente con `subagent_topics` no vacío:
   - Normalizar topics: `{t.lower() for t in subagent_topics}`.
   - Calcular `score = len(message_tokens & topic_set)` (intersection cardinality).
   - Capturar `matched_topics = sorted(message_tokens & topic_set)` para trazabilidad/log.
3. Seleccionar el subagente con `score` máximo > 0.
4. **Tie-break**: si dos subagentes empatan en score > 0, elegir el de mayor `len(subagent_topics)` (subagente más especializado pierde frente al más amplio en empate; alternativa: orden estable por `name` ascendente — documentar la elección y testarla).
5. **Fallback** (sin overlap): retornar `RoutingDecision(selected_subagent=None, score=0, matched_topics=[], fallback_used=True)`. El supervisor decide qué hacer con el fallback (responder genérico, o re-lanzar al subagente "default" si hay marcado tal — en T008 no hay flag de default; el fallback retorna None y el caller decide).

**Edge cases obligatorios cubiertos en tests**:
- `message=""` → `RoutingDecision(None, 0, [], fallback_used=True)`.
- `subagents=[]` → mismo fallback.
- Solo supervisor sin subagents en la lista → ValueError("supervisor needs ≥1 subagent") O fallback documentado — preferir ValueError porque es estado degenerado.
- Acentos: "vacación" vs "vacacion" — el seed usa `vacaciones` sin tilde; los tests deben validar matching ASCII y documentar en docstring que la normalización de acentos se difiere a un follow-up si surge necesidad real (ahora YAGNI).

## Riesgos y decisiones previas

### Riesgos T008-específicos

- **R-T008-1 (medium)**: el módulo nombrado `deepagents_runtime.py` puede generar la expectativa de que llama a `deepagents.create_deep_agent` ya hoy. Mitigación: docstring del módulo aclara que el entry point es el *nombre* del acceptance pero la integración real con `create_deep_agent` queda detrás de la interfaz `SubagentExecutor` y se materializa en P02-S08-T001 cuando exista LLM/DB/endpoint.
- **R-T008-2 (low)**: tie-break en routing puede ser ambiguo y los tests pueden depender del orden del JSON. Mitigación: tie-break determinista documentado (mayor `len(subagent_topics)`, luego `name` ascendente) + test específico que cubre el caso.
- **R-T008-3 (low)**: el JSON fixture vive en `data/verification/mcp_agents/agents.json` y los tests dependen de él. Si T005 cambia el fixture, los tests se rompen. Mitigación: tests cargan el JSON real (no copy local), validan el shape con `AgentSeed.model_validate`, y los topics esperados se extraen del fixture en tiempo de test (no hardcodean valores).
- **R-T008-4 (medium)**: `subagent_topics` no incluye exactamente "vacaciones" como token (sí lo hace — `"vacaciones"` está literal). Para "langchain question": el seed usa `langchain` y `langgraph` como topics, así que mensajes con esas palabras hacen match. Documentar en el test que las verificaciones del registry verbatim se traducen a los tokens reales del fixture.
- **R-T008-5 (medium)**: el agentes JSON declara `mcp_server_name` para subagents, pero los servers MCP no están migrados. El runtime debe **ignorar** `mcp_server_name` en T008 (no resolver tools). Documentar en docstring + test que `mcp_server_name` se carga pero no se usa hasta P02-S08-T001.
- **R-T008-6 (low)**: deepagents importable a nivel de import del módulo. Si `import deepagents` falla en CI por desinstalación accidental, todo el módulo cae. Mitigación: `import deepagents` solo en `_executor.py` dentro de la implementación real, no en T008. T008 sólo usa la *forma* de SubAgent dict de manera documental — sin import directo.
