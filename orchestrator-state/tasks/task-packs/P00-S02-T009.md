## 0. Goal (one line, user-visible)

API keys (`AIza…`, `sk-…`, Bearer tokens) must NEVER appear in stdout/stderr or persisted log sinks, even when `ENABLE_VERBOSE_LOGGING=true`.

## 1. Scope decision (planner-USER level)

Scope is **monofile**: `backend/app/core/logging.py` — same module that already implements the CWE-532 fix from P00-S02-T004 (Rich `show_locals=False` + `_REDACTED_KEYS`). This slice extends that defense by suppressing the **third-party `httpx`/`httpcore` stdlib loggers** that bypass our `_redaction_processor` (it only scrubs structlog `event_dict` keys; foreign loggers route through `ProcessorFormatter.foreign_pre_chain` but the **log message body itself** — `"HTTP Request: GET https://…?key=AIza…"` — is a single string and is not key-scrubbed).

Tests live in `backend/tests/test_logging.py` (existing file from T004). New tests added are an in-scope `write_set_extension` per the precedent class — see §11. Do NOT widen scope to silence other libs (`urllib3`, `boto3`, `litellm`, `langchain`); the FU implicates only `httpx`. Other libs do not currently leak api keys via URL — confirm with grep before declaring done; if a leak surfaces in another lib, register a follow-up.

## 3. Acceptance (verbatim from work-item YAML)

> grep -E 'AIza|sk-[A-Za-z0-9]{20}' verbose_logs.txt returns 0 matches when ENABLE_VERBOSE_LOGGING=true and endpoint makes a real Gemini API call

**Operationalised** (preserves original verb in registry; planner translation per task-packing rule):

A1. With `verbose=True`, exercising `GeminiDirectClient.list_models()` against an endpoint whose URL contains `?key=AIza…`, the captured stdout/stderr text MUST NOT contain the substring `AIza` (the leak signature).

A2. Same condition for OpenAI Bearer tokens (`sk-[A-Za-z0-9]{20}` regex) — defense-in-depth: even though OpenAI passes the key in `Authorization` header (not the URL), httpx's INFO-level "HTTP Request" line could in theory surface it if the URL were ever rewritten to embed it. Verify the line never contains real-looking key signatures.

A3. With `verbose=False` (production default — root logger at WARNING), httpx INFO logs are already silent. Add a regression test confirming this stays silent — CWE-532 defense-in-depth.

A4. The fix MUST NOT silence `WARNING`/`ERROR` from `httpx` — those carry actionable failure info and contain no keys (we already log `error_class` + `upstream_status` only).

## 9. Implementation strategy

### 9.1 Chosen approach: stdlib `logging.getLogger("httpx").setLevel(WARNING)` (+ `httpcore` if researcher confirms)

Place inside `configure_logging()` AFTER the root-logger setup (`root_logger.setLevel(log_level)`) and BEFORE setting `_configured = True`. Do this in BOTH branches (verbose=True and verbose=False — explicit is safer; in verbose=False the root is already WARNING but pinning the named logger guards against future changes to root level).

**Rationale**:
- Smallest possible change: 2-3 lines.
- Idempotent: `configure_logging()` is already idempotent; this stays inside that guard.
- No new deps.
- Respects WARNING/ERROR from httpx (A4) — those still propagate.
- Does NOT touch `provider_clients.py` (defense-in-depth: app-level `_safe_base_url()` stays as primary scrub for app logs; transport-level WARNING gate is the new third layer).

### 9.2 Rejected alternatives (1 line each)

- **httpx event_hooks to scrub URL** — rejected: requires plumbing through `httpx.AsyncClient(event_hooks=...)` in every client class (GeminiDirectClient, OpenAIDirectClient, LiteLLMProxyClient + any future ones) → write_set widens out of scope, repeat-yourself violation.
- **`logging.Filter` on `httpx` logger that regex-scrubs `?key=…`** — rejected: more code surface, regex maintenance, false-positive risk; same outcome as setLevel(WARNING) for the leak vector but heavier.
- **Globally set httpx to WARNING via `logging.basicConfig(...)` extra arg** — rejected: basicConfig is already used; adding logger-level overrides via `disable_existing_loggers` would conflict with structlog's foreign_pre_chain. The named-logger approach is orthogonal and clean.
- **Strip URL query string in httpx via custom `AsyncHTTPTransport`** — rejected: invasive; would also strip query strings from response bodies/repr() and surprise developers. We don't own httpx's internal logging contract.

### 9.3 Pseudo-diff (developer guide, ~2-3 lines added)

Inside `configure_logging()` after `root_logger.setLevel(log_level)` and before `_configured = True`:

```python
# CWE-532 third-layer fix (P00-S02-T009): suppress third-party stdlib
# loggers that emit request URLs containing api_keys (Gemini ?key=AIza…).
# httpx INFO logs the full URL; we keep WARNING/ERROR for actionable failures.
# Researcher must confirm logger names ('httpx', possibly 'httpcore.*') for
# httpx 0.28.1 — this list is the upper bound; trim if researcher narrows it.
for _noisy in ("httpx", "httpcore"):
    logging.getLogger(_noisy).setLevel(logging.WARNING)
```

Plus update the module docstring header (BEFORE/AFTER pattern + cite this slice + cite the FU id) — same format as the T004/T002 docstring updates already in the file.

### Reproducing the leak deterministically (developer guide)

The leak vector is **the request URL log line emitted by httpx's INFO logger**, not the response body. Therefore:

- A `MockTransport` reproduces the leak path because httpx logs the **outbound** request before the transport runs. The mock just prevents a real network call.
- Confirm the leak BEFORE applying the fix: temporarily comment out the new `setLevel(WARNING)` lines, run T1; it MUST fail with `AIza-FAKE-SENTINEL-KEY-...` in captured output. This proves the test catches the regression. Then restore the fix and rerun; it MUST pass.
- The developer should record this BEFORE/AFTER capture in `orchestrator-state/tasks/evidence/P00-S02-T009/` (two files: `leak-before-fix.txt`, `leak-after-fix.txt`). Captures must NOT include any real secret.
