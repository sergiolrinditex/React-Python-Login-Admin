## 0. TL;DR para el developer

Construir el primer Celery worker del proyecto (`celery -A app.worker worker`) y dos tasks en `backend/app/workers/`:

1. `tasks_documents.extract_and_chunk(document_id, version_id)` — descarga el binario de S3/MinIO via `boto3`, calcula checksum, extrae texto con `pypdf` o `python-docx`, parte con `langchain-text-splitters` y persiste filas en `document_chunks`. Actualiza `vectorization_jobs.progress` de 0→50.
2. `tasks_embeddings.embed_chunks(document_id, version_id)` — para cada chunk genera embedding via LiteLLM proxy local (`POST {LITELLM_BASE_URL}/embeddings`), persiste `document_embeddings(chunk_id, embedding vector(1536), model_id)` y actualiza `vectorization_jobs.progress` 50→100, `status='done'`. En cualquier excepción → `status='failed'`, `error=<mensaje>`, `finished_at=now()` dentro de transacción.

Encadenamiento Celery: usa `chain(extract_and_chunk.s(...), embed_chunks.s(...))` para que el segundo task solo arranque tras `extract_and_chunk` `SUCCESS`. El endpoint que crea el job (P02-S06-T001) llamará a `chain.apply_async()`.

Tests reales con Postgres + Redis + pgvector + LiteLLM local (mismo docker compose que el resto del proyecto). Sin mocks de la lógica de negocio.

### 2.1 Acceptance expandido (testeable, numerado)

A1. **Job lifecycle**: dado un `vectorization_jobs(id=J, document_id=D, status='pending', progress=0)`, después de ejecutar `chain(extract_and_chunk.s(D, V), embed_chunks.s(D, V)).apply_async()`, la fila evoluciona a `status='done', progress=100, error IS NULL, finished_at IS NOT NULL`.
A2. **Chunk persistence**: para un documento PDF de N páginas con texto extraíble, después de `extract_and_chunk` existe ≥1 fila en `document_chunks(document_id=D, version_id=V, chunk_index=0..K-1, content<>'', metadata::jsonb)`. K = ceil(total_chars / chunk_size_for_collection_vertical) con overlap.
A3. **Version creation**: si el documento aún no tiene `document_versions`, el worker crea la versión `version=1, storage_key=<source_uri or path>, checksum=<sha256 hex of bytes>`. Si existe versión previa y el checksum coincide, reusa la última versión (idempotencia). Si el checksum cambia, crea `version=max+1` y deja los chunks viejos intactos (el retriever filtra por `version_id`, así que el reindex no contamina resultados).
A4. **Embedding persistence**: para cada `document_chunks` de la versión actual, existe exactamente una fila en `document_embeddings(chunk_id=C, embedding vector(1536), model_id=<default embeddings model>)`. Vector dim DEBE ser 1536 (truncar/expand/reject según `ai_models.capabilities` — ver D-DIM más abajo).
A5. **Progress updates**: el worker emite UPDATEs intermedios:
    - tras leer bytes desde S3/MinIO: `progress=10`
    - tras extraer texto: `progress=25`
    - tras chunkear: `progress=50`
    - cada N chunks embebidos (N=ceil(total/10), mínimo 1): `progress = 50 + 50 * (embedded/total)` (clamp 50..99)
    - al terminar: `progress=100, status='done', finished_at=now()`
    Cada UPDATE es su propia transacción (no se hace en un bulk al final) para que el frontend pueda hacer polling y ver progreso real.
A6. **Failure path**: cualquier excepción no controlada (extract failure, embed API 5xx tras retries, DB constraint violation) → un UPDATE final marca `status='failed', error=<str(exc)[:1024]>, finished_at=now()` dentro de su propia tx. No deja `progress` a medio escalar sin marcar `failed`. La task Celery NO re-raise por encima del marcado (consume el exception tras persistirlo) salvo en casos transitorios donde se quiera autoretry — ver D-RETRY.
A7. **No PII en logs**: ningún log contiene `content` de chunks, bytes binarios del documento, vectores de embedding completos, ni `LITELLM_MASTER_KEY`. Logs solo IDs (job_id, document_id, version_id, chunk_count), tamaños (bytes_read, chars_extracted, chunks_created, embeddings_persisted), latencias (ms) y status.
A8. **BEFORE/AFTER logging** (no negociable): cada etapa loguea:
    - `vectorization.<stage>.before` con `job_id`, `document_id`, params relevantes
    - `vectorization.<stage>.after.ok` con `job_id`, `latency_ms`, métrica de salida
    - `vectorization.<stage>.after.error` con `job_id`, `error_type`, `latency_ms`
    `ENABLE_VERBOSE_LOGGING=true` muestra todo; `false` muestra solo `*.error` + warning. Pattern: copiar exactamente lo que hace `app/rag/retriever.py` (`_VERBOSE = os.getenv("ENABLE_VERBOSE_LOGGING","false").lower() == "true"`).
A9. **Transactional safety**: las INSERTs masivas de `document_chunks` se hacen en un único `session.commit()` por documento. Las INSERTs de `document_embeddings` se hacen en batches (recomendado: batch_size=32 ó igual al embeddings batch_size del proveedor); cada batch es su propia tx para que un fallo a mitad no pierda los embeddings ya persistidos. Si un batch falla → rollback de ese batch + UPDATE de `vectorization_jobs.status='failed'` + commit del fallo.
A10. **Idempotent reentry**: si el worker se invoca de nuevo para el mismo `(document_id, version_id)` con `vectorization_jobs.status='done'`, NO-OP (loguea `vectorization.skip.already_done` y retorna sin tocar nada). Si `status='failed'`, recomienza el flujo (crea nueva versión solo si el checksum cambió o si el caller lo fuerza).
A11. **Real text extraction**: `pypdf.PdfReader` para `.pdf`; `python-docx` (`from docx import Document`) para `.docx`. Para otros mimetypes → `status='failed', error='UnsupportedMimeType: <ext>'`.
A12. **Chunk size por collection vertical**: `documents.collection_id → rag_collections.vertical`. Default chunk_size=1000 chars, chunk_overlap=150 chars (LangChain `RecursiveCharacterTextSplitter`). Si la collection tiene `rag_collections.metadata->>'chunk_size'` o `'chunk_overlap'`, prevalece. Si el documento no tiene collection (NULL), usa defaults.
A13. **Embedding model resolution**: el worker resuelve el modelo de embeddings haciendo `SELECT id FROM ai_models WHERE model_type='embeddings' AND enabled=true AND is_default=true LIMIT 1`. Si no hay default activo → `status='failed', error='NoDefaultEmbeddingsModel'`. Esto fuerza al admin a configurar el modelo antes de indexar (P02-S05-T001 ya creó el endpoint para activar modelos).
A14. **LiteLLM call**: `POST {LITELLM_BASE_URL}/embeddings` con body `{"model": "<model_id_string from ai_models.model_id, NOT UUID>", "input": [chunk1, chunk2, ...]}`, header `Authorization: Bearer {LITELLM_MASTER_KEY}`. Si el provider devuelve 401/403 → `status='failed', error='LiteLLMAuth'`. Si 5xx → retry exponential backoff (max 3, base=2s) — luego falla.
A15. **Constraint pgvector 1536**: si el provider devuelve un vector con dim ≠ 1536, el worker rechaza el batch entero con `status='failed', error='EmbeddingDimMismatch: got=<N> expected=1536'`. NO se trunca ni se pad ceros silenciosamente. Se documenta para P02-S05-T002 (Model test) que debe validar dim antes de activar el modelo.

### 3.4 Celery app shape (worker.py)

```python
# backend/app/worker.py — Celery app entry point
# Required by docker-compose.yml worker service:
#   command: celery -A app.worker worker --loglevel=info
import os
from celery import Celery

_BROKER = os.getenv("REDIS_URL", "redis://localhost:6379/0")
_BACKEND = os.getenv("CELERY_RESULT_BACKEND", _BROKER)

app = Celery(
    "hilo_workers",
    broker=_BROKER,
    backend=_BACKEND,
    include=[
        "app.workers.tasks_documents",
        "app.workers.tasks_embeddings",
    ],
)
app.conf.update(
    task_acks_late=True,                  # avoid losing tasks on worker crash
    task_reject_on_worker_lost=True,
    worker_prefetch_multiplier=1,         # one-at-a-time for vectorization (CPU/IO heavy)
    task_default_retry_delay=2,           # seconds; exponential via Task autoretry_for
    task_serializer="json",
    accept_content=["json"],
    result_serializer="json",
    timezone="UTC",
    enable_utc=True,
)
```

`celery -A app.worker worker` discovers tasks via `include=[...]`. The Celery app is module-level so `app.worker.app` is importable from anywhere (tests use `app.send_task()` or invoke tasks directly with `.apply_async()` / `.apply()` for eager mode).

### 3.5 Task signature contract (lo que el developer DEBE implementar)

```python
# app/workers/tasks_documents.py
from app.worker import app

@app.task(
    bind=True,
    name="hilo.vectorization.extract_and_chunk",
    autoretry_for=(IOError, ConnectionError),
    retry_backoff=True, retry_backoff_max=30, max_retries=3,
)
def extract_and_chunk(self, document_id: str, version_id: str | None = None) -> dict:
    """Extract text, create version (if needed), chunk, persist document_chunks.

    Returns dict with: {job_id, document_id, version_id, chunks_created, status}.
    On any non-retriable failure, marks vectorization_jobs.status='failed' and
    re-raises ONLY if Celery autoretry should kick in (IOError/ConnectionError).
    Other errors are persisted and consumed.
    """
```

```python
# app/workers/tasks_embeddings.py
from app.worker import app

@app.task(
    bind=True,
    name="hilo.vectorization.embed_chunks",
    autoretry_for=(httpx.HTTPError,),
    retry_backoff=True, retry_backoff_max=30, max_retries=3,
)
def embed_chunks(self, prev: dict | None = None, document_id: str | None = None, version_id: str | None = None) -> dict:
    """Read document_chunks for (document_id, version_id), call LiteLLM /embeddings,
    persist document_embeddings, update vectorization_jobs to status='done'.

    When chained, `prev` is the dict returned by extract_and_chunk.
    Returns {job_id, document_id, version_id, embeddings_created, status}.
    """
```

Encadenamiento:

```python
from celery import chain
from app.workers.tasks_documents import extract_and_chunk
from app.workers.tasks_embeddings import embed_chunks

result = chain(
    extract_and_chunk.s(str(document_id), str(version_id)),
    embed_chunks.s(),                  # receives prev dict, also reads document_id/version_id from prev
).apply_async()
```

### 3.6 LiteLLM client (in-task, NO llm_gateway module yet)

```python
import httpx, os
LITELLM_URL = os.getenv("LITELLM_BASE_URL", "http://litellm:4000")
LITELLM_KEY = os.getenv("LITELLM_MASTER_KEY", "")  # never log this

def call_embeddings(model_id_string: str, inputs: list[str]) -> list[list[float]]:
    """POST {LITELLM_URL}/embeddings → returns list of 1536-d vectors."""
    resp = httpx.post(
        f"{LITELLM_URL.rstrip('/')}/embeddings",
        headers={"Authorization": f"Bearer {LITELLM_KEY}"},
        json={"model": model_id_string, "input": inputs},
        timeout=httpx.Timeout(60.0, connect=10.0),
    )
    resp.raise_for_status()
    body = resp.json()
    return [item["embedding"] for item in body["data"]]
```

LiteLLM proxy soporta el OpenAI embeddings shape (`POST /embeddings` body `{model, input: str|list[str]}` → `{data: [{embedding: [...]}, ...]}`). El `model_id_string` en LiteLLM tiene el formato `<provider>/<model_id>`, pero como aquí leemos `ai_models.model_id` (text), el admin debió haber configurado el string correcto (e.g. `openai/text-embedding-3-small`).

## 5. Front → Back → DB contract

### 5.1 Wiring real

```
[Future P02-S06-T001 endpoint]
  POST /api/v1/admin/rag/documents/{id}/index
    │
    ├─→ INSERT INTO vectorization_jobs (document_id, status='pending', progress=0) RETURNING id
    │
    └─→ chain(
          extract_and_chunk.s(document_id, version_id_or_none),
          embed_chunks.s(),
        ).apply_async()
        │
        ▼ (worker process picks up)
        backend/app/workers/tasks_documents.py::extract_and_chunk
          ├─ get_job_by_document_id(document_id) → row J  [or fail "JobNotFound"]
          ├─ UPDATE J SET status='running', progress=5
          ├─ SELECT * FROM documents WHERE id=document_id  → doc
          │     if doc.collection_id IS NULL → status='failed', error='NoCollection', return
          ├─ resolve storage: doc.source_uri starts with "s3://" → boto3.get_object
          │                                "minio://" → boto3 with endpoint_url override
          │                                "file://"  → local path (tests)
          ├─ bytes = read_object(...); checksum = sha256(bytes)
          ├─ UPDATE J SET progress=10
          ├─ if no version or version.checksum != new_checksum:
          │     INSERT document_versions (document_id, version=max+1, storage_key, checksum)
          │     version = new row
          │ else:
          │     version = latest row
          ├─ text = extract_text(bytes, mime)  [pypdf or python-docx]
          ├─ UPDATE J SET progress=25
          ├─ chunks = RecursiveCharacterTextSplitter(chunk_size, overlap).split_text(text)
          ├─ INSERT INTO document_chunks(document_id, version_id, chunk_index, content, metadata) [bulk]
          ├─ UPDATE J SET progress=50
          └─ return {job_id, document_id, version_id, chunks_created}
        │
        ▼ (chained, fires only on SUCCESS of previous)
        backend/app/workers/tasks_embeddings.py::embed_chunks
          ├─ get_job_by_document_id(document_id) → row J
          ├─ SELECT id, model_id FROM ai_models WHERE model_type='embeddings' AND enabled AND is_default
          │     if none → UPDATE J SET status='failed', error='NoDefaultEmbeddingsModel', return
          ├─ SELECT id, content FROM document_chunks WHERE document_id=D AND version_id=V ORDER BY chunk_index
          ├─ for batch in chunks_grouped_by_32:
          │     vectors = call_embeddings(ai_model.model_id, [c.content for c in batch])
          │     validate each len(v) == 1536 (else fail)
          │     INSERT INTO document_embeddings (chunk_id, embedding, model_id) [bulk]
          │     UPDATE J SET progress = 50 + 50 * (embedded/total)
          └─ UPDATE J SET status='done', progress=100, finished_at=now()
```

### 5.2 SQL queries puntuales (el developer puede copiar)

```sql
-- get_or_create_version (in-tx, FOR UPDATE on documents to serialize concurrent reindex)
SELECT id FROM documents WHERE id=:doc_id FOR UPDATE;

SELECT id, version, checksum FROM document_versions
  WHERE document_id=:doc_id ORDER BY version DESC LIMIT 1;

-- if checksum matches → reuse; else:
INSERT INTO document_versions (document_id, version, storage_key, checksum)
  VALUES (:doc_id, :next_version, :storage_key, :checksum)
  RETURNING id;

-- bulk insert chunks
INSERT INTO document_chunks (document_id, version_id, chunk_index, content, metadata)
  VALUES (:doc, :ver, :idx, :content, :meta::jsonb);

-- resolve default embeddings model
SELECT id, model_id FROM ai_models
  WHERE model_type='embeddings' AND enabled=true AND is_default=true LIMIT 1;

-- bulk insert embeddings (1 row per chunk; chunk_id is PK in document_embeddings)
INSERT INTO document_embeddings (chunk_id, embedding, model_id)
  VALUES (:chunk_id, :emb::vector, :model_uuid);

-- progress update
UPDATE vectorization_jobs SET progress=:p WHERE id=:job_id;

-- final
UPDATE vectorization_jobs
  SET status='done', progress=100, finished_at=now() WHERE id=:job_id;

-- failure
UPDATE vectorization_jobs
  SET status='failed', error=:err, finished_at=now() WHERE id=:job_id;
```

### 5.3 SQLAlchemy session pattern

Sigue el patrón de `app/db/session.py` (sync `Session`) ya usado por retriever, auth, chat. El worker NO usa async porque Celery con eventlet/gevent suma complejidad innecesaria para CPU-bound text work. Sync session por task, abrir/cerrar dentro del task body. Para acceder al engine: `from app.db.session import get_engine` (o el patrón equivalente que ya existe).

### 5.4 vectorization_jobs status machine

```
pending → running → done
                  ↘ failed
```

Transiciones permitidas:
- `pending → running`: al inicio de `extract_and_chunk`.
- `running → done`: al final de `embed_chunks`.
- `running → failed`: cualquier excepción durante extract/chunk/embed.
- `pending → failed`: si la pre-condición falla (NoCollection, JobNotFound, etc.).

NO se permite `done → running` ni `failed → running` desde el worker; el reindex crea un job nuevo (responsabilidad del endpoint P02-S06-T001).

Stages a instrumentar: `start`, `read_storage`, `extract_text`, `version_create`, `chunk_text`, `chunks_persist`, `model_resolve`, `embed_batch`, `embeddings_persist`, `finalize`, `failure`.

`request_id`: el worker NO recibe HTTP `X-Request-ID` directamente. Si el chain se invoca sin request_id (e.g., desde test), el worker genera un `uuid.uuid4().hex` propio y lo loguea como `worker_request_id`. Sin re-using request_id real entre HTTP y worker queda más limpio el debug.

Esto desperdicia los embeddings ya hechos pero es correcto/seguro. Si en el futuro se quiere "resume" (continuar desde el chunk K), requiere refactor — fuera de scope.
