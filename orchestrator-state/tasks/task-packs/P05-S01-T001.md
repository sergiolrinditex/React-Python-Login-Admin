§10.2 TECHNICAL_GUIDE — Auth strategy (citas literales):

- JWT propio. Access token Bearer corta vida (config 1800s observado en P03-S01-T005).
- Refresh token en cookie `HttpOnly; Secure; SameSite=Lax; Path=/api/v1/auth` (RFC 6265
  §5.4, contrato fijado en `_REFRESH_COOKIE_PATH` de `backend/app/auth/routers/_helpers.py`).
- Password: Argon2.
- Claims: `sub, email, roles, preferred_language, employee_profile_id, iat, exp, jti`.
- Guards: `require_user`, `require_admin`, `require_auditor`, `require_role`.

§6.4 Navigation Contract: Rutas públicas incluyen `/auth/sign-in`, `/auth/2fa`. Rutas
empleado: `/chat`. Deep links respetan auth guard.

§6.2 Endpoints involucrados:

- `POST /api/v1/auth/sign-in` → `{data:{mfa_required, mfa_challenge_token, expires_in}}`
  o `{data:{access_token, user}}` si MFA no requerido. Errores 400/401/423. Escribe
  `audit_logs` (LOGIN_OK / LOGIN_FAIL).
- `POST /api/v1/auth/2fa/verify` `{challenge_id, code}` → `{data:{access_token, user}}`
  + set-cookie refresh. Errores 400/401/410. Escribe `mfa_totp_secrets read`,
  `refresh_tokens insert`, `audit_logs` (MFA_VERIFY_OK / MFA_VERIFY_FAIL).
- `GET /api/v1/users/me` → `{data:UserProfile}`. Errores 401. Lee `users`,
  `employee_profiles`.

§10.3 Schema de tablas tocadas (resumen):

- `users(id UUID PK, email UNIQUE, password_hash TEXT Argon2, preferred_language CHECK IN ('es','en','fr'))`.
- `employee_profiles(user_id UUID PK FK, employee_id, brand, society, center, country, department, metadata JSONB)`.
- `refresh_tokens(id UUID PK, user_id FK, token_hash, expires_at, revoked_at)`.
- `mfa_totp_secrets(user_id UUID PK FK, secret_encrypted, enabled BOOLEAN)`.
- `audit_logs(id UUID PK, actor_user_id FK ON DELETE SET NULL, action, entity_type, entity_id, metadata JSONB, created_at)`.

§8 Logging: structlog con `request_id, user_id_hash, role, route, latency_ms, status_code`.
ENABLE_VERBOSE_LOGGING controla el detalle (true → flow completo; false → warn+error solo).

§6.4 Error format: códigos canónicos `AUTH_INVALID_CREDENTIALS`, `AUTH_MFA_REQUIRED`,
`AUTH_SESSION_EXPIRED`, `AUTH_FORBIDDEN`. Envelope siempre `{data, meta:{request_id}, errors:[{code,message,field,details}]}` (verificado en vivo con probe a /sign-in).

### Reglas de seguridad (instrucciones.md + .claude/rules/01-non-negotiables.md)

- BFF pattern: el frontend recibe access_token vía POST /sign-in y /2fa/verify, y lo
  guarda solo en memoria del cliente. NUNCA en localStorage / sessionStorage. Refresh
  va por cookie HttpOnly.
- Aggregate-401 anti-enum: 401 al sign-in nunca diferencia "usuario no existe" vs
  "password incorrecto" en el body. Mensaje genérico siempre. Verificado en vivo (code
  `AUTH_INVALID_CREDENTIALS` con body genérico).
- Audit_logs: LOGIN_OK / LOGIN_FAIL / MFA_VERIFY_OK / MFA_VERIFY_FAIL persistidos con
  `request_id` correlacionado al header `X-Request-ID`.
- Logging PII-clean: ni tokens, ni passwords, ni códigos TOTP, ni email plano deben
  aparecer en logs back/front (instrucciones.md "Logging").

## 13. Riesgos y decisiones previas

### Riesgos identificados (planner)

| Id | Riesgo | Mitigación |
|---|---|---|
| R1 | TOTP rota cada 30s; entre paso 1 (sign-in) y paso 3 (verify) puede caducar. | Generar TOTP justo antes del verify (`pyotp.TOTP(...).now()`), no antes del sign-in. Si verify falla por ventana, retry una vez. |
| R2 | El refresh cookie en dev (sin HTTPS) puede no llevar flag `Secure`; navegadores lo aceptan en localhost. | Documentar en verification-report.md; verificar contra `_set_refresh_cookie` en backend si la ausencia de `Secure` es intencional para http://localhost. |
| R3 | `audit_logs.action` puede ser `auth.sign_in` (snake) o `LOGIN_OK` (legacy). | Detectar en vivo con `SELECT DISTINCT action FROM audit_logs WHERE created_at > now() - INTERVAL '1 hour'` y documentar el nombre exacto observado. NO asumir. |
| R4 | `users.last_login_at` puede no existir en el schema (no aparece en §10.3). | Si la columna no existe, omitir esa aserción y notar en report; usar `audit_logs` LOGIN_OK como source de truth para "último login". |
| R5 | `ENABLE_VERBOSE_LOGGING` se cambia con env var → restart de uvicorn obligatorio. Si el back está reload-only con archivo, env var no se relee. | Documento "stop+start back" en S5 explícitamente. |
| R6 | Bootstrap `--only auth` puede fallar si la migración `0001_auth` se ha downgrade-ado por test_migrations_0001_auth.py (problema conocido P04-S01-T010 H3). | Si bootstrap falla con tablas inexistentes, ejecutar `cd backend && alembic upgrade head` antes y reintentar. |
| R7 | `data/verification` puede estar bajo `data/` en lugar de `backend/data/` — el comando `python -m app.verification_data.bootstrap --source data/verification` se ejecuta desde `backend/`, por lo que la ruta relativa es `data/verification` desde `backend/`. | Verificar `ls backend/data/verification` antes de ejecutar bootstrap; si no existe, usar ruta absoluta o ajustar. Tras planner probe, `data/verification/users/employee_primary.json` existe en worktree root → bootstrap probablemente espera path relativo al cwd `backend/`. |
| R8 | El path real de los logs back/front depende de cómo se invocaron los procesos. /tmp/back.log puede no existir. | Detectar primero con `ls -la /tmp/back*.log 2>/dev/null && ps aux | grep -E "uvicorn|vite"`. Si los logs no se capturaron, restart con redirección explícita: `nohup uvicorn ... > /tmp/back.log 2>&1 &`. |
