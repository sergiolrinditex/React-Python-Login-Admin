## §A. Resumen ejecutivo

Slice **foundational** de Phase 0 que entrega la maquinaria de "datos reales/proporcionados" sobre la que descansan **todos los journeys** (J100–J105) y todas las gates productivas (`/verify-slice`, `/verify-journey`, tests de integración con DB real). Construye tres piezas indivisibles:

1. **`backend/app/verification_data/`** — paquete Python con `bootstrap.py` invocable como `python -m app.verification_data.bootstrap --source data/verification [--only <group>] [--dry-run]`. Lee fixtures, los valida (Pydantic), y los carga **idempotentemente** en Postgres usando UPSERT (`INSERT ... ON CONFLICT ... DO UPDATE` o equivalente SQLAlchemy 2.x). Si las tablas destino aún no existen (porque el schema lo añadirán slices futuras), salta ese grupo con WARN y continúa — never silently skip, always log + exit-code mapping.
2. **`backend/alembic/`** — infraestructura Alembic (alembic.ini en `backend/`, `alembic/env.py`, `alembic/versions/` vacío). Sin migrations todavía (las añade P01-S01-T001). Esto permite que `dev-restart.sh --reset` ejecute `alembic downgrade base && alembic upgrade head` como ciclo de reset, incluso cuando `head == base`.
3. **`data/verification/`** — directorio de fixtures source-of-truth, con la estructura por grupos exigida en §6.5 técnico (`auth/`, `rag_chat/`, `history/`, `admin_ai/`, `rag_docs/`, `mcp_agents/`). Esta slice entrega **el contrato de fixtures + las cabeceras mínimas usadas por los grupos auth/history**. Las cargas semánticas de `rag_chat`, `admin_ai`, `rag_docs`, `mcp_agents` necesitan tablas que aún no existen y por tanto sus fixtures viven aquí (validadas y listas) pero el loader los marca `deferred_until_schema_ready` y los activará automáticamente en cuanto exista la tabla.
4. **`scripts/dev-restart.profile.sh`** — rellenar el contrato `db_reset()` que el script genérico `scripts/dev-restart.sh --reset` ya invoca. Hoy es neutral y lanza `fail`. Esta slice lo conecta a Postgres real (`docker-compose.yml` ya levanta `postgres:17-alpine`) + Alembic + bootstrap.

**Decisión arquitectónica clave** (resuelta en §E): **Opción C refinada** — loader funcional + Alembic infra vacía + fixtures completas + carga condicional por grupo (no por presencia de tabla en compile-time sino runtime, vía `inspect()` de SQLAlchemy). Esta decisión está justificada con citas literales de §15 instrucciones + §6.5 técnico + §10.3 tablas + STACK_PROFILE en §E.

## §E. Decisión arquitectónica resuelta (A/B/C)

**Decisión: Opción C refinada** — *Loader + Alembic infra vacía + fixtures completas + carga condicional por presencia runtime de tabla*.

**Cita textual que la soporta**:

1. §15 instrucciones literal: `"carga exclusivamente datos reales/proporcionados entregados por People Tech: usuarios, employee profiles, documentos People ES/EN/FR, colecciones RAG, proveedores LiteLLM de verificación, MCPs sandbox autorizados y agentes de prueba"`. El verbo **carga** implica DB. Pero §15 NO obliga a que el schema lo cree esta slice — eso es responsabilidad de las migrations futuras.
2. STACK_PROFILE.yaml literal: `migrate_cmd: alembic upgrade head`. Sin Alembic infra, este comando falla. Por tanto la slice DEBE entregar Alembic infra.
3. §6.5 técnico literal en la columna *Slices/Journeys*: la fila J100 dice `"J100, P01-S02, P03-S01"`, J101 dice `"J101, P02-S03, P02-S04, P03-S02"`, etc. Las cargas semánticas se activan en P01-S02+ y P02-S03+. P00-S02-T003 es el **frame**; las semánticas se activan progresivamente. Esto confirma que la slice actual NO debe crear las 7 tablas.
4. CHECKLIST `Step 1.1 — DB auth baseline → Auth/profile/audit migration` (línea 87) demuestra que las tablas auth (`users`, `employee_profiles`, `mfa_totp_secrets`, `refresh_tokens`, `password_reset_tokens`, `audit_logs`, `roles`, `permissions`, `user_roles`) son **P01-S01-T001**. CHECKLIST `Step 2.1 — Data model and migrations → AI/chat/RAG/MCP/agents schema` (línea 98) confirma que el resto son **P02-S01**. Si esta slice creara las 7 tablas, **invade scope de P01-S01-T001 y P02-S01**, violando DAG dependencies (`Depends on: P00-S02-T002` no incluye permiso de schema).

**Por qué NO Opción A** (solo fixtures + validador, sin DB):
- §15 verbo "carga" implica side-effect en DB.
- AC1 idempotencia no es testeable sin DB (¿idempotente en qué sentido?).
- STACK_PROFILE seed_cmd queda no-op.
- §6.5 columna "Reset/Cleanup" describe operaciones SQL ("truncate conversations/messages/citations"), incompatible con un loader sin DB.

**Por qué NO Opción B** (crear las 7 tablas aquí):
- Invade scope de P01-S01-T001 y P02-S01 (DAG dependency violation).
- Las 7 tablas tienen FKs a otras 8+ tablas no listadas en el registry (mfa_totp_secrets, refresh_tokens, audit_logs, document_versions, document_chunks, document_embeddings, ai_provider_credentials, ai_models, ai_model_tests, llm_usage_logs, mcp_credentials, mcp_tools, mcp_resources, mcp_prompts, mcp_agent_bindings, agent_runs, mcp_tool_invocations, mcp_approvals). Crear las 7 referenciadas sin sus dependencias rompe integridad referencial.
- pgvector vector(1536) requiere la extensión `vector`, cuya instalación es decisión de P01 (image switch a `pgvector/pgvector:pg17` — PROGRESS.md infra status lo confirma como "deferred to P01").

**Por qué SÍ Opción C refinada**:
- Cumple §15 (loader + carga + reset bloqueante) entregando el verbo "carga" como condicional-runtime (`if table exists → UPSERT else WARN deferred`). El día que P01-S01-T001 cree `users` y `employee_profiles`, la PRÓXIMA invocación del bootstrap **ya las carga sin tocar este código**.
- Cumple STACK_PROFILE (Alembic infra disponible, `alembic upgrade head` es no-op exit-0 hasta que P01 añada una migration).
- Respeta DAG (no invade scope de P01 o P02).
- Mantiene el contrato `dev-restart.sh --reset` funcional desde día 1.
- Es **forward-compatible**: la slice P01-S01-T001 no necesita modificar el bootstrap, solo añade migration y los grupos `auth`/`history` se activan automáticamente.

## §N. Notas para developer / validator / tester

### N.1 — Developer

- **Empieza por §G capa 1 (Alembic infra)**. Verifica `alembic upgrade head` exit 0 antes de seguir.
- **Capa 4 (fixtures) antes de capa 6 (tests)** — los tests dependen de fixtures válidas.
- **No hashees passwords en el JSON**. Los JSON contienen `password_plain: "..."`; el loader hashea al cargar usando `crypto.hash_password()`. Esto preserva idempotencia: si re-hashearas en cada run, Argon2 produce hashes distintos (salt distinto) y UPSERT actualiza siempre ⇒ trivializa AC1. **Solución**: el UPSERT debe usar `ON CONFLICT (email) DO UPDATE SET password_hash = EXCLUDED.password_hash, updated_at = now()` solo si `password_plain` cambió. **Pero**: detectar cambio sin recolar es imposible si Argon2 cambia el hash siempre. **Solución correcta**: hashear con `argon2.PasswordHasher.hash(plain)` y NO incluir `password_hash` en el SET del ON CONFLICT — solo en el INSERT branch. Para UPDATE branch, verificar `verify(stored_hash, plain)` y solo re-hashear si falla. **Implementar este patrón** para que AC1 pase.
- **Patrón P-9 planner MEMORY**: usa `restart: on-failure` y no inventes nuevos servicios — aquí ya tienes Postgres listo.
