## Approved scope (HUMAN GATE — option (a) scaffolding-only)

The user approved building the **scaffolding** for the verification bundle while People Tech delivery is still pending. Concretely:

1. **Loader CLI** `backend/app/seeds/bootstrap_verification_data.py`:
   - Module-runnable: `python -m app.seeds.bootstrap_verification_data --source <dir> [--only <ns>]`.
   - Six allowed `--only` namespaces (exact match TECHNICAL_GUIDE §6.5 column 4): `auth`, `rag_chat`, `history`, `admin_ai`, `rag_docs`, `mcp_agents`. Without `--only` ⇒ load all six.
   - **Idempotent**: re-running converges to identical state; no duplicate rows. Use upsert by natural key (`users.email`, `ai_providers.name`, `mcp_servers.name`, `agents.name`, `rag_collections.name`, `documents.title+language`).
   - **Fail-fast**: missing `data/verification/` directory ⇒ exit non-zero with clear message; missing required JSON field within a file the run actually consumes ⇒ exit non-zero pointing at file + field. Pydantic models per namespace are the validator (one schema module per namespace).
   - **Tolerant of missing tables** (this slice predates P01-S01-T001 and P02-S01-T001): when a target table does not exist, log a structured WARN with the namespace and table name, skip that namespace's writes, continue. Critically: this is NOT a free pass — JSON validation still runs against every loaded fixture before the table check, so structurally invalid data still fails. Exit code policy:
     - All consumed namespaces had every fixture validated AND either persisted OR cleanly skipped due to missing tables ⇒ exit 0.
     - Any fixture failed JSON/Pydantic validation ⇒ exit 1 with field-level error.
     - `--source` directory missing ⇒ exit 2.
     - Required namespace's bundle directory missing required files ⇒ exit 1.
   - **Logging**: every namespace BEFORE/AFTER (`structlog` `event="seed.namespace.start" namespace=auth count=N` / `event="seed.namespace.done" namespace=auth persisted=K skipped_missing_table=M`); never log secret material; hash emails via SHA-256 (first 12 chars) before logging if any email surfaces in error messages. Respects `ENABLE_VERBOSE_LOGGING` flag.

2. **`data/verification/` directory contract** (one subfolder per namespace + a top-level `MANIFEST.json` listing the namespace files):
   - `data/verification/users/employee_primary.json` — fields per §6.5 row J100 (email, full_name, password_plain_for_seed, employee_profile{...}, mfa_enabled).

### Logging

- Every CLI entry point + every loader namespace block logs BEFORE/AFTER under `ENABLE_VERBOSE_LOGGING`. Pattern matches P00-S02-T002 precedent: structured fields only. Example:
  - BEFORE: `logger.info("seed.namespace.start", namespace="auth", fixture_count=2)`
  - AFTER: `logger.info("seed.namespace.done", namespace="auth", persisted=2, skipped_missing_table=0, duration_ms=...)`
- No PII: do NOT log raw emails. Hash via SHA-256 first 12 chars when an email needs to surface in an error message.
- No secret material: even `synthetic-` credentials must be masked (e.g., `field=api_key value="synth..."`). Mask after the first 4 chars.
- WARN level (not DEBUG) for the "table missing" branch — visible at default level so operators understand why a namespace was skipped.

### Security

- **Synthetic guard**: every provider/MCP credential field MUST be prefixed `synthetic-` (`synthetic-sk-test-deadbeef`, `synthetic-mcp-token-abcd`). The loader rejects with exit 1 any credential value that does NOT start with `synthetic-`. This is defense-in-depth: when People Tech later delivers real material, that real material will land via a separate file or env-var pathway, NOT via copy-paste into the synthetic JSON. The follow-up "Replace synthetic verification bundle with People Tech delivery" will lift or relocate this guard under explicit human approval.
- **TOTP secrets**: generate per-fixture using `pyotp.random_base32()` (researcher confirms whether pyotp is already a transitive dep or needs explicit declaration). Mark in the JSON `synthetic_totp: true`.
- **No real emails**: use `@hilopeople.com` placeholders (`employee.primary@hilopeople.com`, `admin.peopletech@hilopeople.com`); document in `MANIFEST.json` that this domain is reserved for the verification bundle and never used for productive flows.
- **Encryption**: this slice does NOT encrypt provider credentials. The `ai_provider_credentials` table lands in P02-S01-T001 and P02-S02-T001 owns the encryption service. The seed loader writes plaintext synthetic placeholder credentials; this is acceptable because they're synthetic. When P02-S02-T001 lands, T003's loader will need a follow-up to encrypt-on-write — document this in the loader's `admin_ai` namespace docstring (do NOT register a formal follow-up YAML now; P02-S02-T001 will trigger it organically).
