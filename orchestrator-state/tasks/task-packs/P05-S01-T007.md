## Goal (one line — what the user sees)

F5 on `/chat` with a valid HttpOnly refresh cookie keeps the user on `/chat` after **exactly one** successful `POST /api/v1/auth/refresh` (HTTP 200, `Set-Cookie: refresh_token=…; HttpOnly; Secure; SameSite=Lax; Path=/api/v1/auth; Max-Age=2592000`). No spurious redirect to `/auth/sign-in`. No `AUTH_SESSION_EXPIRED` in the network panel under happy path.

## Bug root-cause (verified by reading code in this worktree)

The frontend has TWO independent code paths that can call `POST /api/v1/auth/refresh`, and only one of them is single-flight protected:

1. `httpClient.ts::_doRefresh()` (line 59–98) is properly de-duplicated by the module-level `let _inflight: Promise<string> | null = null`. Concurrent 401 interceptions reuse the same Promise (`auth T16` in `auth.test.tsx` already proves this).
2. `authRepository.ts::AuthRepository.refresh()` (line 707–742) uses a **raw `fetch`** to `${API_BASE}/api/v1/auth/refresh` and does **not** touch `_inflight`. This method is called by `AuthProvider.tsx::hydrate()` (line 124) on every mount.

On `/chat` reload:

- `AuthProvider` mounts → `hydrate()` → `authRepo.refresh()` → network call #1 to `/auth/refresh`.
- Roughly the same tick, the React tree (chat shell or any other authenticated query) issues a protected `authFetch` whose first response is `401` because the in-memory access token is empty post-reload → `httpClient._doRefresh()` → network call #2 to `/auth/refresh`.
- Both requests carry the **same** refresh cookie. The backend (`backend/app/auth/...` per TECHNICAL_GUIDE §10.2 + ADR P01-S02-T011) rotates the refresh token on success: the winner gets a new `Set-Cookie` and revokes the previous token; the loser arrives with the now-revoked cookie and gets `401 AUTH_SESSION_EXPIRED` → `onAuthFailure()` → `clearAccessToken()` → user redirected to `/auth/sign-in`.

Evidence: `orchestrator-state/tasks/handoffs/P05-S01-T001.md` `## verify-slice` row "/chat → F5" (line 278) marks `FU` and links the FU; `orchestrator-state/tasks/evidence/P05-S01-T001/verify-network-summary.json` + `verify-07-reload-result.png` show both POST `/auth/refresh` in the Chrome DevTools network panel.

## Recommended fix shape (Clean Arch + KISS + DRY)

Extract the single-flight registry into a small, dependency-free module that both code paths can share. This keeps Clean Architecture intact (single owner of the in-flight Promise, no presentation→data coupling) and avoids the alternative of forcing `AuthRepository` to depend on `httpClient` (which would create a circular feel since `httpClient` already imports `accessTokenStore` and `logger` and is logically "below" the repository in Clean Arch but operationally calls `/auth/refresh`).

**Option A (recommended)** — new module `frontend/src/features/auth/data/refreshSingleFlight.ts`:

- Exports `refreshAccessToken(opts: { onAuthFailure?: () => void; reason: "interceptor" | "hydrate" | "manual" }): Promise<string>`.
- Holds the module-private `let _inflight: Promise<string> | null = null`.
- Performs the actual `fetch(REFRESH_URL, { method: "POST", credentials: "include", headers: { "X-Request-ID": uuid } })`.
- On 200 → `setAccessToken(newToken)` + `logVerbose("auth.refresh.singleflight.ok", { request_id, token_len, reason })` + resolve with token.
- On non-200 → `clearAccessToken()` + `onAuthFailure?.()` + `logWarn("auth.refresh.singleflight.failed", { status, request_id, reason })` + throw `AuthSessionExpiredError`.
- On network error → `logError("auth.refresh.singleflight.network", { error, reason })` + throw mapped `NetworkError`.
- Joining caller logs `auth.refresh.singleflight.joined` with `reason`.
- Exposes `_resetSingleFlight()` for tests (mirrors existing `_resetInflight`).

Then:

- `httpClient.ts::_doRefresh()` becomes a thin wrapper: `return refreshAccessToken({ onAuthFailure, reason: "interceptor" });`. The local `_inflight` and the raw `fetch` are deleted; `_resetInflight()` becomes a re-export of `_resetSingleFlight()` for back-compat with existing tests (or is renamed; pick whichever keeps the test diff smallest).
- `authRepository.ts::AuthRepository.refresh()` becomes a thin wrapper that calls `refreshAccessToken({ onAuthFailure: this._onAuthFailure, reason: "hydrate" })` and converts the `Promise<string>` / throw into the existing `Result<string>` envelope. The raw `fetch` block (lines 708–741) is replaced. The `AuthSessionExpiredError` branch and the logging contract are preserved.
- `AuthProvider.tsx::hydrate()` (line 124) does NOT change — it keeps calling `repo.refresh()`. The de-duplication is now invisible from above.

**Option B (rejected, documented for traceability)** — make `AuthRepository.refresh()` call `authFetch(REFRESH_URL, { method:"POST", __authNoRetry:true })`. Rejected because (a) `authFetch` injects a Bearer header by default — `_shouldInjectBearer` must be honored and the public refresh endpoint is in `NO_AUTH_PREFIXES`, so behavior is technically fine, but (b) it forces `authRepository` (data layer) to depend on `httpClient` (data layer sibling) for its own primary endpoint, creating a subtle coupling and an indirection that hides the single-flight, and (c) it doesn't actually de-duplicate the cold-mount race because `authFetch` only routes through `_doRefresh` on a 401 interceptor path, not on the explicit `/auth/refresh` POST. Reject and use A.

## Risks (and how to mitigate)

- **R1 — Regression on `/auth/refresh` failure**: If the new module's error path forgets to call `onAuthFailure()`, sessions silently linger as `hydrating`. Mitigation: explicit unit test on the 401-from-refresh path; reuse the existing `auth T17` shape.
- **R2 — Concurrent logout race**: If a user clicks logout while a refresh is in-flight, the in-flight Promise may still resolve and call `setAccessToken()` after `clearAccessToken()`. Mitigation: keep the existing post-await guard in `_doRefresh` (`onAuthFailure` callback wired to `_onAuthFailure` of `AuthRepository` which goes through `AuthProvider.onAuthFailure` → `clearAccessToken+setStatus("unauthenticated")`). Add an assertion in the test: if `clearAccessToken()` is called during the refresh window, the post-resolve `setAccessToken` is acceptable but the subsequent `getAccessToken()` returns `null` again because the React effect path will not consume it (status is already unauthenticated). If the developer detects a real ordering bug, treat as `needs_debug`, not FU.
- **R3 — Test ordering**: `_inflight` is module-private state; tests must `_resetSingleFlight()` (or equivalent) in `beforeEach`/`afterEach`. The existing tests already do this for `_resetInflight`. Mitigation: keep symmetric API.
- **R4 — Logging key rename breaks log analysis**: Existing slices reference `auth.http.refresh.*` and `auth.repo.refresh.*`. The validator may flag missing keys. Mitigation: prefer **adding** the new `auth.refresh.singleflight.*` keys at the new module, and keep one outer log line at each old call site for back-compat (`auth.http.refresh.delegated`, `auth.repo.refresh.delegated`).
- **R5 — `Result<string>` vs throw**: The new module throws; `AuthRepository.refresh()` returns `Result<string>`. Mitigation: thin try/catch wrapper in the repository that maps `AuthSessionExpiredError` → `{ok:false, error: AuthSessionExpiredError}` and `NetworkError` → `{ok:false, error: NetworkError}`. Keep the public IAuthRepository contract intact.
- **R6 — `useEffect` strict-mode double-invoke**: React 18 dev StrictMode mounts effects twice. The current `cancelled` flag in `AuthProvider` (line 119/126/166) handles that. The new single-flight will additionally make the second invocation a no-op network-wise even outside StrictMode. Net positive; no mitigation needed.
