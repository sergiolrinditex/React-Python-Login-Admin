1. `GET /health` sigue respondiendo `{data:{status:"ok",version,uptime}}` HTTP 200 — backward compatible con stub T001; los 4 tests previos deben seguir verdes.

3. `GET /ready` responde `{data:{db:<state>,redis:<state>,litellm:<state>}}`:

   - HTTP 503 con el mismo envelope si DB o Redis fallan; el campo del componente roto debe contener `status == "error"` y un `error` string corto SIN PII/credenciales.

### Driver Postgres — DECISIÓN ABIERTA (researcher hook)

`requirements.txt` NO pinea `asyncpg` ni `psycopg[binary]`. SQLAlchemy 2.x exige un driver real para `create_engine("postgresql+psycopg://…")` o `create_async_engine("postgresql+asyncpg://…")`. Sin driver pinned, el `/ready` endpoint puede:

- (a) Conectarse vía `sqlalchemy.create_engine` + driver default (psycopg2 binary si está disponible transitiv vía alguna dep, lo cual **NO está garantizado**).
- (b) Conectarse "raw" vía `socket` (anti-patrón, frágil).
- (c) **Recomendado**: declarar `psycopg[binary]==3.x` como write_set extension en `backend/requirements.txt`. Documentar como extensión justificada del write_set canónico de T002 — patrón ya usado en T001/T003 (validator aprobado).

El developer debe RESOLVER esto en su primera lectura del pack:
- Opción default: añadir `psycopg[binary]` (versión exacta a confirmar por researcher; PyPI actual `psycopg[binary]==3.2.x`) y usar sync SQLAlchemy (`from sqlalchemy import create_engine, text`).
- Async (`asyncpg`) es alternativa pero introduce complejidad async-engine que no aporta valor en una slice scaffold.

**Researcher hook**: pin exact `psycopg[binary]` version compatible con `sqlalchemy==2.0.49` y `python:3.12-slim-bookworm` (compose backend Dockerfile base). Anotar en `orchestrator-state/memory/official-doc-notes/P00-S02-T002-*.md` si discrepa.

def test_ready_success(monkeypatch):
    fake_engine = ...  # devuelve engine cuya conexión ejecuta SELECT 1 exitosamente
    fake_redis = ...   # cliente cuyo .ping() devuelve True

### Contratos a diseñar / decisiones de implementación

1. **¿`/health` se queda en `main.py` o se migra a `api/router.py`?**
   - **Recomendación planner**: migrar al router para evitar dos sitios con la misma route. `main.py` queda lean (solo crea `app` y monta routers). Esto es buen Clean Architecture y prepara para que P01 monte `/api/v1/auth/*` con el mismo pattern.
   - Si el developer prefiere preservar `main.py:55-83` por mínima diff, es aceptable mientras los 3 probes terminen siendo descubrible y consistentes.

2. **Engine SQLAlchemy lifecycle**:
   - Crear un engine global a nivel de módulo en `api/router.py` (o exportado por `app/core/db.py` — pero ese módulo no existe en write_set; **mantener el engine inline en router.py** evita expansion). `engine = create_engine(os.getenv("DATABASE_URL"), pool_pre_ping=True, pool_size=2)`. `pool_pre_ping=True` evita falsos negativos por conexión muerta tras tiempo idle.
   - El engine NUNCA se cierra explícitamente en handlers — FastAPI lo deja vivo durante el lifecycle. SQLAlchemy gestiona el pool.

3. **Redis client lifecycle**:
   - `redis.Redis.from_url(os.getenv("REDIS_URL"), decode_responses=True, socket_timeout=2, socket_connect_timeout=2)`. Timeout corto para que un Redis caído no cuelgue el probe.

4. **Timeout de ping**: ambos pings con timeout ≤ 2s. Total handler ≤ 5s. Si excede, el probe debe devolver 503 con `error: "timeout"` (sin stack trace al cliente).

5. **Cuerpo del response `/ready`** (planner-spec):
   ```json
   {
     "data": {
       "db": {"status": "ok"},
       "redis": {"status": "ok"},
       "litellm": {"status": "unknown"}
     }
   }
   ```
   o en degraded:
   ```json
   {
     "data": {
       "db": {"status": "error", "error": "connection refused"},
       "redis": {"status": "ok"},
       "litellm": {"status": "unknown"}
     }
   }
   ```
   `litellm.status: "unknown"` por ahora — `httpx` no está en runtime deps; la fila §6.2 declara que `/ready` solo ejecuta "DB/Redis ping" (no LiteLLM). Si en el futuro se quiere LiteLLM ping, requiere `httpx` runtime — fuera de scope de T002.
