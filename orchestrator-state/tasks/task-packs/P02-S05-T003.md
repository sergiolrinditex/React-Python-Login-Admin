  - **Planner recommendation: U.1a**, mirrors `app/auth/repository.py:131` precedent for `EmailAlreadyExistsError` and respects Clean Architecture. Developer may pick a smaller variant (raise generic `IntegrityError` from service and let router catch) if it keeps file sizes / cognitive complexity lower — document choice in handoff §D-SVC-409.

- **U.2 — Concurrent-PATCH test mechanism**. TestClient is synchronous; to drive a true race the test must spawn two threads or two processes hitting the live ASGI transport. Two viable patterns:
  - **U.2a** `concurrent.futures.ThreadPoolExecutor` with 2 workers, each calling a separate `TestClient` instance against the same FastAPI app. SQLAlchemy `Session` is bound per-request via `Depends(get_db_session)`, so two requests genuinely race at the DB. Race window: between `repo.find_previous_default_id` and the final `session.commit()`. The partial unique index makes the second commit fail. (Precedent: `test_auth_refresh.py` race test pattern — see `T08` in P01-S02-T003 handoff for linearizability discussion.)
  - **U.2b** Direct DB-level test: bypass the endpoint and assert that two raw `INSERT INTO ai_models … is_default=true` statements (or two `UPDATE`s in two sessions committing simultaneously) raise `UniqueViolation`. This is simpler but does not exercise the endpoint envelope. Less valuable.
  - **Planner recommendation: U.2a** (endpoint-level race) is the canonical acceptance-test. Add U.2b as a complementary DB-level invariant test (small, fast, deterministic) — optional. Developer may simulate the race by manually transitioning model_A.is_default=true via direct SQL (committed), then issuing a single PATCH on model_B with `{is_default:true}` AND **deferring the UPDATE that clears model_A** — but the canonical app-layer flow already clears the previous default in the same transaction, so the race only manifests under TRUE concurrency. Keep U.2a as the primary mechanism.

### I.5 Risks

- **R1 (data tie-break) — RESOLVED**: confirmed `data/verification` does NOT seed any `ai_models` rows (let alone with `is_default=true`). Migration `0003` upgrade will not fail on preexisting violators. If a developer's local DB has manually-inserted `is_default=true` duplicates from earlier exploratory testing, the migration's `CREATE UNIQUE INDEX` will fail with `23505 unique_violation`. **Recovery procedure** (document in migration docstring): `DELETE FROM ai_models WHERE ctid NOT IN (SELECT MIN(ctid) FROM ai_models WHERE is_default=true GROUP BY model_type)` is NOT in the migration body (deterministic data deletion is not appropriate at infra level); instead, the migration fails fast with a clear log line and the developer cleans up manually. Acceptable for a dev-only situation in P02; would need a different strategy for prod (none in this project today).
- **R2 (error contract regression)**: the PATCH endpoint currently returns `500 INTERNAL_ERROR` on any unhandled exception (`router.py:177-183`). Without the new catch, an IntegrityError would surface as 500. The 409 branch must take precedence over the catch-all. Risk: a debugger / future refactor could swap the order and re-introduce 500. Mitigation: include T26 assertion that the failure path returns **exactly** `409 AI_MODEL_DEFAULT_CONFLICT`, never 500.
- **R3 (logging hygiene)**: the IntegrityError's `exc.orig.args` may contain SQL fragments. Do NOT log `exc.orig.args` verbatim in production; log only `constraint_name='ai_models_default_per_type_uidx'` + `pgcode='23505'`. Verify by tracing both verbose modes in tester evidence.
- **R4 (regression on T01..T25)**: the existing 25 tests must all still pass. Specific risk on T18 (multi-model defaults flow): the existing service path issues `UPDATE … SET is_default=false WHERE id != target` THEN sets `target.is_default=true`. Inside a single transaction, Postgres defers partial-unique-index check until commit (default behavior — index is enforced at row write, BUT the previous default's row is updated to `false` before the target's row is updated to `true` in the same tx, so at no point are there two `is_default=true` rows for the same `model_type` in the same tx). **Verify by inspection** that `apply_patch` (`repository.py:152-160`) executes the clearing UPDATE BEFORE the attribute assignment on `model` — confirmed in code. T18 will pass unchanged.
- **R5 (alembic env target_metadata)**: `alembic/env.py:62` uses `target_metadata = Base.metadata`. A future `alembic revision --autogenerate` will compare ORM metadata vs DB. If §I.2 §D-ORM-INDEX is NOT applied (the ORM does not declare the index), autogenerate may suggest "drop index ai_models_default_per_type_uidx" in a future migration. Mitigation: either apply §D-ORM-INDEX OR add `compare_indexes=False` for this specific index in `env.py` (heavier, NOT recommended). **Recommendation: apply §D-ORM-INDEX** to keep parity.
