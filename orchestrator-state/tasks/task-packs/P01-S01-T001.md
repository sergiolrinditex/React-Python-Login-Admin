## 4. Plan de migración — tabla por tabla

Notas globales:
- Dialecto target: PostgreSQL 17 (Postgres 17-alpine en compose).
- Tipo PK: `UUID` con default `gen_random_uuid()` (pgcrypto). En SQLAlchemy: `sa.Column(UUID(as_uuid=True), primary_key=True, server_default=sa.text("gen_random_uuid()"))` con `from sqlalchemy.dialects.postgresql import UUID, JSONB, TIMESTAMP`.
- Timestamps: `TIMESTAMPTZ` con `server_default=sa.text("now()")` para `created_at`. `updated_at` similar; la actualización en escritura es responsabilidad de la aplicación (P01-S02-T007). No se añaden triggers en este schema.
- Nombre de constraints/indices: explícito y predecible (`{table}_{col}_key|chk|idx|fk`) para que `alembic downgrade` los borre por nombre y no por inferencia.
- Logging Alembic: ya configurado en `env.py` (BEFORE/AFTER al inicio/fin de migration).

### 4.1 Extensión PostgreSQL

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

`gen_random_uuid()` lo provee pgcrypto. **NO se crea `vector`** aquí — vector pertenece a P02-S01-T001
(la migración AI/chat/RAG/MCP). Razón: §10.3 declara ambas extensiones al inicio del bloque SQL,
pero `vector` no es referenciada por NINGUNA tabla de este migration; introducirla aquí violaría
"una sola responsabilidad por migración" y ataría el rollback de P01 al rollback de la extensión que
luego usa P02. Decisión documentada en handoff §K.

En `downgrade()` **NO se hace `DROP EXTENSION pgcrypto`**: la extensión puede ser usada por otras
piezas (futuras), y dejar `IF EXISTS` no es destructivo. Es práctica estándar Alembic dejar las
extensiones, solo se borran si pertenecen 100% a esta migración. Decisión documentada en handoff §K.

## 5. Orden upgrade() y downgrade()

`upgrade()` debe respetar FKs (tablas hijas después de padres):

1. `CREATE EXTENSION IF NOT EXISTS pgcrypto` (idempotente).
2. `users`
3. `employee_profiles` (FK → users)
4. `roles`
5. `permissions`
6. `user_roles` (FK → users + roles)
7. `refresh_tokens` (FK → users)
8. `mfa_totp_secrets` (FK → users)
9. `password_reset_tokens` (FK → users)
10. `audit_logs` (FK → users, ON DELETE SET NULL)

`downgrade()` orden inverso (hijas antes de padres):

1. `audit_logs`
2. `password_reset_tokens`
3. `mfa_totp_secrets`
4. `refresh_tokens`
5. `user_roles`
6. `permissions`
7. `roles`
8. `employee_profiles`
9. `users`
10. (NO `DROP EXTENSION pgcrypto`)

Cada `op.drop_table(...)` debe ir precedido por `op.drop_constraint`/`op.drop_index` solo si Alembic
no los borra al hacer `DROP TABLE ... CASCADE`. Práctica recomendada: declarar constraints/indices con
nombres explícitos en `upgrade()`, dejar que `op.drop_table` haga CASCADE implícito en `downgrade()`
(comportamiento Postgres). Probar **ambos sentidos** con el verify command oficial:

```bash
alembic upgrade head && alembic downgrade -1 && alembic upgrade head
```

### 6.3 `Base` declarativo

`Base = declarative_base()` (estilo SQLAlchemy 2.x: `class Base(DeclarativeBase): pass`) debe vivir en
un módulo compartido para que `alembic/env.py` pueda hacer `from app.db.base import Base`.

**Decisión recomendada**: crear `backend/app/db/base.py` con la clase `Base`. Esto NO está en el
`write_set` literal de la slice. Se declara como **write_set extension justificada** en el handoff §I:
sin `base.py` no hay sitio canónico para `Base`; importar `Base` desde `user.py` o `auth.py` crea
ciclos de imports si ambos lo importan del otro. Precedente reutilizable: P00-S01-T001 y P00-S01-T003
también crearon files fuera del write_set literal por necesidad estructural (PROGRESS.md
"Recent Decisions" 2026-05-11). Validator approved en ambos casos.

**Alternativa rechazada**: declarar `Base` en `user.py` e importar desde `auth.py`. Crea acoplamiento
"auth depende de user" que no refleja la realidad (ambos son peers del bounded context auth).

`backend/app/db/__init__.py` se crea como marker package vacío con docstring (precedente
`backend/app/core/__init__.py` en P00-S01-T003). También write_set extension justificada.

### 6.4 Path final del módulo `db`

```text
backend/app/db/
  __init__.py            # package marker (write_set extension)
  base.py                # Base = DeclarativeBase  (write_set extension)
  models/
    __init__.py          # exporta todos los modelos (write_set extension)
    user.py              # canonical write_set
    auth.py              # canonical write_set
```

Las extensions están justificadas por necesidad estructural (no por feature creep). Validator debe
aprobarlas vía handoff §I.

### 6.5 Modificación de `alembic/env.py`

Cambiar la línea 51 de:

a:

## 9. Impact analysis — riesgos y decisiones

### 9.1 Riesgos visibles

- **R1 (medium)** — Modelos SQLAlchemy 2.x sintaxis: usar `class Base(DeclarativeBase): pass` y
  `mapped_column(...)` (no `Column(...)` legacy). Si el developer cae en patrón 1.4, mypy y validator
  alzarán bandera. Linkear `https://docs.sqlalchemy.org/en/20/orm/declarative_styles.html` en handoff.
- **R2 (medium)** — `gen_random_uuid()` requiere extensión pgcrypto. Si la extensión no se crea en
  upgrade, la primera fila falla. Test: `alembic upgrade head` debe levantar las 9 tablas + insertar
  una fila ficticia (developer puede dejar test integration que insert+rollback).
- **R3 (low)** — `JSONB DEFAULT '{}'::jsonb` en SQLAlchemy: usar
  `sa.Column(sa.dialects.postgresql.JSONB, nullable=False, server_default=sa.text("'{}'::jsonb"))`.
  El cast `::jsonb` es importante; sin él, Postgres infiere `text`.
- **R4 (medium)** — `users.preferred_language CHECK` en Alembic: usar
  `sa.CheckConstraint("preferred_language IN ('es','en','fr')", name="users_language_chk")` dentro
  de `op.create_table`, NO añadirlo después como `op.create_check_constraint` (mejor legibilidad).
- **R5 (medium)** — `target_metadata` ya `None` en `env.py` con instrucción explícita "P01-S01-T001
  must replace". El cambio de UNA línea sí es necesario; declararlo en handoff §I como write_set
  extension justificada y NO oculta.
- **R6 (low)** — Si `verification_data` bootstrap se ejecuta sobre la DB recién migrada y los datos
  reales/proporcionados ya están cargados de antes (estado contaminado de tests previos), las
  UPSERT idempotentes manejan el caso. NO contaminar el verify command oficial con seed; ejecutar
  seed solo en evidencia adicional.
- **R7 (low)** — `audit_logs.actor_user_id` ON DELETE SET NULL: requiere que la columna sea NULLable.
  Es así por contrato §10.3. No cambiar.
- **R8 (medium)** — `refresh_tokens.user_id` y `password_reset_tokens.user_id` NOT NULL es tighter
  than §10.3 raw DDL. Validador debe aprobar. Si lo rechaza, relajar a NULL pero deja al app gestionar.

### 9.2 Decisiones del planner

| ID | Decisión | Motivo |
|---|---|---|
| D1 | **NO** crear extensión `vector` aquí | No referenciada por ninguna tabla de esta migración; pertenece a P02-S01-T001 |
| D2 | **NO** drop `pgcrypto` en downgrade | Práctica estándar; idempotencia inocua si la extensión queda |
| D3 | Reparto modelos: identity (user.py) + sesión/audit (auth.py) | Bounded context claro; auth_logs es traza de operaciones de identidad → encaja en auth.py |
| D4 | Crear `app/db/base.py` para `Base` (write_set extension justificada) | Evita ciclo user↔auth; precedente P00-S01-T001/T003 |
| D5 | `alembic/env.py` single-line change (write_set extension declarada) | Predicho por el propio file; sin esto no hay autogen ni metadata link |
| D6 | `refresh_tokens.user_id` y `password_reset_tokens.user_id` declarar `NOT NULL` | Tighter than §10.3; sin user no hay token; validador debe aprobar |
| D7 | Índices recomendados: `refresh_tokens(token_hash)`, `refresh_tokens(user_id, revoked_at)`, `password_reset_tokens(token_hash)`, `user_roles(role_id)`, `audit_logs(actor_user_id, created_at DESC)`, `audit_logs(entity_type, entity_id)`, `audit_logs(created_at DESC)` | Patrones de query críticos para P01-S02 y P04-S03 |
| D8 | NO añadir índice `employee_profiles(country)` | YAGNI hasta que P04 admin lo requiera |
| D9 | NO seed inicial de roles aquí | Pertenece al verification_data loader |
| D10 | NO trigger `updated_at = now()` en SQL; la app lo controla (P01-S02-T007) | Mantener DB simple; trigger es complejidad oculta |

### 9.3 Unknowns

- **U1**: ¿Nombres canónicos de los roles iniciales? `admin|auditor|user` es la asunción razonable
  de §10.2 guards. Si verification_data loader (P00-S02-T003) ya seedea otros nombres, alinear.
  **No bloquea T001** (el roles seed no es parte de esta slice), pero será relevante para P01-S02.
- **U2**: ¿Falta `index` explícito en `users.email` además del UNIQUE? Postgres crea uno implícito
  con cada UNIQUE; documentar y no duplicar.
- **U3**: ¿Convención de nombres de constraints? Recomendación: PostgreSQL default
  (`<table>_<column>_key|chk|fkey|pkey`) declarados explícitamente. Si SQLAlchemy declara con
  `naming_convention` en `MetaData(...)`, declararlo en `base.py` y dejar que Alembic los herede.
  Decisión: usar `MetaData(naming_convention=...)` estándar Alembic
  (`https://alembic.sqlalchemy.org/en/latest/naming.html`). Documentar en handoff §B.
