## 7. Migration design notes (binding for developer)

### 7.1 Naming convention (set on `Base.metadata`)

Use the SQLAlchemy 2.x recommended naming convention so Alembic autogenerate produces stable indexed/constraint names across runs:

```python
NAMING_CONVENTION = {
    "ix": "ix_%(table_name)s_%(column_0_name)s",
    "uq": "uq_%(table_name)s_%(column_0_name)s",
    "ck": "ck_%(table_name)s_%(constraint_name)s",
    "fk": "fk_%(table_name)s_%(column_0_name)s_%(referred_table_name)s",
    "pk": "pk_%(table_name)s",
}
```

### 7.2 Indexes — required by `01-non-negotiables.md §Database`

> Indexes on every field used in WHERE / ORDER BY / JOIN.

For the auth baseline:

| Table | Column(s) | Reason | Notes |
|---|---|---|---|
| `users` | `email` | login lookup | unique constraint already covers it (PG creates an index automatically) |
| `employee_profiles` | `employee_id` | unique lookup | unique constraint covers it |
| `user_roles` | `(user_id, role_id)` | composite PK | PK covers both join directions |
| `user_roles` | `role_id` | reverse join (roles → users) | **add explicit `CREATE INDEX user_roles_role_id_idx`** — reverse FK lookup has no auto-index in PG |
| `refresh_tokens` | `user_id` | per-user enumeration | **add `refresh_tokens_user_id_idx`** |
| `refresh_tokens` | `expires_at` (partial: WHERE revoked_at IS NULL) | janitor sweep | **add `refresh_tokens_active_expires_idx`** |
| `mfa_totp_secrets` | `user_id` | PK, covered |
| `password_reset_tokens` | `user_id` | per-user | **add `password_reset_tokens_user_id_idx`** |
| `password_reset_tokens` | `expires_at` | janitor sweep | **add `password_reset_tokens_expires_idx`** |
| `audit_logs` | `(actor_user_id, created_at DESC)` | "show me what user X did, newest first" — required by §3.1 audit rule | **add `audit_logs_actor_created_idx`** |
| `audit_logs` | `created_at DESC` | global newest-first | **add `audit_logs_created_idx`** |
| `audit_logs` | `(entity_type, entity_id)` | "what happened to entity X" | **add `audit_logs_entity_idx`** |

These are explicit `op.create_index(...)` calls in `upgrade()`. Mirror with `op.drop_index(...)` in `downgrade()` BEFORE `op.drop_table()`.

### 7.3 `downgrade()` discipline

Reverse order of creation. Drop indexes → drop tables in dependency order (children first: `audit_logs`, `password_reset_tokens`, `mfa_totp_secrets`, `refresh_tokens`, `user_roles`, `permissions`, `roles`, `employee_profiles`, `users`). Do **not** drop the `pgcrypto` or `vector` extensions (other tools may rely on them; CREATE EXTENSION IF NOT EXISTS is idempotent on upgrade).

### 7.4 What does NOT belong in this migration

- Seeded rows (no `INSERT`s for default roles/permissions). Seed responsibility is P00-S02-T003 + future role-seed slice.
- Argon2 hashing helpers — service code, P01-S02.
- Encryption-at-rest for `secret_encrypted` / `password_hash` — service code, P02-S02-T001. The migration only declares the columns as TEXT.
- Updated-at triggers. The columns default to `now()` at insert; per-row update is application-managed in SQLAlchemy via `onupdate=func.now()`. Postgres triggers add ops debt; only add later if a use case requires it.
- Audit-log enrichment with `ip/user_agent/request_id` — see D1.

### 7.5 Alembic env.py contract

- Async env (template `alembic init -t async backend/alembic` style, but configured to import `app.core.db.get_engine()` so the URL is consistent with the FastAPI app).
- `target_metadata = Base.metadata` (imported from `app.db.models`).
- `compare_type = True`, `compare_server_default = True` for autogen safety in future migrations.
- Read DATABASE_URL from `get_settings()` rather than from `alembic.ini` `sqlalchemy.url` (keeps secret in Pydantic settings, not the ini). Leave the ini's `sqlalchemy.url = ` blank.
- Logging: keep Alembic's `[loggers]` block from the template (it logs to stderr by default; structlog binds via the FastAPI side).

## 13. Risks and watch-outs

| # | Risk | Mitigation |
|---|---|---|
| R1 | Developer "improves" the §10.3 schema (adds columns the technical guide does not specify, e.g. `users.last_login_at`) | Validator must reject any column not in §10.3. Speculative columns belong in P01-S02 (or as an explicit follow-up to amend §10.3 first). YAGNI. |
| R2 | Async vs sync Alembic env confusion | Use `alembic init -t async` template and call `app.core.db.get_engine()`. The Alembic 1.18 docs explicitly support `run_async_migrations` pattern; `official-docs-researcher` should confirm the current pattern (5s cache hit if previously cached). |
| R3 | `pgvector` extension creation collides with image's pre-installed pgvector | `CREATE EXTENSION IF NOT EXISTS vector;` is idempotent — safe. |
| R4 | Downgrade leaves `alembic_version` table | This is correct Alembic behavior; tester assertion: only `alembic_version` survives in `public` after downgrade. |
| R5 | Naming convention not set on `Base.metadata` until after migration is generated | Set the convention BEFORE the first model is declared; otherwise autogenerated FKs/indexes get random names. The first migration must include the convention from the start. |
| R6 | FU-20260508225027 promotion happens during this slice | Out of scope — T001 only consumes `DATABASE_URL` (matches §11.1 already). Do NOT touch `config.py`/`.env.example`. |
| R7 | Developer creates `data/verification` schema migration "to make seed loader green" | Loader is data, not schema. Do not invent a `verification_*` table. The loader's transition from skip→upsert is a runtime behavior, not a schema change. |
| R8 | Test isolation between `test_auth_migration.py` runs | Use a per-test transaction-rollback or `alembic downgrade base` in fixture teardown. Do not let test #2 see test #1's tables. |
| R9 | psql/PG18 column-shape edge case (`gen_random_uuid()` requires pgcrypto in some images, native in pg18) | pg18 ships `gen_random_uuid()` natively, but the §10.3 baseline declares `CREATE EXTENSION IF NOT EXISTS pgcrypto;` for backward compat with pg<13. Keep it — idempotent. |
| R10 | Researcher discrepancy on Alembic 1.18 async pattern | Official-docs-researcher runs in parallel. If a discrepancy lands in `orchestrator-state/memory/official-doc-notes/` mid-implementation, developer pauses, reads it, RESOLVED-tags it, then continues. |
