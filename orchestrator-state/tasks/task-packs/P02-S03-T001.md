## G. Pagination contract (decisión técnica)

**No hay precedente** de pagination en el codebase (auth/users no listan; chat es el primero). El planner propone:

- **Cursor opaco**: `base64url(<ISO-8601 updated_at>|<uuid id>)`. Ej `MjAyNi0wNS0xM1QxMjozNDo1Ni43ODlaLTQ3NzM3M2EyLTEyMzQtNDU2Ny04OWFiLWNkZWYwMTIzNDU2Nw==`.
- **Orden**: `ORDER BY conversations.updated_at DESC, conversations.id DESC` (tiebreaker estable por PK). El índice `conversations_user_updated_idx` cubre el path para `WHERE user_id = ?`.
- **WHERE clause cursor**: `(updated_at, id) < (cursor_updated_at, cursor_id)` decoded.
- **Query params**: `cursor: str | None`, `limit: int = Query(default=20, ge=1, le=100)`.
- **Response meta**: `{pagination: {next_cursor: <base64 del último row si has_more else null>, has_more: bool}}`.
- **Algoritmo**: pedir `limit+1` rows, si llegan `limit+1` → `has_more=True` y descartar el extra; `next_cursor` se construye a partir del último de los `limit` devueltos.
- **Edge cases que el developer DEBE testear**:
  - cursor inválido base64 → 400 `CHAT_CURSOR_INVALID`.
  - cursor decodable pero apunta a UUID inexistente → no es error, simplemente no matcheará y devolverá página vacía con `has_more=false`.
  - `limit=0` o `>100` → 400 (FastAPI Query ge/le ya lo hace).
  - lista vacía → `{data:[], meta:{pagination:{next_cursor:null, has_more:false}, request_id}}`.

Decisión documentada como **D-PAG1** en el handoff. Si el developer prefiere offset pagination, el planner pushback: cursor es estándar para feeds time-ordered (Twitter/GitHub patterns) y evita issues con inserts concurrentes. KISS dice cursor también, porque offset requiere COUNT(*) para `total`.

## H. Decisiones técnicas pendientes para el developer

### H.1 — Título auto-generado para POST sin `initial_message`

**Pregunta**: ¿Qué `title` se persiste cuando el cliente no envía `initial_message`?

**Propuesta planner**:
- Si `initial_message` presente: `title = initial_message[:60].strip()` (truncado a 60 chars; suficiente para mostrar en HistoryPage).
- Si `initial_message` ausente: `title = "Nueva conversación"` literal en castellano. **No usar i18n del backend** — el backend no tiene `t()` runtime; la traducción es responsabilidad del frontend. El cliente puede traducir el title si detecta el literal (anti-pattern: mejor que el frontend lo detecte si NULL/empty title y muestre placeholder traducido). 
- Alternativa más limpia: `title` opcional en POST request y, si ausente sin initial_message, persistir `title = ""` empty string; el frontend (HistoryPage P03-S02-T003) lo renderiza como `t('chat.untitled')`. RECOMENDADA porque mantiene backend i18n-agnostic.

**Decisión final por developer** documentada como **D-TIT1**. Tests deben cubrir ambos paths.

### H.2 — Language default cuando no se envía

**Propuesta**: usar `current_user.preferred_language` (coherente con J102 y con la columna `users.preferred_language`). Si el campo `language` viene en el request, usar ese valor (override explícito). Si no, fallback a preferred_language del usuario actual. Documentado como **D-LANG1**.

### H.3 — Transacción atómica POST con initial_message

**Propuesta**:
```python
with session.begin():  # o usar la sesión transaccional ya provista por get_db_session
    conv = Conversation(user_id=..., title=..., language=...)
    session.add(conv)
    session.flush()  # para tener conv.id antes del Message
    if initial_message:
        msg = Message(conversation_id=conv.id, role='user', content=initial_message, token_count=None)
        session.add(msg)
    session.commit()
```
- Sin assistant message en el POST (eso pertenece al stream T002).
- `updated_at` se actualiza desde `now()` en SQL (server_default ya cubre create, pero al añadir el user message la app DEBE setear `conv.updated_at = now()` para que aparezca arriba en la lista — patrón explicado en el comentario ORM `Conversation.updated_at`).

Documentado como **D-TX1**.

### H.6 — Logging structlog (§8)

Cada función debe loggear BEFORE + AFTER con campos:
- `request_id` (ya extraído por `_get_request_id`).
- `user_id_hash`: usar `hashlib.sha256(str(user.id).encode()).hexdigest()[:16]`. **NUNCA loggear UUID en claro**. Patrón Phase-1 audit.
- `route`: `/api/v1/chat/conversations` o `/api/v1/chat/conversations/{id}`.
- `latency_ms`: medir con `time.perf_counter()` en el router.
- `status_code`: el devuelto por la response.
- NUNCA loggear `content`, `initial_message`, `title`, `language` completos. Solo metadatos (longitud, presencia).

Verbose-off (`ENABLE_VERBOSE_LOGGING=false`): zero `app.chat.*.*` lines on success/4xx. Solo errors no-esperados (5xx) loggean WARNING/ERROR. Documentar como **D-LOG1**.

## M. Riesgos y decisiones previas

### M.1 Riesgos

- **R1 — ResponseMeta extension**: si developer elige opción (b) (ChatResponseMeta local), el frontend tendrá DOS envelope shapes coexistiendo. Recomendado (a). Si validator pushback (b), aceptar.
- **R2 — Cursor pagination correctness**: tiebreaker por UUID es necesario porque `updated_at` puede repetirse con creates concurrentes. Sin tiebreaker, paginación pierde rows o duplica. Test T08 lo cubre.
- **R3 — `user_id NULL` en conversations.user_id**: la ORM permite NULL (`nullable=True` para "anonymous/system conversations"). En esta slice, current_user.id NUNCA es NULL porque get_current_user rechaza unauthenticated. Pero la query `WHERE user_id = ?` debe usar `==` no `is_(...)`. Documentar.
- **R4 — `messages.role` no tiene CHECK constraint**: el ORM dice "enforced at app layer". En T001 los únicos values posibles son `user` (POST con initial_message) y `assistant`/`system` (los crea T002 stream). Para esta slice, hardcode `role='user'` en `create_conversation_with_optional_message`. Documentar.
- **R5 — Conflict group `api:chat`**: T002 (stream) está en mismo grupo. Si alguien abre paralelamente T002 (depende de T001+T004+T005 que NO están done) sería bloqueado por DAG. Verified: T002 no es ready.
