```
| POST | /api/v1/auth/refresh | cookie refresh | {data:{access_token}} | Refresh cookie | 401 | authStore | refresh_tokens | P01-S02-T003 |
```

Request: incoming **cookie `refresh_token`** (set by /sign-in). No JSON body.
Response (success): `{data:{access_token}, meta:{request_id}}`. **No body refresh_token** — the rotated refresh stays in cookie only.
Auth: cookie-only (no Authorization header required — the refresh cookie IS the credential).
Errors: **401** (the only listed code). Map to envelope `code = AUTH_SESSION_EXPIRED` (§6.4 catalog) for missing/expired/revoked/unknown refresh, body byte-identical to prevent token-state enumeration. Other errors:
- Rate limit → 429 with own code `AUTH_REFRESH_RATE_LIMITED` (extension following T002 precedent for SIGNIN; AUTH-* taxonomy in §6.4 is non-exhaustive — same precedent that minted AUTH_SIGNIN_RATE_LIMITED in T002).
- No 400 in §6.2 — Pydantic would not fire because there's no JSON body to validate. The "no cookie" case maps to 401.

### D.1 Front (n/a in this slice — context only)

- Future consumer: `authStore` (P01-S03-T001). Expected client flow: when an access token returns 401, call POST /auth/refresh; if 401, redirect to /auth/sign-in. The browser sends the cookie automatically because Path=/auth covers /auth/refresh.

### D.2 Back — endpoint contract (consolidated)

| Aspect | Contract |
|---|---|
| Method + path | `POST /api/v1/auth/refresh` (mounted at `auth_router` prefix `/auth` + `/api/v1` from `main.py`) |
| Auth | cookie `refresh_token` (NO Authorization header) |
| Request body | empty (no JSON). FastAPI handler signature uses `Request` + `Depends(get_db_session)` only — no Pydantic body model. |
| Cookie read | `request.cookies.get("refresh_token")`. If missing → 401 (same envelope as invalid). |
| Response 200 | `{"data": {"access_token": "<jwt>", "token_type": "Bearer", "expires_in": <int>}, "meta": {"request_id": "<uuid>"}, "errors": []}` |
| Response 200 Set-Cookie | new opaque refresh token. **EXACTLY** the same cookie attrs as sign-in: `HttpOnly; Secure; SameSite=lax; Path=/auth; Max-Age=AUTH_REFRESH_TTL_SECONDS`. |
| Response 401 | `{"data": null, "meta": {"request_id": "<uuid>"}, "errors": [{"code": "AUTH_SESSION_EXPIRED", "message": "Session expired or invalid; please sign in again.", "field": null, "details": null}]}` — **byte-identical** for: missing cookie, malformed cookie, hash not in DB, expired, revoked, user inactive/deleted. |
| Response 429 | `{"data": null, "meta": {"request_id": "<uuid>"}, "errors": [{"code": "AUTH_REFRESH_RATE_LIMITED", "message": "Too many refresh attempts; retry after Ns", "field": null, "details": null}]}` + header `Retry-After: N`. |
| Headers (always) | `X-Request-ID` echoed (request header preserved or generated). |
| Logging | BEFORE/AFTER/ERROR per non-negotiables §Logging; verbose-on shows full flow with `user_id`, request_id, never raw cookie value, never the new opaque token, never the JWT. Verbose-off shows nothing for success/401. 429 logs at WARNING. |
| Concurrency | Same client sending two concurrent /refresh requests with the same cookie → AT MOST ONE may succeed. The losing race returns 401 `AUTH_SESSION_EXPIRED` (token already rotated/revoked). See D-RP3. |

The registry literal write_set is `backend/app/auth/refresh_tokens.py` — a single flat file. The existing architecture (T002 debugger cycle 1) uses `services/<use_case>.py` + `routers/<endpoint_family>.py` packages. A flat 300-LOC file would mix presentation + use case + repository concerns, **violating the same non-negotiables F1/F2 the T002 debugger fixed** (`backend/app/auth/router.py` 466→29 LOC, `service.py` 702→28 LOC).

**Decision**: follow the established T002 pattern, **predeclared WRITE_SET_DRIFT**. New files (all inside `backend/app/auth/**`):
- `backend/app/auth/services/refresh.py` — `RefreshTokenUser` use case + `RefreshResult` dataclass. ≤300 LOC, ≤50 LOC per function.
- `backend/app/auth/routers/refresh.py` — `refresh_router` APIRouter + handler. ≤300 LOC.
- `backend/app/auth/services/__init__.py` — re-export `RefreshTokenUser` (additive 1-line change).
- `backend/app/auth/routers/__init__.py` — re-export `refresh_router` (additive 1-line change).
- `backend/app/auth/router.py` — `auth_router.include_router(refresh_router)` (additive 1-line change; 29→30 LOC aggregator).
- `backend/app/auth/repository.py` — add 3 methods: `find_active_refresh_by_hash`, `revoke_refresh_token`, `revoke_all_refresh_tokens_for_user` (rotation + defensive logout chain). Existing file is 300 LOC at hard cap; adding methods MAY require minor cleanup if it exceeds. If it does, extract `app/auth/repositories/refresh.py` instead (additional predeclared drift).
- `backend/app/auth/errors.py` — add `SessionExpiredError` (401, `AUTH_SESSION_EXPIRED`) and `RefreshRateLimitedError` (429, `AUTH_REFRESH_RATE_LIMITED`). Additive.
- `backend/app/auth/rate_limit.py` — add `check_rate_limit_refresh(ip)` with namespace `REFRESH` reading `AUTH_REFRESH_RATE_PER_MINUTE` / `AUTH_REFRESH_RATE_BURST`. Additive (mirrors T002's SIGNIN addition).

### D-RP2 — Cookie attributes byte-identical to T002

`routers/refresh.py` MUST set the cookie via the shared `_set_refresh_cookie` helper. Verbatim attrs: `httponly=True, secure=True, samesite="lax", path="/auth", max_age=int(os.getenv("AUTH_REFRESH_TTL_SECONDS", "2592000"))`. Lowercase `lax` (Starlette quirk confirmed by T002 researcher Q2). Curl evidence in `/verify-slice` must show: `Set-Cookie: refresh_token=…; HttpOnly; Max-Age=2592000; Path=/auth; SameSite=lax; Secure`.

### D-RP3 — Atomic rotation with concurrency safety

Two concurrent /refresh requests with the same cookie must result in EXACTLY ONE success + ONE 401. Approach (recommend; researcher to confirm — see §M):

1. `SELECT * FROM refresh_tokens WHERE token_hash = :hash AND revoked_at IS NULL AND expires_at > now() FOR UPDATE` (row-level lock via SQLAlchemy `with_for_update()` on PG).
2. If no row → 401.
3. `UPDATE refresh_tokens SET revoked_at = now() WHERE id = :old_id` (within the same tx).
4. Generate new opaque + sha256, `INSERT INTO refresh_tokens (...) VALUES (...)` (within the same tx).
5. Lookup user (within same tx); if user inactive → ROLLBACK → 401 (no rotation visible).
6. Write success audit (within same tx). Commit.
7. Issue new access JWT via `encode_access_token(user)`.
8. Set new cookie via `_set_refresh_cookie`.

If the loser blocks on `FOR UPDATE` until the winner commits, the loser then sees `revoked_at IS NOT NULL` → no row matches the predicate → 401.

`audit_logs(reason='unknown_hash'|'expired'|'revoked')` is written via D-S2 separate-transaction pattern (T002 precedent) so audit survives the rollback in step 5.

### D-RP4 — Replay detection: schema-driven conservative read

`instrucciones.md §3.1` says *"refresh tokens se guardan hasheados y rotan"*. It is **silent on replay-detection** beyond hashing + rotation. The DB schema has no `replaced_by`/`previous_token_id` link, no `family_id` for token-family revocation, no `device_id`. Adding any of these = source-of-truth amendment + new migration, **out of scope for T003**.

**Decision**: do NOT implement "if you receive a revoked-hash, kill the whole user's session". Reasons:
1. Schema does not support it cleanly (`revoke_all_for_user` would nuke ALL active tokens including the legitimate winner of D-RP3's race; no way to distinguish "stolen rotation chain" from "lost race"). False positives would log legitimate users out.
2. Source-of-truth is silent. KISS/YAGNI.
3. Conservative: just rotate-and-revoke per use; rely on short access TTL (1800s) + opaque-token unguessability + cookie HttpOnly+Secure+SameSite=lax as the security perimeter. This matches OWASP refresh-rotation baseline.
4. If future product decision wants RFC 6819 §5.2.2.3 family-revocation, that becomes a follow-up DAG task with its own migration + ADR.

The audit row records `reason='revoked'` when a revoked hash is presented — observability is preserved for future incident response without requiring an enforcement action now.

### D-RP5 — Body never contains refresh

T002 pin (D-RP5 reused): refresh stays in cookie only; response body has only `access_token + token_type + expires_in`. T15-pattern test required to assert "no refresh in body" — see §G.

### D-RP6 — No 400, no Pydantic body

Empty/missing JSON body is normal for /refresh. Cookie absence → 401 (one envelope branch).

### D-RP7 — Verbose logging redaction

Per `01-non-negotiables.md §Security` + `instrucciones.md` line 328: no API key, no token in logs. Forbidden in any log message: raw cookie value, new opaque refresh, new access JWT, token_hash hex strings. Allowed: `user_id`, `request_id`, `ip`, `user_agent`, `email_domain` (after user lookup; never the local-part), `old_token_id` and `new_token_id` (UUIDs of the refresh_tokens rows). T012-pattern grep test required.

## F. Acceptance criteria, broken into tests

Acceptance text from registry: *"Refresh token cookie rotates and old hash is revoked."* Decomposed:

### F.1 Success path (rotation)
- T01 `test_refresh_success_rotates_and_returns_new_access_token`
  - Arrange: sign in real verification user via the real /sign-in handler; capture the refresh cookie.
  - Act: POST /api/v1/auth/refresh with the cookie.
  - Assert: HTTP 200; body has `access_token` (non-empty JWT) + `token_type=="Bearer"` + `expires_in` (int>0). Body has NO `refresh_token` key.
  - Assert: response sets a new `refresh_token` cookie whose value DIFFERS from the input cookie.
  - Assert: new cookie attrs: HttpOnly, Secure, SameSite=lax, Path=/auth, Max-Age=AUTH_REFRESH_TTL_SECONDS.
  - Assert: decoding the new access JWT yields the 8 §10.2 claims (sub, email, roles, preferred_language, employee_profile_id, jti, iat, exp).
  - Assert: in DB, old `token_hash` row has `revoked_at IS NOT NULL`; one new row exists with `revoked_at IS NULL`, expires_at ≈ now+TTL.
  - Assert: audit row exists with `action='auth.refresh'`, `outcome='success'`, `actor_user_id=user.id`, metadata contains `old_token_id` and `new_token_id`.

### F.2 Failure: no cookie sent
- T02 `test_refresh_no_cookie_returns_401_session_expired`
  - POST /refresh without any cookie. → HTTP 401, `code=AUTH_SESSION_EXPIRED`, byte-identical envelope (modulo meta.request_id) to other 401s.
  - Assert: audit row `actor_user_id=NULL, outcome='failure', reason='no_cookie'`.

### F.3 Failure: malformed / unknown hash
- T03 `test_refresh_unknown_token_returns_401`
  - POST /refresh with cookie value `"not-a-real-token-xyz"`. → 401 SESSION_EXPIRED, audit `reason='unknown_hash'`, actor NULL.

### F.4 Failure: expired token
- T04 `test_refresh_expired_token_returns_401`
  - Seed a refresh_tokens row with `expires_at = now - 1s`, `revoked_at = NULL`. Use opaque token whose hash matches.
  - POST /refresh with that opaque value as cookie. → 401, audit `reason='expired'`, actor=user.id.
  - Assert: row is NOT auto-cleaned-up; `revoked_at` may stay NULL (expired ≠ revoked; up to housekeeping).

### F.5 Failure: revoked token
- T05 `test_refresh_revoked_token_returns_401`
  - Seed a refresh_tokens row with `revoked_at = now - 60s`. POST /refresh with matching opaque. → 401, audit `reason='revoked'`, actor=user.id.

### F.6 Failure: user inactive (race with admin disable)
- T06 `test_refresh_inactive_user_returns_401`
  - Seed an active refresh_tokens row, then UPDATE users.status='disabled'. POST /refresh. → 401, audit `reason='user_inactive'`, actor=user.id.
  - Assert: rotation did NOT happen (no new row, old row's revoked_at unchanged OR revoked_at is set but no INSERT — implementation-defined). Recommended: ROLLBACK so neither side-effect leaks.

### F.7 Replay (old hash after rotation)
- T07 `test_refresh_replay_old_token_returns_401`
  - Successful rotation (run /refresh once with cookie A). Then POST /refresh **again with original cookie A**. → 401, audit `reason='revoked'`, actor=user.id.
  - Assert (D-RP4): no other active refresh tokens for the user are revoked. Specifically the rotated cookie B remains usable.

### F.8 Concurrency / race
- T08 `test_refresh_concurrent_requests_only_one_wins`
  - Fire two threads doing /refresh with the same cookie value at the same time (use a `Barrier` to align start). EXACTLY one returns 200, the other 401. DB shows exactly 1 new row + 1 row newly revoked.
  - Acceptable if test is `pytest -p no:cacheprovider` flaky-sensitive — use ≥3 retries with mutex inside the handler eliminating the flakiness.

### F.9 Rate limit
- T09 `test_refresh_rate_limit_returns_429`
  - Set `AUTH_REFRESH_RATE_PER_MINUTE=2, AUTH_REFRESH_RATE_BURST=2`. 3rd /refresh from same IP → 429, `code=AUTH_REFRESH_RATE_LIMITED`, `Retry-After: <int>` header present, audit `reason='rate_limited'`.

### F.10 Verbose-on log redaction
- T10 `test_refresh_no_pii_or_token_value_in_logs`

## M. NEEDS_OFFICIAL_DOCS assessment

**NEEDS_OFFICIAL_DOCS: yes** — narrow scope, 2 concrete questions for the researcher.

PyJWT 2.12.1, Argon2id, HttpOnly/Secure/SameSite cookie attrs, structlog, and FastAPI Depends patterns are already RESOLVED by `orchestrator-state/memory/official-doc-notes/P01-S02-T002-pyjwt-cookies-argon2.md` (no new note needed for those).

**New uncertainty** centers on atomic refresh rotation under contention with SQLAlchemy 2.0 + PostgreSQL 17:

### M.1 Question 1 — `with_for_update()` semantics on the active dialect
"In SQLAlchemy 2.0.49 against PostgreSQL 17 (psycopg3), is `session.query(RefreshToken).filter(RefreshToken.token_hash == h, RefreshToken.revoked_at.is_(None), RefreshToken.expires_at > func.now()).with_for_update().first()` sufficient to guarantee that two concurrent transactions both reading the same row will serialize, such that ONLY ONE can subsequently `UPDATE refresh_tokens SET revoked_at = now() WHERE id = :id` and the OTHER, upon retry of the same SELECT, sees zero rows (because the predicate `revoked_at IS NULL` now fails)? Specifically:
  (a) Default PG isolation is READ COMMITTED — does `FOR UPDATE` still block the second SELECT until the first commits?
  (b) Does the second transaction's SELECT re-evaluate the predicate after unblocking, returning zero rows, or does it return the locked-then-stale row at its statement snapshot?
  (c) Is `with_for_update(skip_locked=False, nowait=False)` (defaults) the correct flag set, or do we want `nowait=True` to fail-fast?"

### M.2 Question 2 — Transaction isolation for the audit/rotation atomicity
"In the same setup, if step 5 of the rotation (read User row, check status=active) detects an inactive user and we want to ROLLBACK so neither old-token-revocation nor new-token-INSERT becomes visible, but we want the rejection audit row to persist: is the D-S2 pattern (separate short-lived session that COMMITs the audit independently) the canonical SQLAlchemy 2.0 way, or is there a Postgres-specific advisory-lock or SAVEPOINT approach that's preferred? The constraint: audit row persists even if main tx rolls back; main tx must remain atomic (all-or-nothing rotation)."
