## 4. Business rules y UX (instrucciones.md + UX_CONTRACT.md)

### Business rules (instrucciones.md §3.1 Auth motor + §3.2 Feature Login propio)

- "Solo emails corporativos permitidos" → server enforces via `CORPORATE_EMAIL_DOMAINS` env var allowlist (e.g. `inditex.com, inditex-sandbox.com, zara.com, ...`). Frontend MUST also surface this rejection (400 `AUTH_SIGNUP_NON_CORPORATE_EMAIL` → field `email`).
- "Aceptación legal en sign-up" → `legal_acceptance` field is a literal `true` boolean. Service-layer enforced (NOT Pydantic), so 400 `AUTH_SIGNUP_LEGAL_NOT_ACCEPTED` is returned via project envelope. UI requires explicit checkbox toggle.
- "Contraseña mínima" → Password policy is server-side (Argon2 + value object). Policy: **min 12 chars, max 256, ≥1 letter AND ≥1 digit**. Returned as 422 `AUTH_SIGNUP_INVALID_PAYLOAD` field `password`. Frontend SHOULD client-side enforce min 12 + letter + digit to give immediate feedback (better UX), but the server is the final authority.
- "Todo login/logout/reset queda auditado" → server writes `audit_logs` rows; frontend has no responsibility here, but PII/log discipline applies on the client side too (NEVER log password, NEVER log full email — domain only).
- Roles assigned at sign-up: `employee` by default (server side). Frontend does not select role.

### Personas (UX_CONTRACT §2)

Target persona: **Empleado Inditex** — quiere registrarse con su email corporativo, móvil-first, recibir feedback claro de errores.

### Required UI states (UX_CONTRACT §3 row /auth/sign-up + instrucciones §3.2 Login propio)

- `loading` — submit in flight; CTA disabled + "Creando cuenta…" label; inputs disabled.
- `error_network` — fetch failed (offline, DNS, TypeError) → red inline copy `signUp.errors.network`.
- `error_validation` — 400 (legal_not_accepted, payload invalid) OR 422 (password policy) OR client-side zod → field-level errors via `aria-invalid` + `aria-describedby` + `role="alert"`.
- `permission_denied` — 400 `AUTH_SIGNUP_NON_CORPORATE_EMAIL` (email domain not allowlisted) → inline field-error on `email`. 429 `AUTH_SIGNUP_RATE_LIMITED` → disabled submit + "Demasiados intentos. Espera N segundos."
- `success` — 201 created → user can now sign in. Two viable next-actions:
  - (a) Navigate to `/auth/sign-in` with a flash success message (KISS — server returns `mfa_required:false`, no access_token, so user must sign in to bootstrap session).
  - (b) Show a success screen with "Continuar a iniciar sesión" CTA (more editorial).
  - **Recommended: (a)** — keeps the flow linear, matches T001's redirect pattern, and instrucciones §3.2 next-action says "sign-in con MFA navega a /auth/2fa; éxito final navega a /chat" → SignUpPage's natural next action is /auth/sign-in (NOT /auth/2fa, since `mfa_required:false` from sign-up).
- `empty` — **N/A** for forms per instrucciones §3.2 (UX_CONTRACT §3 lists it but instrucciones §3.2 explicitly says "empty no aplica por formulario"). Validator approved this as N/A for T001; same applies here.

## 5. API contract — Front → Back → DB

### POST /api/v1/auth/sign-up (TECHNICAL_GUIDE §6.2 + backend/app/auth/schemas.py + errors.py)

**Request body** (JSON):

```ts
interface SignUpRequest {
  email: string;            // RFC 5322 valid; server checks corporate domain allowlist
  password: string;         // server policy: min 12, max 256, ≥1 letter, ≥1 digit
  full_name: string;        // 1-200 chars after server-side strip
  legal_acceptance: boolean;// MUST be literal true; service-layer enforced
}
```

**Success response** (`HTTP 201` + `X-Request-ID` echoed):

```ts
interface SignUpResponseSuccess {
  data: {
    user_id: string;         // UUID
    mfa_required: false;     // always false for new employee sign-ups (forward-compat flag)
  };
  meta: { request_id: string };
  errors: [];                // always empty on 201
}
```

**No access_token, no refresh cookie set on sign-up** (unlike sign-in MFA-success). User must now go through `/auth/sign-in` to bootstrap session. This is intentional — sign-up only creates the account; sign-in creates the session.

**Error envelope** (4xx/5xx):

```ts
interface ErrorEnvelope {
  data: null;
  meta: { request_id: string };
  errors: [{
    code: string;
    message: string;
    field?: string;            // for field-level errors (email, password, full_name, legal_acceptance)
    details?: Record<string, unknown>;
  }];
}
```

**Error code matrix** (from `backend/app/auth/errors.py`):

| HTTP | Code | Field | Trigger | UI state | i18n key (proposed) |
|------|------|-------|---------|----------|--------------------|
| 400 | `AUTH_SIGNUP_NON_CORPORATE_EMAIL` | email | Email domain not in `CORPORATE_EMAIL_DOMAINS` allowlist | permission_denied → field-level inline error on email | `signUp.errors.nonCorporateEmail` |
| 400 | `AUTH_SIGNUP_LEGAL_NOT_ACCEPTED` | legal_acceptance | `legal_acceptance` is false or missing | error_validation → checkbox-adjacent error | `signUp.errors.legalNotAccepted` |
| 409 | `AUTH_SIGNUP_EMAIL_TAKEN` | (no field — anti-enumeration) | Email already exists OR cannot be created (generic) | error_validation → toast/inline copy | `signUp.errors.emailTaken` |
| 422 | `AUTH_SIGNUP_INVALID_PAYLOAD` | password (typically) | Password policy violation (too short, no letter, no digit) | error_validation → field-level inline error on password | `signUp.errors.passwordPolicy` |
| 422 | `AUTH_SIGNUP_INVALID_PAYLOAD` | full_name OR email | Pydantic v2 RFC 5322 syntax fail; empty strip; oversize | error_validation → field-level | `signUp.errors.payloadInvalid` |
| 429 | `AUTH_SIGNUP_RATE_LIMITED` | (none) | Too many sign-ups from this IP in window | permission_denied → disabled submit + "Espera N segundos" | `signUp.errors.rateLimited` (with `{{seconds}}` interpolation) |
| 5xx | (generic) | (none) | Backend error (500-503) | error_network or error_validation depending on UX choice | `signUp.errors.serverInternal` |
| TypeError | (NETWORK_ERROR client) | (none) | fetch rejected (offline/DNS) | error_network → "Sin conexión. Comprueba tu red." | `signUp.errors.network` |

**Note**: 409 returns NO field intentionally (anti-enumeration — server never reveals whether email is taken vs blocked). Frontend should show a generic copy like "No se pudo crear la cuenta con ese email. Si ya tienes cuenta, intenta iniciar sesión." This balances UX vs security.

**Anti-enumeration discipline** (echoes T001 D-T001-AGGREGATE-401): different codes (409 vs 422 vs 400) DO leak some signal, but field-omission for 409 plus generic copy keeps the security floor consistent.

## 6. Verification data contract (TECHNICAL_GUIDE §6.5 J100 row)

For T002 the verification data row is the J100 auth-login row, but **sign-up itself creates NEW users**, not the verification user. Strategy:

- For sign-up happy-path verification: **use a fresh ad-hoc corporate email** that is NOT the `employee.verification@inditex-sandbox.com` (that user already exists, would return 409). Example: `signup.test+<timestamp>@inditex-sandbox.com`. The domain `inditex-sandbox.com` IS in `CORPORATE_EMAIL_DOMAINS`.
- For "non-corporate email" rejection: any `*@gmail.com` or `*@example.com`.
- For "email taken" 409: re-submit the existing `employee.verification@inditex-sandbox.com`.
- For "legal not accepted": submit valid email/password but `legal_acceptance: false`.
- For "password policy": submit short or letter-only password.
- For "rate limit": submit ≥ N requests within the window (env var `AUTH_SIGNUP_RATE_PER_MINUTE` controls; check `.env.example`).
- Cleanup after verify: optional `DELETE FROM users WHERE email LIKE 'signup.test+%';` to prevent test-row accumulation (tester/verify-slice's call, NOT this slice's responsibility).

**Verification data load command** (already runs in dev-restart): `python -m app.verification_data.bootstrap --source data/verification --only auth`.

## 15. Decisions register (for handoff §Decisions)

Suggested anchors the developer should fill in during implementation:

- **D-T002-EMAIL-CORP** — Client-side: only zod `.email()` (RFC 5322 syntax). Domain allowlist is server-side authoritative. Frontend surfaces the 400 rejection in field error. Rejected: hardcoding the `CORPORATE_EMAIL_DOMAINS` list in the frontend (would drift; sync nightmare; allowlist is env-var-controlled on backend).
- **D-T002-PASSWORD-PRE-VALIDATE** — Mirror server policy (min 12, ≥1 letter, ≥1 digit) in zod for UX. Server is still authoritative; client pre-check is a UX hint only.
- **D-T002-LEGAL-LITERAL-TRUE** — zod `.literal(true)` for `legal_acceptance` so unchecked checkbox shows field error before submit.
- **D-T002-CHECKBOX-INLINE** — Inline editorial checkbox in SignUpPage.tsx (no new base component). Hairline border, no radius. KISS + YAGNI. (Validator may REC extracting later — that's a follow-up, not this slice's scope.)
- **D-T002-SUCCESS-REDIRECT** — `success → navigate("/auth/sign-in", { state: { flash: "account_created" } })`. NOT auto-login (would require access_token in sign-up response, which the contract does not provide).
- **D-T002-NO-NEXT-PARAM** — sign-up does NOT support `?next=` since post-sign-up the user MUST sign in. Preserve any incoming `?next=` by passing it through to `/auth/sign-in?next=<value>` so deep-link survives the round trip.
- **D-T002-409-NO-FIELD** — On 409 EMAIL_TAKEN, do NOT highlight the email field (anti-enumeration). Show a generic toast/inline copy.
- **D-T002-PII-LOGGING** — `email_domain` only in logs (split after `@`). Password length numeric metadata only. NEVER log full email, password, full_name (PII).
- **D-T002-USESIGNUP** — Hook extracted (`useSignUp.ts`) for testability, mirror of `useSignIn.ts`. SignUpPage becomes a thin layout + state-binding component.
- **D-T002-PROACTIVE-EXTRACT** — Preemptively apply T001 validator REC-1: keep `getSignUpErrorI18nKey` and `useSignUpSchema` inside the page initially BUT if file size approaches 350 LoC during implementation, extract to sibling helpers. Avoids future revise-slice.
