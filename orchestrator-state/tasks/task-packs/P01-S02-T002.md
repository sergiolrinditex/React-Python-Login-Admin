## E. Front ‚Üí Back ‚Üí DB contract

### E.1 Request

```
POST /api/v1/auth/sign-in
Headers: Content-Type: application/json, X-Request-ID? (optional)
Body: { "email": "...", "password": "..." }
```

Pydantic:
- `email: EmailStr` (RFC 5322 syntactic; do NOT call `CorporateEmail`).
- `password: str`, `min_length=1`, `max_length=512` (guard; do NOT call
  `Password` policy ‚Äî login must accept legacy/short passwords if the hash
  matches).

### E.2 Success ‚Äî no MFA (HTTP 200)

```
{
  "data": {
    "mfa_required": false,
    "access_token": "<JWT>",
    "token_type": "Bearer",
    "expires_in": <int seconds>
  },
  "meta": { "request_id": "..." },
  "errors": []
}
```

Set-Cookie response header: `refresh_token=<opaque_hex>; HttpOnly; Secure; SameSite=Lax; Path=/auth; Max-Age=<int>`.
- `Path=/auth` per ¬ß6.4 Navigation Contract scope and ¬ß10.2.
- `Secure` always (even in dev ‚Äî local browser will accept on localhost).
- `Max-Age` from `AUTH_REFRESH_TTL_SECONDS` env (see ¬ßF.7).

DB side effects (one transaction):
- INSERT `refresh_tokens(user_id, token_hash=sha256(opaque), expires_at, revoked_at=NULL)`.
- INSERT `audit_logs(action='auth.sign_in', outcome='success', actor_user_id=user.id, metadata={request_id, ip, user_agent})`.
- Optional: if `needs_rehash(user.password_hash)` ‚Üí UPDATE `users.password_hash` (same transaction). T001 pinned param policy already exceeds OWASP; this is just forward-compat for a future params bump.

### E.3 Success ‚Äî MFA challenge (HTTP 200)

When `mfa_totp_secrets.enabled = true` for the user:

```
{
  "data": {
    "mfa_required": true,
    "mfa_challenge_token": "<short-lived JWT>",
    "expires_in": <int seconds>
  },
  "meta": { "request_id": "..." },
  "errors": []
}
```

- NO `access_token` in body.
- NO `Set-Cookie: refresh_token=...`.
- `mfa_challenge_token` is a short-lived JWT (claim `purpose: "mfa_challenge"`,
  `sub: user.id`, TTL 5 min) signed with the same JWT key as the access token,
  consumed by P01-S02-T006 `/api/v1/auth/2fa/verify`. See ¬ßF.6.
- DB side effects: INSERT `audit_logs(action='auth.sign_in', outcome='mfa_challenge_issued', actor_user_id=user.id, metadata={request_id, ip, user_agent})`. NO `refresh_tokens` row yet ‚Äî that is issued only by `/2fa/verify` (T006).

### E.4 Errors

| Case                                              | HTTP | code                          | field    | side effects                                                                                 |
|---------------------------------------------------|------|-------------------------------|----------|-----------------------------------------------------------------------------------------------|
| Unknown email                                     | 401  | `AUTH_INVALID_CREDENTIALS`    | ‚Äî        | INSERT audit `outcome='failure', reason='unknown_email'`, **dummy verify** (timing-equalised) |
| Known email + wrong password                      | 401  | `AUTH_INVALID_CREDENTIALS`    | ‚Äî        | INSERT audit `outcome='failure', reason='wrong_password', actor_user_id=user.id`              |
| Account locked (N failures within window)         | 423  | `AUTH_ACCOUNT_LOCKED`         | ‚Äî        | INSERT audit `outcome='blocked', reason='account_locked', actor_user_id=user.id`              |
| User `status != 'active'` (disabled/pending)      | 401  | `AUTH_INVALID_CREDENTIALS`    | ‚Äî        | INSERT audit `outcome='failure', reason='user_inactive', actor_user_id=user.id` (NEVER reveal which) |
| Missing field / payload malformed                 | 422  | (FastAPI default `{detail}`)  | ‚Äî        | NO audit row (request never reached the service); matches T001 precedent for missing fields  |
| Empty email / empty password                      | 400  | `AUTH_INVALID_PAYLOAD`        | email/password | INSERT audit `outcome='failure', reason='invalid_payload'`                              |
| Rate limit exceeded                               | 429  | `AUTH_SIGNIN_RATE_LIMITED`    | ‚Äî        | Retry-After header; INSERT audit `outcome='blocked', reason='rate_limited'`                  |
| Internal error                                    | 500  | `AUTH_SIGNIN_INTERNAL_ERROR`  | ‚Äî        | Log error with stack; no audit row required                                                  |

Error body shape is the project envelope (`{data:null, meta:{request_id}, errors:[{code, message, field}]}`). Use `_error_response` from `router.py`.

**Critical**: codes for 401 unknown-email and 401 wrong-password are IDENTICAL
(`AUTH_INVALID_CREDENTIALS`) to prevent user enumeration. Reasons differ ONLY
in the audit log metadata (server-side compliance trail), never in the response
body.

## F. Decisions & DECISION_REQUIRED gates

All security primitives below are **in-scope** for this slice (the acceptance
explicitly pins "verifies Argon2 password, returns MFA challenge or access
token, does not leak user existence" and the ¬ß6.2 row pins 423). They do NOT
go through follow-up.

### F.1 (DECISION_REQUIRED ‚Äî PROPOSED) User-enumeration mitigation

TECHNICAL_GUIDE does not pin a concrete pattern; it only mandates "does not leak user existence" (acceptance) and "AUTH_INVALID_CREDENTIALS" (¬ß6.4).

**Proposed default**: aggregate-401 + dummy-Argon2-verify.
- If `find_by_email(email)` returns `None`, compute `verify_password(_DUMMY_HASH, password)` in a try/except so the wall-clock time of the "unknown email" path is statistically indistinguishable from the "wrong password" path. The same Argon2 verify-time dominates both branches.
- `_DUMMY_HASH` is a module-level constant computed once at module import: `hash_password("dummy-equaliser-not-a-real-password-12345")`. Stored in `password.py` next to the real helpers.
- Always return `{errors:[{code:"AUTH_INVALID_CREDENTIALS"}]}` 401, regardless of branch.
- Audit logs differentiate `reason='unknown_email'` vs `'wrong_password'`
  (server-side only; never in HTTP response).

Precedent: T001 sign-up already uses a dummy-hash on the duplicate-email branch (`service.py:208 hash_password(password_plain)`). T002 extends the pattern to the lookup branch.

Status: **proposed default ‚Äî confirm OK with main-orchestrator OR developer goes ahead with this pattern**. No alternative is safer in this stack.

### F.2 (DECISION_REQUIRED ‚Äî PROPOSED) MFA challenge token

TECHNICAL_GUIDE ¬ß6.2 declares `mfa_required` boolean but does not pin the challenge token. ¬ß10.2 declares "JWT propio".

**Proposed default**: short-lived JWT (HS256 with `JWT_PRIVATE_KEY`, OR RS256 if asymmetric key pair available ‚Äî see ¬ßF.5).
- Claims: `sub=user.id`, `purpose="mfa_challenge"`, `iat`, `exp` (5 min), `jti` (UUID4).
- Returned in response body as `data.mfa_challenge_token`; no cookie.
- Consumed by P01-S02-T006 `/api/v1/auth/2fa/verify` (must verify `purpose=="mfa_challenge"` and `exp`).
- TTL: 300 s (5 min) via `AUTH_MFA_CHALLENGE_TTL_SECONDS` env (default 300).

Alternative considered & rejected: opaque-random + DB row in `mfa_challenges` table. Rejected because (a) it would require a new table not in 0001 migration (out of write_set), (b) stateless JWT challenge tokens are standard for 5-min ephemeral flows.

Status: **proposed default ‚Äî confirm OK with main-orchestrator OR developer goes ahead**.

### F.3 (DECISION_REQUIRED ‚Äî PROPOSED) 423 Account locked policy

¬ß6.2 pins 423 in the error column. CHECKLIST acceptance does not pin the threshold.

**Proposed default ‚Äî minimal viable lockout**:
- Lockout threshold: **5 consecutive failed sign-in attempts** for the same `user_id` within a **15 minute rolling window**, derived by querying `audit_logs WHERE actor_user_id = :uid AND action='auth.sign_in' AND metadata->>'outcome' = 'failure' AND created_at > now() - interval '15 minutes'`.
- Lockout duration: 15 minutes from the 5th failure (then the count resets when window slides).
- On lockout, return 423 with `code='AUTH_ACCOUNT_LOCKED'`. No PII in response.
- Lockout is per-user (by `user_id`), not per-IP (per-IP is already covered by rate-limit ¬ßF.4). This protects against credential-stuffing of a single account from multiple IPs.
- Lookup-only-when-user-exists: lockout query is gated by `find_by_email != None`. Unknown emails cannot trigger lockout (would otherwise be an enumeration oracle).
- Configurable: `AUTH_SIGNIN_LOCKOUT_THRESHOLD` (default 5), `AUTH_SIGNIN_LOCKOUT_WINDOW_SECONDS` (default 900).
- TODO(P02-S02-T001): replace audit-log scan with a Redis counter when Redis-backed rate-limit lands. Document this TODO in `service.py`.

Status: **proposed default ‚Äî confirm OK with main-orchestrator OR developer goes ahead**. Without a 423 implementation this slice would fail the ¬ß6.2 error contract and the acceptance.

### F.6 Access token TTL (no decision required, default proposed)

### F.7 Refresh token storage (no decision required)

- Opaque random token: `secrets.token_urlsafe(48)` ‚Üí ~64-char URL-safe string. Issued exactly once; never logged.
- Stored hashed: `sha256(opaque).hex()` ‚Üí `refresh_tokens.token_hash`. Per ¬ß10.2 "guardado hasheado" and ¬ß12 invariant. SHA-256 is sufficient for a high-entropy random token (no Argon2 needed ‚Äî it's already 48 bytes of randomness, not a low-entropy human input).
- Cookie attributes (already pinned by ¬ß10.2): `HttpOnly; Secure; SameSite=Lax; Path=/auth; Max-Age=2592000`.
- T003 (refresh) rotates: on `/refresh` use, current row gets `revoked_at=now()`, new row inserted, new cookie set.

### F.8 Logging redaction (no decision required)

Per ¬ß10.5 + non-negotiables ¬ßSecurity. Redacted tokens for `password`, `token`, `access_token`, `mfa_challenge_token`, `refresh_token`, `JWT_PRIVATE_KEY`, `secret`, `secret_encrypted`. Log only:
