## §8. Impact analysis

### Modules candidates

- `backend/app/core/db.py` — **READ-ONLY** for T011. T012 already enforces `echo=False`. T011 must NOT touch this file (declared in `write_set` for safety/scope expansion only; the actual code is fine).

- `backend/tests/integration/test_signup_no_argon2_hash_in_logs.py` — **NEW**. Implements the FU's literal verify command as a deterministic integration test.

### Existing tests that already partially cover this

- `backend/tests/integration/test_db_engine_no_secret_leak.py::test_engine_echo_is_false` (T012 T1) — guards the engine attribute. Does NOT touch users/argon2.
- `backend/tests/integration/test_db_engine_no_secret_leak.py::test_echo_false_vs_echo_true_fernet_leak` (T012 T2) — proves `echo=False` closes the bind-param leak channel for **Fernet**. The same channel is shared with argon2; T011's new test makes that explicit for the auth/users path.
- `backend/tests/integration/test_auth_signup.py::test_auth_signup_logging_no_password_leak` (T001 T8) — asserts `\bpassword\b` and `\blegal_acceptance\b` are absent from `capsys` stdout. Does NOT assert `$argon2id$` absence and does NOT capture the `sqlalchemy.engine` named logger separately.

### Unknowns / open questions (none blocking)

- **Q1**: should the T011 test ALSO bind a capturing handler to the `sqlalchemy.engine` logger (defense-in-depth), or is `capsys` over `sys.stdout` sufficient? **Planner answer**: capture both. T012's test pattern (`_CapturingHandler` attached to `sqlalchemy.engine`) is the gold standard; reuse it. `capsys` over stdout is necessary because `configure_logging(verbose=True)` reroutes the root logger to a `StreamHandler(sys.stdout)`, so structlog and stdlib both end up on stdout.
- **Q2**: do we need `VERIFICATION_GEMINI_API_KEY`? **No** — that's only required for the admin_ai bundle (T012 T3). T011's flow exercises auth, not admin_ai. The DB skip guard is the only gate (`_skip_no_db` if 5433 not reachable).
- **Q3**: does adding `password_hash` to `_REDACTED_KEYS` break any existing test? **No** — grep shows no test/code logs `password_hash=...` as a structlog key today. The redaction only activates if/when someone adds such a log call in the future.

### Riesgos

- **R1 (low)**: tester could observe environmental flakiness (5433 not reachable in some CI environments → test skipped, not failed). Mitigated by the `_skip_no_db` guard pattern from T012/T001. Acceptable behavior.
- **R2 (low)**: structlog's `_REDACTED_KEYS` uses `key.lower() in _REDACTED_KEYS` so adding `"password_hash"` covers `password_hash`, `Password_Hash`, `PASSWORD_HASH`. No collision with existing keys (none start with `password_h`).
- **R3 (low)**: developer might be tempted to ALSO scrub `$argon2id$` substrings inside log message bodies via a regex processor. **DO NOT do this in T011** — it expands scope, costs CPU on every log call, and the engine-level fix already closes the channel. Out of scope.
- **R4 (very low)**: if a future code change re-enables `echo=True` (e.g. someone copies `echo=settings.enable_verbose_logging` from a tutorial), T012's `test_engine_echo_is_false` catches it AND T011's new test catches it. Two independent regression guards.
