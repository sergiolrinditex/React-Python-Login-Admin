## H. Acceptance criteria detallados

**Forgot endpoint (`POST /api/v1/auth/forgot-password`)**

1. Email válido → 200 `{data: {sent: true}}` SIEMPRE (incluido email inexistente, deshabilitado, fuera de dominio corporativo, formato inválido tras validación inicial).
2. Payload inválido (no email) → 400 con `code: AUTH_INVALID_PAYLOAD`, `errors:[{field:"email", code:"invalid"}]`. **NO 422** (alineación con T001 sign-up que normaliza Pydantic a 400).
3. **Anti-enumeration timing**: el tiempo de respuesta para email existente vs no existente debe ser indistinguible (±50 ms). Patrón: si el email no existe, ejecutar igualmente `password.verify_with_dummy_fallback`-equivalente (es decir, generar token+hash dummy y descartarlo, en lugar de short-circuit).
4. **Anti-enumeration body**: respuesta literal byte-equal en ambos casos.
5. Rate limit per IP: bucket `FORGOT_PASSWORD` (default 3/min, env `AUTH_FORGOT_RATE_PER_MINUTE`) → 429 con `Retry-After`.
6. Token: `secrets.token_urlsafe(32)` (256 bits entropy). El raw token va al email. La DB guarda `token_hash = sha256(raw)` (sin sal — el token tiene 256 bits de entropy, hash es solo para defensa-en-profundidad ante dump de tabla).
7. TTL: `AUTH_RESET_TOKEN_TTL_SECONDS` (default 3600 = 1h). `expires_at = now() + TTL`.
8. Audit row síncrono en MISMA transacción (patrón D-S2 de T001/T002). Action `auth.password_reset.requested`. Metadata: `request_id`, `ip_hash`, `email_domain`, `user_found: bool` (interno; NO se devuelve).
9. Email enviado por `app.mail.sender.send_reset_email` con plantilla localizada según `user.preferred_language` (es/en/fr). Si user no existe → NO se envía email (pero el audit row queda con `user_found: false` y NO se contabiliza para Resend quota — el silencio es el objetivo).
10. **Modo dev sin API key** (oficial-docs-researcher confirmará patrón): si `RESEND_API_KEY` vacío y `MAIL_MODE=dev`, el sender escribe el email a `orchestrator-state/dev-logs/mail-outbox.jsonl` y devuelve OK; tests integration usan ESTE modo y leen el outbox. NO se manda nada por red en CI.
11. Logging BEFORE/AFTER bajo `ENABLE_VERBOSE_LOGGING`: BEFORE log `auth.forgot.start email_domain=*** request_id=...`, AFTER `auth.forgot.done user_found=BOOL sent=BOOL latency_ms=N`. **Verbose-off** = ZERO líneas `app.auth.forgot.*` en path success/429. PII safety: el log NUNCA contiene el email completo, sólo el dominio + hash. NUNCA el raw token.

**Reset endpoint (`POST /api/v1/auth/reset-password`)**

1. Token válido + password válido → 200 `{data: {reset: true}}`.
2. Token inexistente → **410 Gone** `code: AUTH_RESET_TOKEN_INVALID`. (§6.2 fila 260 dice 400/410; usamos 410 para token inválido/usado y 400 sólo para payload inválido.)
3. Token expirado (`expires_at < now()`) → **410 Gone** `code: AUTH_RESET_TOKEN_EXPIRED`.
4. Token usado (`used_at IS NOT NULL`) → **410 Gone** `code: AUTH_RESET_TOKEN_INVALID` (envelope byte-equal con caso 2 para no diferenciar).
5. Payload inválido (token vacío, password vacío, password < 12 chars, password sin upper/digit/symbol) → 400 `code: AUTH_INVALID_PAYLOAD` con `errors[]` por campo.
6. Rate limit per IP: bucket `RESET_PASSWORD` (default 10/min, env `AUTH_RESET_RATE_PER_MINUTE`) → 429.
7. **One-use enforcement bajo concurrencia**: `SELECT ... FOR UPDATE` sobre `password_reset_tokens.id` antes de marcar `used_at`. Patrón ya validado en T003 refresh rotation. Dos POST simultáneos con el mismo token → uno gana (200) + el otro 410.
8. **Session invalidation**: marcar `revoked_at = now()` en TODOS los `refresh_tokens` activos del `user_id`. Si el usuario tenía 3 sesiones abiertas, las 3 quedan revocadas. La cookie del cliente actual NO se toca aquí (no hay cookie en esta request).
9. **Password hash**: Argon2 vía `app.auth.password.hash_password`. Verificación de re-hash NO aplica (la password es nueva).
10. Audit row `auth.password_reset.completed` con metadata `{token_id, sessions_revoked_count, request_id}`. PII safe.
11. Transacción atómica: UPDATE users.password_hash + UPDATE password_reset_tokens.used_at + UPDATE refresh_tokens.revoked_at + INSERT audit_logs → COMMIT. Si CUALQUIER paso falla → ROLLBACK completo.
12. Logging BEFORE/AFTER análogo al forgot (verbose-off = silencio en path 410/429). Nunca loggear `token`, `password`, `token_hash`.

**Global**

- Localización del email: usar `frontend/src/i18n` como referencia conceptual, pero el backend tendrá su propio mini-i18n para emails en `backend/app/mail/templates/` (es/en/fr). Las claves de copy quedan documentadas en `instrucciones.md §6` (i18n) — no se duplican en el pack.
- `X-Request-ID` header del middleware existente → debe propagarse a logs y a audit.metadata.request_id (ya patrón establecido).
- `OWASP top 10` compliance: anti-enumeration (A07), rate-limit (A04), hashed tokens en DB (A02), Argon2 (A02), no PII en logs (A09).

### M.2 Anti-enumeration — la decisión más sutil

- Body byte-equal: **obligatorio**.
- Timing: tolerancia 100 ms. Si en CI no podemos garantizar, documentar en handoff que la métrica P95 es < 100 ms y dejarlo en lugar de assertion estricta (P-14 memory: tests flaky en CI).
- Outbox-no-send para usuario inexistente: el TOKEN sí se genera + hash dummy descartado para nivelar timing, pero NO se inserta en DB ni se envía. Audit row sí queda (`user_found: false`).

### M.3 Modo dev sin Resend API key (lo que official-docs-researcher debe confirmar)

- Patrón propuesto: `MAIL_MODE=dev` → outbox JSONL local en `orchestrator-state/dev-logs/mail-outbox.jsonl`. Tests integration usan este modo. CI no necesita API key.
- Patrón alternativo: `RESEND_API_KEY=` vacío + sandbox mode del SDK. Necesita validación del researcher (¿el SDK soporta sandbox sin red?).
- Decisión por defecto: outbox JSONL local. Mucho más simple y reproducible. Si el researcher dice que Resend Python SDK tiene un sandbox oficial idiomático (`resend.api_key = None` + `resend.dry_run=True`, o similar), evaluar y elegir.

### M.4 Riesgos enumerados

1. **Resend SDK volátil** (ecosystem >1.x, planner memory `instrucciones.md §10.1 fila 52 = pendiente`). MITIGACIÓN: official-docs-researcher pin de versión + smoke test mínimo.
2. **Anti-enum timing flaky en CI** — MITIGACIÓN: 3 reintentos + mediana, no assert estricto.
3. **Session invalidation puede sorprender al usuario** si tiene varios dispositivos. MITIGACIÓN: documentar en email template ES/EN/FR “se cerraron todas tus sesiones por seguridad”.
4. **One-use enforcement bajo concurrencia** — MITIGACIÓN: `FOR UPDATE` ya validado en T003.
5. **Email no llega y usuario re-pide**: rate-limit FORGOT 3/min, pero token TTL 1h. Decisión: cada forgot genera un token NUEVO; los viejos NO se invalidan (no es necesario, expiran solos). Documentado en service.
6. **PII en logs**: usar formato `email_domain=***` no `email=user@x`. Test T07 cubre.
7. **Audit row con `user_found: false`** es información sensible (revela enumerabilidad por DBA). MITIGACIÓN: audit logs son acceso auditor/admin; aceptable. Documentado.
8. **Old refresh cookie del cliente actual NO se invalida desde aquí** (no hay cookie en request). Es correcto — la cookie viva morirá cuando intente refresh y el `refresh_tokens.revoked_at` esté set → 401 normal. Documentar.
9. **Concurrencia `revoke_all_active`** + emisión de nuevo refresh por sign-in pos-reset: orden temporal seguro porque reset COMMIT antes de devolver 200, y sign-in es una request posterior.
10. **Bootstrap idempotency** del seed user — ya validado en T001/T002/T008. Sin riesgo nuevo.

### M.5 Decisiones recurrentes a documentar en handoff

- Body envelope `{data:{sent:true|reset:true}}` byte-equal.
- HTTP codes: 200 / 400 / 410 / 429. Sin 401 (endpoint público), sin 422 (normalizado a 400).
- Logging contract verbose-on/verbose-off.
- Mail outbox JSONL es runtime-only en `orchestrator-state/dev-logs/` (ya gitignored como el resto).
