## A. Resumen ejecutivo

**Qué**: `wait_for db_health` declara Postgres UP en cuanto `pg_isready` (ejecutado
DENTRO del contenedor postgres via `compose exec`) responde OK. Pero
`alembic upgrade head` corre desde el HOST y conecta a `127.0.0.1:5432` — el
forward del runtime de contenedores (Rancher Desktop sobre todo, también
Docker Desktop) puede tardar 0.5–3 s más en abrir el port-forward del host
después de que el contenedor reporte ready.

**Dónde**: race observable solo en runs back-to-back de `bash scripts/dev-restart.sh --reset`
porque el primer run deja el port-forward calentado; el segundo run hace
`compose down -v` + `compose up` y vuelve a la ventana de race.

**Por qué T008 lo dejó visible y no lo arregló**: T008 reemplazó el
`warn + return 0` por hard-fail en el bootstrap del seed. Ese hard-fail
ahora propaga el `OperationalError: connection refused` de alembic como
exit code != 0 en lugar de silencio (correcto). El race subyacente (contenedor
vs host TCP) es una bug independiente con write_set idéntico (`scripts/dev-restart.profile.sh`)
y conflict_group idéntico (`tooling:dev-restart`) — eso es justamente lo que
disparó la decisión de FU (P-12 MEMORY pattern, "bundling = KISS violation").

**Forma del fix**: añadir un probe host-side reusable (Bash /dev/tcp ya existe
como `_tcp_probe`) que valide `127.0.0.1:5432` TCP-open, después de
`_compose_pg_ready` y antes de declarar `db_health` UP. Endurecer `db_health`
para que devuelva 0 sólo cuando AMBOS pasen; mantener fallback TCP existente
para entornos sin compose CLI; subir el timeout efectivo a 60 s si fuera
necesario y emitir mensaje claro de fallo.

## B. Acceptance literal (del registry / FU YAML)

1. **Two consecutive 'bash scripts/dev-restart.sh --reset' commands both exit 0
   and persist users>=1**, with no manual sleep between them.
2. **wait_for db_health returns only after both container-internal pg_isready
   AND a real host-side connection (127.0.0.1:5432 with valid credentials)
   succeed within 60s.**
3. **If host-side connection fails after 60s the script fails non-zero (no
   silent fallback).**
4. **Negative control**: `docker compose stop postgres` + `bash scripts/dev-restart.sh --reset`
   debe fallar dentro de 60 s con un mensaje claro tipo "Postgres did not become ready".

> Nota planner: la acceptance #2 dice "host-side connection with valid credentials".
> La interpretación KISS (recomendada) es: TCP-open en `127.0.0.1:5432` es prueba
> suficiente de que el port-forward está activo. El handshake con credenciales
> añade complejidad (necesita `pg_isready -h` en host o psycopg) y el siguiente
> paso de la pipeline (`alembic upgrade head`) ya hace ese handshake real y fallaría
> ruidosamente si las credenciales son malas — `alembic` es la verificación natural.
> Si el developer ve riesgo (p. ej. port-forward abierto pero auth no lista), puede
> elegir el path psycopg/`pg_isready -h`. Plan §F.2 enumera ambas opciones.

## F. Diagnóstico técnico y forma del fix

### F.1 — Mapa de la race en el código actual

```
scripts/dev-restart.profile.sh

  146 _tcp_probe()        # YA EXISTE — Bash /dev/tcp, host-side. Reusar.
       host=$1 port=$2
       (exec 3<>/dev/tcp/$host/$port) 2>/dev/null && exec 3<&- 3>&-

  152 _compose_pg_ready() # Container-internal: compose exec ... pg_isready -U -d
       # ✗ Devuelve OK antes de que el port-forward del runtime esté listo.

  161 _ensure_infra_essential()
       compose up -d postgres redis
       wait_for db_health 30 "Postgres"   # 30 s, suficiente normalmente

  280 db_health()
       if _compose_pg_ready: return 0     # ← problema: gana esta rama,
                                          #   port-forward host puede no estar
       elif _tcp_probe localhost 5432: return 0   # fallback ya existe pero
                                                   # solo se evalúa si compose
                                                   # fails
       elif no compose CLI: return 2
       return 1

  305 db_reset()
       compose down -v
       compose up -d ...
       wait_for db_health 60 "Postgres"   # 60 s, ya elevado en T008
       cd backend && alembic upgrade head # ← FALLA aquí con ECONNREFUSED
                                          #   porque db_health volvió 0 mientras
                                          #   el port-forward host aún no abría
```

Race window observada: ~0.5–3 s en Rancher Desktop, intermitente en Docker
Desktop. `compose down -v` + `compose up` reinicia el daemon de port-forward
del runtime, y esa es la ventana donde el contenedor está listo pero el host
aún no enchufado.

### F.2 — Opciones de fix evaluadas

| # | Opción | KISS | Cobertura acceptance | Complejidad | Recomendación |
|---|--------|------|----------------------|-------------|---------------|
| 1 | Reutilizar `_tcp_probe(localhost, 5432)` AND con `_compose_pg_ready` dentro de `db_health` | ✅ Alta | TCP-open suficiente; alembic upgrade hace el handshake credencial real 1 s después | Baja — 3 líneas | **RECOMENDADA** |
| 2 | `pg_isready -h 127.0.0.1 -p 5432 -U hilo -d hilo_dev` (host CLI) | Media | Cubre credenciales también | Necesita `pg_isready` instalado en host (no garantizado en macOS sin brew) | Backup si #1 insuficiente |
| 3 | psycopg connect from host | Baja | Cubre todo | Necesita psycopg en venv, latencia mayor, side-effect en pool | Descartada (over-engineering) |
| 4 | `nc -z 127.0.0.1 5432` | Alta | Igual que #1 | Necesita `nc` en host (no garantizado) | Inferior a #1 (Bash /dev/tcp es nativo) |

**Decisión recomendada**: opción 1. `_tcp_probe` ya está implementado, es Bash
nativo (cero deps), funciona en macOS/Linux por igual, y el handshake credencial
es responsabilidad de `alembic upgrade head` (que va inmediatamente después y
fallará ruidoso si las creds son malas — y esto NO es el bug que estamos
arreglando).

### F.3 — Forma concreta del cambio (developer reference, no mandatoria)

```bash
# scripts/dev-restart.profile.sh

# Helper nuevo (opcional, AND inline también vale)
_host_pg_ready() {
  # Probe host-side TCP forward to postgres. Returns 0 if reachable.
  _tcp_probe "${HILO_POSTGRES_HOST}" "${HILO_POSTGRES_PORT}"
}

db_health() {
  # 0 = postgres reachable BOTH container-internal AND host-side
  # 1 = postgres should be up but not responding (one or both probes failing)
  # 2 = unknown (no compose CLI)
  if ! command -v curl >/dev/null 2>&1; then :; fi

  # Fast path: ambos probes deben pasar.
  if _resolve_compose >/dev/null \
     && _compose_pg_ready \
     && _host_pg_ready; then
    return 0
  fi

  # Fallback: si no hay compose CLI, sigue siendo válido un host TCP probe
  # (postgres podría estar corriendo nativo en el host).
  if ! _resolve_compose >/dev/null 2>&1; then
    if _host_pg_ready; then
      return 0
    fi
    return 2
  fi

  return 1
}
```

Y considerar subir `wait_for db_health 30` (línea 176, en
`_ensure_infra_essential`) a `60` para alinearlo con el de `db_reset` (línea
335). 30 s era suficiente cuando sólo se probaba container-internal; ahora
que se exige el port-forward del host también, 60 s da margen seguro y
sigue por debajo del SLA "fail-within-60s" del negative control.

> **Importante para el negative control**: el negative control hace
> `compose stop postgres` y espera fallo en ≤ 60 s. Si `wait_for db_health`
> sigue siendo 30 s en `_ensure_infra_essential` y 60 s en `db_reset`, ambos
> casos están cubiertos. Si el developer sube `_ensure_infra_essential` a 60 s,
> sigue cumpliendo el SLA (≤ 60 s, no == 60 s).

### F.4 — Mensaje de error claro (acceptance #4)

`wait_for` ya emite `warn "${name} did not respond within ${timeout}s"`.
Con `name="Postgres"` y `timeout=60` el mensaje será literalmente:
`Postgres did not respond within 60s`. El acceptance pide "Postgres did not
become ready" — semántica equivalente. Si el validator quiere literalidad
exacta, el developer puede añadir un `fail "Postgres did not become ready
within 60s (host TCP $HILO_POSTGRES_HOST:$HILO_POSTGRES_PORT closed)"` en
`db_reset` línea 336, en lugar de confiar en el warn genérico de `wait_for`.
La línea 336 (`fail "Postgres did not become ready within 60s."`) ya existe
en T008-current — el developer puede simplemente enriquecerla con detalle
del host TCP.

## G. Impact analysis

### G.1 — Consumidores de `db_health` y `wait_for db_health` en el repo

```
scripts/dev-restart.sh:159  → "required_fn" check (firma must existir)
scripts/dev-restart.sh:171  → print_status: case sobre el rc {0, 2, *}
scripts/dev-restart.sh:193  → --soft loop "all healthy" check
scripts/dev-restart.profile.sh:176  → _ensure_infra_essential, wait_for 30s
scripts/dev-restart.profile.sh:335  → db_reset, wait_for 60s
```

- El dispatcher (`dev-restart.sh`) sólo lee el exit code; con la nueva
  semántica "0 = ambos probes pasan" el comportamiento es estricto pero
  retrocompatible: nunca era válido reportar UP si alembic iba a fallar.
- `print_status` distingue {0 UP, 2 UNKNOWN, * DOWN}. La nueva implementación
  mantiene los tres valores (2 sólo cuando no hay compose CLI Y no hay host
  TCP); está OK.
- `--soft` (`back_health && front_health && db_health`) se vuelve un poco más
  estricto: si el port-forward host se cae sin tirar el contenedor, ahora reporta
  DOWN en vez de UP. Esto es mejor (refleja la realidad para el cliente alembic).

### G.2 — Timeouts y SLA

| Sitio | Antes | Propuesto | SLA acceptance |
|-------|-------|-----------|----------------|
| `_ensure_infra_essential` wait_for | 30 s | 60 s (recomendado) | n/a (no es el flujo del race) |
| `db_reset` wait_for | 60 s | 60 s (sin cambio) | "≤ 60 s success o fail" ✅ |
| `wait_for` step interno | 1 s sleep entre intentos | sin cambio | n/a |

### G.3 — Logging discipline

- `_tcp_probe` no requiere logging adicional (no toca credenciales).
- El `warn`/`fail` mensaje en `wait_for` y `db_reset` ya cumple. No hay password
  en juego (Bash /dev/tcp no necesita auth).
- `ENABLE_VERBOSE_LOGGING` no aplica (este es shell, no Python).
- No usar `set -x` ni `echo $POSTGRES_PASSWORD`. Mantener el `.env` opaco.

### G.4 — File-size discipline

`scripts/dev-restart.profile.sh` actual: 362 líneas. Está cerca del límite
~300 LOC para una single-responsibility file. El fix añade **a lo sumo 10 LOC**:
1 helper `_host_pg_ready` (3 líneas) + 2 líneas en `db_health` para el AND
+ posible mensaje de error enriquecido. Aceptable; no obliga a split.

Si el developer prefiere mantenerlo ultra-corto, puede inline `_tcp_probe`
directamente en `db_health` sin crear `_host_pg_ready` — ambas formas son
equivalentes; la versión con helper documenta mejor la intención.

### G.5 — Idempotencia

El fix debe ser idempotente entre runs. `_tcp_probe` y `_compose_pg_ready` son
puras; ningún side-effect. `db_health` ya es side-effect-free salvo el
`compose exec` que lanza un proceso transitorio dentro del contenedor.
Acceptance #1 (two consecutive --reset both exit 0) es la prueba directa.

### G.6 — Compatibilidad con setup-from-scratch.sh

`scripts/setup-from-scratch.sh` invoca el dispatcher; ya pasa por el camino
nuevo automáticamente. No requiere cambios en setup-from-scratch.

### G.7 — Negative control

Cuando `docker compose stop postgres` está aplicado:
- `_compose_pg_ready` → fails (container down).
- `_host_pg_ready` (TCP probe) → fails (no port-forward sin contenedor).
- `db_health` retorna 1.
- `wait_for db_health 60 "Postgres"` itera 60 s, cada iteración 1 s, y al final
  emite warn + return 1.
- `db_reset` línea 336 `fail "Postgres did not become ready within 60s."`.
- Exit code != 0. ✅ acceptance #4 cumple.

## H. Riesgos abiertos (referencia main-orchestrator R1..R4)

- **R1 (race residual en hardware lento)**: incluso con probe host-side,
  algunos laptops viejos pueden tardar > 60 s en abrir el port-forward tras
  `compose down -v + up`. Mitigación: subir timeout a 90 s sólo si el negative
  control se cumple en <60 s (no expandir SLA del fail). Validator/tester
  pueden cuantificar en evidencia.
- **R2 (Rancher vs Docker Desktop semántica)**: el `_resolve_compose` devuelve
  `docker compose` para ambos; el comportamiento del port-forward es distinto
  entre runtimes pero el probe TCP cubre ambos. No requiere branching por
  runtime.
- **R3 (compose exec sin TTY)**: `compose exec -T postgres pg_isready ...`
  podría devolver 0 con stderr ruidoso en algunos runtimes. Ya silenciado
  con `>/dev/null 2>&1`. Sin cambios.
- **R4 (port hijack)**: si otro proceso (postgres nativo, otro container)
  está escuchando 127.0.0.1:5432 a la vez que el compose, `_tcp_probe` daría
  un falso positivo. Esto NO es bug de este slice — es un pre-condición de
  entorno limpio. El acceptance no lo cubre. Documentar como "known limitation,
  fuera de scope" y dejarlo para FU si reaparece. `stop_orphan_on_port` ya
  cubre el caso de back/frontend; no extendido a 5432 deliberadamente.
