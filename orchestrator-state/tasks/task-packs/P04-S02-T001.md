## Reglas de negocio y UX (instrucciones.md + UX_CONTRACT.md)

- **instrucciones.md §3.1 RAG admin** (verbatim quote): "solo admins suben documentos; cada documento tiene idioma y vertical; no se indexan documentos sin colección; embeddings se recalculan por versión; vectorización se ejecuta en Celery."
  - Implications for the UI:
    - Page is admin-only (RequireRole guard). Employee gets bounced to `/chat`.
    - Upload form **requires** `title`, `language` (`es|en|fr` selector), `collection_id` (selector populated from `GET /api/v1/admin/rag/collections`). All three are server-enforced required Form fields.
    - "no se indexan documentos sin colección": index button is enabled only when `document.collection_id != null`. Backend will return 422 otherwise.
    - "embeddings se recalculan por versión" + "vectorización en Celery": after re-upload of same sha256, backend returns 200 dedup or new document_version row; index is async, UI must POLL the document's `status` to surface `processing` → `indexed` / `failed`.
- **instrucciones.md §3.2 LAS FEATURES**: this slice is part of "Admin RAG documentos y colecciones" feature. Acceptance requires loading/empty/error/permission/success **plus** transient `uploading` and `indexing`.
- **Personas** (UX_CONTRACT §2): "People Tech Admin" persona only. Auditor is RO but uses `/admin/audit`, not this page.
- **UX_CONTRACT §3 Screen inventory row** (verbatim):
  | Route | Page | Journey | Required UI states | Real data contract |
  |---|---|---|---|---|
  | `/admin/rag/documents` | `RagDocumentsPage` | J104 | **loading, empty, uploading, indexing, error_network, error_validation, permission_denied, success** | "PDF/DOCX real/proporcionado desde `data/verification` y colección activa" |
- **Required UI states — 8 mandatory** (resolve §6.1 ∪ UX_CONTRACT §3):
  1. **loading** — initial list fetch and per-doc refresh. `aria-busy="true"` skeleton or live region.
  2. **empty** — admin landed but `data: []`. Wordmark + body copy ("No hay documentos indexados") + upload affordance.
  3. **uploading** — in-flight `POST /api/v1/admin/rag/documents`. Show progress affordance (textual percentage or indeterminate StatusDot). Disable form re-submit. NEVER simulate progress when backend is the source of truth — use `fetch` upload progress only if available, otherwise show indeterminate.
  4. **indexing** — at least one row has `status ∈ {uploaded → processing}` and is being polled. StatusDot + TrackedLabel "INDEXANDO" (i18n). Poll via TanStack `refetchInterval` (recommended 3–5s) for documents in non-terminal state; stop polling when all are `indexed|failed`.
  5. **error_network** — `ChatNetworkError`-equivalent for RAG (call it `RagNetworkError`). Inline + retry CTA. Surfaced on list, upload and index failures.
  6. **error_validation** — RFC field-level mapping from 422 errors: invalid `title`, unsupported `language`, missing `collection_id`. Inline near offending field via `aria-describedby`. 413 `RAG_DOCUMENT_TOO_LARGE` maps to a validation error tied to the file input.
  7. **permission_denied** — `ForbiddenView` for 403 surface (e.g. if a token momentarily lost its admin role). 401 is handled upstream by `authFetch` single-flight refresh + RequireAuth; not surfaced here.
  8. **success** — list rows update, dedup case surfaced as "Existente: <title>" toast/inline note, index `202` flips row to indexing+spin.
- **Accessibility** (UX_CONTRACT §7 + rule 01-non-negotiables §Accessibility):
  - Every interactive control labelled. File input has explicit `<label>` and visible focus.
  - Tap targets ≥ 44×44 px. Contrast AA. Keyboard-only flow end-to-end (Tab to file, Enter on dropzone, Tab to fields, Enter on submit).
  - Dynamic status changes announced via `aria-live="polite"` (e.g. "Documento subido. Indexando.").
  - Never convey status with color alone — combine StatusDot + TrackedLabel uppercase text.
- **i18n** (rule 01-non-negotiables §i18n + UX_CONTRACT §5): all visible text uses `t("rag:documents.*")` (extends existing namespace block at index.ts lines 173 / 315 / 457). No hardcoded ES strings.
- **Security** (rule 01-non-negotiables §Security): never log filename PII; log `language`, `size_kb_bucketed` (e.g. <100, <1000, <25000), `mime_type`, `content_type`, `dedup_hit`, `request_id`. Never log token/cookie content. Access token in memory only (no `localStorage`), refresh via HttpOnly cookie — already provided by `authFetch`.

## Riesgos y decisiones previas

- **R-1 (HIGH) — Real PDF fixture missing**. `backend/tests/data/verification/people_docs/politica_vacaciones_es.pdf` does NOT exist on disk, and the `rag_docs` bootstrap loader is explicitly deferred (`reason="documents deferred (no load impl yet)"`). Mitigations:
  - Developer/tester/slice-verifier generate a small valid PDF at runtime. Easiest path: `python -c 'from reportlab.pdfgen.canvas import Canvas; c=Canvas("/tmp/policy.pdf"); c.drawString(72,720,"Política de vacaciones — verification fixture"); c.save()'` if `reportlab` is in backend deps, OR use a 1-byte-correct minimal PDF skeleton (8 lines) committed nowhere (live in `/tmp` only). Document path + sha256 in handoff.
  - DO NOT create FU for this: it is not in this slice's `Write set`, but it is also not blocking — the page can be exercised end-to-end with a runtime-generated PDF, which is what `/verify-slice` will use. If during execution it becomes clear the journey verification (P05-S01-T005) actually needs a real persisted PDF, that's an out-of-scope FU for that journey slice, NOT for P04-S02-T001.
- **R-2 (HIGH) — RBAC gating + admin user MFA state**. `data/verification/users/admin_peopletech.json` carries `roles:["people_admin"]` and no `mfa.enabled` field. The auth flow is JWT + optional MFA. Verify during dev that admin sign-in returns `mfa_required:false` (or include a 2FA step in the verification script if it does). If MFA is on for admin → slice-verifier must complete TOTP using `data/verification/auth/mfa_primary.json` (which is the employee secret — admin would need its own; check before verify).
- **R-3 (MEDIUM) — Multipart upload via FormData**. NEVER set `Content-Type: multipart/form-data` manually in `authFetch`; the browser must add the boundary. `authFetch` already sets `credentials:"include"` and Bearer. Confirm `authFetch` does NOT force JSON content-type when body is `FormData` (likely it sets `Content-Type: application/json` only when body is non-FormData). If it does, pin §D-RAGDOC-AUTHFETCH-FORMDATA — the developer must either (a) verify `authFetch` already branches on body type or (b) add the smallest safe fix to skip JSON header for FormData. Either way, do this in feature data layer (no edit to `authFetch` unless necessary; if needed, treat as WRITE_SET_DRIFT and propose).
- **R-4 (MEDIUM) — Polling cadence + cost**. `refetchInterval` of 3s during indexing is fine for verify, but for production traffic with many docs it can saturate. Decision: gate `refetchInterval` to **only** activate when at least one visible row is in non-terminal state (`uploaded|processing`); stop when all rows are `indexed|failed`. TanStack v5 idiom: `refetchInterval: (query) => hasInflight(query.state.data) ? 3000 : false`. Document as §D-RAGDOC-POLLING-GATED.
- **R-5 (MEDIUM) — Verification data contract gap, propagate to verify-slice**. As above (R-1), the verification data contract row for J104 in TECHNICAL_GUIDE §6.5 claims a PDF and a collection should be loaded by `--only rag_docs`. The collection IS loaded (via `rag_chat`); the PDF is not. Slice-verifier MUST document in `## verify-slice → DATA_SETUP` whether it uploaded a runtime-generated PDF, the sha256, and that the journey closure (P05-S01-T005) will need a separate FU if a persisted real fixture is required.
- **R-6 (LOW) — UX_CONTRACT 8 states includes `indexing` AND `uploading`** while TECHNICAL_GUIDE §6.1 lists only 7 (omits `indexing`). The UX_CONTRACT is the canonical UX source; implement BOTH. Pin §D-RAGDOC-STATES-EIGHT.
