### F.3 UserProfile DTO (NEW ‚Äî DISCREPANCY-2 below)

The TECHNICAL_GUIDE ¬ß6.2 says `{data: UserProfile}` but ¬ß6.3 only defines separate `User` and `EmployeeProfile` Pydantic models. T007 must pin `UserProfile` as the **merged shape** consumed by both ChatHomePage and AccountPage. Pin it here:

```python
class UserProfileEmployeeFields(BaseModel):
    employee_id: str
    brand: str
    society: str
    center: str
    country: str
    department: str
    # extra_metadata intentionally omitted from response ‚Äî internal use
    # Reason: prevents leaking jsonb metadata to UI by default; if a future
    # screen needs it, expose a specific dot-path, not the full blob.

class UserProfile(BaseModel):
    id: uuid.UUID
    email: EmailStr
    full_name: str
    status: Literal["active","inactive","pending","locked"]
    preferred_language: Literal["es","en","fr"]
    roles: list[str]                       # ["employee"] | ["admin"] | ...
    employee_profile: UserProfileEmployeeFields | None  # admin users have no profile
    created_at: datetime                   # ISO-8601 UTC
    updated_at: datetime                   # ISO-8601 UTC

class UserProfileResponse(BaseModel):
    data: UserProfile
    meta: ResponseMeta
    errors: list[Any] = []
```

Field rationale:

- `status`: needed so the frontend can render a "your account is disabled" banner without a separate call. Already in the DB; cheap to expose.
- `roles`: required by `Account` & `Chat` for role-gated UI. Sourced from `user.user_roles[].role.name` relationship; defaults to `["employee"]` if empty (mirrors `encode_access_token` line 105).
- `employee_profile`: nullable. The admin verification user (`admin.peopletech@inditex-sandbox.com`) does NOT have an `employee_profile` row ‚Äî confirmed by `data/verification/users/admin_peopletech.json` (no `employee_profile` key). For that user the response is `"employee_profile": null`. AccountPage must render the "no profile" state gracefully (UX permission_denied / empty).
- `password_hash` is **NEVER** included. Hard rule.
- `extra_metadata` (the `metadata` JSONB column) is omitted by default ‚Äî see commentary.
- `id`, `created_at`, `updated_at` are exposed as UUID + ISO-8601 strings (Pydantic v2 default for datetime).

## H. DISCREPANCIES (must reconcile before coding)

### DISCREPANCY-1 ‚Äî PATCH /me/language response: 200 or 204?

- TECHNICAL_GUIDE ¬ß6.2 row 263: `Response: {data:UserProfile}` ‚Üí implies **200 with body**.
- Other PATCH-like endpoints: POST /auth/reset-password returns `{data:{reset:true}}` 200; POST /auth/logout returns 204 No Content.
- Decision: **200 with full UserProfile body**. Reasons:
  1. ¬ß6.2 explicitly says `{data: UserProfile}` for this row.
  2. AccountPage will want to reflect the new state immediately without a refetch.
  3. Idempotent identity: caller can trust the server-confirmed value (not just echo).
- Resolution: no source-of-truth amendment needed; ¬ß6.2 is consistent. Pinning here for the developer.

### DISCREPANCY-2 ‚Äî UserProfile schema not defined in ¬ß6.3

- ¬ß6.3 defines `User` and `EmployeeProfile` as separate Pydantic models, but ¬ß6.2 row 262 references an undefined `UserProfile`.
- The existing `MfaUserDto` (`backend/app/auth/schemas.py:400-417`) returns `{id, email, preferred_language, roles}` ‚Äî INSUFFICIENT for GET /me (missing employee fields, full_name, status, created_at, updated_at).
- Pinning UserProfile shape in ¬ßF.3 above. This IS the contract T007 ships.
- Resolution: developer pins UserProfile in `backend/app/users/schemas.py`. No source-of-truth amendment required (the ¬ß6.2 row says "UserProfile" ‚Äî we are giving it a concrete definition). **Flag in handoff ¬ßI "schema contract pinned": link to this task pack ¬ßF.3 so future devs find it.**

### DISCREPANCY-3 ‚Äî Admin user has no employee_profile

- `data/verification/users/admin_peopletech.json` has no `employee_profile` key.
- `data/verification/users/employee_primary.json` has a full one.
- The DB schema `employee_profiles.user_id` is FK NOT NULL but `users` may not have a row in `employee_profiles` (1:0..1 relationship).
- Decision: GET /me returns `"employee_profile": null` for the admin user. AccountPage P03-S02-T004 must accept null and not render the empty employee block.
- **Test T01-A (admin)** required: GET /me as admin ‚Üí `employee_profile=null`, `roles=["admin"]`.

### DISCREPANCY-4 ‚Äî `encode_access_token` checks `user.employee_profile_id` but the ORM exposes `user.employee_profile` (relationship)

- `tokens.py:108-110`: `if hasattr(user, "employee_profile_id") and user.employee_profile_id:`. This is always False today (the attribute does not exist on the User ORM). The JWT claim `employee_profile_id` is therefore always `None`.
- T007 does NOT depend on this claim (it loads the profile from DB via the `sub` claim, not from JWT).
- **Not a T007 fix.** Document for future researcher / debugger. The JWT claim is currently effectively unused.
- Resolution: NOT IN SCOPE. Flag in handoff as residual risk R-Future.

### DISCREPANCY-5 ‚Äî `Account` UI state `error_validation` for /me

- UX_CONTRACT line 27 says `/account` requires `error_validation` state. Validation only happens on PATCH /me/language. GET /me has no body ‚Üí no validation. AccountPage's error_validation state is only triggered by the PATCH path. T007's job is to make PATCH return a 400 with `errors[0].field=="language"` and a localized message so AccountPage's react-hook-form can highlight the field. Confirmed by ¬ßF.2 above.
- Resolution: no contract change. Test T05 covers this.
