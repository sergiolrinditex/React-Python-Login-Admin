### §3.4 TECHNICAL_GUIDE §6.2 — Endpoint contract (verbatim row)

> POST `/api/v1/auth/sign-in` · Request `{email,password}` · Response
> `{data:{mfa_required,access_token?}}` · No auth · Errors `400,401,423` ·
> Consumer SignInPage/J100 · Side effects `refresh_tokens, audit_logs` · Slice
> `P01-S02-T002`.

**Authoritative response shape (verified against `backend/app/auth/routers/sign_in.py`
+ `schemas.py`, since the guide's `access_token?` shorthand is ambiguous):**

```jsonc
// 200 success (no MFA) — refresh cookie also set as Set-Cookie HttpOnly
// SignInResponseSuccess
{
  "data": {
    "mfa_required": false,
    "access_token": "eyJ...",           // JWT HS256 (AUTH_ACCESS_TTL_SECONDS default 1800)
    "token_type": "Bearer",
    "expires_in": 1800
  },
  "meta": { "request_id": "<uuid>" },
  "errors": []
}

// 200 MFA challenge — NO Set-Cookie, NO access_token
// SignInResponseMfaChallenge
{
  "data": {
    "mfa_required": true,
    "mfa_challenge_token": "eyJ...",    // short-lived JWT (AUTH_MFA_CHALLENGE_TTL_SECONDS)
    "expires_in": 300
  },
  "meta": { "request_id": "<uuid>" },
  "errors": []
}

// 4xx envelope
{ "data": null, "meta": { "request_id": "<uuid>" },
  "errors": [{ "code": "AUTH_INVALID_CREDENTIALS|...", "message": "...", "field": null, "details": null }] }
```

Error codes the SignInPage must map to user-facing copy:
- `400` → `AUTH_SIGNIN_VALIDATION` (rare; client zod should catch first) → `error_validation`.
- `401` → `AUTH_INVALID_CREDENTIALS` (aggregate-401: unknown email + wrong password
  return BYTE-IDENTICAL body — must NOT reveal which) → `error_validation` with
  generic "Email o contraseña incorrectos".
- `423` → `AUTH_ACCOUNT_LOCKED` (5/900s sliding window) → `permission_denied` with
  "Cuenta bloqueada temporalmente, intenta de nuevo más tarde".
- `429` → `AUTH_SIGNIN_RATE_LIMITED` (`AUTH_SIGNIN_RATE_PER_MINUTE` default 20, IP-scoped,
  `Retry-After` header) → `permission_denied` with rate-limit copy.
- `500` → `AUTH_SIGNIN_INTERNAL_ERROR` → `error_network` fallback copy.
- `TypeError`/`AbortError`/no Response → `error_network` (network/offline).

### §3.10 Open contract decisions (NOT in docs — developer must record in handoff)

These are gaps in source-of-truth the developer fills with a documented decision:

- **D-T001-CHALLENGE-TRANSPORT**: How does the `mfa_challenge_token` reach
  TwoFactorPage? Options: (a) `navigate('/auth/2fa', { state: { challenge_token, expires_in } })`
  via react-router v7; (b) `sessionStorage` with timed clear; (c) querystring (REJECTED
  — token must not appear in URLs/history/logs). Recommendation: **option (a)**
  (router state); the SignInPage hands it via `navigate(target, { state })` and
  TwoFactorPage reads via `useLocation().state` with a defensive fallback that
  redirects back to `/auth/sign-in` if state is missing (e.g. tab opened directly
  with `/auth/2fa`).
- **D-T001-NEXT-FORWARD**: When the response is MFA challenge, forward `?next=`
  to `/auth/2fa?next=<safe>` so the deep link survives the 2-step flow.
- **D-T001-USERFETCH-ON-SUCCESS**: The no-MFA response contains `access_token` but
  NOT `user`. Therefore, SignInPage must (after `setAccessToken(access_token)`)
  call `repo.fetchMe(token)` and only then call `signInAccepted(token, user)`.
  Alternative considered: signInAccepted accepts a deferred user — REJECTED
  (would require changing the AuthProvider contract, larger blast radius).
- **D-T001-PASSWORD-MIN**: `instrucciones.md` says "contraseña mínima" without a
  number; the backend `SignInRequest` allows `min_length=1` (it does NOT enforce
  policy on sign-in — only on sign-up). Frontend zod schema MUST reject empty +
  trim, but MUST NOT enforce sign-up policy on sign-in (it would block legitimate
  legacy passwords). Use `password.min(1)` with a generic "obligatorio" message.
  Document the asymmetry.
- **D-T001-EMAIL-CORP**: "email corporativo" — backend does NOT enforce a domain
  allowlist (P01-S02-T002 has none). Frontend uses zod `.email()` only; do NOT
  add a domain regex (out of scope; would be a follow-up if/when the contract
  declares allowlisted domains).
- **D-T001-LOCALE-AT-VIEW**: SignInPage is rendered before authentication, so we
  use the i18next current detected language (browser/i18n-browser-languagedetector,
  fallback `es`). Post-login, AccountPage syncs to `preferred_language`. No
  cross-cutting language store change in this slice.

## §7. Logging contract

`frontend/src/features/auth/data/logger.ts` already provides `logVerbose / logWarn / logError`
gated by `import.meta.env.VITE_ENABLE_VERBOSE_LOGGING`. Reuse it.

Prefix every event with `auth.signin.<phase>`:
- `auth.signin.submit.start` BEFORE the call. **Allowed metadata**: `email_domain` (split `@`, never the local part), `password_len` (number), `request_id` (UUID generated client-side).
- `auth.signin.submit.ok_no_mfa` AFTER success. **Allowed**: `token_len`, `expires_in`, `request_id`. **Never**: the token, the email local part.
- `auth.signin.submit.mfa_required` AFTER MFA branch. **Allowed**: `challenge_token_len`, `expires_in`, `request_id`.
- `auth.signin.submit.invalid_credentials` WARN. **Allowed**: `status=401`, `request_id`.
- `auth.signin.submit.locked` WARN. **Allowed**: `status=423`, `request_id`.
- `auth.signin.submit.rate_limited` WARN. **Allowed**: `status=429`, `retry_after`, `request_id`.
- `auth.signin.submit.network` ERROR (no exc_info dump in browser console — message only).
- `auth.signin.submit.unexpected` ERROR with status code only.

Hard rules (instrucciones §3 + non-negotiables §Logging):
- **NEVER log password.**
- **NEVER log access_token, refresh cookie, mfa_challenge_token in full.** Only `.length`.
- **NEVER log full email** (the local part is PII). Domain only.
- `request_id` (UUID v4 generated client-side; backend echoes it in `X-Request-ID` response header) must travel through every event for end-to-end correlation.
- Toggle `VITE_ENABLE_VERBOSE_LOGGING=true` shows full flow; `false` shows only warning+error. **No code change** required to flip — only env.
