## Why this matters (real production bug)

Any developer running the canonical onboarding flow:

```bash
cp .env.example .env
bash scripts/dev-restart.sh   # internally does: set -a; source .env; set +a
```

hits exit 127 with `People: command not found`. Bash treats `MAIL_FROM_NAME=Hilo People` as `MAIL_FROM_NAME=Hilo` followed by an attempt to execute the command `People`. Pydantic-settings parses fine when reading `.env` directly (it does NOT shell-source), but `dev-restart.sh` and any operator running `source .env` for ad-hoc inspection, debugging, or curl-with-real-key flows is blocked. The fix is one line and reversible.

1. **Scope bleed (primary risk)**: developer touches other lines of `.env.example` ("while we're here"). Mitigation: hard rule above + 2-line `git diff` review by validator + `git diff --stat` showing exactly `1 file changed, 1 insertion(+), 1 deletion(-)`.
2. **CRLF / encoding drift**: editor adds BOM or converts LF→CRLF. Mitigation: developer checks `file .env.example` shows `ASCII text` (not `UTF-8 Unicode (with BOM)` or `with CRLF line terminators`) before handoff.
3. **Quote style**: must use double quotes (`"Hilo People"`), not single quotes. The verification command `grep 'MAIL_FROM_NAME="Hilo People"' .env.example` literally checks for double quotes. Single quotes would also work for bash but would fail the verification gate.
4. **No app-code regressions expected**: pydantic-settings via `python-dotenv` correctly strips both `"..."` and `'...'` wrappers when parsing `.env`, so backend config (`backend/app/core/config.py`) keeps reading `MAIL_FROM_NAME=Hilo People` as a Python str of value `Hilo People` (no leading/trailing quote characters). No backend test should break.
