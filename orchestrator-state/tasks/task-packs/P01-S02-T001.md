### §6.3 Sign-up flow (use case sequence)

```
SignUpUserUseCase.run(req: SignUpRequest, request_meta: RequestMeta) -> SignUpResponseData:
  1. log BEFORE: "auth.sign_up.start" (email_masked, request_id)
  2. assert legal_acceptance is True            → else LegalAcceptanceMissingError (422)
  3. validate corporate-email rule              → else NonCorporateEmailError (422)
  4. validate password strength                 → else WeakPasswordError (422)
  5. check duplicate                            → if exists: EmailAlreadyExistsError (409, non-leaky)
  6. hash password (argon2)                     → password_hash
  7. INSERT into users (email, password_hash, full_name, status='active', preferred_language='es')
  8. INSERT into employee_profiles (user_id, employee_id=hash-derived, brand='Hilo', society='Hilo People', center='HQ', country='ES', department='', metadata={})
  9. INSERT into audit_logs (actor_user_id=user_id, action='auth.sign_up', entity_type='user', entity_id=user_id, metadata={ip, user_agent, request_id, resource:'auth.signup'})
  10. log AFTER: "auth.sign_up.ok" (user_id, mfa_required=true, request_id)
  11. return SignUpResponseData(mfa_required=True, user_id=...)
```

Hard rules (project non-negotiables apply):
- **NEVER** log `password`, `password_hash`, `legal_acceptance`. Bind email as `email_masked` (e.g. `s.l***@gmail.com`).
- **NEVER** include the password or password_hash in the audit_log metadata.
- Never `catch` generic Exception in service.py; catch typed domain errors. Repository can catch `IntegrityError` to map to `EmailAlreadyExistsError` (race-condition safety net for the unique-email constraint).

### §6.4 Validation rules (concrete numbers — no ambiguity)

- **email**: Pydantic `EmailStr` (RFC-5322). Plus the corporate-email rule (R3 below).
- **password**: minimum 12 characters, must contain at least one uppercase letter, one lowercase letter, one digit, and one symbol. Implementation: simple regex helpers in `service.py` (no extra dependency). Error code `AUTH_WEAK_PASSWORD`. Source: `01-non-negotiables.md §Security` ("Passwords hashed with bcrypt/argon2") + the project's existing seed loader uses 14-character passwords; 12 is a conservative production minimum.
- **full_name**: 1..200 chars after `.strip()`. Empty/whitespace-only → 422.
- **legal_acceptance**: must be `True`. Pydantic `bool` does not enforce truthiness; check explicitly in service.py. Error code `AUTH_LEGAL_ACCEPTANCE_REQUIRED`.
- **preferred_language**: NOT in request (defaults to `'es'` from §10.3 server_default; user can change via `PATCH /api/v1/users/me/language` in T007).

## §9. Open decisions answered (from prompt)

### R3 — Corporate email rule

**Source-of-truth answer**: instrucciones.md says only "email corporativo" without enumerating domains. The verification employee `s.lopezrap+employee@gmail.com` is a Gmail address (intentional for dev). A naive blocklist of free providers (`gmail.com|hotmail.com|yahoo.com|outlook.com`) would BREAK the verification user.

**Pragmatic answer (USER-APPROVED implicit by precedent)**: introduce a config-driven allowlist `CORPORATE_EMAIL_DOMAINS` (env var, comma-separated). Empty = permissive (dev/test). Non-empty = strict allowlist. Production deployment populates the env with the company's real domain(s). This is the same shape as `MCP_ALLOWLIST_DOMAINS` (§11.1) — the project already uses config-driven domain whitelists.

Implementation in `service.py`:
```python
def _validate_corporate_email(email: str, allowed_domains: list[str]) -> None:
    if not allowed_domains:
        return  # permissive in dev
    domain = email.rsplit("@", 1)[-1].lower()
    if domain not in allowed_domains:
        raise NonCorporateEmailError(domain=domain)
```

Documented in `.env.example` and TECHNICAL_GUIDE §11.1 amendment in a follow-up doc-slice (NOT this slice).

### D3 — `mfa_required` value in response on first sign-up
