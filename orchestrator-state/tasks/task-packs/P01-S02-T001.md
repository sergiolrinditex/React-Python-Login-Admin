Response envelope is the project-wide standard from ¬ß6.2 header: `{data, meta, errors}`. Error item shape: `{code, message, field, details}`.

### C.2 HTTP status code pin (planner decision, ¬ßH)

201 Created on success ‚Äî this slice CREATES a resource (the user). The ¬ß6.2 cell does not pin a number; planner pins **201** because:
- the route creates a new persistent row,
- the response includes `user_id`,
- 200 OK is reserved for idempotent reads / non-creating successes elsewhere in ¬ß6.2 (refresh, logout-204, etc.).

### C.3 Error code semantics (¬ß6.4 + planner mapping)

¬ß6.4 lists project-wide auth codes. For sign-up specifically, planner pins these mappings inside the envelope's `errors[].code`:

| HTTP | `code` (envelope) | Cause |
|---|---|---|
| 422 | `AUTH_SIGNUP_INVALID_PAYLOAD` | Pydantic validation failed (missing field, wrong type, password too short, full_name empty, `legal_acceptance != true`). Pydantic 422 with `errors[].field` populated per field. |
| 400 | `AUTH_SIGNUP_NON_CORPORATE_EMAIL` | Email syntactically valid but domain NOT in allowed corporate set. Returned BEFORE checking existence ‚Äî does not depend on DB. |
| 409 | `AUTH_SIGNUP_EMAIL_TAKEN` | Email already exists in `users`. **Generic 409 message** ("an account with that email exists or cannot be created"). DO NOT leak whether the email was in DB vs blocked by policy when both fail ‚Äî return the more specific 4xx for the case that matched. Per ¬ß11.2 non-negotiables (no user enumeration). |
| 400 | `AUTH_SIGNUP_LEGAL_NOT_ACCEPTED` | `legal_acceptance` is `false`. Optionally fold into 422; planner pins 400 here so frontend can distinguish quickly. |

Note: ¬ß6.4 lists `AUTH_INVALID_CREDENTIALS` and `AUTH_MFA_REQUIRED` but NOT sign-up-specific codes. The planner mints `AUTH_SIGNUP_*` codes to fit the existing prefix family. Validator should approve as a non-breaking extension.

### C.4 Auth strategy (¬ß10.2)

> "JWT propio. Access token Bearer de vida corta. Refresh token en cookie HttpOnly Secure SameSite=Lax, guardado hasheado y rotado. Password hashing con **Argon2**. Claims: `sub`, `email`, `roles`, `preferred_language`, `employee_profile_id`, `iat`, `exp`, `jti`. Guards: `require_user`, `require_admin`, `require_auditor`, `require_role`."

Implication for T001: **sign-up does NOT issue JWT, does NOT set refresh cookie**. It returns `{mfa_required, user_id}` so the frontend can route to the 2FA setup screen if `mfa_required=true` (admin) or sign-in flow otherwise. Token issuance lives in sign-in (T002) and 2FA verify (T006). This avoids accidental "logged-in just by registering" patterns and keeps sign-in as the single token-issuing entry point.

`mfa_required` value: `false` for new employees (no admin role granted at sign-up). The field is in the contract for forward-compatibility with admin onboarding flows.

### C.5 DB schema relevant fields (¬ß10.3)

**`users`** (FK targets the columns this slice writes):
- `id UUID PK` (server_default `gen_random_uuid()` via pgcrypto)
- `email TEXT NOT NULL UNIQUE` ‚Äî DB-level uniqueness enforces 409 even on race. Catch `UniqueViolation`.
- `password_hash TEXT NOT NULL` ‚Äî Argon2id PHC string.
- `full_name TEXT NOT NULL`
- `status TEXT NOT NULL DEFAULT 'active'` ‚Äî sign-up creates `active` users (no email-verify step in V1).
- `preferred_language TEXT NOT NULL DEFAULT 'es'` ‚Äî CHECK in (`es`,`en`,`fr`). Default `'es'` ‚Äî request body does NOT include language for sign-up (frontend can patch later via P01-S02-T007 `PATCH /users/me/language`). Planner pins: do not accept language in sign-up payload to keep contract narrow.
- `created_at`, `updated_at` ‚Äî server-defaulted via `now()`.

**`employee_profiles`** (request body ¬ß6.2 does NOT include employee fields, but the ¬ß6.2 "Tables/side effects" cell DOES list `employee_profiles`):
- ¬ß6.2 contract: `{email, password, full_name, legal_acceptance}` ‚Äî no employee fields.
- ¬ß10.3 schema: `employee_profiles` rows require `employee_id`, `brand`, `society`, `center`, `country`, `department` (all NOT NULL).
- **DECISION (planner, ¬ßH.1)**: T001 creates a `users` row + writes an `audit_logs` row, but does **NOT** create an `employee_profiles` row. Rationale: ¬ß6.2 request payload has no employee fields and we cannot invent valid values. The employee profile is enriched later by HR onboarding (out of MVP V1 scope) or by the verification_data loader for sandbox users. This matches the M1 verification script which seeds employee_profile from `data/verification`, not from sign-up. The registry "Tables/side effects" cell stays consistent because `employee_profiles` IS impacted in J100 (read by `GET /users/me` P01-S02-T007), just not written by T001.
  - Validator must approve this scope reading. If they disagree, the alternative is to create a placeholder `employee_profiles` row with NULL placeholders ‚Äî but that violates ¬ß10.3 NOT NULL constraints. There is no third option without expanding the ¬ß6.2 request contract.

**`audit_logs`** (write-only this slice):
- `actor_user_id` = the freshly created user's UUID (NULL for failed sign-up attempts where no user exists).
- `action` = `"auth.sign_up"` (planner pin ‚Äî matches ¬ß14 audit_actions naming pattern `auth_*`; using dot-form for consistency with the ¬ß14 mention of `auth_sign_in`, `auth_2fa_verified`).
- `entity_type` = `"user"`, `entity_id` = new user's UUID.
- `metadata` JSONB = `{request_id, ip, user_agent, outcome: "success"|"rejected", reason?: "<code>"}`. NEVER include password, email plaintext in metadata when outcome is `"rejected"` due to non-corporate email ‚Äî for that case log domain only (e.g. `metadata.rejected_domain="gmail.com"`) to support compliance review without leaking employee PII.

## F. Decisions resolved by planner

These are recorded so the developer does not re-debate them.

### F.1 Email validation: `pydantic.EmailStr` + custom domain validator
- Use `pydantic.EmailStr` for syntactic email validation (RFC 5322 + DNS-safe).
- That requires the `email-validator` runtime lib. It is **NOT pinned yet** in `backend/requirements.txt`. Pinning it IS a WRITE_SET_DRIFT (¬ßI.1) acknowledged in advance ‚Äî the developer adds `email-validator==<latest stable confirmed by researcher>` to both `pyproject.toml` and `requirements.txt`.
- Layer a custom field validator (or a separate domain rule) that rejects non-corporate domains. Implement the allowlist as a config-driven list `CORPORATE_EMAIL_DOMAINS` (env var, comma-separated, default `inditex.com,inditex-sandbox.com,zara.com,zarahome.com,massimodutti.com,bershka.com,pullandbear.com,stradivarius.com,oysho.com,lefties.com`). Empty value ‚Üí reject all ‚Üí fail-closed. Document in the response envelope `errors[].field="email"` and `code=AUTH_SIGNUP_NON_CORPORATE_EMAIL`.
- The verification data user `employee.verification@inditex-sandbox.com` MUST match the default allowlist. Default includes `inditex-sandbox.com` ‚Äî ‚úÖ.

### F.2 Password policy
- Minimum 12 chars, must contain at least: 1 letter and 1 digit. KISS ‚Äî no symbol requirement (industry trending away from arbitrary symbol rules). Max length 256 (Argon2 PHC strings handle up to ~1 KB; cap server-side to prevent DoS via huge passwords).
- The frontend should mirror the rule client-side, but the backend is canonical.
- The verification data user password `VerifyPass2024!` satisfies the rule (>=12, has letter+digit; symbol allowed but not required).

### F.3 Argon2 parameters (defaults are OWASP 2024)
- Use `argon2.PasswordHasher()` with library defaults: `time_cost=3, memory_cost=64*1024 KiB = 64 MiB, parallelism=4, hash_len=32, salt_len=16`. These are the OWASP 2024 minimums for Argon2id (researcher should confirm ‚Äî see ¬ßM.3 question).
- Always `ph.check_needs_rehash(stored_hash)` on verify (relevant in T002, not here). For T001 just `ph.hash(plain)`.
- Never log the resulting hash. Never log the input password (already required by ¬ß10.5).

### F.4 Rate-limit policy on this endpoint ‚Äî **implement now, configurable**
- Plataforma de rate-limit (P02-S02-T001) NO existe. Mandato del main-orchestrator: implementar AHORA con un middleware local activo y configurable.
- Implementaci√≥n m√≠nima inside `app/auth/`:
  - In-memory token-bucket keyed by IP (from `request.client.host` + `X-Forwarded-For` first value if present). Production deployment will replace with Redis backend in P02-S02-T001; the in-memory limiter is acceptable for V1 single-instance dev and stops obvious bot floods.
  - Default: 10 sign-up attempts per IP per 60 seconds, configurable via env var `AUTH_SIGNUP_RATE_PER_MINUTE` (int, default `10`). 0 disables (NOT recommended; document in code).
  - Exceeded ‚Üí respond `429 Too Many Requests` with `errors[].code="AUTH_SIGNUP_RATE_LIMITED"`, `Retry-After` header.
- This is a write-set extension only if it lives outside `app/auth/`. Planner pins it inside `app/auth/rate_limit.py` so it stays inside the declared `Write set`. It will be REPLACED by the platform middleware in P02-S02-T001 ‚Äî document that explicitly in `rate_limit.py` module docstring with a `TODO(P02-S02-T001): replace with platform middleware`.
- Tests must cover: 10 OK, 11th returns 429; counter window resets after 60s (use `time.monotonic` mockable).
- This is a real, active control. Not a stub. Not "configurable but off by default". DEFAULT IS ON.

### F.5 Audit log fire pattern
- Synchronous within the sign-up transaction. Same DB session, same transaction. Rationale: ¬ß10.5 requires AFTER log + audit; a failed audit insert must roll back the user creation (otherwise we'd have a user with no audit trail ‚Äî violates ¬ßSecurity/Audit log in non-negotiables).
- For failed sign-ups (non-corporate email, taken email), audit still fires but in its own short transaction (no user row to commit). `actor_user_id` NULL for rejections where user does not exist.
- Async fire-and-forget audit (e.g. Celery) is NOT required by ¬ß10.5 and adds operational complexity. KISS.

### F.6 Transaction strategy
- One DB transaction: `BEGIN ‚Üí INSERT users ‚Üí INSERT audit_logs ‚Üí COMMIT`.
- On `UniqueViolation` from `users.email`: rollback, then open a SHORT transaction to write the audit `auth.sign_up rejected reason=EMAIL_TAKEN`, then return 409. Never include the existing user's id in the response or audit (no user enumeration).

### F.7 NO `employee_profiles` row created at sign-up (cross-reference ¬ßC.5)
- Already discussed in ¬ßC.5. Final pin: T001 does not write `employee_profiles`. If frontend (P03-S01-T002 SignUpPage) needs employee fields collected at registration time, that is a ¬ß6.2 request-schema amendment owned by source-of-truth, not this slice.

### F.8 Response codes already pinned
- 201 success; 400 non-corporate email / legal not accepted; 409 email taken; 422 payload invalid; 429 rate-limited.
