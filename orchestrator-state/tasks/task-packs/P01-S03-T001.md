### C.4 §10.2 Auth strategy + token storage policy (LITERAL — critical)

From TECHNICAL_GUIDE §10.2: "JWT propio. Access token Bearer de vida corta. Refresh token en cookie `HttpOnly; Secure; SameSite=Lax; Path=/api/v1/auth`, guardado hasheado y rotado. Password hashing con Argon2. Claims: `sub`, `email`, `roles`, `preferred_language`, `employee_profile_id`, `iat`, `exp`, `jti`."

From `instrucciones.md §11.2 Paquetes prohibidos` (non-negotiables 01 §security): tokens **NEVER** in `localStorage` ni `sessionStorage`.

**Operational frontend contract (this slice MUST implement)**:

1. **Access token: in-memory only** (closure or React state variable inside `AuthProvider`). Lost on page reload — that's the desired behavior; `AuthProvider` calls `/api/v1/auth/refresh` on mount to rehydrate from the HttpOnly cookie.
2. **Refresh token: never touched by JS.** Set by backend as `HttpOnly` cookie; only sent automatically by browser to URLs prefixed `/api/v1/auth`. The frontend NEVER reads it, NEVER stores it, NEVER logs it.
3. **No `Authorization` header on `/auth/refresh` and `/auth/logout`** — the cookie is the credential. Use `credentials: 'include'` on every fetch.
4. **`Authorization: Bearer <access_token>` on `/api/v1/users/**` and ALL future protected endpoints** — injected by the HTTP client interceptor reading from the in-memory `AccessTokenStore`.
5. **401 interceptor with single-flight refresh**: on any 401 from a protected endpoint, queue the request, fire one `/auth/refresh`, on success replay queued requests with the new token, on failure clear `AccessTokenStore` and redirect to `/auth/sign-in?next=<current_path>`.
6. **Logout**: POST `/api/v1/auth/logout` → clear `AccessTokenStore` → invalidate all queries → redirect to `/auth/sign-in`.

### C.7 Access-token JWT claims (informative — frontend does NOT decode)

Per §10.2: `sub` (user_id UUID), `email`, `roles` (list[str]), `preferred_language` (es|en|fr), `employee_profile_id`, `iat`, `exp`, `jti`.

**The frontend MUST NOT decode the JWT to obtain identity.** Identity comes from `GET /api/v1/users/me`. Decoding the JWT in the client couples the UI to the server's signing algorithm and bypasses the `inactive`/`locked` status the server returns. Treat the access token as an opaque bearer string.

### H.5 Risks (planner-visible)

| Risk | Mitigation |
|---|---|
| R-A: Concurrent 401s trigger N simultaneous /auth/refresh calls (thundering herd) | Single-flight refresh: one in-flight promise, queued retriers await it. Researcher Q2. |
| R-B: Page reload on protected route — guard fires /me; AuthProvider fires /refresh — race | AuthProvider's hydration MUST resolve BEFORE guards run permission check. | R-C: `?next=` open-redirect (attacker crafts `?next=https://evil.com`) | `redirectAfterAuth.ts` rejects any `next` value that does not start with `/`, contains `//`, contains `://`, or matches `\\\\`. Default safe target on invalid: `/chat`. Tested. |
| R-D: Logout race: user clicks logout while a query is in-flight | useLogout: 1) call /logout, 2) clear AccessTokenStore, 3) `queryClient.clear()`, 4) navigate. Steps 2–4 run EVEN IF /logout returns 401. |
| R-E: 401 interceptor recursion (refresh itself returns 401 → infinite loop) | Mark `/auth/refresh` and `/auth/logout` calls with `__authNoRetry` flag; interceptor checks and short-circuits. |
| R-F: SPA navigation bypasses guards if declared as plain children instead of route wrappers | Guards use `<Route element={<RequireAuth><Outlet/></RequireAuth>}>` layout pattern. Researcher Q1 confirms canonical v7 form. |
| R-G: Page reload between sign-in success and a guarded navigation loses the in-memory access token (CORRECT behavior) — test could miss this | Test plan §K covers reload-with-cookie → AuthProvider rehydrates via /refresh. |
| R-H: Test failures unrelated to auth | Only `-t auth` is the SLICE gate; full suite must remain green but is informative. |

## P. Security guardrails (hard non-negotiables — copy verbatim into developer mind)

1. **NEVER** `localStorage.setItem(<anything token>)` or `sessionStorage.setItem(...)`. If a snippet from training data contains either, REMOVE IT.
2. **NEVER** decode the JWT in the client. Treat as opaque string.
3. **NEVER** log the token value. Log only `len=64` or `present=true|false`.
4. **NEVER** log password, secret, api_key, raw cookie. Redact at the log source.
5. Refresh cookie is HttpOnly → `document.cookie` will NOT show it. Tests do not need to "read" it.
6. ALL fetches MUST include `credentials: 'include'` to send/receive cookies; even non-auth endpoints — defense in depth. Do NOT mix-and-match.
7. `X-Request-ID` header — every fetch MUST include it (UUID v4). Generate at HTTP-client level; log only after request completion.
8. `?next=` open-redirect guard MUST be a function with unit tests (T13). Do NOT inline a regex in the consumer.
9. Logout MUST clear AccessTokenStore + `queryClient.clear()` EVEN IF backend returns 401. The user's intent ("log me out") wins over backend errors.

## Q. Logging requirements (non-negotiables §logging)

`AuthProvider` and the HTTP client log BEFORE/AFTER every meaningful operation:

- `auth.provider.hydrate.start` (BEFORE /refresh)
- `auth.provider.hydrate.ok` (AFTER /refresh + /me success — log `user_id` only, NOT email)
