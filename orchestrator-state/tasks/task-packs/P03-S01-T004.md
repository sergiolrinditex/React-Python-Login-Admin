## 4. Required decisions (developer must anchor each in code with `§D-T004-*`)

### D-T004-EMAIL-MASK — pure deterministic email masking

Spec:

- Function signature: `function maskEmail(email: string): string`.
- Pure, deterministic, side-effect-free, unit-testable.
- Algorithm (no PII leak): take everything up to the first `@`; mask the local part keeping the first 1 char and appending `***`; preserve the `@domain` exactly as received.
- Edge cases (must be covered by unit tests):
  1. Standard email `john@acme.com` → `j***@acme.com`.
  2. Empty local part `@acme.com` (invalid but defensive) → `***@acme.com`.
  3. Missing `@` (e.g. `not-an-email`) → return literal `"***"` (no exception thrown, no PII leaked).
  4. Empty string `""` → return literal `"***"`.
- Lives inline in `ResetSentPage.tsx` as a non-exported helper. Justification: it's tiny (~6 lines), used in one place, has no reuse signal yet. If a second consumer appears, extract to `frontend/src/features/auth/presentation/maskEmail.ts` in a future slice (YAGNI now).
- Anchor with `§D-T004-EMAIL-MASK` in the function docstring.

### D-T004-NO-STATE-FALLBACK — state-missing rendering

Spec:

- When `useLocation().state?.email` is `undefined`, `null`, `""`, or not a `string`, the page renders the **fallback variant**:
  - Title: same (`auth:reset_sent.title`).
  - Body: `auth:reset_sent.body.fallback` — generic copy: "Si la dirección está registrada en Hilo, recibirás un correo con instrucciones." (and EN/FR equivalents).
  - CTA: same `<Link to={ROUTE_AUTH_SIGN_IN}>` with text from `auth:reset_sent.actions.back_to_sign_in`.
- **No redirect.** The user lands here legitimately via deep link / browser refresh / back-button after T003 already navigated forward. Redirecting them backward would create a loop or destroy product trust.
- **No error UI.** This is a legitimate sub-variant of success, not an error.
- Anchor with `§D-T004-NO-STATE-FALLBACK`.

### D-T004-ANTI-ENUM-UI — anti-enumeration parity with T003

Spec:

- The page copy + visual treatment are **byte-identical** for the "known email" path and the "unknown email" path (both flow through the same `navigate("/auth/reset-sent", { state: { email } })`). The page itself does not know whether the email is registered — it just renders what state it received.
- The only visual variance is between **state-present** (renders masked email line) and **state-missing** (omits masked email line). Both render the same title, body-shell, and CTA.
- Document in the file header that this aligns with `§D-T003-ANTI-ENUM-UI` and TECHNICAL_GUIDE §6.2 `/auth/forgot-password` returns identical `{data:{sent:true}}` for both cases (400/429 only on validation / rate-limit, never on "user not found").
- Anchor with `§D-T004-ANTI-ENUM-UI`.

### D-T004-NO-PII-LOG — PII-safe logging

Spec:

- Zero `console.*` calls that print the email or any portion of it (no domain alone is fine, but never the full address or local part).
- If verbose logging is added at all (gated by `import.meta.env.VITE_ENABLE_VERBOSE_LOGGING === "true"`), it logs only:
  - `email_present: boolean` (is state present at all),
  - `email_domain: string | "<none>"` (substring after first `@`, or sentinel),
  - `email_local_len: number | 0` (length of local part, or 0).
- Matches T003 §D-T003-PII-LOGGING.
- Anchor with `§D-T004-NO-PII-LOG`.

### D-T004-PUBLIC-ROUTE — public route, no auth guard

Spec:

- Route is mounted OUTSIDE `<RequireAuth>` (mirrors `/auth/forgot-password`, `/auth/sign-in`, `/auth/sign-up`).
- TECHNICAL_GUIDE §6.4 Navigation Contract is explicit: `/auth/reset-sent` is a public route.
- If an authenticated user navigates here (e.g., testing the flow from /account or via direct URL), the page renders normally. There is no redirect to `/chat` because the user may legitimately be testing the recovery flow. This deviates from typical "authed users get redirected from /auth/*" patterns — document why in the page header docstring.
- Anchor with `§D-T004-PUBLIC-ROUTE`.

### D-T004-A11Y — accessibility contract

Spec:

- Single `<h1>` for the page title (i18n `auth:reset_sent.title`).
- The success status region uses `role="status"` + `aria-live="polite"` so screen readers announce the confirmation on arrival.
- CTA is a real `<Link>` (from `react-router`) pointing to `ROUTE_AUTH_SIGN_IN`. Not a `<button>`, not a `<div onClick>`. Real navigation, real keyboard support, real focus ring.
- Tap target ≥ 44 × 44 px for the CTA (use `SolidCTA` base component which already meets this).
- WCAG AA contrast: rely on `--color-ink` on `--color-bg` / `--color-paper` exclusively. No hardcoded color literals.
- No information conveyed by color alone.
- Anchor with `§D-T004-A11Y`.
