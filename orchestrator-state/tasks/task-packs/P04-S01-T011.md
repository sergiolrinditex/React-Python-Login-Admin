**Root cause (confirmed)**:
1. `test_admin_ai.py` imports first → unconditional `os.environ["ENCRYPTION_KEY"] = KEY_A` (its own Fernet.generate_key()).
2. `test_admin_ai.py`'s autouse fixture (L335) calls `reset_fernet_cache()` per test, but the LAST state at end-of-module is `KEY_A` primed in the cache (T18 in `test_admin_ai.py` performs a roundtrip with KEY_A via `f = Fernet(_TEST_FERNET_KEY.encode())` at L466).
3. `test_audit_endpoint.py` and others run between, none overrides because KEY_A is already set in env.
4. `test_mcp_registry.py` imports → at L79 **overrides** `ENCRYPTION_KEY = KEY_B` (its own Fernet.generate_key()). Imports `reset_fernet_cache` at L88 but never calls it after the override. The lru_cache still holds `Fernet(KEY_A)`.
5. `test_T06` runs → POST creates server with `auth.api_key.secret=secret_plain`. Server-side calls `app.mcp.service_register.encrypt_secret(secret_plain)` → `_get_fernet()` returns cached `Fernet(KEY_A)` → stores ciphertext encrypted with KEY_A.
6. Test reads the row, builds `f = Fernet(_TEST_FERNET_KEY.encode())` = `Fernet(KEY_B)`, calls `f.decrypt(enc.encode())` → `cryptography.fernet.InvalidToken`.

**Confirming evidence**: PROGRESS.md L9 mentions T06 fails ONLY in full-suite ordering and never in isolation. In isolation, no earlier module primes the cache, so `_get_fernet()` reads `KEY_B` from env on first call inside the request. In full-suite, `KEY_A` is already cached.

### 4.4 Candidate fix surface

**Strongly preferred — test-only minimal fix** (1 line in `test_mcp_registry.py`):

Add a `reset_fernet_cache()` call immediately after the `ENCRYPTION_KEY` override at line 79–80, BEFORE `from app.main import app` on line 87. Since `reset_fernet_cache` is imported from `app.security.encryption` on line 88 (after `app.main`), the cleanest sequence is:

```python
# After line 82 (MFA key setup) and the allowlist pop on line 85:

# Override prior cache before importing app.main (other test modules in this
# session may have already primed _get_fernet() with their own KEY at first
# encrypt/decrypt call). The app.main import chain does NOT call encrypt/decrypt
# at module load, but we reset eagerly to make the cache reflect THIS module's
# ENCRYPTION_KEY for any subsequent encrypt/decrypt call from the server.
from app.security.encryption import reset_fernet_cache  # noqa: E402
reset_fernet_cache()

from app.main import app  # noqa: E402
from app.auth import rate_limit as _auth_rl_module  # noqa: E402
```

This mirrors the proven pattern in `test_chat_stream.py:121-124` and `test_admin_ai.py:103+335` autouse fixture.

**Alternative (acceptable but slightly redundant)** — add an autouse session-scope fixture that resets the cache once at session start:

```python
@pytest.fixture(autouse=True, scope="module")
def _reset_fernet_cache_for_module():
    """Module-level: ensure THIS module's ENCRYPTION_KEY (line 79) is
    the active Fernet, not a cached key from an earlier test module
    in the same session (root cause of test_T06 InvalidToken)."""
    reset_fernet_cache()
    yield
```

Either solution touches only `test_mcp_registry.py`. **`backend/app/security/encryption.py` does NOT need to change** — the existing `reset_fernet_cache()` helper is sufficient. The `allowed_paths` entry for `encryption.py` exists in the registry to authorize a defensive change there if developer judges one necessary (e.g., adding a stronger docstring example), but a no-op there is the preferred outcome.

**Forbidden**: do NOT remove or relax the `@lru_cache(maxsize=1)` on `_get_fernet` — that is the production contract D-ENC1.

## 8. NEEDS_OFFICIAL_DOCS decision

**Recommendation: NO.**

Rationale:
- `cryptography.fernet.Fernet` API is stable since cryptography 1.x; no breaking changes in 48.0.0. The `decrypt(token).decode()` path that raises `InvalidToken` on key mismatch is documented and unchanged across all minor versions of cryptography==48.x.
- `functools.lru_cache(maxsize=1)` is stdlib-stable; cache invalidation via `.cache_clear()` (used by `reset_fernet_cache`) is documented stdlib behavior.
- Pytest fixture ordering is stable in pytest==9.0.2 (module imports happen at collection time, in alphabetical-by-file order with the default configuration; per-test autouse fixtures run before each test). No researcher consultation adds information beyond what is already in the existing test fixtures of `test_admin_ai.py` and `test_chat_stream.py`, which already encode the correct pattern.

If, during implementation, the developer discovers that the cache poisoning happens via a path the planner did not trace (e.g., a celery/redis client that loads the key elsewhere), they MAY escalate to `official-docs-researcher` with the 1–5 concrete questions framing. But the prior probability is low; the leading hypothesis is fully consistent with all observable evidence.
