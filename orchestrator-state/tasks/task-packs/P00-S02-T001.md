**Compose YAML invariants (Rancher Desktop friendly):**

- **No** `version:` al inicio (deprecado en spec v2).
- Usar `depends_on` con `condition: service_healthy` para encadenar arranques.
- Volumes **nombrados**, no bind-mounts a paths fuera de `$HOME` (Rancher Desktop monta sólo paths del HOME por defecto).
- Healthchecks declarados con `test`, `interval`, `timeout`, `retries`, `start_period`.
- Networks: usar la default; no declarar networks custom salvo necesidad.
- `restart: unless-stopped` en servicios long-running; `restart: "no"` en sidecars one-shot.
- NO usar `extra_hosts: host.docker.internal:host-gateway` (problemático en containerd/nerdctl).
- Env vía `env_file: .env` + `environment:` para overrides explícitos.

## G. Runtime constraint: Rancher Desktop (moby OR containerd)

El humano usa **Rancher Desktop**. El YAML debe ser **compose v2-spec puro** para que funcione tanto si Rancher está en `dockerd (moby)` (donde `docker compose` resuelve) como en `containerd` (donde se usa `nerdctl compose`). El tester:

1. Detecta el runtime:
   ```bash
   if docker compose version >/dev/null 2>&1; then COMPOSE="docker compose"
   elif nerdctl compose version >/dev/null 2>&1; then COMPOSE="nerdctl compose"
   else echo "Rancher Desktop not running" >&2; exit 2; fi
   ```
2. Usa `$COMPOSE config` para validar el YAML (parseo + interpolación).
3. Para el smoke productivo (`up -d`), el tester puede deferirlo si T002/T003 aún no han cerrado.
