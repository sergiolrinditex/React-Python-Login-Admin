### useMe vs useAuth().user — RESOLVED

- `useAuth().user` is populated at AuthProvider hydration (mount-time `refresh` → `fetchMe`) and on sign-in. It is the **single source of truth** for the current user on the client.
- The previous (stripped) implementation introduced `useMe` (TanStack Query `['user','me']` with `initialData` from `useAuth().user`, staleTime 60s) so the AccountPage could re-fetch the profile after a language change without re-mounting AuthProvider.
- Recommendation for this slice (lighter, fewer files): do **NOT** introduce a new `useMe`. Use `useAuth().user` directly for read, and on PATCH success either:
  1. Setter pattern — call `i18n.changeLanguage(lng)` optimistically and let the server-returned new profile flow back into `AuthProvider` via a small new method **OR**
  2. Re-fetch via a one-off `repo.fetchMe(token)` inside the `useLanguagePicker` mutation success handler and stash result locally for re-render (acceptable for one screen).
- If the developer prefers strict Clean Architecture (`useMe` query) they MUST add it inside `frontend/src/pages/account/` (allowed by write_set glob `frontend/src/pages/account/**`) — but a presentation-folder hook is acceptable because the data path goes through `authRepository`. Document the choice in the handoff (`§D-T007-USERREAD-CHOICE`).
