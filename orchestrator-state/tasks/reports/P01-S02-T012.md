## Decisions

| Decision | Rationale | Source doc ref |
|----------|-----------|---------------|
| Option 1: thin `_host_pg_ready()` wrapper over `_tcp_probe` | KISS — reuses existing Bash builtin `/dev/tcp` probe; no new deps (no `pg_isready -h`, no `psycopg`, no `nc`). Chosen from task-pack §F.2 Option 1 vs 2 trade-off analysis. | `HILO_PEOPLE_TECHNICAL_GUIDE §4.3` (tooling stack); task-pack §F.2 |
| AND-gate in `db_health`: both `_compose_pg_ready` AND `_host_pg_ready` must pass | Closes the 0.5–3s Rancher Desktop port-forward race window. Container-internal `pg_isready` passes before host-side TCP is forwarded, causing alembic ECONNREFUSED on host. Two-probe AND eliminates the race. | Task-pack §B, §F.1; FU-20260512052820 root cause analysis |
| Timeout raised 30→60s in `_ensure_infra_essential` | Aligns both `wait_for db_health` call sites (essential + db_reset) to 60s SLA ceiling; gives host TCP probe margin on slow hardware. `db_reset` was already 60s. | Task-pack §D; validator ARCHITECTURE note |
| No-compose branch: host-TCP-only sufficient | When compose is absent (native postgres), only the host TCP probe can be checked; no container-internal probe available. Correct behavior per three-valued `db_health` semantics {0 UP, 1 DOWN, 2 UNKNOWN}. | Handoff §A, validator ARCHITECTURE note |
