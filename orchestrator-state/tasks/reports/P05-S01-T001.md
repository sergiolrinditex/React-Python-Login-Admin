## Open items / Risks / Follow-ups

### FU-20260519091359-test-mfa-add-jwt-private-key-env-setup-in-mfa-us
- **Severity**: low
- **Kind**: test / missing_coverage
- **Origin**: tester run (P05-S01-T001)
- **Summary**: `test_mfa_verify_invalid_signature_returns_401_aggregate` fails when `test_mfa.py` is run in isolation because `JWT_PRIVATE_KEY` is set at module level in `test_auth_signin.py` and leaked via pytest collection order. Live backend behavior is correct (tampered JWT → 401 AUTH_MFA_CODE_INVALID). Fix: self-contained env guard at top of `test_mfa.py`.
- **Why not debugger**: `P05-S01-T001` write_set is strictly `evidence/journeys/J100/**`; `test_mfa.py` is outside write_set.
- **Write set**: `backend/tests/integration/test_mfa.py`
- **Acceptance**: `pytest tests/integration/test_mfa.py` runs 17/17 in isolation.
- **Status**: proposed — does NOT block this close; does NOT block J100 verify; low severity.

### FU-20260519121908-auth-store-deduplicate-concurrent-refresh-token-
- **Severity**: high
- **Kind**: bug / out_of_scope
- **Origin**: slice-verifier browser reproduction (F5 reload flow on /chat)
- **Summary**: On `/chat` reload, frontend fires two simultaneous `POST /auth/refresh` requests with the same refresh_token cookie. First call (200) rotates the cookie; second call carries the now-revoked cookie and gets 401, causing logout on every F5. Fix: singleton/mutex pattern in token refresh interceptor.
- **Why not debugger**: `P05-S01-T001` write_set is `evidence/journeys/J100/**`; the defect is in `frontend/src/**`.
- **Write set**: `frontend/src/**`
- **Journey refs**: J100 — this defect is the primary reason J100 verification is deferred.
- **Acceptance**: F5 on `/chat` stays at `/chat` after exactly 1 successful `POST /auth/refresh`.
- **Status**: proposed, severity=high — blocks `/next-wave` for tasks referencing J100; does NOT block this slice close. Promote or waive via `/promote-followup FU-20260519121908-auth-store-deduplicate-concurrent-refresh-token-` or `/register-followup waive <ID>`.
