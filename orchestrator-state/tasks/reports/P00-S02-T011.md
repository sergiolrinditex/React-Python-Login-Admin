## Summary

This slice closed a critical dev-workflow gap: after P01-S01-T002 renamed `PROVIDER_ENCRYPTION_KEY` to `ENCRYPTION_KEY` in `.env.example` and `config.py`, the actual dev `.env` was never updated and still carried the legacy name with the placeholder string `dev-encryption-key-placeholder` — an invalid Fernet base64 key. Two real failures were observed during `/verify-slice P00-S02-T006`: (a) `bootstrap_verification_data --only auth` failed immediately with "ENCRYPTION_KEY env var is required for MFA TOTP encryption"; (b) POST `/api/v1/admin/ai/providers/{id}/discover-models` returned 502 CryptoError because Fernet construction with the invalid base64 key failed on first decrypt.

The fix adds an idempotent `ensure_encryption_key()` function to `scripts/setup-from-scratch.sh` that (1) creates `.env` from `.env.example` if absent, (2) removes the deprecated `PROVIDER_ENCRYPTION_KEY` line, (3) generates a fresh Fernet key via `cryptography.fernet.Fernet.generate_key()` if `ENCRYPTION_KEY` is missing, a placeholder, or legacy. The generated key is NEVER echoed; only masked last-4 chars are logged. Two security incidents were discovered and remediated during the cycle: a Fernet master key leaked into `ledger.jsonl` via bash literal commands (rotated + redacted over 2 debugger cycles) and a redundant plaintext-key write in `dev-restart.profile.sh` (removed). The slice required 3 full validator/tester/debugger cycles before reaching verified closure.

## Architectural decisions

- **Idempotent key generation over migration script**: `ensure_encryption_key()` runs every time `setup-from-scratch.sh` is called; it is safe to call multiple times (key unchanged on re-run if valid). This avoids a separate migration step that would be easy to skip. Ref: `setup-from-scratch.sh` docstring, TECHNICAL_GUIDE §11.1.
- **Key never echoed, only masked last-4**: logging pattern `****${last4}` used consistently across `setup-from-scratch.sh`, `dev-restart.profile.sh`, and evidence files. Prevents inadvertent logging of the master Fernet key. Ref: non-negotiables §Security (claves de proveedores externos NUNCA en texto plano en logs).
- **In-process Python for key generation**: `Fernet.generate_key()` via Python subprocess; avoids exposing the key in the shell's process list or bash command capture (the PostToolUse hook records bash commands verbatim). Ref: debugger MEMORY.md "Secret hygiene in bash commands".
- **Transitional backend fallback `ENCRYPTION_KEY → PROVIDER_ENCRYPTION_KEY`**: `backend/app/core/security.py:_resolve_fernet_key` has a transitional fallback in place since P01-S01-T002. It is intentional and will be retired in P02-S02-T001 when productive provider credentials are introduced. Ref: TECHNICAL_GUIDE §11.1 (new note).
