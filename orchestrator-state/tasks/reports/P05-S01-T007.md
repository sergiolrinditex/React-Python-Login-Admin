## Problem Statement

On `/chat` page reload (F5), two independent code paths both issued `POST /api/v1/auth/refresh` using the same HttpOnly refresh cookie:

1. `AuthProvider.hydrate()` → `AuthRepository.refresh()` (cold-mount lifecycle call)
2. React tree `authFetch` → 401 interceptor → `httpClient._doRefresh()` (race from first render before hydrate completes)

The backend's rotating-refresh-token scheme revokes the first token immediately on use. The second caller arrived with a now-revoked cookie, triggering `AUTH_SESSION_EXPIRED` and redirecting the user to `/auth/sign-in` unexpectedly.

## Decisions

| Decision | Rationale | Source doc |
|----------|-----------|------------|
| Option A (shared `refreshSingleFlight.ts` module) over Option B (authRepository calls authFetch) | Option B would introduce coupling between repository and HTTP client layers and would not de-duplicate the cold-mount race. Option A is a pure data-layer utility with no cross-layer imports. | Task pack §Technical decision |
| `_inflight` assigned synchronously before first `await fetch()` | This is the critical correctness invariant: the second concurrent caller checks `_inflight` synchronously before the first `await` has suspended, guaranteeing coalescence. | `refreshSingleFlight.ts` line 106; `TECHNICAL_GUIDE §6.5` |
| `_resetInflight()` as thin re-export of `_resetSingleFlight()` | Preserves backward compatibility for existing `auth.test.tsx` and `httpClient.test.ts` without requiring coordinated rename across test files. | Validator review AC7 / rule `01-non-negotiables.md §DRY/KISS` |
| `Result<string>` public contract preserved on `AuthRepository.refresh()` | `IAuthRepository` is the domain port; changing its return type would be a breaking domain-layer change outside the scope of this bug fix. | Task pack §Contract map; `01-non-negotiables.md §API contract` |
