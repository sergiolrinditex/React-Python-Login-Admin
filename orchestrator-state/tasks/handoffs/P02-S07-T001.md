#### CRITICAL-2 — MCP protocol non-compliance: missing `initialize` handshake will fail against real MCP servers

**File**: `backend/app/mcp/client.py` (lines 186-276, `discover()`)
**Issue**: `discover()` directly POSTs `tools/list`, `resources/list`, `prompts/list` without the mandatory MCP `initialize` lifecycle request. The official MCP specification (lifecycle, §6 of 2025-06-18 spec) explicitly states that the server SHALL NOT respond to non-`initialize` requests before the initialization phase completes. A spec-compliant MCP server will reject every call with a protocol error.
**Why tests don't catch it**: T12, T14, T15, T16, T17 mock `app.mcp.client.discover` directly with `patch(...)` and return canned dicts. The mock bypasses the missing handshake. This is permitted (third-party transport) for test realness, but it leaves the actual `client.discover` path completely untested against a real MCP server. The slice's J105 verify-slice setup expects `data/verification/mcp_agents/servers/sandbox_readonly.json` at `http://localhost:8080/mcp` — the moment that real sandbox responds spec-compliantly, the sync will return `502 MCP_SERVER_UNREACHABLE`.
**Why this is in-scope, not a follow-up**: this is a defect in `client.py` (already in `Write set`) implementing the very acceptance criterion the task pack lists: "MCP server registration encrypts auth, **sync discovers tools**, tools disabled by default". The slice cannot honor "sync discovers tools" against a real server without the handshake.
**Fix (debugger, single function, within write_set)**: in `client.py` before the three discovery calls, perform the MCP initialization handshake:
```python
def _initialize_session(endpoint, headers, timeout) -> dict[str, str]:
    """MCP §6 lifecycle — initialize the session and capture Mcp-Session-Id header.

    Returns the updated headers dict (with Mcp-Session-Id if returned by the
    server). Required before tools/list, resources/list, prompts/list per
    MCP spec 2025-06-18 §6.1.
    """
    payload = {
        "jsonrpc": "2.0",
        "id": str(uuid.uuid4()),
        "method": "initialize",
        "params": {
            "protocolVersion": "2025-06-18",
            "capabilities": {},
            "clientInfo": {"name": "hilo-people", "version": "1.0"},
        },
    }
    # POST, check 2xx, capture Mcp-Session-Id from response headers if present,
    # return updated headers dict for subsequent calls. Map errors to
    # McpServerUnreachableError as in _json_rpc_call.
```
Then invoke `_initialize_session` from `discover()` before the first `tools/list` call. Update T12/T14/T15/T17 mocks to also accept the new lifecycle (or — better, since `discover` is the only function mocked — leave those tests unchanged; the mock continues to bypass the protocol detail). Add at least ONE new test that exercises the real client against an `httpx` mock (no patching of `client.discover`), driving an `initialize` request and checking the server received the JSON-RPC method `initialize` first.

If you accept the engineering judgement that "this slice's tests don't need to exercise the real handshake because /verify-slice is the gate" — that's defensible — but the handshake code must still be added now, because validator-merge means the next slice will inherit this client and any P02-S08+ work will be blocked the moment a real MCP server is wired.
