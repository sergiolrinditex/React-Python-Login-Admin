## Actions performed

### 1. Schema audit (pre-implementation)

Compared `backend/alembic/versions/0003_admin_ai_providers_models.py` DDL columns against the old
loader INSERT statements. Discovered the loader was inserting 11 phantom columns:

- `ai_providers`: old code tried `api_key`, `is_active`, `description` — none exist in migration 0003
- `ai_provider_credentials`: old code did not insert here at all despite §10.3 requiring Fernet-encrypted secret
- `ai_models`: old code tried `capability` (not in DDL), missed `auto_discovered`

Also confirmed:
- No UNIQUE constraint on `ai_providers.name` (only `uq_ai_models_provider_id_model_id` on ai_models)
- `auth_type` mapping: `litellm → master_key`, all others → `api_key`
- `app.core.security.encrypt_secret()` exists and must be reused (not duplicated)

### 2. Complete rewrite of `admin_ai.py`

Key design decisions:

- `_upsert_provider()`: SELECT-then-INSERT (no UNIQUE on name) — idempotent without conflict
- `_upsert_credential()`: DELETE+INSERT — rotation pattern for secrets; clean slate per run
- `_upsert_model()`: ON CONFLICT (provider_id, model_id) DO UPDATE — uses the real UNIQUE index
- `encrypt_secret()` imported from `app.core.security` — single source of Fernet key resolution
- `provider_id_by_name: dict[str, uuid.UUID]` built during provider phase, used in model phase
- Orphan models (unknown `provider_name`) → WARNING log + skip, not exception
- `status` mapping: `is_active=True → 'active'`, else `'draft'`
- `capabilities` JSONB column populated from seed `model_type` list

Security: api_key value NEVER logged; only `token_length` hint logged after encryption.

## R-rows (risks and constraints)

**R1 — Fernet key for tester**:
The dev `.env` has `PROVIDER_ENCRYPTION_KEY=dev-encryption-key-placeholder` which is NOT a valid
Fernet key. Tester must inject a real key:
```bash
PROVIDER_ENCRYPTION_KEY=$(python3 -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())")
```
The loader and tests handle this gracefully: loader falls back to `settings.encryption_key` chain;
tests with `SKIP_IF_NO_FERNET` skip cleanly if key is invalid.

**R2 — No conflict with other slices**:
T010 is in `admin_ai_endpoints` conflict group; no other in-flight task is in this group currently.
Write set limited to `backend/app/seeds/loader/admin_ai.py`.

**R3 — Reranker capability mapping**:
`AiModelSeed` may have `capability="reranker"` but migration 0003 has no `reranker` model_type.
Loader maps unknown capabilities to `"unknown"` with a WARNING log. This is an accepted limitation
until T011 or a schema extension is planned.

**R4 — No UNIQUE index on ai_providers.name**:
The §10.3 DDL has no UNIQUE constraint on `ai_providers.name`. Loader uses SELECT-then-INSERT.
If a concurrent seed run races, it could insert a duplicate provider. Acceptable for seed data;
production multi-tenant scenarios would need a UNIQUE index (out of scope for T010).

**D1 — auto_discovered column in migration but not in §10.3 DDL**:
The field `auto_discovered BOOLEAN NOT NULL DEFAULT false` is in migration 0003 but NOT yet
back-ported to the §10.3 Technical Guide DDL. The loader correctly uses `auto_discovered` from
migration 0003 as the source of truth. Technical Guide needs an update in a future doc-sync pass.
