**D1 — Flat shape preserved on /health (Discrepancy)**

**D2 — Redis/LiteLLM placeholders (Discrepancy)**
TECHNICAL_GUIDE §6.2 declares `{data:{db,redis,litellm}}`. Clients don't exist yet.
Decision: expose `redis: {status:"not_implemented", reason:"client_not_wired"}` and same for litellm.
NOT removed, NOT faked as "ok". Forward-compatible shape: downstream slices flip status to "ok"/"fail"
without changing the response structure.

**D3 — request_id middleware (Discrepancy, resolved)**
`app.core.logging.py:17` docstring: "middleware wired in P00-S02-T002". Implemented as
`@app.middleware("http")` (FastAPI-idiomatic form per official doc note Pattern 1).
Uses `structlog.contextvars.clear_contextvars()` + `bind_contextvars()` + `clear_contextvars()`
pattern (per official doc note Pattern 4 verified by researcher).

**Exception handling in /ready**
Catches `SQLAlchemyError` first (covers all asyncpg-wrapped DB errors per official doc note Pattern 3).
Bare `Exception` as last-resort fallback only — asyncpg.PostgresError subclasses are NOT always
wrapped by SQLAlchemy in edge cases. The catch is justified by the readiness contract: a failed probe
must return 503, never 500. Exception class name is logged for future refinement.

**_sanitize_db_error()**
Error detail is stripped of DSN components (postgresql://, localhost, change-me, password, token,
secret) before inclusion in the response body. Prevents credential leakage via the /ready body.
Verified: test_ready_returns_503_when_db_fails asserts that "postgresql" and "change-me" are NOT
in the detail string.

**mypy / structlog BoundLogger API**
Structlog's BoundLogger.debug/error/warning accept the event as the first positional arg. Passing
`event=` as a keyword triggers mypy "multiple values" error. Solution: use event as positional arg,
extra context as keyword args. Applied throughout router.py.

**@app.middleware("http") vs BaseHTTPMiddleware**
Chose `@app.middleware("http")` decorator form over `add_middleware(BaseHTTPMiddleware)` because:
(a) official docs list it as the FastAPI-idiomatic approach; (b) avoids mypy incompatible override
error on `dispatch()`'s `Response` return type annotation in BaseHTTPMiddleware.

**Real DB test credentials**
Default `.env.example` / `config.py` fallback uses `change-me` password. Compose postgres uses
`hilopeople_dev_pwd`. Test_3 (DB-up) run with `DATABASE_URL` env override pointing to correct
credentials. This is documented as the acceptance gate for the slice.

**api/__init__.py write_set extension**
Work-item YAML lists 3 paths. `api/__init__.py` added as a Python language requirement (cannot
import from a subpackage without __init__.py). Documented here as write_set technicality.

### Initial hypothesis
DSN/password leak under `/ready` 503 logs is caused by `_logger.error("...", exc_info=True)` in `_probe_db()`. Structlog's verbose chain runs `structlog.dev.set_exc_info` + `ConsoleRenderer` → `RichTracebackFormatter` (defaults to `show_locals=True`). The asyncpg/SQLAlchemy connect path stores `cparams = {host, user, password, port, database}` and `ConnectionParameters(...)` as frame locals, so the traceback rendering writes those values verbatim to stdout/stderr.

### Confirmed root cause
Reproduced exactly with a controlled test: `_FakeAsyncConn.__aenter__` raises `OperationalError("connection refused")` while binding `cparams = {...secrets...}` and `dsn` as frame locals. With `exc_info=True` set, the captured pytest output shows the Rich traceback panel printing the cparams dict verbatim — `'host': 'hostxyz-debugger-fix'`, `'password': 'topsecret_pwd_xyz_8f3c1'`, `'port': '54399'`, etc. With `exc_info=True` dropped, the traceback is not rendered and only the structured fields `error_class` + `db_detail` (sanitized) reach the log.

The structured-only path is sufficient for production debugging: an SRE sees the exception class name (e.g. `OperationalError`, `gaierror`, `ConnectionRefusedError`) plus the sanitized `db_detail`. Stack traces remain available in dev via Python's default exception printing on uncaught errors; for richer in-log traces, the global structlog config must use `RichTracebackFormatter(show_locals=False)` — that change is tracked in follow-up `FU-20260509044829-disable-structlog-rich-traceback-show-locals-glo` and is OUT OF SCOPE for T002 (write_set excludes `app/core/logging.py`).

### 8. Lessons captured (will be appended to validator/MEMORY.md)

1. **structlog `exc_info=True` + Rich is a CWE-532 vector.** The redaction processor scrubs `event_dict` keys only. `RichTracebackFormatter` defaults to `show_locals=True`, which renders Python frame locals — bypassing the redactor entirely. Any DB-connect, HTTP-client, or secret-decoder failure path will leak credentials when its exception is logged with `exc_info=True`. Mandatory check on every slice that adds a `_logger.error(..., exc_info=True)` line.
2. **Test realness for log-leak regressions requires deliberate frame-local binding.** A test that monkeypatches a generic `OperationalError("simulated")` does NOT exercise the leak path. The regression test must build a fake call site that binds secret values as locals at the failing frame, then assert on captured stdout/stderr. Pattern: `_FakeAsyncConn.__aenter__` with `cparams = self.cparams; dsn = self.dsn  # noqa: F841 — bound as local on purpose`.
3. **In-scope CWE fix vs out-of-scope architectural fix.** The right call is to (a) close the immediate leak inside the slice's existing write_set (drop `exc_info=True` at the call site) AND (b) register a HIGH-severity follow-up for the architectural root cause (`RichTracebackFormatter(show_locals=False)` global tuning in `app/core/logging.py`). Touching `core/logging.py` from a `health-endpoints` slice would have been illegal scope expansion. The followup must be `severity: high` to block the next wave until promoted/waived.
