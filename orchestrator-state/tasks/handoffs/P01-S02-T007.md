### Concrete fixes for debugger (same TASK_ID + write_set)

**F1 — `backend/app/users/deps.py:133` — broad `except Exception` violates §Error handling.**
```python
# Current
except (jwt.InvalidTokenError, Exception):
    logger.warning(...)
    return _emit_401(request_id)

# Should be (per §G.2 planner pack):
except (jwt.InvalidTokenError, ValueError):
    logger.warning(...)
    return _emit_401(request_id)
```
`jwt.MissingRequiredClaimError`, `jwt.DecodeError`, `jwt.InvalidSignatureError`, `jwt.InvalidAudienceError` etc. are all subclasses of `jwt.InvalidTokenError`. `ValueError` is what `decode_token` raises for purpose mismatch (tokens.py:208) — though the explicit `if decoded.get("purpose"): return _emit_401(...)` step at line 140 also handles that case, so `ValueError` belongs in the catch list for defense-in-depth. Catching bare `Exception` is the §Error handling non-negotiable violation that needs to go.

Functional impact: trivially zero (test suite already covers the failure modes that matter and they map to `jwt.InvalidTokenError` or are pre-handled). This is a code-quality fix, not a behavior fix.

### Findings (non-blocking, document only)

**N1 — `audit.py:133` — broad `except Exception` is acceptable here**. Audit write failure must not surface to the client as 500; the catch wraps a tightly-scoped DB session block and logs ERROR with `exc_info=True`. Same pattern as `LogoutAuditWriter`. Keep as is.

**N2 — `update_user_language.py:104-118` — D-S2 audit flow is sequential, not failure-resilient**. The audit row is written AFTER `session.commit()` on the success path. Strict D-S2 would write the audit BEFORE the commit so a rollback still records the attempt. The planner pack §G.15 explicitly mentions "Write audit FIRST via D-S2 independent session so it persists even on rollback" but the docstring in the file claims this while the code does the opposite. **Not a blocker** because: (a) the audit writer uses an independent session, so it would persist independently anyway, and (b) the success path is currently the only path that writes audit (failure paths after token decode for inactive user are handled at deps level which emits 401 with no audit). For the inactive-user PATCH path (T22), the deps.py 401 short-circuits before the service is reached — no audit row is written — which contradicts §G.15 ("Failure-audit reasons: 'unauthorized' (only when user identity could be determined post-decode)"). This is informational; the pack §J does not test the inactive-user audit row, so no test fails. Document as residual risk; debugger can address in same TASK_ID if main-orchestrator requests it.

**N3 — `test_users_me.py` T25/T26 assert no WARNING/ERROR on success path but the planner pack §J wording is "ZERO INFO app.users.*"**. Since the implementation uses `logger.debug` (not `info`) the test passes trivially and the intent (no INFO spam at verbose=false) is satisfied by the code, so the spirit of the requirement is met. Document only.

**N4 — `repositories/user_profile.py:131` — `session.expire_all()` is a hammer**. It works (forces a re-SELECT) but invalidates the entire session cache. A targeted `session.refresh(user)` after re-fetching would be more surgical. Not a defect; performance is irrelevant at this slice's call volume.

**N5 — `services/get_current_user_profile.py:82-83` — `# type: ignore[arg-type]` for `status` and `preferred_language`**. The ORM declares these as `Mapped[str]` but Pydantic expects `Literal["active","inactive","pending","locked"]` / `Literal["es","en","fr"]`. The DB-level CHECK + app-level invariants make this safe at runtime; the type-ignore is honest. Could be improved with a cast or per-field validator, but not required.

- hypothesis: broad `except (jwt.InvalidTokenError, Exception)` at backend/app/users/deps.py:133 violates `.claude/rules/01-non-negotiables.md` §Error handling ("Never catch generic Exception/Error. Catch typed domain errors") and contradicts task pack §G.2 pin.
- root_cause: developer caught `Exception` defensively to keep the anti-enum 401 envelope byte-equal across every unexpected decode failure. The intent (don't leak which decoder branch crashed) is correct, but the typing is wrong: every legitimate decode failure mode either inherits from `jwt.InvalidTokenError` (DecodeError, InvalidSignatureError, MissingRequiredClaimError, InvalidAudienceError, InvalidIssuerError, etc.) or surfaces as a `ValueError` from `app.auth.tokens.decode_token` (raised for `purpose` mismatch — defense-in-depth even though Step 3 also explicit-checks `purpose`). Catching `Exception` swallows unrelated runtime bugs (TypeError, AttributeError, OSError) that should bubble up to the global error boundary and produce a 500, not a silent 401.
