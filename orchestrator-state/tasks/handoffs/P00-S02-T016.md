### Root cause confirmed
H1 was correct: `pytest-asyncio asyncio_mode=auto` creates a new event loop per test function. The `app.core.db._engine` singleton is created on the first handler invocation with DB access, binding it to loop A. Subsequent tests run on loop B and asyncpg raises "Task got Future attached to a different loop".

The bug ONLY manifests when `.env.local` is loaded (activating `VERIFICATION_GEMINI_API_KEY`), because without it, the Gemini tests skip — leaving only 1 test that touches DB (`test_discover_models_404_unknown_provider`), so there's only one loop and no reuse.

### Fix applied — Plan A (preferred)
Moved the `reset_db_engine_singleton` fixture from per-file usage to `backend/tests/integration/conftest.py` as a shared `autouse=True`, function-scoped, synchronous fixture.

The fixture:
1. Before each test: sets `db_module._engine = None` and `db_module._session_factory = None`
2. After each test (teardown): disposes the engine if created, suppresses all exceptions

This forces the app to create a fresh engine inside each test's own event loop.

### Decision: autouse vs opt-in
**Decision: autouse** — applied globally to all `integration/` tests.

Rationale:
- The bug affects any integration test that calls a FastAPI handler touching DB (structural issue, not local to one file).
- `test_auth_signup.py` already has its own copy (idempotent — setting `None` twice is a no-op; no double-dispose risk).
- Tests that don't use the FastAPI handler/singleton (seed tests, migration tests, direct-engine tests) are unaffected: setting `_engine = None` is a no-op if it was never set.
- Impact audit in task pack §13: all 9 sibling test files verified as neutral or positive.

### test_auth_signup.py
NOT modified (write_set protected). The existing local `reset_db_engine_singleton` in that file runs in addition to the conftest fixture — idempotent, harmless.
