## Risks / open points (initial)

1. **pydantic version constraint (KNOWN)**: Locked to 2.12.5 by litellm. If litellm upgrades and relaxes
   the pydantic constraint, we can bump pydantic to 2.13.x+. Not a current blocker.

2. **redis version constraint (KNOWN)**: Locked to 6.4.0 by celery→kombu. When celery releases with
   kombu that supports redis>=7, we can upgrade. Not a current blocker.

3. **structlog ConsoleRenderer in test output**: structlog in verbose mode uses ANSI escape codes.
   Tests don't set `ENABLE_VERBOSE_LOGGING=true` so this is not an issue for CI.

4. **deepagents pulls langchain-anthropic transitively**: not declared in pyproject but installed.
   This is acceptable — transitive dep of deepagents. No direct import in app code.

5. **`python-multipart` import name change**: In newer versions, the import is `import python_multipart`
   not `import multipart`. Test uses `importlib.import_module("multipart")` which works with the
   currently installed 0.0.27 (still exports `multipart` module). Watch this on upgrade.

6. **db.py eagerly calls `_get_engine()` at module level**: The `engine` export at the bottom of db.py
   triggers engine creation on import (for Alembic config access). This is safe in T003 (no real DB;
   engine creation succeeds without a live connection), but in tests that import `app.core.db`,
   `DATABASE_URL` must be set or default will be used. Smoke test uses `model_construct()` to avoid this.

### Findings (in priority order)

#### Finding 1 — YAGNI / over-eager export at `backend/app/core/db.py:118` (BLOCKING for changes_requested; 1-line fix)

```python
# backend/app/core/db.py line 118 (last line of file)
engine: AsyncEngine = _get_engine()
```

Issue: this runs `_get_engine()` at module import despite the module docstring (line 11) asserting "This module does NOT connect to the database at import time." The function technically does not connect (it only builds the engine; pool opens on first query), but the module-level export is speculative — the comment claims it is "for Alembic async config in P01-S01-T001," yet Alembic is not in this slice. Tests cope by using `Settings.model_construct()` (line 116 of `test_dependency_smoke.py`) to dodge env reads triggered by the eager call — that hack is itself the symptom.

The task pack `Impact analysis` explicitly said: "Do not call `engine.connect()` at import time" and "~60-80 lines. **No live connect.**" The current implementation is 118 lines and exposes a module-level `engine` that the next slice has not asked for yet.

Recommended fix (debugger-sized, ~3 line edit): replace the module-level export with a lazy public accessor.

```python
# Replace this:
engine: AsyncEngine = _get_engine()

# With this:
def get_engine() -> AsyncEngine:
    """Public accessor — lazy. Will be used by P01-S01-T001 alembic env.py."""
    return _get_engine()
```

This preserves the public name discoverability (Alembic can still import `get_engine`), drops the import-time side effect, and makes the test no longer need `model_construct()` workaround.

Severity: low (does not break tests, does not connect, is documented in handoff Risks #6). But because the fix is trivial AND the dev openly acknowledges the smell, I prefer fixing in-slice via `debugger` rather than a follow-up.
