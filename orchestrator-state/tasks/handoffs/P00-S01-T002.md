### Important decisions

1. **react-i18next v17 + i18next v26 instead of v15/v25**: npm 11 strict mode fails
   with ERESOLVE because react-i18next@15.x declares peerOptional `typescript@"^5"`
   which conflicts with our typescript@6.0.3. react-i18next@17 declares
   `typescript@"^5 || ^6"` as optional peer and requires `i18next >= 26.0.10`.
   This is API-compatible for the providers shell; T005 i18n namespace code is
   unaffected. Discrepancy documented in
   `orchestrator-state/memory/official-doc-notes/P00-S01-T002-frontend-libs.md`.

2. **@hookform/resolvers v5 + zod v4**: hookform/resolvers 5.x supports RHF ^7.55+;
   zod 4.x is current stable (major from 3.x). Both are compatible and are the npm
   latest-stable choices for a greenfield 2026-05 project.

3. **I18nextProvider wrapping order (outer→inner)**: I18nextProvider is outermost,
   then BrowserRouter, then QueryClientProvider. Rationale: t() needed everywhere
   including error boundaries that may wrap the Router; BrowserRouter above
   QueryClientProvider so route-aware mutation callbacks can call useNavigate.

4. **i18n init inline**: `i18n/index.ts` belongs to T005. T002 inlines a minimal
   i18next init with empty resources so I18nextProvider mounts without crashing.
   i18next v26 with no async backend is synchronous: `isInitialized` is `true`
   immediately after `.init()` (no Suspense/async wait needed for tests).

5. **Providers.tsx logging**: `import.meta.env.VITE_ENABLE_VERBOSE_LOGGING` gates
   `console.info`. In test environments Vitest passes env vars correctly.
   The inline logger (<20 lines) avoids creating `shared/` (DRY exception justified
   because this is single-file usage only; shared/ is for 2+ features).

6. **T001 scaffold completion (WRITE_SET_DRIFT)**: T001 only committed
   `frontend/package.json`. `vite.config.ts`, `vitest.config.ts`, `tsconfig.json`,
   `tsconfig.node.json`, `index.html` are declared in TECHNICAL_GUIDE §4 structure
   and are required to run `npm test`. Without them the verification command fails.
   Created them in T002 as scaffold completion; documented as drift in this handoff.

7. **package-lock.json at worktree root**: npm workspace hoisting puts the lockfile
   at the workspace root (`<worktree>/package-lock.json`) not `frontend/package-lock.json`.
   This is correct npm workspace behavior. The write_set declared `frontend/package-lock.json`
   but the actual artifact is the root-level file. Documented as drift below.
