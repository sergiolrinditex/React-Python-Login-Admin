## Scope
- Goal of the task: Fix the F5 reload race condition on /chat that causes spurious logout. Both `AuthProvider.hydrate()` (via `AuthRepository.refresh()`) and `httpClient._doRefresh()` (via 401 interceptor) were independently calling `POST /api/v1/auth/refresh` with the same HttpOnly refresh cookie, causing the backend to revoke the first token before the second request could complete.
- Files changed: see FILES_CREATED / FILES_MODIFIED below.

### Technical decision (Option A — KISS)

Extracted a new shared module `refreshSingleFlight.ts` that owns a single `let _inflight: Promise<string> | null = null`. Both `httpClient._doRefresh()` and `AuthRepository.refresh()` now delegate to `refreshAccessToken({ onAuthFailure, reason })`. On F5 reload, `AuthProvider.hydrate()` calls `repo.refresh()` → `refreshAccessToken({ reason: "hydrate" })` at t=0. Simultaneously, the React tree fires an `authFetch` that gets 401 → `_doRefresh()` → `refreshAccessToken({ reason: "interceptor" })` at t≈0. The second caller finds `_inflight !== null` (set synchronously before the first `await fetch()`), logs `auth.refresh.singleflight.joined`, and returns the same Promise. Only ONE POST /api/v1/auth/refresh is made.

Option B (making `authRepository.refresh()` call `authFetch`) was rejected per the task pack (creates coupling, doesn't actually de-duplicate the cold-mount race).
