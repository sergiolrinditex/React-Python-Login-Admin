## Debugger fix
- AGENT: debugger
- TASK_ID: P02-S02-T001
- OUTCOME: fixed
- NEXT_STATUS: validator_tester_pending
- TIMESTAMP: 2026-05-13T10:00:00+02:00
- hypothesis: Tester reported module-level `_JWT_KEY = os.getenv("JWT_PRIVATE_KEY", "")` in `backend/tests/unit/test_security.py` captured an empty string at collection time, causing TestPermissions T03-T06 to mint tokens signed with empty key and be rejected with `invalid_token`.
- root_cause: The tester's hypothesis was directionally correct but incomplete. The real chain is: when `pytest backend/tests -k security` is run from the worktree root, `backend/tests/test_health.py` (root-level test, not in `unit/`) is imported during collection BEFORE `backend/tests/unit/conftest.py` loads `.env`. `test_health.py` imports `app.main` which transitively imports `app.auth.tokens`, capturing `_JWT_KEY = ""` at module load time. The cached value in `sys.modules` is then used by `decode_token` at request time, regardless of when `.env` is loaded later. So the test-side `_JWT_KEY` is one half of the problem; the cached `app.auth.tokens._JWT_KEY` is the other half. In isolation (`pytest backend/tests/unit/test_security.py`) both halves are empty → tokens encode+decode with the same empty key → tests pass. In the acceptance gate the unit conftest fires for some imports but the app side has already snapshotted empty → encode and decode get out-of-sync.
- fix_applied: Two changes in the authorized write_set `backend/tests/unit/test_security.py` (no production-code edits):
  1. Replaced module-level `_JWT_KEY` / `_JWT_ALG` with lazy getters `_get_jwt_key()` / `_get_jwt_alg()` so the test side is robust to collection order and always reads the current env.
  2. Added `_sync_app_jwt_key()` (called at the top of `_mint_token`) that patches `app.auth.tokens._JWT_KEY` and `app.auth.tokens._JWT_ALGORITHM` back to the env value before encoding. This restores encode/decode symmetry without modifying production code (which is outside this slice's write_set).
- verification_rerun:
  - `pytest backend/tests/unit/test_security.py -v` → 18/18 PASS (3.82s).
  - `pytest backend/tests -k security -v` → 18 PASSED, 167 deselected, 3 FAIL outside scope (pre-existing test_users_me::TestGetMeSecurityShape failures in production-code territory).
  - `ruff check backend/tests/unit/test_security.py` → All checks passed.
- in_scope_defect: yes — write_set declared in task pack (`backend/tests/unit/test_security.py`); no source-of-truth amendment, new endpoint, journey, or scope expansion required.
- evidence: `orchestrator-state/tasks/evidence/P02-S02-T001/debug-cycle-1/` (pytest-unit-isolated.log, pytest-acceptance.log, ruff.log, summary.md).

### Is patching `app.auth.tokens._JWT_KEY` from tests acceptable?

YES — accepted as the correct in-scope fix. Reasoning:

1. The root cause (`_JWT_KEY = os.getenv("JWT_PRIVATE_KEY", "")` captured at import time in `backend/app/auth/tokens.py:57`) is a PRE-EXISTING defect from P01-S02-T002. It is NOT a regression of P02-S02-T001.
2. The defect does NOT manifest in production: `uvicorn` loads `.env` before workers import `app.main`, so `_JWT_KEY` is populated correctly at startup. The bug only surfaces in test collection when an integration test module (`backend/tests/test_health.py`) is imported before `backend/tests/unit/conftest.py` has loaded `.env`.
3. The task pack's `write_set` for P02-S02-T001 does NOT include `backend/app/auth/tokens.py`. Touching it from this slice would be out-of-scope drift requiring its own FU.
4. The fix is narrowly scoped: `_sync_app_jwt_key()` is invoked from `_mint_token()` only — a single test-only path. No production code path executes it. No global monkeypatch fixture installed.
5. The debugger's `residual_pre_existing` ALREADY flags the underlying `_JWT_KEY` import-time capture as a candidate for a future production-code refactor. Out-of-scope debt is visible, not hidden.

Counter-argument considered: "tests patching production-module attributes hide a real bug." Real in general; here the bug is documented residual, production runtime is unaffected, and the alternative (fix `tokens.py` from this slice) breaks write_set discipline. Smallest safe scope wins.
