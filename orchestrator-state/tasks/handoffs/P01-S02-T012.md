## Scope

  `alembic upgrade head` (host-side connection) in `scripts/dev-restart.profile.sh`. The race

  alembic step with `ECONNREFUSED`.

Note on acceptance #4: `--reset` restarts postgres via `compose down -v && compose up -d`, so
`compose stop postgres` before `--reset` is overridden by the reset itself. The negative control
was tested by directly invoking `db_health` from bash with postgres stopped/paused, confirming
the probes fail correctly. The `wait_for db_health 60 "Postgres"` + `fail "Postgres did not
become ready within 60s."` at db_reset:L336 is the hard-fail path that triggers when postgres
can't start within 60s (e.g., compose up itself fails or postgres crashes on startup).

### F. Risks / open points

- **R1 (residual race on very slow hardware)**: if a laptop's port-forward takes >60s, both
  `wait_for` calls (essential + reset) would timeout and fail non-zero with the clear message.
  This is correct behavior per the acceptance. No risk added by this fix; timeout is appropriate.
- **R4 (port hijack)**: if another process listens on 127.0.0.1:5432, `_host_pg_ready` would
  return 0 while the real postgres is down. Documented in task pack §H as known limitation,
  out-of-scope for this fix. `stop_orphan_on_port` already handles back/frontend ports.
- **zsh incompatibility**: `/dev/tcp` is a Bash builtin, not available in zsh. The script has
  `#!/usr/bin/env bash` shebang and must always be run via `bash scripts/dev-restart.sh`, not
  sourced in zsh. This is a pre-existing constraint (T008 pattern); not introduced by T012.
