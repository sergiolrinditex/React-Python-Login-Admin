### Important decisions

1. **Migration chain update (WRITE_SET_DRIFT)**: The existing migration 0003 had `down_revision = "0001"`. Inserting 0002 between 0001 and 0003 would create two heads (0002 and 0003 both branching from 0001), causing `alembic upgrade head` to fail with "Multiple head revisions". To maintain a single head and a linear chain (0001→0002→0003), I updated 0003's `down_revision` from `"0001"` to `"0002"`. This is outside the strict write_set but is mechanically required for correctness. See §WRITE_SET_DRIFT.

2. **asyncpg INET decode**: asyncpg 0.31.0 returns `ipaddress.IPv4Address` objects (not `str`) when reading INET columns (official-doc-note 2026-05-10, RESOLVED). All test assertions use `str(row.ip)` for safe string comparison. ORM annotation kept as `Mapped[str | None]` (runtime-safe; SQLAlchemy does not enforce annotations at runtime).

3. **ORM type for ip**: Task pack says `Mapped[str | None]` is fine — asyncpg accepts `str` on INSERT (via `inet_encode`). Only SELECT needs `str()` wrapper. No conversion to `ipaddress.IPv4Address` in ORM (per task-pack §Open questions resolved).

4. **PostgreSQL 18 INET information_schema type**: PG18 reports `data_type='inet'` for INET columns in `information_schema.columns`, not `'USER-DEFINED'` (which older PG versions may report). Test 1 accepts either: `data_type == 'inet' OR udt_name == 'inet'`.

5. **No indexes added**: Migration 0002 adds zero indexes. All 4 new columns are nullable scalars with no FK; no consumer queries exist yet. Adding indexes prematurely was explicitly prohibited by R1 risk note in task-pack.

6. **GDPR naming reconciliation**: `entity_type` (DB table name, e.g. "user") vs `resource` (REST-style path, e.g. "POST /api/v1/auth/sign-up") are semantically distinct — both kept as separate columns per §Naming reconciliation in task-pack.
