## Root cause + fix narrative

### Defect 1 — cwd-relative `--source` path (the bug)

**Location:** `scripts/dev-restart.profile.sh`, `db_reset()` function, the seed block (old lines 347–362).

**Sequence:** `db_reset()` runs `cd "${HILO_BACKEND_DIR}"` (which sets `cwd = <repo>/backend/`) and then calls:

```bash
"${py}" -m app.verification_data.bootstrap \
    --source data/verification
```

With `cwd=backend/`, the relative path `data/verification` resolves to `<repo>/backend/data/verification`, which does not exist. The `bootstrap.py` module aborts with "verification data directory not found". Because the failure was caught by the stale warn-only branch (Defect 2), `db_reset()` returned 0 anyway and users were silently not loaded.

**Fix (Option A from task pack):** Changed `--source` to an absolute path anchored on `${ROOT_DIR}`:

```bash
"${py}" -m app.verification_data.bootstrap \
    --source "${ROOT_DIR}/data/verification"
```

`${ROOT_DIR}` is exported by `scripts/dev-restart.sh` (line 45/54) before sourcing the profile — stable across all cwd positions. The `cd "${HILO_BACKEND_DIR}"` is preserved so Python `-m` module resolution still uses `backend/` as its working directory.

A comment block was added inline explaining the rationale and tagging it with the slice ID:

```
# --source uses an absolute path derived from ROOT_DIR so that cwd=backend/
# does not cause the relative path data/verification to resolve to
# backend/data/verification (which does not exist). P01-S02-T008 fix.
```

### Defect 2 — stale warn-only branch (the silencer)

**Location:** Same block, failure branch (old lines 354–359).

**Sequence:** The original failure branch contained:

```bash
warn "verification_data bootstrap failed — see ${BACK_LOG}."
warn "This is expected until P00-S02-T004 closes (FU-20260511145446)."
return 0
```

This comment was written when P00-S02-T004 was still open (the loader `:meta::jsonb` cast bug). That task is now `done` (PROGRESS.md confirms). Every legitimate seed failure is now a real problem, not a known-pending bug. The `return 0` after warn made `--reset` exit 0 even when no users were loaded — acceptance criterion (a) "exits 0 must be earned by real success" was violated.

**Fix:** Replaced the warn-only branch with a hard-fail matching the project non-negotiables (real error handling, no silent fallbacks):

```bash
fail "verification_data bootstrap failed; see ${BACK_LOG}."
```

Added an AFTER log on success:

```bash
log "verification_data bootstrap complete."
```

The file header comment for `db_reset` was also updated from the stale "warn-only on seed failure: P00-S02-T004 covers ..." phrasing to "hard-fail on seed failure: data/verification/ must exist and be loadable; any bootstrap error aborts --reset with non-zero exit."
