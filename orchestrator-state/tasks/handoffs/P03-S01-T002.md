## Decisions

### §D-T002-AUTH-PORT
Extend `IAuthRepository` with `SignUpRequest` + `SignUpOutcome` types and `signUp()` method.
Follows the exact same structural pattern as T001's `signIn()` addition. Domain layer has no external imports.

### §D-T002-AUTH-DATA
`AuthRepository.signUp()` uses plain `fetch()` (not `authFetch`) for public endpoint — no Bearer token needed.
Mirrors `signIn()` shape: `credentials:'include'`, `X-Request-ID`, full error map per status code.
BEFORE log: email_domain + password_len + has_full_name + legal_accepted. AFTER: user_id only.

### §D-T002-AUTH-ERRORS
7 sign-up typed error classes added: `NonCorporateEmailError`, `LegalNotAcceptedError`, `EmailTakenError`,
`PasswordPolicyError`, `SignupRateLimitedError` (with `retryAfter`), `SignupValidationError`, `SignupInternalError`.
Naming follows `signup` lowerCamel convention (differs from `signin` convention for prior classes).

### §D-T002-EMAIL-CORP
Client-side: zod `.email()` (RFC 5322 syntax only). Domain allowlist is server-authoritative (env-var `CORPORATE_EMAIL_DOMAINS`).
Frontend surfaces the 400 `AUTH_SIGNUP_NON_CORPORATE_EMAIL` rejection as form-level error in `permission_denied` state.
Rejected: hardcoding domain list in the frontend (env-var drift, sync nightmare).

### §D-T002-PASSWORD-PRE-VALIDATE
Mirror server policy (min 12, ≥1 letter, ≥1 digit) in zod for UX (immediate red feedback as user types).
Server is still the final authority; client pre-check is a UX hint only.

### §D-T002-LEGAL-LITERAL-TRUE
`z.boolean().refine(v => v === true, ...)` for `legal_acceptance` so unchecked checkbox shows a field error
before submit attempt. SignUpRequest type uses `legal_acceptance: true` (literal).

### §D-T002-CHECKBOX-INLINE
Inline editorial checkbox (styled `<input type="checkbox">`) inside `SignUpPage.tsx`.
No new shared design-system component. KISS + YAGNI — checkbox is layout primitive only.
Tokens used: `border: "var(--hairline) solid var(--color-ink)"`, `borderRadius: "var(--radius, 0)"`,
`accentColor: "var(--color-ink)"`.
Rejected: extract `EditorialCheckbox` to shared design-system (YAGNI until 2nd page needs it).

### §D-T002-SUCCESS-REDIRECT
`success → navigate(ROUTE_AUTH_SIGN_IN, { replace: true, state: { flash: "account_created" } })`.
NOT auto-login (sign-up response has no access_token; session requires a separate sign-in flow).

### §D-T002-NO-NEXT-PARAM
Sign-up does NOT consume `?next=`. It passes any incoming `?next=` through to `/auth/sign-in?next=<value>`
so deep-link survives the round trip. User must sign in to bootstrap session.

### §D-T002-409-NO-FIELD
On 409 `AUTH_SIGNUP_EMAIL_TAKEN`, do NOT highlight the email field (anti-enumeration).
Server returns `field: null` intentionally. Frontend shows generic copy:
"No se pudo crear la cuenta con ese email. Si ya tienes cuenta, intenta iniciar sesión."
Test T06 asserts email input does NOT have `aria-invalid="true"` after 409.

### §D-T002-PII-LOGGING
BEFORE: `email_domain` only (split after `@`), `password_len` numeric, `has_full_name` boolean, `legal_accepted` boolean.
NEVER log: full email, password, full_name.
AFTER (success): `user_id` UUID (non-PII, safe to log).
Test T14 in `useSignUp.test.tsx` actively asserts this invariant.

### §D-T002-USESIGNUP
Hook extracted (`useSignUp.ts`) for testability + DRY, mirrors `useSignIn.ts` shape.
`SignUpPage` becomes thin layout + state-binding component; orchestration stays in the hook.

### §D-T002-PROACTIVE-EXTRACT
`signUpHelpers.ts` extracts `useSignUpSchema()` + `SignUpFormValues` + `getSignUpErrorI18nKey()` + `SignUpErrorInfo`
from `SignUpPage.tsx` to keep the page under the ~300 eff. LoC cap (measured: 295 eff. lines after extraction).
Applies T001 validator REC-1 preemptively, avoids future revise-slice.
