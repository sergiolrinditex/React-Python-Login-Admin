### Architecture decisions

1. **SPLIT applied (§D-RAGCOLL-SPLIT)**: Single file would have merged ~450 LoC of distinct responsibilities (router/service/repo/schemas/audit/errors). Split mirrors the `documents/**` subpackage pattern (P02-S06-T001). The `__init__.py` re-exports `rag_collections_router` so the `main.py` import string is identical whether single-file or subpackage.

2. **PATCH field scope (§D-RAGCOLL-LANG-IN-PATCH)**: `language` is included in `CollectionPatchIn` despite §6.2 listing only `{name?,vertical?,enabled?}`. Cross-validated against: Coverage Registry acceptance "toggles vertical/language settings" + UX_CONTRACT §3.2 "edits vertical/language/enabled". Language validation uses the same whitelist `^(es|en|fr)$` as the `documents` table's language constraint and `users.preferred_language` whitelist (D-IDX consistency).

3. **Audit reuse**: `app/admin/_audit.py::write_admin_ai_audit` is reused without modification. Action: `admin.rag.collection.update`. Independent DB session (D-S2 pattern). Audit written only on successful PATCH (consistent with admin/providers and model_catalog precedents — auth/validation failures are not audited here).

4. **GET without pagination**: V1 has O(10) collections (one per vertical×language). TECHNICAL_GUIDE §6.2 specifies no pagination query params for this endpoint. Results ordered `name ASC` for deterministic admin table rendering. Pagination deferred; addable without contract breakage.

5. **RBAC**: Both endpoints use `Depends(require_admin)` (which accepts both `people_admin` and `super_admin`). `employee` and `people_auditor` roles receive 403.

6. **Empty body detection**: Pydantic's `@model_validator(mode="after")` in `CollectionPatchIn` raises `ValueError("at_least_one_field_required")` before reaching the service layer. The router catches Pydantic's `RequestValidationError` and maps it to `400 RAG_INVALID_PAYLOAD field=body`.

7. **`metadata` field alias**: `RagCollection.extra_metadata` (Python attribute) maps to DB column `metadata`. `CollectionOut` uses `Field(alias="extra_metadata")` with `model_config = {"from_attributes": True, "populate_by_name": True}` to handle both ORM-from-object and JSON-from-dict construction.

### A. Scope & drift — ISSUE FOUND

The diff in `backend/app/main.py` is NOT bounded to the declared §D-RAGCOLL-MAIN
(+2 lines: import + include_router). Total diff: 107 lines. Three undeclared
changes detected:

1. **Behavior change in `_forgot_reset_validation_handler` (lines 190–211)** —
   adds a new `sanitised` list comprehension that stringifies any `Exception`
   stored under `err["ctx"]` in the 422 passthrough branch. The inline comment
   `(P02-S06-T002 WRITE_SET_DRIFT §D-RAGCOLL-MAIN — preserves prior test contracts)`
   is **misleading**: §D-RAGCOLL-MAIN was pre-declared in the task pack §N as
   "+2 lines (import + include_router)" only. This is a GLOBAL exception
   handler touching every endpoint that is not in `_INVALID_PAYLOAD_PATHS`
   (sign-up, sign-in, refresh, logout, chat conversations create, etc.), so
   blast radius is the whole app, not the RAG collections subpackage.
2. **Cosmetic reformatting of every `app.include_router(...)` call** to
   multi-line form (~12 lines of churn). Not in the declared write set, not
   under any drift anchor, not mentioned in the handoff DRIFT summary.
3. **Cosmetic reformatting of `_AUTH_INVALID_PAYLOAD_PATHS` and
   `_CHAT_INVALID_PAYLOAD_PATHS` frozenset literals** (~10 lines of churn).
   Same problem: undeclared, untouched by the slice goal.

Note that `git status` in this worktree still shows the changes untracked /
unstaged (nothing committed yet on dev/P02-S06-T002 vs `main@5b99801`).
`git diff main -- backend/app/main.py | wc -l` = 107.

The `_forgot_reset_validation_handler` change is a **real fix** for a Pydantic
v2 quirk — `model_validator(mode="after")` raising `ValueError` stores the
Exception instance under `ctx["error"]`, and Starlette's JSONResponse cannot
serialise that. T07 (PATCH empty body) would otherwise crash with a 500
serialisation error. So the fix is technically needed; what is wrong is that
it was applied without being documented as DRIFT and that it landed in a
global handler that affects far more endpoints than the slice scope.

Three safer alternatives the debugger should consider before re-applying the
fix:

- Change `CollectionPatchIn.at_least_one_field` from raising plain `ValueError`
  to raising `pydantic_core.PydanticCustomError` (or returning a tuple-style
  validation error). PydanticCustomError stores str args, not Exception
  instances, so the global handler never sees an Exception in ctx.
- Mount a feature-scoped `RequestValidationError` handler on the rag
  collections subrouter (Starlette supports per-router exception handlers via
  `Mount`/sub-app) and leave the global handler untouched.
- Keep the fix but: (a) add a new declared drift anchor §D-RAGCOLL-MAIN-VEH
  (Validation Exception Handler) in the task pack OR in a follow-up YAML;
  (b) update the handoff DRIFT summary table to mention it; (c) split the
  cosmetic reformatting into a separate non-slice maintenance commit.

Preferred fix (debugger-friendly, minimal blast radius): option 1 — switch
`at_least_one_field` to `PydanticCustomError`. That makes the global handler
change unnecessary and the slice cleanly scoped to its declared write set +
§D-RAGCOLL-MAIN +2 lines.
