### Important decisions (+ doc source reference)
- **POSIX sh only** (`#!/bin/sh`, `set -eu`, no `set -o pipefail`, `[ ]` not `[[ ]]`, no `local`, no arrays): compose entrypoint forces `/bin/sh` on the Alpine-based `minio/mc:latest` image (task pack §3 + §7.1 constraint).
- **eval-based `require_env()`**: portable env var validator without bashisms. POSIX `sh` has no native way to reference `${!varname}` — `eval "val=\${$1:-}"` is the portable alternative.
- **Retry 5×2s around `mc alias set`**: task pack §9.5 R4 identifies the S3-API warm-up race; retry loop inside the script absorbs the window without looping the container (compose `restart: "no"` is preserved).
- **`mc alias set local`**: alias name `local` is ephemeral (lives in `/root/.mc/config.json` inside the one-shot container). Chosen to be short and collision-free with the host.
- **Endpoint `http://minio:9000` hardcoded**: task pack §7.1 item 3 explicitly mandates not making it configurable to avoid over-engineering. If the service name ever changes, compose file and this script change together in the same diff.
- **`MINIO_ROOT_PASSWORD` never echoed**: `mc alias set` receives the password as a positional arg; stdout/stderr is redirected to `/dev/null 2>&1` during the alias call to prevent creds leaking into compose logs (non-negotiables §Security + §Logging).
- **`mc mb --ignore-existing`**: idempotent acceptance literal (task pack §2 acceptance row). This flag returns exit 0 when the bucket already exists (stable behavior since MinIO mc RELEASE.2022+; researcher Q2 targets confirmation).
- **Logs on stdout** using `==> ` prefix: matches the project convention in `dev-restart.profile.sh` and `setup-from-scratch.sh`. Compose captures stdout and shows it in `docker compose logs minio-init`.
- **Errors on stderr** via `warn()`/`fail()`: errors visible in compose logs even if stdout is piped elsewhere.

### Findings (informational, not blocking)
- The script suppresses `mc alias set` stdout+stderr via `>/dev/null 2>&1` to protect creds. If a real failure happens (bad credentials, unreachable host), the actual `mc` error message is lost — only the generic `[ERROR] mc alias set failed after 5 attempts` reaches the operator. This is the correct security tradeoff (creds protection wins over diagnostics), but tester should be aware that debugging a real `mc alias set` failure may require a temporary `unset` of the redirect during root-cause analysis. **Not a blocker.**
- The `local` alias name (`ALIAS_NAME="local"`) is fine inside an ephemeral sidecar (alias lives only in the container's `/root/.mc/config.json`). No conflict with any host-side `mc` alias because the sidecar is short-lived and the config file is not mounted.
- Researcher Q3 (ready-gate via `mc ready`) was not adopted; instead the developer used a generic retry loop on `mc alias set`. Both approaches are valid; the retry loop is more conservative and works on older `mc` versions where `mc ready` may not exist. **Acceptable.**
