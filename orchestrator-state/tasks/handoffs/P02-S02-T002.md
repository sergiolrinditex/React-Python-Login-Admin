### §K Important decisions

**D1 — Option A (lru_cache + raise on empty):** Mirrors `app/security/encryption.py:_get_fernet()` exactly. The cache never pins `""` because the getter raises `RuntimeError` before returning. First successful call is cached; subsequent calls use cached value without re-reading `os.getenv`. Chosen over Option B (no cache) for codebase consistency.

**D2 — At-import warnings preserved (D-TK7):** Lines 62-73 of original `tokens.py` are preserved in the new form: `_startup_key = os.getenv("JWT_PRIVATE_KEY", "")` → conditional warnings → `del _startup_key`. No closure kept; no interference with the lazy cache. Prod/dev startup visibility maintained.

**D3 — Only `_JWT_KEY` converted to lazy (§F.2 / D-TK6):** `_JWT_ALGORITHM`, `_ACCESS_TTL`, `_MFA_CHALLENGE_TTL` kept at-import. KISS + YAGNI: no fixture rotates these at runtime in any test. Future slices can convert if needed.

**D4 — test_users_me.py symmetric fix with self-healing fallback:** `_get_jwt_key()` in `test_users_me.py` falls back to a test-only key when `JWT_PRIVATE_KEY` is absent, writes it into `os.environ`, and calls `_clear_jwt_key_cache()` so the app-side lazy getter re-reads the same key. This ensures both mint-side (test) and decode-side (ASGI app) use the same key even when pytest runs without env exported. Avoids needing a new `conftest.py` in `tests/integration/` (which would be outside write_set).

**D5 — Verbose logging gate in `_get_jwt_key()`:** `tokens.jwt_key.resolved bytes=N` fires only when `ENABLE_VERBOSE_LOGGING=true` (inline check in getter). Never logs the key value — only its byte length. `tokens.jwt_key.cache_cleared` uses `logger.debug` (controlled by logger level, not ENABLE_VERBOSE_LOGGING flag — consistent with rest of tokens.py where cache operations are debug-level infrastructure).

### §I.2 WRITE_SET_DRIFT — `backend/tests/unit/test_security.py`

**Drift declared (Option I.3.b from task pack):**

The T001 debugger workaround in `test_security.py:204` did `_toks._JWT_KEY = _get_jwt_key()`. After the refactor, `_JWT_KEY` no longer exists as a module attribute — the getter `_get_jwt_key()` is the only entry point. Assigning to `_toks._JWT_KEY` would create a dangling attribute that `app.auth.tokens._get_jwt_key()` ignores, making token decode fail with `InvalidSignatureError`.

**Fix applied (5 lines changed):** `_sync_app_jwt_key()` body replaced — instead of `_toks._JWT_KEY = _get_jwt_key()`, now calls `from app.auth.tokens import _clear_jwt_key_cache; _clear_jwt_key_cache()`. Docstring updated to explain the migration. `_mint_token` still calls `_sync_app_jwt_key()` first (unchanged call site).

**Justification:** Fixing this inside P02-S02-T002 is KISS (5 lines, same conflict_group `auth:tokens`, zero new routes/tables/journeys). Not fixing it would leave 4 `TestPermissions` tests broken, which is a regression introduced by this slice's removal of `_JWT_KEY`. The validator has explicit authority to accept or reject this drift per task pack §I.3.b.
