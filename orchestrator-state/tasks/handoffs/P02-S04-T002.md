### File size verdict for `tasks_embeddings.py` (305 LoC)

**ACCEPT — tilde wiggle.** The non-negotiable rule allows ~300 hard cap and explicitly states "self-contained components that contain only layout, local state, lifecycle" may reach ~300. `tasks_embeddings.py` contains exactly two functions: `call_embeddings()` (29 LoC including docstring) and the task body `embed_chunks()` (~158 LoC). The 5 LoC overage above ~300 is entirely docstring + section banner comments — the actual code density is normal Celery task orchestration (linear stages: resolve model → fetch chunks → loop batches with per-batch try/except for HTTPStatusError + dim validation + persist + progress). Splitting the httpx client (`call_embeddings`) into a separate `embeddings_client.py` would gain only ~30 LoC and add an import boundary for a function used in exactly one place — that's premature abstraction (YAGNI). The task body itself **cannot be split without losing the linear stage flow** that is the readable backbone of the task. Recommend no change.

> Caveat: the embed_chunks **task body function** is ~158 LoC, which exceeds the per-function "≤50 LoC" soft target. This is a known production pattern for Celery tasks orchestrating multi-stage pipelines with per-stage logging and per-stage commits. The same shape applies to `extract_and_chunk` (~163 LoC). Splitting into private helpers (e.g. `_run_extract`, `_run_chunk`, `_run_persist`) would multiply state-passing and obscure the linear narrative. **Soft flag, not a blocker** — this is the standard trade-off for Celery task entrypoints.
