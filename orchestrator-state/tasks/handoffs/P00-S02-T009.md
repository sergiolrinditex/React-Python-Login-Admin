### Important decisions

1. **Named-logger approach** (`logging.getLogger("httpx").setLevel(logging.WARNING)`) chosen over event_hooks, Filter, or custom transport. Rationale: smallest possible change (~3 lines), no write_set widening, no new deps, idempotent inside the existing `_configured` guard. See task pack §9.2 rejected alternatives.

2. **Applied UNCONDITIONALLY** — not inside `if verbose:` branch. In verbose=False mode the root is already at WARNING, but the explicit pin guards against future commits that change root level without re-pinning httpx (defense-in-depth, risk R2).

3. **Both `httpx` and `httpcore`** pinned to WARNING. Researcher confirmed these are the correct logger names for httpx 0.28.1. `httpcore` covers DEBUG transport lines that could surface headers in pathological future versions (risk R1).

4. **MockTransport** used for T1/T3/T4. The leak vector is the REQUEST URL logged by httpx BEFORE the transport runs — a fake transport still exercises the exact logging path. Google Gemini is an external API we do not control; mocking is explicitly permitted per `01-non-negotiables.md §Tests are REAL`.

5. **async test functions** (`async def test_*`) used for T1, T3, T4, T9 because `httpx.AsyncClient` requires an async context. `pytest-asyncio asyncio_mode=auto` (set in `pyproject.toml`) handles this automatically — same pattern as `test_admin_ai_discover_models.py`.

6. **`_restore_logging_state_with_httpx()`** extends the existing T004 pattern to also reset httpx/httpcore logger levels. Without this, T009 tests would leave the httpx logger at WARNING in subsequent tests that might check default levels.

7. **T5 (real Gemini test)** skipif-gated on `VERIFICATION_GEMINI_API_KEY` presence. Follows same guard as T006's `test_discover_models_gemini_real`. Skipped in CI without the key.

8. **Did NOT touch** `backend/app/features/admin_ai/provider_clients.py` (owned by T006) or `backend/app/main.py` (call site). Write_set drift: none.
