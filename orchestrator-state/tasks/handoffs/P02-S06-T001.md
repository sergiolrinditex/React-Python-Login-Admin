### Q2 — MIME sniffing: hand-rolled magic bytes vs python-magic

**Answer:** A hand-rolled magic bytes sniffer is **sufficient but must be layered** per OWASP's defense-in-depth recommendation. OWASP File Upload Cheat Sheet (2026) states file signature validation "should not be used on its own, as bypassing it is pretty common and easy" — combine it with extension allow-list + server-side content re-validation. For PDF: `%PDF` (bytes `25 50 44 46`) at offset 0 is reliable. For DOCX: `PK\x03\x04` (ZIP magic `50 4B 03 04`) at offset 0, then check central directory for `word/document.xml` — **however**, there are critical DOCX pitfalls:

- **Encrypted DOCX** (password-protected): also a ZIP (`PK\x03\x04`) but contains `EncryptionInfo` and `EncryptedPackage` instead of `word/document.xml`. A magic-bytes check on the outer bytes would pass it as DOCX even though it cannot be processed.
- **OOXML strict variant**: uses `word/document.xml` but in a slightly different namespace — most parsers handle this but your zip-inspection must not require exact `word/document.xml` path (use `startswith("word/")` or similar).
- **Polyglot files**: a file can be simultaneously valid PDF and valid ZIP; rely on the first 4 bytes + structural check, not just outer magic.

`python-magic` (libmagic OS dependency): No CVEs found in NVD for python-magic in the 2024–2026 range from the NVD search. However, `libmagic` itself has had historical CVEs (e.g., via `file` command). **Recommendation:** Do NOT add `python-magic` — it introduces a native OS dependency (`libmagic`/`file` package) that complicates Docker builds and CI, and adds OS-level attack surface. The hand-rolled approach with `zipfile.ZipInfo` inspection + `word/document.xml` presence check is production-grade if also gated by extension + `python-docx` parse attempt (which will fail gracefully on encrypted/invalid files). Use `pypdf` and `python-docx` (already in requirements.txt) as the second-layer validator — a parse attempt that raises an exception is more reliable than magic bytes alone.

- Source: https://cheatsheetseries.owasp.org/cheatsheets/File_Upload_Cheat_Sheet.html (2026-05-13)
- Source: OWASP recommends defense-in-depth; no specific libmagic recommendation found.

### Q3 — FastAPI UploadFile chunked reading + body-size enforcement

**Answer:** The exact API for streaming-read with byte counter on `UploadFile` is:

```python
chunk = await file.read(1024 * 1024)   # reads up to 1 MiB at a time
```

`UploadFile.read(size)` is an async method that reads `size` bytes from the underlying `SpooledTemporaryFile` via a threadpool. Loop until `chunk == b""` while accumulating a byte counter. FastAPI 0.135.2 / Starlette does **NOT** expose a built-in `MaxRequestBodySizeMiddleware` by name. What Starlette does expose is `max_part_size` on `request.form()`, but that is a per-part cap on multipart parsing, not a total body size gate. For total body size enforcement, the correct approach is either:

1. **Middleware (preferred)**: wrap the `receive` callable in a custom ASGI middleware that counts `body` chunk sizes and returns HTTP 413 before the route handler sees the data. Starlette's own docs show the `LoggedRequestBodySizeMiddleware` pattern as the extension point.
2. **Per-route counter** (fallback): accumulate bytes in the chunked `await file.read(chunk_size)` loop, raise `HTTPException(413)` when the counter exceeds the limit, then `await file.close()`.

**Recommendation:** Use the ASGI middleware approach — it enforces the limit before multipart parsing starts, which is safer. Starlette 0.49+ has no built-in `MaxRequestBodySizeMiddleware`; it must be custom-written (10–15 lines).

- Source: https://github.com/kludex/starlette (Starlette docs — UploadFile, request.form, middleware pattern, 2026-05-13)
- Source: https://fastapi.tiangolo.com/reference/uploadfile (UploadFile.read async method, 2026-05-13)

### Q4 — boto3 1.43.6 put_object against MinIO

**Answer:** `put_object` is a **single-part upload** regardless of file size — it performs one HTTP PUT. The `ContentType` parameter is **optional** but should be passed explicitly (MinIO stores it and returns it in GET responses; if omitted it defaults to `application/octet-stream`). For files under 25 MiB, `put_object` with explicit `ContentType` is the correct approach — no multipart threshold concern. The `TransferConfig(multipart_threshold=...)` only applies to high-level methods (`upload_file`, `upload_fileobj`), not to `put_object` directly. boto3 with `endpoint_url` pointing to MinIO is the **standard, documented integration pattern** for S3-compatible object stores; MinIO implements the AWS S3 API. KMS/SSE is not required for MinIO RELEASE.2025-09-07 (MinIO uses its own SSE-S3 if configured, but it is not mandatory). **Confirmed: keep boto3**, do NOT replace with the `minio` Python SDK — it is inconsistent with the existing worker `_storage._s3_client` pattern.

```python
s3_client = boto3.client(
    "s3",
    endpoint_url=settings.S3_ENDPOINT_URL,      # e.g. http://minio:9000
    aws_access_key_id=settings.S3_ACCESS_KEY,
    aws_secret_access_key=settings.S3_SECRET_KEY,
    region_name="us-east-1",                    # MinIO ignores region but boto3 requires it
)
s3_client.put_object(
    Bucket=settings.S3_BUCKET_DOCUMENTS,
    Key=storage_key,
    Body=file_bytes,
    ContentType=detected_mime_type,             # explicit — do not omit
)
```

No MinIO-specific quirks with `endpoint_url` for `put_object` on files under 25 MiB. One known quirk: MinIO enforces `Content-MD5` integrity on some operations — boto3 handles this automatically.

- Source: https://docs.aws.amazon.com/boto3/latest/reference/services/s3/client/put_object.html (2026-05-13)
- Source: https://github.com/boto/boto3/blob/develop/docs/source/guide/s3.rst (TransferConfig, 2026-05-13)

### Q5 — Celery apply_async from FastAPI async endpoint

**Answer:** `apply_async` is a **synchronous blocking call**. It publishes a message to the broker (Redis) via Kombu's synchronous socket I/O — it does NOT use `async`/`await`. Celery 5.6.3 has **no async-native variant** of `apply_async`; no `async def apply_async` exists in the codebase and no asyncio-native dispatch mechanism was added in 5.3–5.6 (changelog confirmed). The Celery changelog for 5.3–5.6 contains no asyncio-native task dispatch entries.

**Risk:** When called directly from an `async def` FastAPI route, `apply_async` blocks the event loop thread for the duration of the broker TCP round-trip (typically 1–5 ms on local Redis, but up to >10 ms on tail latency or under load). This stalls all other concurrent requests on that event loop iteration.

**Recommendation:** Wrap in `asyncio.to_thread()`:
```python
import asyncio

@router.post("/admin/rag/documents/{id}/index")
async def enqueue_index(id: UUID):
    result = await asyncio.to_thread(index_task.apply_async, args=[str(id)])
    return {"job_id": str(result.id)}
```
`asyncio.to_thread` (Python 3.9+, confirmed available on Python 3.12 in this stack) runs the blocking call in the default thread pool executor without blocking the event loop. This is the correct pattern for calling any synchronous blocking I/O from `async def` routes.

- Source: https://docs.celeryq.dev/en/stable/userguide/calling.html (confirmed no async variant, 2026-05-13)
- Source: https://docs.celeryq.dev/en/stable/ changelog 5.3–5.6 (no asyncio dispatch, 2026-05-13)
- Source: https://docs.python.org/3/library/asyncio-task.html#asyncio.to_thread (Python 3.9+)
