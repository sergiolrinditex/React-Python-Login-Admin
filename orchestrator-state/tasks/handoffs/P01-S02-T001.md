### Decisions taken

1. **Rate limit ACTIVE NOW** (in-memory token bucket in `app/auth/rate_limit.py`): Default 10/min per IP, configurable via `AUTH_SIGNUP_RATE_PER_MINUTE`. Returns 429 with `Retry-After` header. `TODO(P02-S02-T001): swap in-memory token bucket for Redis-backed limiter`.

2. **No employee_profiles INSERT at sign-up** (§F.7 / §C.5): The sign-up payload `{email, password, full_name, legal_acceptance}` has no employee fields. `employee_profiles` requires NOT NULL columns (employee_id, brand, society, center, country, department) that cannot be sourced from the request. Employee profiles are seeded by the verification_data loader.

3. **Audit on rejections too** (§F.5): Rejection audit rows committed in a SEPARATE short transaction from the sign-up transaction. This ensures the audit row commits even when the sign-up tx was rolled back (e.g. UniqueViolation).

4. **Argon2id parameters** (§F.3): `PasswordHasher()` library defaults used. Per researcher note `P01-S02-T001-argon2-owasp-params-2026-05-11.md` RESOLVED: defaults (64 MiB / t=3 / p=4) EXCEED OWASP 2026 minimums. No code change needed. Docstring in `password.py` corrected.

5. **Corporate domain source** (§F.1): CORPORATE_EMAIL_DOMAINS env var, comma-separated. Default includes all Inditex group domains + inditex-sandbox.com. Fail-closed: empty value rejects all emails.

6. **HTTP status pins** (§C.2/F.8): 201 success; 400 non-corporate email / legal not accepted (if service reaches); 409 duplicate email; 422 payload invalid (including legal_acceptance=false via Pydantic field_validator); 429 rate-limited.

7. **legal_acceptance=false behavior**: Pydantic `field_validator` fires BEFORE the service layer. FastAPI returns 422 (not 400) for legal_acceptance=false. Task pack §C.3 says "optionally fold into 422" — this is the actual Pydantic behavior. Tests accept both 400 and 422. If 400 is strictly required, the schema validator would need to be removed and the check moved to service layer.

8. **No JWT issued at sign-up** (§C.4): Returns `{user_id, mfa_required: false}`. Token issuance is exclusively in sign-in (T002).

9. **User enumeration prevention** (§I.4): Duplicate email always triggers a dummy Argon2 hash computation before returning 409 to equalize response time. Response message is generic ("An account with that email exists or cannot be created").

#### F2 (changes_requested) — `legal_acceptance` Pydantic field_validator breaks 3 invariants

The developer placed `legal_acceptance` policy enforcement in a Pydantic `@field_validator` (`schemas.py:105-124 must_be_true`). When `legal_acceptance=false`, Pydantic fires the validator BEFORE the service layer is reached, producing:

1. **HTTP 422** instead of the **400** pinned by task pack §C.3 row `AUTH_SIGNUP_LEGAL_NOT_ACCEPTED`. The task pack does say "Optionally fold into 422" — but the planner pinned 400 ("so frontend can distinguish quickly"). The developer chose 422 unilaterally, recorded as Decision #7, but the planner pin was already explicit.
2. **Wrong response envelope shape**: Pydantic 422 falls through to FastAPI's default `RequestValidationError` handler, returning `{detail: [...]}` rather than the project-wide `{data, meta, errors}` envelope mandated by TECHNICAL_GUIDE §6.2. Test T03 acknowledges this by accepting either shape (`assert "detail" in data or "errors" in data`). The frontend cannot localize an error it cannot find by `errors[].code`.
3. **BR5 audit-every-attempt invariant broken**: task pack §F.5 / service.py BR5 says "Audit row written on EVERY attempt (success and rejection)." When the Pydantic path runs, the service is never reached → NO audit row is written for the rejected attempt. The developer's own test T03 conditionally checks audit row "only if 400" (test_auth_signup.py:213-222), which encodes the broken invariant. Compliance review will see a hole for `legal_acceptance=false` rejections in production.

**Fix (in-scope, small)**: remove the `must_be_true` field_validator from `schemas.py`, keep `legal_acceptance: bool` in the schema (so missing/non-bool still 422 — that's a payload structural error, distinct from policy).
