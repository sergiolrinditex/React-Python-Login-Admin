### Findings

**Finding #1 — In-scope defect: stale `current` after successful language change (changes_requested).**

- File: `frontend/src/features/auth/presentation/useLanguagePicker.ts`, line 104–108.
- Problem: `current` is derived purely from `useAuth().user.preferred_language`. After a successful PATCH the hook calls `i18n.changeLanguage()` (so UI text re-renders) but does NOT update any local state. `useAuth().user.preferred_language` only refreshes on the next AuthProvider hydration. Result: the radiogroup `aria-checked={lang === currentLanguage}` (AccountPage line 274) still reports the OLD language as checked. User-visible bug: click "English", text changes to English, but "Español" remains highlighted in the radiogroup.
- Why this is in-scope: both files are inside the declared write_set (`useLanguagePicker.ts` is on the `useLanguagePicker*` glob, AccountPage is under `frontend/src/pages/account/**`). The fix is small: introduce a local `pendingCurrent` / `confirmedCurrent` state in the hook that gets set on PATCH 200 and is preferred over `user.preferred_language` until next hydration. The task pack §D-T007-USERREAD-CHOICE explicitly listed this as one of the two acceptable paths ("Setter pattern — let server-returned new profile flow back into local state OR re-fetch via repo.fetchMe in mutation success handler"). Neither was implemented.
- Why tests miss it: AP05 mocks `useLanguagePicker` to return `current: "en"` directly, bypassing the real hook. LP4 only asserts `i18n.language === "en"` and `error === null`; it does NOT assert `result.current.current === "en"`. The debugger fix should add `expect(result.current.current).toBe("en")` after a successful setLanguage.
- Classification: `in_scope_defect`. Resolve via `debugger` cycle, not FU.

**Finding #2 — Marginal state gap: GET /users/me network error not rendered / `NetworkErrorView` dead code (changes_requested).**

- File: `frontend/src/pages/account/AccountPage.tsx`.
- Problem: `NetworkErrorView` (and an implied `onRetry`) are imported (line 62) but never rendered anywhere in the JSX. ### D1 — stale `current` after successful language change
- hypothesis: `current` was derived purely from `useAuth().user.preferred_language`, and AuthProvider does not expose a `setUser` API. After a successful PATCH the hook called `i18n.changeLanguage()` (so user-visible text re-rendered), but the radiogroup `aria-checked={lang === currentLanguage}` kept reading the stale `user.preferred_language`. The previous validator's Finding #1 reproduced this exactly.
- root_cause: confirmed — `useLanguagePicker.ts` lines 103–108 (old version) lacked any local state for the server-confirmed language. `currentUserQuery` is conceptually `useAuth().user` in this codebase (no separate TanStack query — §D-T007-USERREAD-CHOICE), and `AuthProvider` only refreshes `user.preferred_language` on next hydration (or via `signInAccepted` which is unrelated). Result: every PATCH 200 success left the hook reporting the OLD language until next mount.
- option_applied: **A — setter pattern** (preferred per task pack §D-T007-USERREAD-CHOICE second bullet, "let server-returned new profile flow back into local state"). Introduced a local `confirmedCurrent: Language | null` state in `useLanguagePicker`. On PATCH 200 success the hook calls `setConfirmedCurrent(result.value.preferred_language)`. - option_rejected: B — deriving `current` from `i18n.language`. Rejected because i18n is updated optimistically BEFORE the PATCH completes; if the PATCH fails and i18n is reverted asynchronously, the radiogroup would flicker between two values during the round-trip. - why_not_a_new_useAuth_setter: introducing `useAuth().setUser(...)` would touch `AuthProvider.tsx`, which is out of T007 write_set. Option A keeps the fix entirely inside the declared `useLanguagePicker*` glob.
- defensive_fallback: if the server echoes a malformed `preferred_language` not in `SUPPORTED_LANGUAGES`, the hook trusts the requested language (`setConfirmedCurrent(language)`) instead of leaving `current` stale.
