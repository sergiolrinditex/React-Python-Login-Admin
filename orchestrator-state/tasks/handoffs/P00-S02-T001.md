### Important decisions

1. **pgvector image**: Used `pgvector/pgvector:pg18-bookworm` (researcher confirmed pg18 is available with pgvector 0.8.2 preloaded; Bookworm = Debian stable variant).

2. **Worker placeholder**: Python `time.sleep(2**31)` (not `sleep infinity` shell) to satisfy PID 1 logging discipline. Service is included by default (no profile gate) so `docker compose up` will show worker as running/placeholder.

3. **Frontend profile gate**: Frontend service is marked with `profiles: ["frontend"]` so default `docker compose up` skips it until T004 delivers `tsconfig.json`. The Dockerfile is correctly authored and will work once T004 lands.

4. **LiteLLM healthcheck**: Uses `/health/liveliness` endpoint. If LiteLLM main-stable tag changes the path, tester will report it.

5. **MinIO console**: Bound to `127.0.0.1:9001:9001` only (not 0.0.0.0) per security checklist (R3).

6. **DATABASE_URL in compose**: Backend env uses `postgres` (compose service name) as hostname, not `localhost`. `.env.example` keeps `localhost` for host-side dev (alembic from host).

7. **No `host.docker.internal`**: All inter-service references use compose service names (`postgres`, `redis`, `litellm`, `minio`, `backend`).

8. **env_file vs env**: Compose uses `env_file: .env` to pass vars to containers. The `.env` file is gitignored and excluded from all `.dockerignore` files.

9. **No `latest` tags**: All images pinned to exact version/tag. Never `:latest`.

10. **Structlog + stdout**: Backend container relies on structlog already wired in T003 logging to stdout. `PYTHONUNBUFFERED=1` set in Dockerfile to ensure no buffering.

11. **PG18 volume mount path**: PostgreSQL 18 changed data directory layout — volume must be mounted at `/var/lib/postgresql` (not `/var/lib/postgresql/data`). PG18 auto-creates `/var/lib/postgresql/18/main/`. See: https://github.com/docker-library/postgres/pull/1259

12. **LiteLLM healthcheck uses Python urllib**: LiteLLM image has no `curl` or `wget`. Healthcheck uses `python3 -c "import urllib.request; urllib.request.urlopen('http://localhost:4000/health/liveliness')"`. Also requires `start_period: 120s` (image takes ~2min to fully initialize).

13. **Worker healthcheck disabled**: Worker service inherits `HEALTHCHECK CMD curl http://localhost:8000/health` from backend Dockerfile; this always fails (worker has no HTTP port). Override with `healthcheck: disable: true` in compose override. Real healthcheck (`celery inspect ping`) lands in P02-S04-T002.

14. **Postgres host port 5433**: Port 5432 was occupied by local PostgreSQL instance on dev machine. Mapped to 5433 on host. Inter-container comms still use port 5432 via `postgres` hostname.

#### Hallazgo 2 — Frontend nginx config not wired into the image (production gap, non-blocking now)

`frontend/Dockerfile` does NOT `COPY infra/nginx/default.conf` into the image (it can't — build context is `./frontend`, config lives at `./infra/nginx/`). Effect when frontend is eventually built (after T004 has tsconfig.json, which it now does): the running container will fall back to the default nginx-unprivileged config, lacking:
- SPA fallback (`try_files $uri /index.html`) → React Router deep links return 404.
- `/api/v1/` reverse proxy to backend → frontend cannot talk to backend without external proxy.
- Security headers (X-Content-Type-Options, X-Frame-Options, X-XSS-Protection).

Why **not** blocking right now: frontend is gated behind `profiles: [frontend]`; default `docker compose up` skips it; the slice's tester scope is the 6 non-gated services and `bash scripts/dev-restart.sh --check`. Acceptance is met as defined.

Why this is a real production gap: the FE Dockerfile asserts in its docstring "SPA: try_files $uri /index.html for client-side routing" and "Proxy: /api/v1 requests forwarded to backend service" but the image as built does NOT enforce either. The first slice that consumes the frontend image (P05/P06 release) will fail any deep-link or API call test. **Required action:** register a follow-up to either (a) add a bind-mount in compose `frontend` service for `./infra/nginx/default.conf:/etc/nginx/conf.d/default.conf:ro`, or (b) move `infra/nginx/default.conf` into `frontend/nginx/default.conf` so the build context can `COPY` it. Option (b) is cleaner for K8s/Helm portability.

This is registered as a **soft finding** for now — the closer or main-orchestrator should `register-followup propose --severity medium --kind wiring` so it is on the radar before P05/P06. Validator will not block the slice on it because the gate (default `up` excludes frontend) honestly defers consumption.
