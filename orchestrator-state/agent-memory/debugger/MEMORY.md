## P00-S01-T001 (2026-05-08) — Stale dependency pins on bootstrap manifests

- **Symptom**: `/verify-slice` flagged 13 of 14 declared dependency pins (PyPI + npm) as significantly behind current stable. `01-non-negotiables.md#dependencies` requires actually-current pins.
- **Root cause**: developer pinned to "current stable at bootstrap time" using stale memory; the official-docs-researcher safety-net pass at developer-time only flagged FastAPI and did not catch the rest. By the time the slice reached verify-slice, pins were 1–2 majors behind across the board.
- **Fix applied**: text-only diff in `backend/pyproject.toml` and `frontend/package.json` — same write set ("package manifests") declared by the TASK_ID. No installs, no other files touched. Bumped: fastapi 0.115.12→0.136.1, uvicorn 0.34.2→0.46.0, ruff 0.11.10→0.15.12, mypy 1.16.0→2.0.0, pytest 8.3.5→9.0.3, pytest-asyncio 0.25.3→1.3.0, react 18.3.1→19.2.6, vite 6→8, vitest 3→4, typescript 5.8.3→6.0.3, etc.
- **Pattern**: at scaffolding phase, deps are DECLARED but NOT installed. Pin-bump fix is low-risk because real install/compat verification happens 2 slices later (T002 = npm install, T003 = pip install + dependency_smoke). Major-version compat (pytest 9, pytest-asyncio 1.x, mypy 2.x, vite 8, react 19, ts 6) is the install-slice's problem.
- **Recurring gotcha**: `01-non-negotiables.md#dependencies` says "Pin exact versions" — it does NOT say "pin to whatever you remember was stable last quarter". Trust ONLY a registry query at the moment of writing the manifest, never training-data memory for AI/ML or fast-moving JS ecosystems.
- **Anti-pattern**: re-querying registries inside debugger when verify-slice already provided verified values. Apply the verified diff verbatim — re-querying wastes a debug cycle and risks drift between debug pass and the verify report.

## P00-S01-T003 (2026-05-08) — Eager module-level engine + docstring drift

- **Symptom**: validator `OUTCOME: changes_requested` (tester was already `pass`). Two findings: (a) `backend/app/core/db.py` exported `engine: AsyncEngine = _get_engine()` at module top-level, contradicting its own docstring claim of "no import-time side effects" and forcing tests to use `Settings.model_construct()` as a workaround; (b) `config.py` docstring claimed `pydantic 2.13.4` while the actual pin (forced by litellm transitive constraint) is `2.12.5`.
- **Root cause (a)**: speculative "future-proofing for Alembic in P01-S01-T001" added a module-level export that ran the factory at import time. The function does not connect to the DB, but it does build the engine, instantiate the pool object, read settings, etc. — all of these are import-time side effects that the docstring promised the module would NOT have. YAGNI violation: the consumer (Alembic env) does not exist yet, and when it lands it can call a public accessor.
- **Root cause (b)**: developer wrote the docstring before resolving the pin conflict, then forgot to update it after litellm forced the downgrade. Pure documentation drift.
- **Fix applied**: replace `engine: AsyncEngine = _get_engine()` (last line of file) with a public `get_engine() -> AsyncEngine` lazy accessor that delegates to the existing memoized `_get_engine()`. Docstring for the module updated to describe `get_engine()` instead of `engine`. Single character fix in `config.py` docstring. **Tests untouched** — they only imported `get_session`, never the deleted `engine` symbol.
- **Pattern — lazy-singleton in Python at module level**: when a module needs to expose a "singleton" expensive resource (DB engine, HTTP client, Redis connection, LLM client), prefer this triple:
  ```python
  _resource: T | None = None  # private cache, NOT a public symbol

  def _get_resource() -> T:  # private builder + memoizer
      global _resource
      if _resource is None:
          _resource = build_expensive_thing()
      return _resource

  def get_resource() -> T:  # public lazy accessor
      """Public lazy accessor — first call builds, all calls return same."""
      return _get_resource()
  ```
  Never `resource: T = _get_resource()` at module top-level. That builds the resource at import time and breaks (a) tests that need to override config before first use, (b) cold-start performance, (c) the documented invariant that "import is side-effect-free".
- **Pattern — SQLAlchemy async testing without a live DB**: smoke tests should `from app.core.db import get_session` and assert the symbol is callable. Do NOT call `get_session()` (it requires a live engine + pool). Do NOT call `_get_engine()` from tests unless the test fixture injects a working DSN (e.g. SQLite in-memory aiosqlite). The smoke pattern is import-only, exercise-real-symbol-elsewhere.
- **Pattern — pin-version drift in docstrings**: any docstring or comment that names a version number is a future debt. Either omit the version (`Dependencies: pydantic-settings, pydantic`) or accept that you must update both `pyproject.toml` AND every docstring/comment when bumping. There is no `ruff` rule that catches this; only manual review or grep audits do. For T003 this caught 1 drift; future audits should `grep -rn "pydantic 2\." backend/app/` etc. before the validator does.
- **Anti-pattern that caused this**: docstring written aspirationally ("This module does NOT connect at import time") while code does have an import-time side effect of building the engine. If the docstring describes a contract, the code must enforce it. Use a quick `python -c "import app.core.db; print('ok')"` with a deliberately broken DATABASE_URL to verify the contract (it should still print 'ok' if there are no import-time side effects beyond settings read).

## P00-S02-T002 (2026-05-09) — CWE-532 leak via structlog Rich traceback frame locals

- **Symptom**: `/verify-slice` flagged a BLOCKER security finding. Hitting `/ready` with a bad DSN (port 5499, no listener) under `ENABLE_VERBOSE_LOGGING=true` produced an stdout log file containing `hilopeople_dev_pwd × 14`, `localhost × 18`, `5499 × 16`, `postgresql+asyncpg × 5`. Response body itself was sanitized correctly; the leak was ONLY in stdout/stderr logs. Validator+tester missed it because (a) validator audited only the structured event_dict keys, (b) tester's DB-down test used a hand-crafted `OperationalError("simulated connection failure")` whose monkeypatched call site has clean frame locals.
- **Root cause**: `_logger.error("...", error_class=..., exc_info=True)` in `_probe_db()`'s except branches. In verbose mode, `app/core/logging.py` configures structlog with `structlog.dev.set_exc_info` + `structlog.dev.ConsoleRenderer()`. ConsoleRenderer's default exception formatter is Rich's `RichTracebackFormatter`, which has `show_locals=True` by default. asyncpg/SQLAlchemy bind `cparams = {host, user, password, port, database}` and `ConnectionParameters(...)` as frame locals on the connect path — Rich renders those locals verbatim to the output stream.
- **Fix applied**: drop `exc_info=True` from both except branches in `_probe_db()`. Add structured field `db_detail=detail` (sanitized via `_sanitize_db_error()`) so operators still get a usable error message. Defense in depth: same comment in `app/main.py` request_id middleware exception path (the path didn't use `exc_info=True` already, but the pattern is poison and easy to reintroduce). NO changes to `app/core/logging.py` — that file is out of scope for the debugger fix; the global `RichTracebackFormatter(show_locals=False)` tuning is tracked in follow-up `FU-20260509044829-disable-structlog-rich-traceback-show-locals-glo`.
- **Pattern — `exc_info=True` is POISON when ConsoleRenderer + RichTracebackFormatter are used with default `show_locals=True`**. Any function in any except branch that handles exceptions whose stack contains sensitive locals (DSN, password, JWT, API key, request payload with PII) WILL leak those locals to logs in verbose mode. The structlog `_redaction_processor` does NOT redact traceback locals — it only operates on the event_dict. Solution: either (a) drop `exc_info=True` everywhere and rely on structured fields + class name (the defensive default for prod), OR (b) configure `RichTracebackFormatter(show_locals=False)` globally and document it.
- **Pattern — regression test for CWE-532 traceback-locals leak**: build a fake engine/client whose call path raises an exception while binding the secret values as **frame locals** (not in the exception message). Capture stdout/stderr via `capsys`, assert the secret values are NOT present. To force `configure_logging(verbose=True)` past its idempotent guard: `core_logging._configured = False; core_logging.configure_logging(verbose=True); ... ; finally: core_logging._configured = prev`. This test FAILS with `exc_info=True` and PASSES without it — a true behavioral regression test.
- **Anti-pattern — "richer logs are always better"**: developers add `exc_info=True` because it gives them a stack trace in dev. But once `ConsoleRenderer` + Rich tracebacks are in the chain, `exc_info=True` is a credentials-leak time bomb. The structured `error_class=type(exc).__name__` field is enough for prod observability; debugging can use `logger.exception()` deliberately in known-safe code paths after auditing what locals exist in that stack frame. Default to OFF.
- **Validator/tester gotcha**: a mock-based DB-down test that constructs `OperationalError("clean message")` directly will NEVER catch this bug — the monkeypatched call has clean locals. Real reproduction requires either (a) a fake engine whose connect() builds and raises with the secrets as locals, or (b) hitting a real DB-down with `ENABLE_VERBOSE_LOGGING=true` and grepping the log file. Add the former to your test suite for any error-path that touches infra (DB, Redis, S3, LLM provider).
- **Acceptance commands at this slice (T002 debugger fix scope)**:
  1. `pytest tests/test_health.py -v` → 9/9 PASS (with `DATABASE_URL` overridden to compose port 5433).
  2. `pytest tests/ -v` → 48/48 PASS.
  3. `ruff check app/main.py app/api tests/test_health.py` → clean.
  4. `mypy app/api app/main.py tests/test_health.py` → clean.
  5. Live verbose=true uvicorn against `DATABASE_URL=postgresql+asyncpg://baduser:topsecret_pwd_xyz_8f3c1@hostxyz:54399/dbxyz` + `curl /ready` → 503 + log file count of `topsecret_pwd_xyz_8f3c1` MUST be 0.
  6. Negative test sanity: temporarily re-add `exc_info=True`, run test #9 → MUST FAIL.

## P00-S02-T003 (2026-05-09) — File-size cap violation: loader.py 690 lines (2.3× cap) → package split

- **Symptom**: validator `OUTCOME: changes_requested` (tester was already `pass` 11/11 + 62/63). Single finding: `backend/app/seeds/loader.py` is 690 lines = 2.3× the 300-line hard cap defined in `01-non-negotiables.md §File size`. The file packs six independent namespace use cases (auth/rag_chat/history/admin_ai/rag_docs/mcp_agents). The planner pack itself anticipated the split (§Predicted writes line 96: "Per-namespace loader functions (split when approaching ~250 lines)") but the developer shipped them in a single file.
- **Root cause**: violation of "1 use case per file". Six independent use cases collapsed into one module, ~110 lines per loader plus ~50 lines of shared infrastructure (LoadReport dataclass, `_hash_email` helper, module docstring). No business-logic bug — purely structural.
- **Fix applied**: convert `backend/app/seeds/loader.py` (file) → `backend/app/seeds/loader/` (package). One file per namespace + one `_common.py` for the two genuinely shared symbols (`LoadReport`, `_hash_email`). Public API re-exported verbatim from `__init__.py` so `from app.seeds.loader import load_auth, ...` works exactly as before. Each function moved verbatim — zero refactor of SQL, logging, dataclass updates, exit codes. 6 external callers (1 CLI + 5 test sites) unchanged.
- **Pattern — file-size cap fix is a package split, not a behavior change**: when a single module collapses many use cases, the smallest safe debugger fix is `module.py` → `module/{__init__.py, use_case_a.py, use_case_b.py, ...}`. Re-export the public API from `__init__.py` so callers stay unchanged. This is mechanical, low-risk, and the validator's exact requested outcome.
- **Pattern — KISS/YAGNI for the `_common.py` module**: factor out only the symbols **actually** shared by ≥2 of the new files. In this slice that meant `LoadReport` (used by all 6) + `_hash_email` (used by 2). Did NOT factor: a try/timing/log decorator (boilerplate is repetitive but each use site has subtle namespace-specific fields and tables; abstracting would obscure them). Resist the urge to "improve while you're in there" — debugger scope is "smallest safe fix", not "refactor opportunity".
- **Pattern — per-file logger names are a free win on package splits**: when each new namespace file does its own `_logger = get_logger(__name__)`, log lines now carry per-namespace logger names (`app.seeds.loader.auth`, `app.seeds.loader.history`, ...) instead of a single `app.seeds.loader` for everything. Operationally better (per-namespace filtering becomes possible) and zero impact on tests using `structlog.testing.capture_logs()` which captures regardless of logger name.
- **Anti-pattern — leaving `loader.py` alongside `loader/`**: Python prefers packages over modules but `__pycache__/loader.cpython-*.pyc` will silently shadow the new package after the split if not purged. Always: (1) delete `loader.py`, (2) `rm -rf backend/app/seeds/__pycache__`, (3) re-run smoke import. Failure mode: tests pass on first run but mysterious `ImportError` on second invocation when bytecode is reloaded.
- **Pattern — capture imports BEFORE/AFTER as evidence**: `grep -rn "from app.seeds.loader\|from app.seeds import loader" backend/ > evidence/debugger-cycle1-imports-before.log` before editing, same after editing, then diff. Hard objective evidence that the public API is unchanged. Beats narrative claims in handoffs.
- **Validator/tester gotcha**: a file-size violation is NOT detectable by `pytest`, `ruff`, or `mypy`. Only `wc -l` + the human eye + the file_size gate in the validator catch it. Adding `wc -l` to the developer's per-slice verification checklist would have prevented this debug cycle — log into PROGRESS as a future improvement.
- **Acceptance commands at this slice (T003 debugger fix scope)**:
  1. `wc -l backend/app/seeds/loader/*.py` — every file ≤300 lines (target ≤200). Largest in this fix: 157 lines (admin_ai).
  2. `grep -rn "from app.seeds.loader\|app.seeds.loader" backend/` — same 6 external hits as before.
  3. `python -c "from app.seeds.loader import load_auth, load_rag_chat, load_history, load_admin_ai, load_rag_docs, load_mcp_agents, LoadReport; print('imports ok')"` → exit 0.
  4. `pytest tests/integration/test_seed_*.py -vv` → 11/11 PASS.
  5. `pytest tests/ -v` → 62/63 (1 pre-existing T002 fail unchanged; zero new regressions).
  6. `ruff check app/seeds/` + `mypy app/seeds/ --ignore-missing-imports` → clean.

## P00-S02-T003 cycle 2 (2026-05-09) — Path-resolution wiring bug surfaced ONLY by live `--reset` end-to-end

- **Symptom**: `/verify-slice` ran the work-item acceptance command `bash scripts/dev-restart.sh --reset` end-to-end live for the FIRST time. The dispatcher printed "Reset complete" with overall `exit 0`, but the structured log captured `FileNotFoundError: Verification bundle directory not found: <root>/backend/data/verification` from the seed CLI. Validator and tester had both passed cycle 1 (validator approved post-loader-split; tester ran 11/11 + `bash -n` syntax + `--check` status — but **never invoked `--reset` live**). The bug was completely invisible to lint, typecheck, integration tests and `bash -n`. It only surfaced when a real verify ran the actual command end-to-end.
- **Root cause**: `scripts/dev-restart.profile.sh::db_reset` does `cd "${ROOT_DIR}/backend"` (correct — needed so `python -m app.seeds.bootstrap_verification_data` resolves the importable module via the venv) and then passes the loader the relative literal `--source data/verification`. Python's `os.path` resolves the relative path against CWD = `${ROOT_DIR}/backend`, so the loader looks for `backend/data/verification/` — which does not exist. The actual bundle is at the repo root: `${ROOT_DIR}/data/verification/`. The loader correctly exits 2 with `FileNotFoundError`, but the dispatcher's `|| warn` swallows the non-zero exit and the overall reset reports success.
- **Fix applied** (cycle 2): one functional line + comment block. `--source data/verification` → `--source "${ROOT_DIR}/data/verification"` (Option A: explicit absolute path; keeps existing `cd backend/` so the module resolves correctly). The `info` log line was updated in lockstep so the printed command matches the executed command. File grew by 7 comment lines documenting WHY the absolute path is mandatory (operator-debuggability invariant: future devs should see the trap at the call site, not have to discover it via end-to-end reproduction).
- **Pattern — relative paths after `cd` are silently broken when the bundle is sibling, not child**. When a script does `cd <subdir>` to set up some import/runtime context (venv activation, npm cwd, alembic working dir), any relative argument passed afterwards resolves under that subdir. If the resource is a **sibling** of the subdir (typical for `data/`, `fixtures/`, `docs/` at the repo root), the relative path will silently resolve to a non-existent location. Fix: pass the bundle as an absolute path `${ROOT_DIR}/<resource>` (or `$(git rev-parse --show-toplevel)/<resource>`). Cheaper than refactoring out the `cd`.
- **Pattern — `|| warn` is a credentials-leak time bomb's cousin**. T002's lesson was "`exc_info=True` + ConsoleRenderer leaks frame locals". This slice's lesson is "`|| warn` swallows the exit code AND silently masks failures". Both are anti-patterns that look like "good defensive engineering" but actually destroy observability. The dispatcher pattern `cmd || warn "non-blocking"` is fine when the failure is genuinely non-fatal AND the operator is supposed to see the warning AND the exit code is propagated up. It is NOT fine when (a) the operator scrolls past the warning amid 30 lines of "==> Reset complete" output, or (b) the dispatcher's overall exit code is then taken as a green light. **Defense**: if you write `|| warn`, also write a `tail -1` or grep on the log file in the closing diagnostic; or escalate the warn to a non-fatal error with a distinct exit code.
- **Pattern — the gap between "tester passed bash -n" and "verify-slice ran live"**. The tester's coverage included `bash -n scripts/dev-restart.profile.sh` (syntax check) and `bash scripts/dev-restart.sh --check` (status probe), both of which are correct safety nets but neither of which executes the `db_reset()` body. `--reset` was deliberately not run in the tester to avoid affecting compose state. Net effect: the bug was completely undetectable until a human-gated reset. **Defense**: when a slice's acceptance command is genuinely destructive (`--reset`, `migrate down`, `drop schema`), the tester pipeline cannot be expected to run it. The verify-slice human gate IS the natural firing pin. Document this gap in the task pack so the planner doesn't expect the tester to catch it. (T003's task pack already anticipated this in §4: *"If the existing flow surfaces UX gaps (silent skip when a namespace is missing? Confusing exit code on tableless DB?), surface them in the handoff and either (a) adjust db_reset() minimally or (b) register a follow-up."* — the planner saw this risk; the verify-slice was the right place to fire on it.)
- **Anti-pattern — re-touching the loader package**. Cycle 1 fixed the loader.py file-size split; cycle 2's bug was in a wholly different file (the shell dispatcher). Resist the urge to "improve" the loader while you're in there. Each debug cycle has ONE root cause and ONE write target. Touching unrelated files contaminates the cycle and makes the diff hard to review. T003 cycle 2's diff is exactly: 1 file, +9 −2 lines.
- **Anti-pattern — broadening scope to fix the next-surfaced error**. Once the FileNotFoundError was gone, `InvalidPasswordError` (port mismatch) surfaced. That is **FU-20260508225027** — already tracked, severity=medium, non-blocking, and explicitly out of scope per the orchestrator brief. Don't fix it. Don't touch `.env`. Don't touch `backend/app/core/config.py`. Stop at the original boundary. The follow-up will land via its own promotion path with explicit human approval.
- **Acceptance commands at this slice (T003 debugger cycle 2 fix scope)**:
  1. `bash -n scripts/dev-restart.profile.sh` → exit 0 (syntax check after edit).
  2. `bash scripts/dev-restart.sh --reset` end-to-end → no `FileNotFoundError: backend/data/verification`. Either exit 0 (if DSN reachable) or exit non-zero with `InvalidPasswordError` (FU-20260508225027 unchanged). Capture full log.
  3. `cd backend && source .venv-verify-t003/bin/activate && DATABASE_URL='postgresql+asyncpg://hilopeople:hilopeople_dev_pwd@127.0.0.1:5433/hilopeople_dev' pytest tests/integration/test_seed_idempotency.py tests/integration/test_seed_namespaces.py tests/integration/test_seed_missing_bundle.py -vv` → 11/11 PASS (regression safety net; the fix is in shell, not Python, so this is a sanity check).
  4. `wc -l scripts/dev-restart.profile.sh` → ≤300 (hard cap unchanged; comment-only delta of +7 lines).

Pure data-fidelity bug; no logic, schema, regex, or test-runner issue. - **Pattern — defense-in-depth tests must NEVER use real credentials as fixtures**. The whole point of an "is this a real key?" rejection test is to assert that real keys would be rejected. Using a real key inside the test source defeats the goal at the file-system level — the test passes the assertion but fails the broader "we don't ship credentials" invariant. Use a synthetic value that is **structurally identical** to the regex (same prefix, same length class) but **clearly fake** to a human reviewer (all-A body, repeating-char body, dead-giveaway substring like `EXAMPLE` or `DUMMY`). 40 chars of `A` after `AIza` says "synthetic" to anyone who looks; a real-looking entropy-rich body does not.
- **Pattern — security grep coverage must include `backend/tests/**`**. Validator's audit-grep scoped to `data/verification/**` was a reasonable default for THIS slice (the slice's primary write set was `data/verification/`), but credentials can leak into any text file under repo root. The minimal next-iteration improvement is to add `backend/tests/`, `frontend/tests/`, `scripts/`, `.github/`, `docs/` (excluding source-of-truth examples that document the regex), and `*.md` to the validator's standard secret-scan set. Better: a project-level `git secrets`/`gitleaks` precommit + CI hook so this kind of leak is caught automatically the first time it appears.
- **Pattern — synthetic-value choice that satisfies regex but is unmistakably fake**. For `^AIza[A-Za-z0-9\-_]{35,}$`: `AIzaSyAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA` (40). For `^sk-[A-Za-z0-9]{32,}$` OpenAI: `sk-EXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLE` (38). For Anthropic `^sk-ant-[A-Za-z0-9-]{40,}$`: `sk-ant-EXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLE` (48). The principle: prefix-correct + length-class-correct + body that screams "synthetic" to a human eye. ALL-`A` and `EXAMPLE` repetition both work.

- **Anti-pattern — assuming "untracked == safe"**. The leaked key was in an UNTRACKED test file, so it never reached `origin/main`. But it WAS in (a) the developer's local working tree, (b) the orchestrator-state ledger as part of past tool-call payloads, (c) read in by parallel terminals via Claude tool calls. Once a productive credential is transcribed into ANY local file (even gitignored), the integrity assumption is broken — ALWAYS recommend rotation, regardless of git tracking status. The cost of rotation (a 30-second GCP console click + redeployment of fixtures referencing it) is trivial compared to the cost of an undetected breach.
