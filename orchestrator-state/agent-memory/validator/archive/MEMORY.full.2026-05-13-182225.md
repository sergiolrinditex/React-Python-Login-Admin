## Recurring patterns observed (good)

- **Write_set extensions when transparent**: developer declares extensions explicitly in a "Write set extension request" table with path + justification + risk-if-not-created (T001, T003, T001 infra). When the justification matches the task pack §D allow-list, approving is straightforward. Drift becomes a finding only when extensions are silent or undocumented.
- **Multi-stage Dockerfiles done right**: builder + runtime split with non-root user (uid 1001), `--no-install-recommends`, `rm -rf /var/lib/apt/lists/*`, layer-cache-friendly COPY of manifests before code. Pattern verified clean in P00-S02-T001 backend/Dockerfile.
- **Compose v2-spec hygiene**: no `version:` key, no `extra_hosts: host-gateway`, named volumes only, healthchecks with full `test/interval/timeout/retries/start_period`. Rancher Desktop (moby + containerd) compatibility maintained.
- **Reconciliation flow with researcher**: when researcher flags a discrepancy (warn-only), developer/orchestrator reconciles, marks `RESOLVED:` on each note, and appends a "Reconciliation" section to the handoff. Validator then verifies the actual code-level change matches the reconciliation prose (NERDCTL CAVEAT block presence; `nginx:stable-alpine` literal in Dockerfile FROM line). This is the right loop.

## Project conventions discovered

- **Service name vs image name**: `redis` is the service name but the image is `valkey/valkey:8-alpine`. The service name must remain `redis` to preserve `REDIS_URL=redis://redis:6379/0` DNS resolution. This is a hard invariant fixed by the human in task pack §C — don't suggest renaming.
- **Worker as same-image-different-command**: Celery worker reuses `backend/Dockerfile` via `build:` block in compose and overrides `command:`. This is the canonical DRY pattern for FastAPI+Celery in this stack. Don't expect a separate worker Dockerfile.
- **R5 acceptance interpretation**: "worker boot locally" is met at service-declaration level when the worker's target module (`app.worker`) lands in a future slice (P02-S04-T002). `restart: on-failure` is the documented mitigation. Don't reject for "service crashes on up" — that's expected.
- **`.env.example` ext pattern**: docker-compose.yml service requirements (POSTGRES_USER/PASSWORD/DB, MINIO_ROOT_USER/PASSWORD, LITELLM_MASTER_KEY) are valid write_set extensions when each new key is justified as a compose service requirement and TECHNICAL_GUIDE §11.1 doesn't pre-declare it. Document with comments inside the file linking to the consumer service.
- **`:latest` tags on sidecars**: `minio/mc:latest` is acceptable when the service is one-shot (`restart: "no"`) and has no schema coupling. `nginx:stable-alpine` is acceptable for the SPA serving layer (tracks current 1.30.x line; no schema coupling). Don't reject reflexively — check schema-coupling first.
- **Worktree path issue (R8)**: `hook_write_scope_guard.py` blocks Edit/Write on paths under `.claude/worktrees/` even though they are legitimate slice artifacts. Workaround used across T001/T003/T001-infra: developer/validator uses `Bash` heredoc (`cat >> path <<'EOF'...EOF`) instead of the Edit/Write tools. Document this in the handoff as R8.

## Security findings patterns (none critical yet)

The `sk-` prefix on LiteLLM matches the gateway's documented requirement. Comments warn to rotate in shared/staging/production. No real keys leaked.

## Errors to watch for in future slices

- **Hardcoded credentials in compose `environment:`** — always reject. Force `${VAR}` interpolation or `env_file:`.
- **`extra_hosts: host.docker.internal:host-gateway`** — breaks containerd. Reject; use service DNS instead.
- **Bind-mounts to paths outside repo workdir / `$HOME`** — Rancher Desktop will silently fail to mount. Prefer named volumes; single-file binds inside repo workdir are OK.
- **`apt-get install` without `--no-install-recommends`** — image bloat and supply-chain expansion. Reject.
- **Generic `Exception` capture without re-throw or context log** — production-quality rule; reject.
- **Tokens/refresh tokens in `localStorage`/`sessionStorage`** — non-negotiable. Force httpOnly cookies + BFF.
- **Source maps shipped in production builds** — reject.

## Auth slice patterns (P01-S02-T001, 2026-05-11)

- **Pydantic field_validator vs service-layer policy gate — 3 invariants**: When a business policy (e.g. `legal_acceptance must be True`, `email must be corporate`) is enforced via Pydantic `@field_validator` on the schema, three things break: (a) HTTP code becomes 422 not the task-pack-pinned 4xx (e.g. 400), (b) response envelope becomes FastAPI's `{detail:[...]}` instead of project `{data, meta, errors}`, (c) the service is never reached, so any audit-row-on-every-attempt invariant is silently broken. Policy gates belong in the **service layer** (raise typed AuthError → router maps to correct HTTP + envelope + audit fires in same flow). Pydantic should only validate STRUCTURAL shape (type, presence, length, regex on syntax, EmailStr). This pattern caught F2 in T001.
- **Domain check correctly placed in service layer**: T001 puts `CorporateEmail(email)` validation in `service.py:152` (NOT in Pydantic schema). Returns 400 with `AUTH_SIGNUP_NON_CORPORATE_EMAIL`, fires audit on rejection. This is the canonical pattern for all future auth slices.
- **Real Argon2 dummy hash for timing equalisation**: T001 calls real `hash_password(password_plain)` on the duplicate-email rejection path (service.py:208), NOT a no-op `time.sleep()`. This is the correct anti-enumeration pattern — actually consume the same CPU/memory cost as the success path. Validate this pattern in T002 sign-in too (failed-password path should also do the hash to equalise vs unknown-email path).
- **In-memory token-bucket rate limit at slice level**: When platform rate-limit infra is deferred to a future slice, implementing a per-IP in-memory token bucket INSIDE the slice's own module (`app/auth/rate_limit.py`) is acceptable when: (a) `TODO(P0X-S0Y-T0Z): replace with platform middleware` is in the module docstring, (b) it stays inside the declared `Write set` (not in `app/middleware/` or `app/core/`), (c) it's configurable via env vars, (d) DEFAULT IS ON (not "configurable but off"), (e) tests cover both pass and 429 paths. T001 hits all five. Pattern reusable for any auth endpoint until P02-S02-T001 lands.
- **WRITE_SET_DRIFT for `main.py` router mount**: API slices NECESSARILY drift into `app/main.py` to mount the new router. This is unavoidable scope drift. Approve when: (a) the diff is minimal (1 import + 1 include_router line), (b) it's flagged with rationale + precedent in handoff §Files, (c) the file docstring is updated to acknowledge the drift. T001 hit all three.
- **Researcher note partial application is a real defect**: When official-docs-researcher writes a note with `RESOLVED:` and a specific action ("correct docstring X to say Y"), the developer must apply the fix at EVERY location where the wrong claim appears, not just one. T001 F1: developer applied the correction to the `_ph` block but left two earlier sections (`Source refs` line + `Decisions` line) with the old wrong wording. Treat as `changes_requested` — small docstring fix.
- **Generic `except Exception` at sub-transaction boundary is acceptable**: `service.py:269 _write_rejection_audit` catches Exception to log+rollback the audit best-effort. The main flow already failed; making the audit write defensive prevents cascading failures. Acceptable boundary alongside the previously-noted CLI boundary and FastAPI top-level handler patterns.
- **309 lines on a router.py with single endpoint**: NOT blocking when (a) the file has ONE responsibility (sign-up presentation), (b) the overage is verbose docstrings + helpers, (c) the natural refactor target (`get_db_session` → `app/db/session.py`) is already foreseen by downstream slices (T002, T005 will reuse it). Recommend (`nit`) extraction in the next slice that touches the area. Apply validator MEMORY learning ("line cap is signal, not rule") consistently — but pair with a clear "what next slice owes us" note.
- **Audit-every-attempt invariant**: When the source-of-truth says audit fires on success AND rejection, validator must check every rejection branch in the flow. Pydantic-validation branches are SILENT bypasses of the service-layer audit fire. Always check: "if Pydantic rejects this case, does it bypass our audit?" — and if yes, push the gate to service. Pattern caught F2 in T001.

## Verification data loader patterns (P00-S02-T003, 2026-05-11)

- **Opción C runtime defer pattern**: `inspect(engine).has_table(t)` before any UPSERT, return `LoadResult(status="deferred", reason="table_missing:<table>")` + `log.warning` (always visible regardless of ENABLE_VERBOSE_LOGGING). This lets the bootstrap stay runnable in P00 while tables are still missing, and auto-activates when P01-S01-T001/P02-S01 create them. Validate this pattern in any "scaffold/bootstrap" slice where the consumer schema lands later.
- **Argon2 idempotency via verify-before-rehash**: Argon2 produces a different hash each call (random salt). The correct UPSERT pattern is:
  - INSERT branch → `ph.hash(plain)` always.
  - UPDATE branch → `ph.verify(stored, plain)` first; only re-hash if VerifyMismatchError or InvalidHashError. This avoids unnecessary DB writes and makes idempotency provable (AC1 row counts stable across runs).
  - Note: argon2-cffi 25.1.0 does NOT have `verify_and_update()`. Use `check_needs_rehash` separately if needed.
- **Test redaction quality bar**: AC9 test asserts on actual sensitive VALUES (`verifypass2024`, `jbswy3dpehpk3pxp`), not on field names. This catches leaks where the field name is sanitized but the value escapes via `str(fixture)` or a generic dict dump. Adopt this pattern for any logging-redaction validation.
- **Tests with importlib.reload**: When the module under test reads env vars at import time (`_VERBOSE = os.getenv(...)`), tests must `importlib.reload(bs_mod); importlib.reload(loader_mod)` after `patch.dict(os.environ, ...)`. Legitimate technique; do NOT confuse with mocking business logic.
- **File-size 300+ for fan-out scaffolding**: `loader.py` 617 lines and `bootstrap.py` 532 lines are 6 nearly-parallel group sections. Two takeaways: (a) approve with explicit caveat that the next slice activating real load logic MUST split into `loader/<group>.py`, (b) the i18n MEMORY learning ("lines are signal, not rule") applies when content is fan-out by group, but the bar is HIGHER for code with branching logic (UPSERT INSERT/UPDATE) than for pure static data. Don't generalize the i18n exemption blindly.
- **TOTP secret plain text in fixture is acceptable for sandbox**, gated by: (1) explicit human acknowledgment at `/verify-slice` (R3 marked CRITICAL in handoff), (2) README.md Security Notice with rotation guidance, (3) the secret is validated but NOT inserted in this slice (the `mfa_totp_secrets` table doesn't exist yet), (4) Fernet `encrypt_secret()` infrastructure is ready for P01-S01-T001 to use. This is a `human_gate_required` pattern, not a `block_now` pattern.
- **Generic `except Exception` at CLI boundary is acceptable**: the rule "never catch generic Exception" targets business-logic try/except, not CLI top-level main entry points that translate any failure into exit-code 3 + stderr message. Don't reject `except Exception` in `main()`, `verify_password()`, or `_get_fernet_key` where the boundary semantic is "any failure → typed user-facing translation".
- **Latent UPSERT bugs in deferred branches**: `loader.py:290` builds JSON via `str(d).replace("'", '"')` — fragile for apostrophes in metadata values. Not reachable in this slice (table missing). Pattern: flag as F5 latent in validator review, push fix to the next slice that activates the deferred branch. Do NOT block the scaffold slice for a defect that cannot run.

## Auth slice patterns — sign-in / second endpoint on the same router (P01-S02-T002, 2026-05-11)

- **The "next slice that touches" exemption is single-shot, NOT recurring**: T001's 309-line router was approved with "next slice will extract `_engine/_SessionLocal/get_db_session` to `app/db/session.py`". T002 DID extract those (good), but then added a second endpoint and grew router.py to 466 lines and service.py to 702 lines. The MEMORY learning "line cap is signal not rule" was conditional on "single responsibility". When a second use case lands, the router/service must split BEFORE the file accretes — not after. Pattern for future auth slices: when a slice adds a new endpoint to an existing module, the developer MUST split the existing file into one-endpoint-per-router-file (or one-use-case-per-service-file) AS PART OF the current slice, not defer it again. The validator must catch this on the second touch, not the third.
- **"One use case per file" is a flat non-negotiable rule, not a heuristic**: `01-non-negotiables.md §File size` explicitly says "1 component per file. 1 use case per file. 1 entity per file." Two `class XxxUser(...)` definitions in one `service.py` is a flat violation — no line-count interpretation needed. Flag unconditionally.
- **Function size discipline**: a ~270-line `execute()` method is the loudest possible signal of hidden sub-responsibilities. For an auth use case the natural split is `_validate_payload`, `_check_lockout`, `_verify_password_path`, `_issue_mfa_challenge`, `_issue_session_tokens`, `_maybe_rehash`. Per non-negotiable: max ~50 lines per function/method. Pair the file-size finding with a function-size finding when both fire on the same change.
- **Lazy imports inside a method are an architecture smell, not a fix**: developer described 6+ lazy imports inside `SignInUser.execute()` as "circular import mitigation". When the imports are all leaf modules (`os`, `hashlib`, `secrets`, `datetime`, `app.auth.password`, `app.auth.tokens`) there is no circular dep — the lazy pattern is hiding scope/file-size pressure. Push imports to module-level and let the file-size violation surface the real defect.
- **Private symbol imports (`from app.auth.password import _DUMMY_HASH`)** are a code smell that signals weak module boundaries. Demand a public name or a public helper. Same for `_SessionLocal` — if `db/session.py` exposes `get_db_session` as the public dependency, the router must NOT bypass it by importing the private factory and instantiating sessions manually.
- **Anti-enumeration acceptance is testable by byte-equality of response bodies**: T05 pattern (`assert w_body["errors"][0]["code"]/message/field == u_body[...]`) is the right shape for any aggregate-401 endpoint. Validator should EXPECT this test to exist on every login-style slice and reject if it doesn't — the response shape is the only thing the attacker sees, so the test must assert response-shape equality, not just code equality.
- **Lockout query must be gated by user-existence check**: `count_recent_signin_failures(user_id, window)` is the right surface, but if the lockout runs BEFORE `find_by_email != None` filtering, you've built an enumeration oracle (known users can be locked out, unknown users can't). Validator: trace the call site, confirm `if user is None` returns 401 before the lockout query fires. T002 §service.py:412-491 had it right; future slices must match.
- **Audit-row table from §F.9 is a checklist**: the task pack pinned a single `auth.sign_in` action with `metadata.outcome` ∈ {success, mfa_challenge_issued, failure, blocked} and `metadata.reason` ∈ {unknown_email, wrong_password, user_inactive, invalid_payload, account_locked, rate_limited}. Validator must trace EVERY branch in the service and confirm an audit row is written. T002 hits all 8 branches. Reuse this checklist for any future endpoint that has multiple rejection paths.
- **PyJWT 2.x + `samesite="lax"` (lowercase) is the canonical pattern for FastAPI**: encode returns `str` in 2.x, `options={"require":["exp","iat","sub","jti"]}` raises on missing claim, `samesite` param value is lowercase `"lax"` (not the HTTP-header `"Lax"`). Researcher confirmed. Lock these as expected patterns.
- **Test directly committed setup data when fixture is rollback-transactional**: when `pg_session` wraps each test in a rollback, but the endpoint under test opens its OWN connection via `_SessionLocal`, the endpoint cannot see uncommitted fixture data. Pattern: use a separate `_SetupSession` that commits, and clean up in an `autouse` fixture teardown via `_created_user_ids` list. This is the correct way to combine "real DB tests" with "rollback isolation for assertions" — do NOT mistake the committed setup connection for "tests aren't transactional".
- **Evidence quality matters but is not blocking**: a curl evidence file captured BEFORE the verification user was bootstrapped (showing 401 where 200 was documented) is an evidence-quality nit, NOT a code defect — the pytest run with `_create_user` already proves the 200 path works. Flag as a nit and ask the debugger to re-capture, don't escalate to blocked.

## Sign-in retest patterns — split package debugger fix (P01-S02-T002 cycle 1, 2026-05-11)

- **`services/` and `routers/` sub-package pattern is the canonical split** for second-endpoint-on-same-feature slices. Once two use cases land on the same feature, structure is `app/<feature>/services/sign_up.py` + `services/sign_in.py` with `services/__init__.py` re-exporting both, and `app/<feature>/routers/sign_up.py` + `routers/sign_in.py` with `routers/__init__.py` re-exporting both, plus `service.py` and `router.py` as 28-/29-LOC compat shims. The compat shim is important: it keeps `from app.auth.service import SignUpUser, SignInUser` working for any latent caller (and for the test file) without forcing a wider rename in the same slice. Validate this pattern on the next sign-in-equivalent slice.
- **`_ReqContext` dataclass for orchestrator-helper sharing**: when an orchestrator method must pass `(request_id, ip, user_agent)` (or any "thread-local-ish" tuple) to 8+ helpers, a frozen `@dataclass(frozen=True)` reduces the helper signatures from 5+ params to 2 and makes the ≤50 LOC cap reachable. The debugger's `services/sign_in.py:_ReqContext` is a textbook case. Reusable pattern for any service that has BEFORE/AFTER audit + multi-step orchestration.
- **Public helper `verify_with_dummy_fallback(stored_hash | None, plain) -> bool` is the right surface** for aggregate-401 anti-enumeration. It hides both the `DUMMY_VERIFY_HASH` constant AND the timing-equaliser branch, so callers cannot accidentally take an unsafe shortcut. The previous pattern (`if user is None: verify_password(_DUMMY_HASH, password)`) leaked the dummy hash by name AND let callers forget the equaliser path. The wrapped helper is the canonical aggregate-401 surface for any future auth-style endpoint.
- **`Depends(get_db_session)` and NOT manual `_SessionLocal()`** is the only acceptable router DB access pattern in FastAPI. The earlier T002 router had `session = _SessionLocal()` + `try/finally session.close()` blocks despite `get_db_session` being available — this duplicated the generator's responsibility. After the cycle-1 fix, both `routers/sign_up.py` and `routers/sign_in.py` use `Depends(get_db_session)`. The router no longer needs to import `_SessionLocal` at all. Reject any future router that manually instantiates `_SessionLocal`.
- **Test file LOC cap exemption**: `test_auth_signin.py` is 801 LOC — well above the 300 cap. The 01-non-negotiables §File size rule targets source code (entities, use cases, repositories, components); a single per-endpoint test file with 16 integration tests + bootstrap fixtures naturally exceeds 300. Accept without flagging when (a) the file has ONE responsibility (one endpoint family), (b) the bulk is pytest functions with per-test fixtures, (c) splitting would just shuffle imports across files. Do NOT generalize this exemption to test helpers or test infrastructure modules — those should still split.
- **In-scope structural refactor is a debugger fix, not a FU**: the cycle-1 work (split service.py + router.py + extract helpers + promote private symbol + tighten test + re-capture evidence) stayed entirely inside the canonical Write set declared by the task pack §I (`backend/app/auth/`, `backend/tests/integration/test_auth_signin.py`, `backend/pyproject.toml`, `backend/requirements.txt`, `.env.example`) — no source-of-truth amendment, no new endpoint/route/table/journey, no `Conflict group` change. Classification `in_scope_defect` is correct; the debugger pass closes all eight findings in one cycle. No FU registered. Reinforces validator MEMORY learning: in-scope refactor is debuggable as long as the resulting tree stays inside the declared Write set.
- **Byte-equality 401 verification with JSON normalisation**: `diff <(jq -S . body1) <(jq -S . body2)` (or the python equivalent) modulo `request_id` is the canonical equality test. Required output: exit 0. The cycle-1 evidence has `request_id` as the only field that differs between unknown-email and wrong-password — code, message, field, details, data, errors-shape are byte-identical. Pattern applies to any aggregate-error endpoint.
