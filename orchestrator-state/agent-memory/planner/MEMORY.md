# Planner — manual memory

## Patterns learned

### Scaffold slices (P00-S01-T001 family)

- For "Repo scaffold + scripts + env" tasks, the Write set declares categories ("package manifests"), not paths. Translate each category into a concrete Allowed Paths list in the task pack so developer cannot drift.
- `backend/app/main.py` "compiles" acceptance is best read as: valid Python syntax + matches the API contract Health spec. Imports may not resolve until the deps slice (T003) lands. Document this explicitly to prevent developer from preinstalling deps.
- Always include an "Out-of-scope" table mapping each adjacent concern to its owner slice. The Coverage Registry is right there in the same Step, so reviewers infer scope greedily — fight that with an explicit table.
- `verify_mode=auto` + low risk + non-journey ⇒ pack must mark `Verification data contract: n/a` to prevent tester/verify from chasing fixtures that don't exist yet.

### Task-packing

- Cite §s of source-of-truth, never copy-paste large blocks. Keep the pack scannable; the developer reads PROGRESS.md + this pack only on a daily basis.
- The five source-of-truth files are present: `instrucciones.md`, `*_IMPLEMENTATION_CHECKLIST.md`, `*_TECHNICAL_GUIDE.md`, `STACK_PROFILE.yaml`, `UX_CONTRACT.md`. Confirm all five exist before writing the extract; missing one ⇒ `CONTEXT_READY: no`.

## Modules that tend to break together

- `backend/app/main.py` ↔ `backend/app/api/router.py` ↔ `backend/app/core/config.py` — split across T001 (stub), T003 (config), P00-S02-T002 (router). Keep each refactor strictly inside its slice's Write set.
- `frontend/package.json` ↔ `frontend/src/app/providers.tsx` ↔ `frontend/src/i18n/index.ts` — three different slices (T001 manifest stub, T002 deps + provider, T005 i18n).

### Dependency-pack slices (P00-S01-T002 family)

- When the registry write set lists **only** the manifest + lockfile + one source file, but Acceptance + Verify command imply tests, the planner MUST amend the write set explicitly in the pack (NEW: `*.test.tsx`, `vitest.config.ts`). Validator gets the heads-up upfront; otherwise it flags drift.
- Frontend test environment: Vitest 4 default env is `node`. Any React DOM mount needs `vitest.config.ts` with `test.environment = 'jsdom'` AND `jsdom` in devDependencies. Without this, smoke tests fail with `document is not defined`.
- `@testing-library/react` is implied by `instrucciones.md §11.0` + `TECHNICAL_GUIDE §2.1` for "tests de componentes" — declare its addition explicitly in the dependency-pack pack so it's not "silent expansion".
- For provider wiring slices, ALWAYS specify the exact export name + signature in the pack. Downstream slices import it; lock it now.
- For library installs against pinned framework majors (React 19, Vite 8, TS 6), the `npm install` step itself is the fail gate. Researcher must report peerDeps live; `--legacy-peer-deps` is forbidden.
- "First provider wired" intentionally excludes Router. Router lands with auth (P01-S03-T001) because `BrowserRouter` is coupled to redirect-after-login behavior. The pack must explicitly say "do NOT add BrowserRouter" or developer will pre-empt P01-S03.
- T001 closer left frontend stack pinned to React 19 / TS 6 / Vite 8 / Vitest 4. T002 inherits these — pack must call them out as "do not regress" so dependency conflict resolution doesn't downgrade them.

### i18n-bundle slices (P00-S01-T005 family)

- For "i18n resources X locales × N namespaces" tasks, the registry write_set declares only `frontend/src/i18n/**` and `frontend/public/locales/**` but the slice ALWAYS needs a surgical edit on `frontend/src/app/providers.tsx` to migrate the inline `i18n.init({...})` (left as a stub by the dependency-pack slice) into the new `i18n/index.ts` module. Document this as `write_set_extension` in the pack with justification — same precedent as T004 Path A. Validator must treat as allowed scope, not drift.
- `instrucciones.md §6` ships productive ES/EN/FR copy as a key×locale table. Treat it as authoritative key inventory: ship verbatim, do not invent extra keys. For namespaces with NO productive copy in §6 (e.g. `mcp`), seed exactly one minimal canonical key per locale that mirrors nearby patterns — empty bundles `{}` would vacuously satisfy a drift test (violating "tests are REAL").
- `journey_refs` in the registry row can mean two different things: "this slice is consumed by these journeys" (infra) OR "this slice closes these journeys" (productive screens). Always cross-check with `python3 -B -S .claude/bin/list_journey_closures.py <TASK_ID> --json`. For infra slices that just provide assets, expect `closing_journeys: []` and record it explicitly so closer does not emit `JOURNEY_PENDING_VERIFY`.
- `tables: <X>` in the registry row of an FE-infra slice is registry inheritance bookkeeping (the journey row that consumed the slice mentioned the table). It does NOT mean the slice touches DB. State this as Out-of-scope explicitly so developer doesn't preempt a migration.
- Loading strategy default for i18next bundles: EAGER imports, not http-backend. Reasons: (a) bundles are tiny; (b) Vitest+jsdom needs no fetch mock; (c) `Suspense` not required; (d) future migration to lazy is trivial because JSON canonicals already live under `public/locales/`. Document this and pin the dep choice — http-backend is intentionally NOT in T002's package.json.
- Browser language detector (`i18next-browser-languagedetector`): defer registration to the language-preference slice (account/language switcher), even though the dep is already installed. The default `lng:'es'` matches DB DEFAULT and DEFAULT_LANGUAGE env. Detector adds UX surface that doesn't belong to an infra slice.
- Drift detector test pattern (real value-add of the slice): recursively flatten each bundle to dotted keys, compare set-equality across `es`/`en`/`fr` per namespace. This is the gate that catches "added a key in ES, forgot EN/FR" in future PRs.
- The verify command `npm --prefix frontend run test -- --run -t i18n` filters by test-name regex, not file path. The describe-block name MUST contain `i18n` (e.g., `describe('i18n bundle structure', ...)`) for the filter to match. Pack must say this explicitly.
- The previous slice (T002) may leave a stale doc-citation in `providers.tsx` (e.g. `instrucciones.md §3.3` pointing at the i18n config when the truth is `§6` + `§1.4`). Surface in the Discrepancies table so the developer fixes it as part of the surgical providers.tsx edit; do not let it ship to T005's commit unfixed.

When the verify command requires a build, the pack MUST record a `write_set_extension` block listing the bootstrap-completion paths (tsconfig, tsconfig.node, vite.config, index.html, main.tsx, the showcase route page itself). Justify with PROGRESS.md "Known Issues" + previous-handoff explicit deferrals. This is "Path A" in the planner pack template.

Path B remains valid post-close as a `/register-followup` to update the registry write_set for reproducibility.
- The `/showcase` route is NOT in UX_CONTRACT screen inventory by design — it is a dev-only design-system surface, not a productive screen. Document `Required UI states: n/a` per state with reasons; otherwise validator/tester chase missing loading/empty/error/permission/success states that don't apply.
- For pure presentational components (Wordmark, TrackedLabel, StatusDot, HairlineTable, MobileFrame, AdminShell), `BEFORE/AFTER` logging applies once they gain a real action handler in a downstream slice — same precedent as T002 `providers.tsx`. Document in the component docstring + handoff "Logging exemption".

Record in the pack `## Discrepancies` table.
- `design_tokens_v1` excludes `tokens|theme|themes|design-system` directory names by default; `frontend.theme_root=frontend/src/shared/styles` is auto-excluded. Sanity-run before adding hex literals so the developer doesn't waste time chasing false negatives.

### Architectural follow-up slices promoted from verify-slice CWE findings (P00-S02-T004 family)

- When verify-slice surfaces a CWE-class issue (CWE-532 leak, CWE-79 XSS surface, etc.) that has a TWO-layer fix — (a) close the immediate leak inside the originating slice's write_set; (b) a global/architectural fix in a different module — the right shape is: in-scope debugger drop at the call site for (a), HIGH-severity follow-up promoted to a new task for (b). Severity must be `high` so it blocks the next wave/claim until promoted/waived. T004 = the (b) for T002's CWE-532. Pack must explicitly cite both layers and tell developer NOT to revisit the call site (it's already fixed and out-of-scope here) — defense in depth means both layers stay intact.
- For `infra:logging` slices, `journey_refs: []` + `verify_mode: human` + `risk_level: high` is the correct shape. Verify is HUMAN even without a journey because the failure mode (secret leak) is invisible to deterministic auto-checks unless we instrument them — and we did, but human read of the captured log is the gold standard. Mark `Verification data contract: n/a` (no journey, no real users) and document the synthetic edge-case fixture inline; cite `orchestrator-contract.json::ux_contract.verification_data_contract.rules` for the synthetic-allowed clause.
- For structlog-related slices, ALWAYS include "researcher must validate constructor signature for `<version>`" in the pack, even when the kwarg is stable — structlog renames helpers across 24.x → 25.x and Rich's traceback formatter has minor breaks across rich 13.x. Five-second cache hit is fine; the alternative is silent test failures that look like "logic bugs".
- Acceptance counts copied from a follow-up YAML may be stale (registry says "8 health tests" when reality is "9 health tests" because a regression guard was added during verify-slice). Cross-check with `wc -l` / `pytest --collect-only` and surface the discrepancy as an "informational" note in the pack so closer report has the correct numbers, not the registry copy.
- Logging changes ALWAYS need to be tested under BOTH ENABLE_VERBOSE_LOGGING values (true → ConsoleRenderer / Rich path; false → JSONRenderer path). One test per mode minimum; the leak vector is mode-specific (Rich's frame locals don't apply in the JSON path today, but a future renderer swap could regress that — defensive testing is cheap insurance).
- Follow-up slices must include a "files NOT to touch" subsection because the in-scope debugger fix from the originating slice (router.py, main.py for T002→T004) is tempting to "improve" but doing so silently widens the write_set and the PreToolUse hook will block. State explicitly: "T002 already fixed (a); this slice ships (b)".

### Health/ops slices (P00-S02-T002 family)

- TECHNICAL_GUIDE §6.2 may declare envelope shape `{data:{...}}` for health endpoints, but a previous slice (T003 in this project) may have shipped flat shape `{status, version, uptime}` because the upstream compose healthcheck or k8s probe expects flat. **Precedent wins**, but the pack MUST surface the discrepancy explicitly in a Discrepancies table and recommend (not register) a follow-up to reconcile docs. Do NOT silently migrate to envelope inside the slice — that breaks downstream healthchecks atomically.
- For multi-probe `/ready` endpoints where some dependencies (Redis, LiteLLM) don't have wired clients yet, return them with explicit `not_implemented` value, not a green "ok" lie and not omitted. This is forward-compatible: downstream slices flip the value to "ok"/"fail" without changing the response shape.
- `app.core.db.get_engine()` is **lazy** (T003 design): builds engine on first call, no import-time side effect. Safe to call from a /ready handler. Tests must use DI (monkeypatch the module global or factory) to simulate DB-down without leaking a half-built singleton.
- request_id middleware is naturally a P00-S02-T002 deliverable per the structlog docstring TODO at `app.core.logging.py:17`. Inline 4–6 lines in `main.py` is preferable over a new module — keeps the work-item write_set tight (3 paths).
- For backend slices with no journey_refs and verify_mode=auto, the Verification Data Contract row is **n/a** — explicitly say so. Tester/verify will otherwise chase fixtures that don't apply.
- `__init__.py` package markers added to satisfy a NEW package directory (`backend/app/api/__init__.py`) are a Python language requirement, not scope expansion. Document them in the pack as a write_set technicality so validator doesn't flag drift.
- Curl verification on `localhost:8000/health` may hit either the compose container (`version: 0.1.0`) or a local dev uvicorn (`version: 0.0.0`) when both are running. Always capture the `version` field in evidence to disambiguate; in-process `httpx.AsyncClient` tests are immune to this shadow.

### Productive-bundle replacement slices (P00-S02-T005 family — synthetic→productive seed swap)

- The slice that swaps a synthetic seed bundle for a productive vendor delivery is rarely "just JSON replacement". Two invariants force a wider scope: (a) the loader's SQL was written against a synthetic schema that pre-dated the real migration (T003 inserted `role, is_active, mfa_enabled` against the §10.3 `status, password_hash, preferred_language` shape); (b) productive credentials must NEVER live in committed JSON, so schemas need a `*_env` resolution layer + bundle_type-aware validators. Pack must declare both as in-scope with `write_set_extension` (precedent: T004 Path A, T005 i18n providers.tsx surgical edit, T003 data/verification bookkeeping).
- Bundle_type propagation pattern: introduce `BundleType = Literal["synthetic","productive"]` in `loader/_common.py`, read from `MANIFEST._bundle_type`, propagate via pydantic `model_validator(mode="before")` context. If a validator sees `bundle_type=None`, REJECT — prevents a future developer from silently bypassing the productive guard. This is R7 in the impact analysis.
- `synthetic-` prefix guard lifecycle: do NOT lift it. The productive variant just *bypasses* it (real keys live in `*_env` references). The guard remains active for the synthetic codepath to catch any future contributor who pastes a real key into a synthetic bundle by accident. Defense-in-depth — same shape as T004's two-layer fix (close-the-leak + global-guard).
- `_REAL_KEY_PATTERNS` regex (sk-..., AIza..., sk-ant-..., Bearer ...) stays in `schemas/admin_ai.py`. In productive bundle, the validator REJECTS plaintext `api_key` matching these patterns even if `_bundle_type=productive` — productive bundles MUST use `api_key_env`. This is the inverse of the synthetic guard: synthetic forces `synthetic-` prefix, productive forbids real-key plaintext.
- Plain backup codes / TOTP secrets / passwords in committed JSON: backup codes get argon2-hashed BEFORE persistence (the loader does NOT receive plaintext from JSON). TOTP secret CAN live plaintext in JSON (it is the only way the user's authenticator app receives it) but must be Fernet-encrypted before DB write, AND documented as "rotated post-slice closure" in the JSON `_comment`. Passwords plaintext live in JSON only as `password_plain_for_seed` (loader argon2-hashes them). Three different security postures, three different mechanisms — pack must enumerate them so validator does not conflate.
- `.env.local` is NOT committed (`.gitignore` line `.env.*` + `!.env.example`); developer uses it for productive secrets at seed time. Verify with `git check-ignore -v .env.local` and capture the output as evidence.
- Productive seed slices that swap data may reveal that downstream features (admin AI dynamic model discovery, deepagents Supervisor runtime) are not yet implemented. The right shape is to (a) declare the contract in the seed (so future bootstrap re-runs do not lose it), (b) open formal follow-ups for the runtime layers, (c) recalibrate Acceptance #3 from "J100..J105 verified end-to-end" → "schema-valid + seed-loadable; per-journey UI verification deferred". Closer applies this to the source-of-truth checklist as part of the slice's amendments.
- ADR section in TECHNICAL_GUIDE §15 may be empty (`(sin ADRs específicos todavía)`); closer is a legitimate place to seed it append-only with ADR-001 + ADR-002 dated when non-obvious decisions land (e.g., LiteLLM dynamic discovery vs hardcoded, deepagents Supervisor vs monolithic).
- Schema reshape MAY be in-scope for a "data" slice: T005 reshapes `AiModelSeed` (adds `name, capability, auto_discovered`, renames `enabled→is_active`) because the synthetic shape is too thin for the productive contract. Document the reshape in the pack §"Schema additions" with explicit "fixture format breaking change; back-compat preserved only for the synthetic guard, not for fixture shape" — validator otherwise flags as scope drift.

### Promoted-followup wiring slices (P01-S01-T004 family — config + env infra)

- When a wiring slice is promoted from a `medium` follow-up that fixes TWO independent root causes (port mismatch + cwd-relative env_file), pack must keep them separate in §"Impact analysis" so the developer can verify each independently. Mixing them in one paragraph hides the second when the first looks fine.
- pydantic-settings `env_file` is **cwd-relative by default**. The idiomatic fix in 2.14.x is `env_file=str(Path(__file__).resolve().parents[N] / ".env")`, NOT `model_config = SettingsConfigDict(env_file=Path(...))` — both work, but `str()` is the most-stable surface across pydantic-settings minor releases. Document the `parents[N]` anchor with a comment noting the directory depth (e.g. `[3] = backend/app/core/config.py → project root`).
- For compose-host port mismatches, ALWAYS state both modes in the pack: Mode A (native dev → host port) vs Mode B (in-compose → service-name port). The compose file's `environment:` block already overrides Mode B, so `.env.example` only needs to ship Mode A as the uncommented value + Mode B as a comment. Documenting both prevents next-slice developers from "fixing" the example back to compose-style and breaking native dev again.
- Journey closure check is mandatory even for "obviously infra" wiring slices. Run `list_journey_closures.py <TASK_ID> --json` and write `closing_journeys: []` explicitly — the closer hook needs this signal to suppress `JOURNEY_PENDING_VERIFY`.
- For test-flipping slices (a previously-failing test must turn green WITHOUT touching the test), pack must cite the test file + line range + skip-mark conditions verbatim. The developer must verify the skipif precondition is satisfied (e.g. `_postgres_reachable()` returns True) BEFORE claiming acceptance #5; otherwise the test stays SKIPPED, not PASSED, and the acceptance is technically unmet.
- Write set may legitimately have ZERO diff for some entries (e.g. `dev-restart.profile.sh` may need no edit if config.py absorbs the fix). Pack should pre-authorize this by listing each write_set entry with a "edit / review-only / no-edit" recommendation; validator then accepts an empty diff for "review-only" entries provided the handoff documents the decision. Same precedent as T002 deferring startup validators.

### Seed/verification-bundle slices (P00-S02-T003 family)

- The registry write_set for a seed slice often lists only the loader module (`backend/app/seeds/**`) + the reset script (`scripts/dev-restart.sh`), but the materialized contract is the data directory (`data/verification/**`). Document this as a `write_set_extension` in the pack with the §15 wording as justification. Same precedent class as T004 Path A and T005 providers.tsx surgical edit.
- `scripts/dev-restart.sh --reset` is **already wired** in the dispatcher and profile from P00-S02-T001 — the profile's `db_reset()` already invokes `python -m app.seeds.bootstrap_verification_data` if importable, with a graceful skip when alembic.ini is missing. T003 should NOT rewrite the dispatcher; verify the existing wiring works, edit surgically only if a real defect surfaces. The registry listing the script in write_set is a bookkeeping convention (the loader and reset cycle co-evolve), not a "must edit" command.
- For seed slices that predate the first migration (P00-S02-T003 ships before P01-S01-T001), the loader MUST be tolerant of "table does not exist" — not a free pass: JSON/Pydantic validation still runs first; only the DB write step is skipped with a structured WARN. Exit-code matrix: `--source` missing ⇒ 2; required fixture invalid ⇒ 1; namespace missing required files ⇒ 1; everything else with possibly-skipped writes ⇒ 0. State this matrix in the pack so the tester can write deterministic assertions.
- Synthetic-vs-prod fixtures policy: when the user-approved scope is "scaffolding-only" pending vendor-delivered data, build a defense-in-depth `synthetic-` credential prefix guard inside the loader's Pydantic schemas — reject any value that does NOT start with `synthetic-`. This catches the failure mode where a future contributor pastes a real key into the synthetic JSON. The follow-up to lift/relocate the guard is registered explicitly when the real bundle is delivered, so the guard's lifecycle is controlled by the same artifact that lands the real data.
- "Prod-like" in registry Acceptance has two readings: **prod-shaped** (field structure, idempotency, exit-code surface — testable now) vs **prod-delivered** (provenance — only when vendor ships). Pack must be explicit about which reading the slice satisfies; otherwise validator will conflate them and either fail-stop on T003 (wrong) or under-validate the future delivery (also wrong).
- Journey scope mismatch: a seed slice is *consumed by* every journey (J100..J105) but closes none of them. Run `list_journey_closures.py <TASK_ID> --json` and record `closing_journeys: []` explicitly so closer does not emit `JOURNEY_PENDING_VERIFY`. The registry's `Journey refs: J100,...,J105` is breadth-of-consumption, not breadth-of-closure.
- Verification Data Contract for the producer slice itself: the slice IS the contract's producer, not a downstream consumer. So the pack lists §6.5 as the primary specification (`--only` namespaces + reset commands), not as a verification gate to satisfy. Validator should not chase fixtures "to verify against §6.5" — the slice's deliverable IS those fixtures.
- TOTP seed values: `pyotp.random_base32()` per fixture; mark `synthetic_totp: true` in JSON. Do not pre-generate a "stable" TOTP because (a) it ages out the moment time-skew tests appear; (b) a stable secret in a public repo defeats the synthetic prefix policy.
- Encryption-at-rest deferral: `ai_provider_credentials.encrypted_secret` is a §10.3 column that lands in P02-S01-T001 and the encryption service in P02-S02-T001. The seed loader writes plaintext synthetic credentials in T003; document this in the loader's `admin_ai` docstring with the explicit deferral pointer. Do NOT register a formal follow-up YAML in T003 — P02-S02-T001 will trigger it organically. Registering early makes the medium-severity FU-list noisy.

### First-migration / DB-baseline slices (P01-S01-T001 family)

- **Alembic bootstrap is implicit**: the registry write_set lists only the version file (`backend/alembic/versions/0001_*.py`) + SQLAlchemy model files, but `backend/alembic/` is empty (`.gitkeep` only) and `alembic.ini` does not exist. The first migration slice OWNS the bootstrap (`alembic.ini`, `env.py`, `script.py.mako`, `db/__init__.py`, `db/models/__init__.py`, `db/models/base.py`). Path A `write_set_extension` precedent (T004 tsconfig + T005 providers.tsx surgical edit) — declare in the pack with explicit justification so validator does not flag drift.
- **Base model file split**: when registry says `models/user.py` + `models/auth.py`, you still need a third file `models/base.py` with `Base = DeclarativeBase()` + naming convention so both files share the same metadata. Do NOT inline Base into one of them (couples the files arbitrarily and breaks the ≤200 line rule when more models land). Surface as a write_set technicality in the pack.
- **§10.3 vs `01-non-negotiables.md` audit_log drift**: §10.3 ships `audit_logs(actor_user_id, action, entity_type, entity_id, metadata, created_at)`; non-negotiable mandates `(user_id, action, resource, timestamp, ip, user_agent, metadata, request_id)`. **Source-of-truth wins for the migration shape** because (a) editing source-of-truth during an active slice is forbidden and (b) the `ip/user_agent/request_id` enrichment depends on the request_id middleware (T002, done) plus a security middleware not yet built. Emit as a `medium` follow-up after closer with explicit "blocks before P02-S02-T001" — non-destructive `ADD COLUMN` later is cheaper than fighting the closer hook now.
- **Journey refs in DB-baseline rows are inheritance bookkeeping, not closures**: P01-S01-T001 row says `J100,J102` because J100/J102 consume those tables. `list_journey_closures.py <TASK_ID> --json` returns `closing_journeys: []`. Same precedent as P00-S02-T003 seed slice. Pack must explicitly say "closer must NOT emit JOURNEY_PENDING_VERIFY" so the closer hook does not pollute `pending_journey_verifications[]`.
- **Index-coverage opinion on the §10.3 baseline**: §10.3's DDL block declares unique constraints (auto-indexed by PG) but NOT explicit indexes for: (1) `user_roles.role_id` (reverse FK lookup); (2) `refresh_tokens.user_id` + `(expires_at WHERE revoked_at IS NULL)` (per-user enumeration + janitor); (3) `password_reset_tokens.user_id` + `expires_at`; (4) `audit_logs.(actor_user_id, created_at DESC)` + `audit_logs.created_at DESC` + `audit_logs.(entity_type, entity_id)`. Non-negotiable §Database mandates "Indexes on every field used in WHERE/ORDER BY/JOIN" — pack must enumerate the explicit `op.create_index(...)` list so developer doesn't ship a half-indexed schema.
- **Both `pgcrypto` AND `vector` extensions belong in the FIRST migration**, even though pgvector is consumed by P02-S01-T001 (later). Reason: the §10.3 baseline block declares both `CREATE EXTENSION IF NOT EXISTS` lines together; keeping them atomic in 0001 makes the foundation reversible (downgrade drops nothing the schema doesn't own). pg18-bookworm image already provides both — `IF NOT EXISTS` is idempotent.
- **Naming convention on `Base.metadata` MUST be set BEFORE the first model is declared**, otherwise autogen produces random index/constraint names. The first migration locks this in for the rest of the project's lifetime.
- **Alembic env.py reuses `app.core.db.get_engine()`** (lazy-singleton from P00-S01-T003), NOT `engine_from_config(alembic.ini)`. This keeps the DSN in Pydantic settings (SecretStr), not in plaintext `alembic.ini`. Leave `sqlalchemy.url = ` blank in the ini.
- **Pre-existing `test_ready_db_ok` infra fail** (PROGRESS.md known issue) is auth-credential drift in dev env, NOT a schema problem. Pack must say "do not fix as part of T001" explicitly — env-var follow-up territory (FU-20260508225027). Otherwise developer will widen the slice trying to make tests "fully green".
