# Planner — manual memory

## Patterns learned

### Schema-only follow-ups that close a §10.3-vs-non-negotiable column drift (P01-S01-T005 family — audit_logs ip/user_agent/request_id/resource)

- **D1-style drift slices have a built-in trap**: the registry Acceptance is written from the *full* perspective ("schema + ORM + service population + integration test via auth endpoint"), but at promotion time the consumer endpoint(s) don't exist yet. The follow-up YAML's `notes[]` will warn ("not before P01-S02 since middleware lands in P02"), but the planner must still defuse the trap: write a §"Waiver acotado" pre-approved by the human, splitting the acceptance into "what T005 delivers now (schema)" + "what defers to P01-S02-T001 (population) with explicit commitment". Without this, validator flags the population gap as scope drift and tester can't write the "via auth endpoint" assertion. Pack must enumerate the deferred clauses verbatim from the registry Acceptance.
- **Migration immutability is a hard constraint**: the predecessor migration (here 0001, committed 99d555b) MUST NOT be edited. The pack §R must spell this out and confirm via `git log <file>` in the impact analysis. The new revision is `down_revision = "0001"` and is purely additive (`add_column` with `nullable=True`, no FK, no server_default). Future planners: this pattern will recur for every column the non-negotiables mandate that §10.3 forgot — log them as "ADD COLUMN nullable" follow-ups, never as schema rewrites.
- **`resource` (REST-style path) vs `entity_type` (DB table name) are SEMANTICALLY DISTINCT**: do not collapse them. `01-non-negotiables.md §Audit log` mandates `resource` as a separate field meaning the HTTP target (`"POST /api/v1/auth/sign-up"`); §10.3 already has `entity_type` for the DB-domain target (`"user"`). T005 adds `resource` as a NEW column alongside the existing `entity_type` (NOT a rename). Pack must explain this dual-axis representation to pre-empt validator questioning.
- **INET as built-in PG type**: `sqlalchemy.dialects.postgresql.INET` exists since SA 1.4; no extension needed (pgcrypto + vector already present from 0001). Type the ORM column as `Mapped[str | None]`, NOT `Mapped[ipaddress.IPv4Address | None]` — asyncpg returns INET as scalar `str` and the conversion adds no value at the schema layer. Population layer (P02-S02-T001 / P01-S02-T001) may revisit if a service needs strong typing.
- **Existing audit_log writers must keep working unchanged after the slice**: `backend/app/features/admin_ai/repository.py:insert_admin_ai_audit_log()` writes audit_logs today and stuffs `request_id` into `metadata` JSONB with a stale comment "P01-S01-T005 adds the columns". Do NOT touch it (write_set is closed). The stale comment becomes informational drift but is OUT OF SCOPE for this slice. Pack §R must call this out so developer doesn't widen the write_set; closer can queue a maintenance follow-up if drift becomes load-bearing.
- **Existing tests touching audit_logs (regression surface)**: tabulate them in the pack so tester knows what MUST stay green. Here: `test_auth_migration.py` (audit_logs assertions on §10.3 shape, FK SET NULL, indexes, metadata JSONB) + `test_admin_ai_discover_models.py:test_audit_log_written` (inserts AuditLog via session.add() with §10.3 fields only). Both survive because new columns are nullable. If either test had `__table_args__` or strict-shape introspection, the pack would need to flag it as breaking-change risk.
- **Integration test design without consumer endpoint**: the "via auth endpoint" clause is deferred, but the schema contract still needs proof. Write 4 tests directly via SQLAlchemy session: (1) information_schema assertions on column presence + nullable=YES; (2) round-trip upgrade→downgrade→upgrade preserves §10.3 rows and drops/recreates new cols; (3) INSERT with all 4 new cols populated → SELECT verifies non-null persistence; (4) INSERT with only §10.3 fields → SELECT verifies new cols default NULL (back-compat guard against future "tighten to NOT NULL" mistakes). Test 4 is the load-bearing guard for the existing admin_ai writer.
- **`task_id_to_index` in registry.json is NOT a spawn budget count**: registry.json line 5993 may show `"P01-S01-T005": 28` — that's the topological index for DAG scheduling, not a spawn count. Real spawn budget lives in `runtime-state.spawns_in_current_slice` (a dict keyed by task_id with integer count). Confused these once → wasted a planner cycle thinking the slice was at budget. Future planners: always read `runtime-state.json`, NOT `registry.json`, for spawn count.
- **Multi-terminal active_task_id**: `runtime-state.active_task_id` may belong to ANOTHER terminal (here it shows `P01-S02-T001`, but this terminal is `P01-S01-T005`). In DAG mode, `CLAUDE_ACTIVE_TASK_ID` env + per-task pack at `task-packs/<TASK_ID>.md` are authoritative. The legacy `orchestrator-state/memory/active-task.md` and singleton `runtime-state.active_task_id` are advisory only — never use them to override the env-scoped task. Pack must say "this terminal owns T005; T001 belongs to a sibling terminal".
- **Closing journey check on a no-journey slice**: `journey_refs: []` in registry → `closing_journeys: []` deterministically → closer must NOT emit `JOURNEY_PENDING_VERIFY` and `/verify-slice §5.bis` does not apply. Pack §"Journey scope" must say so explicitly so closer doesn't pollute `pending_journey_verifications[]`.

### Bug-fix slices promoted from runtime follow-ups (P00-S02-T010 family — loader-vs-schema drift)

- The follow-up TITLE captures the surface-level symptom the FU author tripped over (`api_key column does not exist`). Real-world execution shows the bug is almost always **wider** than the title because seed loaders pre-date their consumer migration by 1-2 phases (T005 loader → T006 migration). Planner MUST run a **full column-by-column audit** of every INSERT against the real `§10.3` DDL + the most recent migration revision file, and tabulate the result in §6 of the pack. Without the audit, developer fixes only the symptom column and the next CLI run fails on the next phantom column. Same anti-pattern as the auth loader pre-T001 known fail mode (planner-memory §98).
- Idempotency clauses (`ON CONFLICT (X)`) reference columns that may not have a UNIQUE constraint. Verify the constraint exists in the migration before trusting the loader's claim of idempotency. Migration `0003` indexed only `ai_models (provider_id, model_id)` UNIQUE; the loader's `ON CONFLICT (name)` was a phantom on both `ai_providers` (no constraint) and `ai_models` (no `name` column). Two failures stacked.
- For DAG-promoted FU slices that touch a write_set listed as `backend/app/seeds/loader.py` when the real path is a package `backend/app/seeds/loader/<ns>.py`, treat it as **path-of-record imprecision** (not scope drift). Document in pack §6.1 + handoff D-row so closer queues a future doc-amendment slice. Do NOT edit the registry mid-slice (forbidden per `00-source-of-truth.md`).
- Reuse cipher helpers (`app.core.security.encrypt_secret/decrypt_secret`) in seed loaders — never duplicate Fernet logic. The endpoint that consumes the credential calls `decrypt_secret(cred.encrypted_secret)`; if the seed loader uses a different code path, the round-trip silently breaks at runtime when keys differ. One helper = one master-key resolution policy = T011's env fix lands once and propagates everywhere.
- FU-promoted slices for `admin_ai_endpoints` conflict group routinely have NO active sibling at claim time (T009 done, T011 planned but only depends on env, T007 frontend-only). Document this explicitly in the pack §R2 so validator does not chase phantom concurrency risks.
- The dev `.env` legacy `PROVIDER_ENCRYPTION_KEY=dev-encryption-key-placeholder` is NOT a valid Fernet key. The seed loader's `Fernet(key_bytes)` raises `ValueError` (config error, not code bug). T011 owns the env fix; T010 must not widen its write_set to `.env` / `setup-from-scratch.sh` (those are T011's). Make the operator export a Fernet key in `.env.local` for the verify-slice run. Pre-empt validator question in pack §9 R1.
- For seed-bug slices, the existing integration test (`test_seed_all_namespaces_table_tolerant_with_real_db`) often passes with the bug present because it asserts only on `report.namespace` and types — NOT on `rows_inserted > 0` or DB row counts. Treat as a "guard against regression" not "coverage proof". Pack §11 must add a NEW dedicated test file mirroring `test_seed_loader_after_migration.py` patterns (savepoint isolation, alembic-head autouse fixture, `@SKIP_IF_NO_DB`, `@skipif(not env_var)`). Without it the bug regresses silently.
- For `auth_type` mapping in `ai_provider_credentials`, derive from `provider_type`: `'litellm'` → `'master_key'`; `'gemini'`/`'openai'` → `'api_key'`. Document in pack so developer does not invent a new column or hardcode `'api_key'` for everything.
- Do NOT register a follow-up for `model_type` enum extensions (e.g. `reranker`) when no concrete row of that kind exists in the bundle yet. Defense-in-depth in code (`reranker → 'unknown' + warn`) is enough. Registering early = noise (planner-memory §127).

### First feature module slices (P00-S02-T007 family — feature-based UI MVP consuming an existing endpoint)

- "Acceptance" copied verbatim from a runtime follow-up YAML can encode unrealizable verbs ("approve/reject before persisting") because the FU author optimized for capturing intent, not implementation reality. The planner's job is to translate the intent into the achievable MVP and ENCODE that translation in §1 of the pack ("pragmatic scope decision USER-APPROVED, encoded here") so developer doesn't try to implement the literal text. Preserve the original verb in the registry — do not edit registry — but re-state operationally in the pack.
- A FE slice that consumes an already-shipped backend endpoint MUST mirror the response schema in TS verbatim (field names, optional/nullable). Do not let developer paraphrase. Cite the backend `schemas.py` file path explicitly.
- Module path drift between `*_TECHNICAL_GUIDE §5 (features/admin-ai/, hyphen)` and `registry write_set (features/admin_ai/**, underscore)` — pick the one that matches the BACKEND module name (always underscore per Python convention). Document explicitly as discrepancy D1; doc amendment goes to next doc-amendment slice.
- `UX_CONTRACT.md §3` requires 5-6 UI states per route, but a *shell* page that does NO fetch can only formally cover `success` (and arguably `empty` for the CTA-to-wizard scenario). The other states are `n/a (no fetch in MVP — covered by wizard)`. Mark this in the page docstring per `05-runtime-write-contract.md §UX and verification-data contract`. Validator must accept the explicit-mark — pre-empt with a discrepancy row.
- For first-time feature modules, the registry write_set is intentionally narrow (`frontend/src/features/<feature>/**`) but **always** needs Path-A extensions: (a) the router file (`main.tsx` or equivalent), (b) the i18n locale bundles for any productive copy. Declare ALL extensions explicitly in §10.2 with per-file justification — same precedent as T004's tsconfig + T005's providers.tsx surgical edit.
- FU follow-ups (e.g. T007 `auth.ts` Bearer stub) MUST include a `// TODO(<next-slice-id>): <action>` comment + R-row in the pack so the next planner grep-finds the deferred work without reading the handoff.
- The first feature module sets the `data/domain/presentation` layering convention for the whole codebase. Lock the directory structure in the pack §11 file plan so subsequent feature modules don't drift into different shapes (some `data/api`, some `api/`, some `client/`).
- For FE slices that consume a backend endpoint with auth, encode the P00 stub auth header in a 1-line feature-local helper (`getAdminAuthHeader()`) rather than scattering `Bearer dev-admin-localhost` across components. P01-S03-T001 will replace one helper instead of N components.
- For the 3-step wizard pattern: warn the developer in the pack that if `<WizardPage>.tsx` approaches 300 lines, split steps into sub-components (`Step1*.tsx`, `Step2*.tsx`, `Step3*.tsx`). Do not pre-split — premature de-composition violates KISS — but flag the watershed.
- For tests in a slice with no MSW dependency, mock `globalThis.fetch` per test instead of adding MSW. Adding MSW is its own dep slice. Document this in §12 testing strategy so validator doesn't request MSW.
- When an FU surfaces issues with seed loaders or dev `.env` (FU-20260509220235, FU-20260509222920), surface them as R-rows with workarounds for verify-slice. Do not let them block T007's verify; document the SQL/env-export recipe inline.

### Scaffold slices (P00-S01-T001 family)

### Dependency-pack slices (P00-S01-T002 family)

- When the registry write set lists **only** the manifest + lockfile + one source file, but Acceptance + Verify command imply tests, the planner MUST amend the write set explicitly in the pack (NEW: `*.test.tsx`, `vitest.config.ts`). Validator gets the heads-up upfront; otherwise it flags drift.
- Frontend test environment: Vitest 4 default env is `node`. Any React DOM mount needs `vitest.config.ts` with `test.environment = 'jsdom'` AND `jsdom` in devDependencies. Without this, smoke tests fail with `document is not defined`.
- `@testing-library/react` is implied by `instrucciones.md §11.0` + `TECHNICAL_GUIDE §2.1` for "tests de componentes" — declare its addition explicitly in the dependency-pack pack so it's not "silent expansion".
- For provider wiring slices, ALWAYS specify the exact export name + signature in the pack. Downstream slices import it; lock it now.
- For library installs against pinned framework majors (React 19, Vite 8, TS 6), the `npm install` step itself is the fail gate. Researcher must report peerDeps live; `--legacy-peer-deps` is forbidden.
- "First provider wired" intentionally excludes Router. Router lands with auth (P01-S03-T001) because `BrowserRouter` is coupled to redirect-after-login behavior. The pack must explicitly say "do NOT add BrowserRouter" or developer will pre-empt P01-S03.
- T001 closer left frontend stack pinned to React 19 / TS 6 / Vite 8 / Vitest 4. T002 inherits these — pack must call them out as "do not regress" so dependency conflict resolution doesn't downgrade them.

- The verify command `npm --prefix frontend run test -- --run -t i18n` filters by test-name regex, not file path. The describe-block name MUST contain `i18n` (e.g., `describe('i18n bundle structure', ...)`) for the filter to match. Pack must say this explicitly.
- The previous slice (T002) may leave a stale doc-citation in `providers.tsx` (e.g. `instrucciones.md §3.3` pointing at the i18n config when the truth is `§6` + `§1.4`). ### Architectural follow-up slices promoted from verify-slice CWE findings (P00-S02-T004 family)

- When verify-slice surfaces a CWE-class issue (CWE-532 leak, CWE-79 XSS surface, etc.) that has a TWO-layer fix — (a) close the immediate leak inside the originating slice's write_set; (b) a global/architectural fix in a different module — the right shape is: in-scope debugger drop at the call site for (a), HIGH-severity follow-up promoted to a new task for (b). Severity must be `high` so it blocks the next wave/claim until promoted/waived. T004 = the (b) for T002's CWE-532. Pack must explicitly cite both layers and tell developer NOT to revisit the call site (it's already fixed and out-of-scope here) — defense in depth means both layers stay intact.
- For `infra:logging` slices, `journey_refs: []` + `verify_mode: human` + `risk_level: high` is the correct shape. Verify is HUMAN even without a journey because the failure mode (secret leak) is invisible to deterministic auto-checks unless we instrument them — and we did, but human read of the captured log is the gold standard. Mark `Verification data contract: n/a` (no journey, no real users) and document the synthetic edge-case fixture inline; cite `orchestrator-contract.json::ux_contract.verification_data_contract.rules` for the synthetic-allowed clause.
- For structlog-related slices, ALWAYS include "researcher must validate constructor signature for `<version>`" in the pack, even when the kwarg is stable — structlog renames helpers across 24.x → 25.x and Rich's traceback formatter has minor breaks across rich 13.x. Five-second cache hit is fine; the alternative is silent test failures that look like "logic bugs".
- Acceptance counts copied from a follow-up YAML may be stale (registry says "8 health tests" when reality is "9 health tests" because a regression guard was added during verify-slice). Cross-check with `wc -l` / `pytest --collect-only` and surface the discrepancy as an "informational" note in the pack so closer report has the correct numbers, not the registry copy.
- Logging changes ALWAYS need to be tested under BOTH ENABLE_VERBOSE_LOGGING values (true → ConsoleRenderer / Rich path; false → JSONRenderer path). One test per mode minimum; the leak vector is mode-specific (Rich's frame locals don't apply in the JSON path today, but a future renderer swap could regress that — defensive testing is cheap insurance).
- Follow-up slices must include a "files NOT to touch" subsection because the in-scope debugger fix from the originating slice (router.py, main.py for T002→T004) is tempting to "improve" but doing so silently widens the write_set and the PreToolUse hook will block. State explicitly: "T002 already fixed (a); this slice ships (b)".
- THIRD-LAYER CWE-532 fixes (P00-S02-T009 family — third-party stdlib logger suppression). When the same module already absorbed two layers of a CWE-532 fix (T002 in-scope `exc_info` drop + T004 global `show_locals=False`), a third leak vector can still exist via THIRD-PARTY stdlib loggers (httpx, httpcore, urllib3, boto3). These bypass the `_redaction_processor` because it scrubs `event_dict` keys, but third-party loggers emit a single string message. Fix is monofile: `logging.getLogger("<lib>").setLevel(WARNING)` inside `configure_logging()` AFTER root setup, in BOTH verbose=True and verbose=False branches (verbose=False is already silent via root=WARNING but explicit is regression-safe). Pack must enumerate the libs (researcher confirms exact logger names live), reject heavier alternatives (event_hooks, Filter, transport-level URL strip) in 1 line each, and ban touching the originating client module (`provider_clients.py`) — defense-in-depth means each layer stays intact.
- For "transport-leak" tests: use `httpx.MockTransport` to reproduce the LEAK PATH deterministically — the leak vector is the OUTBOUND request URL log line emitted by httpx BEFORE the transport runs, so the response body is irrelevant. This satisfies "tests are REAL" because the leak vector IS reproduced; only the upstream provider is mocked, which is allowed for external APIs. Pack must explicitly cite this rationale so validator does not flag the mock as a non-real test.
- The acceptance verb in the FU YAML may say "real Gemini API call" but a deterministic MockTransport test reproduces the SAME leak vector and is CI-portable (no `.env.local` required). Pack must SHIP both: deterministic primary tests (always run) + skipif-gated real-API test (runs when `VERIFICATION_*` key is set). Closer can satisfy the literal acceptance with the gated test when the env is present; otherwise, the deterministic suite + manual uvicorn capture during verify-slice is the human-verifiable evidence.
- Sentinel choice for transport-leak tests: use a value with the leak SIGNATURE prefix (`AIza-`, `sk-`) followed by `FAKE-SENTINEL-` and a deterministic suffix. The grep regex from the FU acceptance (`AIza|sk-[A-Za-z0-9]{20}`) MUST match the sentinel — otherwise the test passes vacuously. Confirm the regex matches the sentinel BEFORE writing the assert.
- BEFORE/AFTER capture as evidence: developer must record `leak-before-fix.txt` (test fails with the sentinel in stdout when the new lines are commented out) and `leak-after-fix.txt` (test passes after restoring the fix) under `evidence/<TASK_ID>/`. This proves the test is not vacuous — same defense-in-depth pattern as a regression guard. Pack states this explicitly so closer report has the artifact.

### Health/ops slices (P00-S02-T002 family)

- TECHNICAL_GUIDE §6.2 may declare envelope shape `{data:{...}}` for health endpoints, but a previous slice (T003 in this project) may have shipped flat shape `{status, version, uptime}` because the upstream compose healthcheck or k8s probe expects flat. **Precedent wins**, but the pack MUST surface the discrepancy explicitly in a Discrepancies table and recommend (not register) a follow-up to reconcile docs. Do NOT silently migrate to envelope inside the slice — that breaks downstream healthchecks atomically.
- For multi-probe `/ready` endpoints where some dependencies (Redis, LiteLLM) don't have wired clients yet, return them with explicit `not_implemented` value, not a green "ok" lie and not omitted. This is forward-compatible: downstream slices flip the value to "ok"/"fail" without changing the response shape.
- `app.core.db.get_engine()` is **lazy** (T003 design): builds engine on first call, no import-time side effect. Safe to call from a /ready handler. Tests must use DI (monkeypatch the module global or factory) to simulate DB-down without leaking a half-built singleton.
- request_id middleware is naturally a P00-S02-T002 deliverable per the structlog docstring TODO at `app.core.logging.py:17`. Inline 4–6 lines in `main.py` is preferable over a new module — keeps the work-item write_set tight (3 paths).
- For backend slices with no journey_refs and verify_mode=auto, the Verification Data Contract row is **n/a** — explicitly say so. Tester/verify will otherwise chase fixtures that don't apply.
- `__init__.py` package markers added to satisfy a NEW package directory (`backend/app/api/__init__.py`) are a Python language requirement, not scope expansion. Document them in the pack as a write_set technicality so validator doesn't flag drift.
- Curl verification on `localhost:8000/health` may hit either the compose container (`version: 0.1.0`) or a local dev uvicorn (`version: 0.0.0`) when both are running. Always capture the `version` field in evidence to disambiguate; in-process `httpx.AsyncClient` tests are immune to this shadow.
